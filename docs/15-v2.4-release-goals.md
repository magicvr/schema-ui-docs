# v2.4 发布目标与门禁

本文档记录 **v2.3 → v2.4** 的 MINOR 发布：标准只读详情组件 `recordView`（ADR-0024）以双重版本/capability 门控进入核心协议。没有 `recordView` 的合法 v2.3 页面保持原有语义；使用新组件的页面必须声明 `meta.protocolVersion: "2.4"` 和 `record.view.load`。

权威 ADR：[0024](./decisions/0024-record-view.md)。迁移见 [migrations/2.3-to-2.4.md](./migrations/2.3-to-2.4.md)。轨道见 [11-next-admin-lifecycle-goals.md](./11-next-admin-lifecycle-goals.md)。

## 1. v2.3 到 v2.4 的边界

- v2.4 页面声明 `meta.protocolVersion: "2.4"`。
- Renderer 可同时支持 `"2.3"` 与 `"2.4"`，但不得为旧页面推断 `recordView`。
- 新增预定义 capability：`record.view.load`。
- 新增组件类型：`recordView`（`recordSource` + `fields[]`；无 children / DataRef / reactions）。
- `recordSource` 与 form 同构；`$context.route` 绑定位置扩展到 `recordView.props.recordSource`。
- L2 字段集下限：`recordView` 必须 `protocolVersion >= "2.4"`，且缺 capability 时 fail-closed。

## 2. v2.4 阻断门禁

### G1. 版本与 capability

- [x] ADR-0024 `accepted`，并列入 `protocol-manifest.json` 的 `authority.semanticSpecs`。
- [x] `record.view.load` 作为预定义 capability 写入 Node、页面 Schema、Renderer 和迁移规范。
- [x] 任何 `recordView` 都由 L2 同时要求 `protocolVersion >= "2.4"` 与 capability。
- [x] 没有 `recordView` 的 v2.3 页面不改变既有语义。
- [x] 根包、lockfile、manifest、validator 与 MCP 兼容声明对齐 `2.4.0` / `2.4`。

### G2. 结构与静态校验

- [x] 组件 DSL 定义 `recordView.props.recordSource` / `fields[]`；`fields[].key` ⊆ `responseMapping`。
- [x] L2 拒绝缺 recordSource/fields、method 非 GET、空 mapping、key 重复/缺失、tag 无 tagMap。
- [x] L3a：`$context.route` 仍禁止 expression 挂载点；允许 form/recordView 的 recordSource 绑定。

### G3. 可观测算法与场景

- [x] `response-mapping` 覆盖 `component: recordView` 显式映射与缺失路径 → null。
- [x] `version-negotiation` 覆盖 `2.4` 接受/拒绝与 `record.view.load` 缺失/具备。
- [x] 扩展示例 [`admin-list-detail-lifecycle.md`](./05-scenarios/admin-list-detail-lifecycle.md) + scenarios 步进。
- [x] 算法类 fixtures 与官方六场景同步 `protocolVersion: "2.4"`。

### G4. 发布制品

- [x] 迁移文档 `2.3-to-2.4` 明确 opt-in 与 form 边界。
- [x] CHANGELOG `v2.4.0` 正式节、协议制品版本和 release target 同步。
- [x] `release:check` 固定 fixture 类别、case 数与 fixture digest；协议制品构建与验证可复现。

## 3. 发布工程门禁

- [x] `package.json` / lockfile / `protocol-manifest.json` 统一为 `2.4.0` / `2.4`。
- [x] 算法类 fixtures 使用 `protocolVersion: "2.4"`；`version-negotiation` 保留历史页面版本输入。
- [x] 链接检查、场景校验、L0-L4、全量 JS/Python conformance、release check 与制品验证通过。

## 4. 版本纪律

v2.4 之后继续遵守 SemVer：文档勘误为 PATCH；兼容字段 / capability 为 MINOR；破坏合法输入或默认可观测结果为 MAJOR。`recordView` 仅在本 MINOR 的显式双重门控下生效。Phase D 其余运营增强（导入导出、异步任务、树表、行内编辑、跨页全选等）必须先经独立 ADR。

## 5. 完成定义

G1–G4 与发布工程门禁全部关闭，且 `npm run release:check` 与 `npm run verify:protocol-artifact` 通过，即可将 `2.4.0` 视为协议制品完成。v2.3 历史门禁保留在 [14-v2.3-release-goals.md](./14-v2.3-release-goals.md)。
