## 发现的问题

### V234（🟢 P2）Node.js `>=18` 支持声明没有最低版本 CI，MCP 测试链已要求 Node 20+

**分类：** CI 盲区 / 支持矩阵不一致

**证据**

1. 根 `README.md` 与 `mcp/package.json` 都声明 MCP 运行时支持 Node.js `>=18`；根 `package.json` 同样声明 `engines.node: ">=18"`。
2. `.github/workflows/mcp-ci.yml` 的 protocol 与 mcp job 均只设置 `node-version: 24`；CD 和 `mcp/Dockerfile` 也只使用 Node 24。没有 Node 18（或最低支持版本）矩阵/兼容 job。
3. `mcp/package-lock.json` 当前锁定 `vite@7.3.6`，其 `engines.node` 为 `^20.19.0 || >=22.12.0`。该包是 `vitest@3.2.7` 的 dev 测试链，因此仓库标准 MCP 测试路径的依赖元数据已经不把 Node 18 列为受支持运行时。
4. `docs/CHANGELOG.md` 曾明确记录“恢复编译产物对所声明 Node 18 下限的兼容性”，说明 Node 18 是有意支持承诺，而非随手元数据。

**影响**

- 生产镜像固定 Node 24，故当前发布镜像不直接受影响；风险集中在源码/本地安装支持承诺与回归发现能力。
- 只在 Node 24 绿灯不能证明 Node 18 运行时仍兼容，未来依赖或语法升级可静默破坏公开下限。

**严重度：** 🟢 P2

### V235（🟢 P2）归档审计仍含真实相对死链，CI 无 Markdown 链接门禁

**分类：** 文档不一致 / CI 盲区

**证据**

1. `docs/audit/archived/0053-2026-07-11-checklist.md` 位于 `docs/audit/archived/`，其中链接 `[docs/CHANGELOG.md](../CHANGELOG.md)` 实际解析到不存在的 `docs/audit/CHANGELOG.md`。
2. 目标文件实际为 `docs/CHANGELOG.md`；从归档目录应使用 `../../CHANGELOG.md`。
3. 对仓库全部 Markdown 相对链接做存在性扫描，当前唯一命中的真实断链就是该项。
4. 根 `package.json`、`scripts/` 与 `.github/workflows/` 均没有 Markdown 相对链接检查步骤；V233 修复的是手工发现的同类死链，现有门禁仍无法防止再次发生。

**影响**

- 0053 的完成证据链接不可点击，削弱审计归档的可追溯性。
- 同类路径回归只能依赖人工复审，0057/V233 的修复没有形成自动防回归门禁。

**严重度：** 🟢 P2

## 环境限制与明确排除

| 观察 | 结论 |
|---|---|
| MCP 77/119 测试失败 | 不计为仓库缺陷。当前容器 Node `v22.22.3` 的最小探针 `spawnSync(node, ['-e', 'console.log("hello")'], { stdio: ['ignore','pipe','pipe'] })` 在子进程退出 0/1 时均得到空 stdout；而直接运行 L2 脚本可输出完整 JSON。MCP 的失败集中表现为依赖子进程 stdout 的 L2/L3a 结果为空/内部错误，证据指向审计执行环境的子进程管道异常。CI 固定 Node 24，0057 在修复分支曾记录 119/119 通过。 |
| `npm audit` 在线失败 | 不计为漏洞发现。失败原因为沙箱网络/代理不可达（`EPERM` / `EAI_AGAIN`），不是 advisory 结果；离线缓存扫描根与 MCP 均为 0，但不能替代一次联网复核。 |
| 历史 `0.3` 文本 | conformance version-negotiation 历史向量与 CHANGELOG 历史版本叙述均属预期，不重开 V227/V231。 |
| `test:conformance:table-state` | 仍为 search-table thin wrapper，兼容别名按 0057 结论保留，不构成重复 fixture 回归。 |
