# 审计报告 0055：v1.0 执行一致性与发布门禁

## 结论

`v0.3.0-rc.1` 已具备较完整的 L0-L4 合法性校验、MCP 读取/校验能力和镜像发布流水线，但尚不能提升到 `v1.0.0`。当前主要风险已经从“字段和文档是否一致”转为“两个独立实现是否会产生相同运行结果”。

本轮基线验证结果：

- `npm --prefix mcp run build` 通过；
- `npm --prefix mcp test` 通过，3 个测试文件、117 个测试全部通过；
- `npm --prefix mcp run smoke:tools` 通过；
- 六个官方场景由 MCP 测试提取 Markdown 中首个 YAML fence，并通过 `protocol.validate_content` 校验；
- 根 `npm run validate -- "docs/05-scenarios/*.md"` 不能直接消费 Markdown 场景；V222 已新增公共 YAML fence 提取器和 `npm run validate:scenarios`，六个场景通过 L0-L4；
- V222 已将根场景校验设为独立 CI job，并在 CD 发布前执行；MCP 测试复用同一提取器；
- `schema-ui-mcp:0.3.0-rc.1` 本地镜像构建与 Docker smoke 通过。

因此，现有绿灯可作为“配置合法性基线”，不能作为“执行一致性门禁”。

## 发现的问题

### V218（🔴）版本协商仍为 fail-open，和 G1 直接冲突

`08-renderer-spec.md` §3.2 仍允许未知版本“按最接近的已知版本解析”，并把缺失 `protocolVersion` 隐式视为 `0.1`。这会让不同 Renderer 根据各自版本集合选择不同解析规则。

**处理结果：** ADR-0009 已删除 `minCompatibleVersion` 推断路径，只接受 `supportedVersions` 中精确声明的 MAJOR.MINOR。缺失版本只能交给调用方显式启用的 legacy adapter，标准 Renderer 入口直接拒绝；14 个框架无关 fixtures 与 JavaScript reference 已通过并接入 CI/CD。

### V219（🔴）query 允许复合值，但没有字节级序列化算法

`04-datasource-contract.md` §3.1 和 Schema 当前允许 string / number / boolean / `null` / 递归对象 / 数组；规范只定义 `$deps` 的空值省略，没有定义数组、对象、字面量 `null`、空字符串、键顺序、重复 key、已有 query 合并和 UTF-8 百分号编码。

**处理结果：** ADR-0010 采用保守方案，四个 query 来源的最终值只允许标量；对象/数组在 L0/L1/L2 静态拒绝。统一 `serializeQuery(baseUrl, sources)` 已由 JavaScript 与 Python reference 实现，并对同一 16 个字节级 fixtures 产生相同结果。

### V220（🔴）分页/排序参数与搜索参数的冲突优先级未闭合

规范称 `page`、`pageSize`、`sort` 为标准参数；搜索模式仅规定“搜索参数优先于 `data.params`”，未定义三者是否保留、搜索字段重名如何处理、提交/清空筛选是否重置页码，以及已有 URL query 的优先级。

**处理方向：** 将 `page`、`pageSize`、`sort` 设为保留参数，L2 拒绝静态 params 和搜索字段使用这些名称。最终合并顺序固定为：已有 URL query < 静态 params < 搜索字段 < Renderer 分页/排序状态；搜索提交和清空筛选均将 `page` 重置为 1，翻页/排序保留当前筛选，排序变化也将 `page` 重置为 1。

**处理结果：** 已由 ADR-0011 接受上述规则；L2 覆盖 `data.params`、`datasources.*.params` 及搜索表单 `field` / `startField` / `endField`，同时保留普通表单、远程选项和行级 query 的独立请求边界。5 个框架无关状态 fixtures 直接断言事件后状态与 ADR-0010 序列化后的最终 URL，并接入 CI/CD。

### V221（🔴）缺少框架无关的执行一致性套件

仓库已有 117 个 MCP 测试，主要验证文档查询、组件契约和 L0-L4 结果；没有 fixtures 固化版本决策、最终 URL、请求体、响应映射、Action 时序、错误状态、上传结果和表达式状态转换。

**处理方向：** 新增 `conformance/fixtures/`、fixture JSON Schema、JavaScript 参考执行器和测试 runner。fixture 只描述协议输入与可观察输出，不包含 React/Vue 组件细节。生产后端语言实现必须消费同一目录或同一不可变制品，禁止复制后形成私有分支。

### V222（🟡）根 L0-L4 与官方场景没有独立 CI 门禁

CI 执行 MCP build/test/smoke 和 Docker smoke，但没有独立调用根校验脚本。官方场景当前由 Vitest 内部 helper 提取 Markdown YAML；根 CLI 对 Markdown 输入失败，且该失败不是协议违规。

**处理结果：** 已提供单一场景提取命令，将六个 YAML fence 生成到临时目录后运行根 L0-L4；CI 增加独立 job，CD 在发布前执行同一命令。MCP 测试与根门禁复用同一个提取器，避免两条解析路径。

### V223（🟡）发布流水线尚未证明 v1 制品可复现

CD 已验证包版本与 Git tag 一致，并发布完整版本、minor、latest 和 SHA 镜像；但尚无迁移文档门禁、CHANGELOG 待发布清空检查、`meta.protocolVersion: "1.0"` 全仓检查、发布后从远端拉取固定镜像的 smoke，以及 fixtures 制品版本校验。

**处理方向：** 增加 `release:check`，在创建 `v1.0.0` tag 前执行版本、文档、Schema、fixtures、迁移说明和 CHANGELOG 检查；推送后从 Docker Hub 拉取 `1.0.0` 和 SHA tag，各执行一次 Docker smoke 并记录 digest。

### V224（🟡）MCP 源码树包含遮蔽 TypeScript 的旧 CommonJS 生成物

`mcp/src` 与 `mcp/tests` 中跟踪了 7 个和 `.ts` 同名的 CommonJS `.js` 文件。这些文件由旧编译命令生成，混用 `require()` 与 `import.meta`；Vitest 会优先加载 `.js`，导致全部测试在收集阶段失败，测试目录中的 `.test.js` 还会作为额外套件直接加载。

**处理结果：** 删除 7 个同目录生成物，`.gitignore` 禁止 `mcp/src/**/*.js` 与 `mcp/tests/**/*.js` 再次写入源码树；标准 `tsc -p mcp/tsconfig.json` 继续只输出到 `mcp/dist/`。当前 MCP build、119 tests 和 tools smoke 全部通过。
