# 审计记录

审计文件采用以下生命周期：

1. **创建：** 审视报告（`*-review.md`）+ 跟踪清单（`*-checklist.md`）同时生成，可选附变更计划（`*-plan.md`）
2. **跟踪：** 跟踪清单中的条目逐步标记 `[x]`
3. **归档：** 所有条目标记为 `[x]` 后，整套文件移入 [`archived/`](./archived/) 目录，并在该目录的 `README.md` 中添加索引记录

## 编号规则

| 规则 | 说明 |
|---|---|
| **全局递增** | 序号 `NNNN` 全局唯一、永续递增，**永不重置**。即使所有活跃审计完成归档，下一份审计也从 `0005` 开始，不会回退到 `0001` |
| **命名格式** | `NNNN-YYYY-MM-DD-{type}.md`，其中 `type` 为 `review` / `checklist` / `plan` |
| **查询最大号** | 创建新审计前，**同时在 `audit/`（活跃区）和 `archived/`（归档区）中查找**最大序号，取二者较大值 + 1。例如当前活跃区最大 `0008`、归档区最大 `0007`，则下一份为 `0009` |

问题编号 `V<n>` 同样全局递增。创建审计前不仅要取最大值，还必须比较“问题定义标题数量”和“唯一 V 编号数量”，防止历史重置被连续数值集合掩盖。历史 0010、0013、0014 曾重复使用 V1-V4；引用这些冲突项时必须写成 `0010/V1`、`0013/V1`、`0014/V4` 等 `NNNN/Vn` 复合形式，不得单独写 `V1`-`V4`。历史正文不重编号，从 V5 起以及当前新增问题继续按全局最大值递增。

> **为什么不能重置？** 序号是审计的唯一身份标识，在文档正文、ADR 引用、CHANGELOG 中均作为交叉引用锚点。重置会导致新旧 `0001` 含义混淆，引用链断裂。

## 约定

- ❌ **不要在根目录 `README.md` 的目录树中列举审计文件清单。** 审计文件增删频繁，每份审计都去更新根目录 README 不现实，且该文件的主要作用是给读者"快速了解项目结构"，而非实时追踪审计状态。
- ✅ 审计的活跃/归档状态以本文件（`audit/README.md`）和 `archived/README.md` 为准。外部文档引用审计时，使用 `docs/audit/archived/NNNN-...` 的完整相对路径即可。
