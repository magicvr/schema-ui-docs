# v0.2 变更计划

> 本计划基于 AI 助手审计报告（2026-07-07）整理，按"直接修复 / 需先补 ADR / 暂不采纳或降级"
> 三类分组。所有条目完成后需同步更新 `CHANGELOG.md`、对应的 JSON Schema 及场景示例。

## 分组原则

| 分组 | 含义 | 处理方式 |
|---|---|---|
| A. 直接修复 | 明确的 bug 或文档矛盾，不涉及协议边界扩展 | 可直接改，走常规 PR 流程 |
| B. 直接新增 | 新增可选字段，向后兼容，不扩展表达式引擎能力边界 | 可直接改，需同步文档+Schema+示例 |
| C. 需先补 ADR | 涉及扩展表达式引擎作用域或协议核心边界 | 必须先写 ADR 并评审通过，再实现 |
| D. 暂不采纳/降级 | 优先级下调或范围收窄 | 记录理由，暂缓 |

## A. 直接修复（bug / 文档矛盾）

### A1. 修复 JSON Schema 中 CSS 禁用词校验逻辑（双轨策略）
- **对应审计条目：** §2.2
- **问题：** `node.schema.json` 中 `"not": {"required": [...]}` 语义错误，实际只禁止"多个禁用属性同时出现"，单个出现无法拦截。
- **决策修正（基于 v0.2-plan-feedback 的 #1）：** 原计划打算完全移除 Schema 层校验、只保留 L4 lint 脚本，判断有误——JSON Schema 校验能自动挂在每次 CI 流程中，而 L4 lint 脚本需要额外配置才会生效；若团队未及时接入 lint，禁用词校验会在空窗期内完全失效。**改为双轨策略：** 同时保留并修复 Schema 层校验（作为自动生效的基础防线）+ L4 lint（作为更细致的深度补充，可覆盖 Schema 表达力之外的场景，如嵌套结构中的禁用词）。
- **变更：**
  ```json
  // 修复后写法：not 配合 anyOf，逐一禁止每个属性单独出现
  "not": {
    "anyOf": [
      { "required": ["margin"] },
      { "required": ["padding"] },
      { "required": ["color"] },
      { "required": ["background"] },
      { "required": ["fontSize"] },
      { "required": ["fontWeight"] },
      { "required": ["border"] },
      { "required": ["borderRadius"] },
      { "required": ["width"] },
      { "required": ["height"] },
      { "required": ["zIndex"] },
      { "required": ["boxShadow"] },
      { "required": ["lineHeight"] },
      { "required": ["letterSpacing"] },
      { "required": ["textAlign"] }
    ]
  }
  ```
- **涉及文件：** `schemas/node.schema.json`（保留并修复，不删除）、`06-validation.md`（明确说明 L1 Schema 校验与 L4 lint 是"两层防线"关系，而非"同一规则的重复实现"；lint 脚本的清单需与 Schema 的 `anyOf` 清单保持同步，任一方新增禁用词需同步更新另一方）。

### A2. 修复 `text` 组件 `data` 支持说明矛盾
- **对应审计条目：** §4.3
- **问题：** `supportsData: false` 与注释"如需动态文本用 `data.value`"自相矛盾。
- **变更：** 明确 `text` 组件 `supportsData: true`，允许 `data.source: static/ref/api`，取值字段固定为 `data.value` 或 API 返回的 `text` 字段（与 `statCard.props.valueField` 保持同样的取值声明方式，见 A3）。
- **涉及文件：** `03-component-registry.md`、`schemas/component-registry.json`。

### A3. 将 `valueField` 从 `DataRef` 移至组件 `props`
- **对应审计条目：** §2.1
- **问题：** `valueField` 仅服务于 `statCard`（及修复后的 `text`），放在通用的 `DataRef` 里破坏了"数据来源"与"取值方式"的边界，且与 `table`（`columns[].field`）、`chart`（`xField`/`yField`）的映射方式不一致。
- **变更：**
  ```yaml
  # 变更前
  data:
    source: api
    url: /api/stats/order-count
    valueField: total

  # 变更后
  data:
    source: api
    url: /api/stats/order-count
  props:
    valueField: total
  ```
- **兼容性：** 破坏性变更（Node 结构语义位置改变）。发布时需在 `CHANGELOG.md` 标注为 MINOR 内的"字段迁移"，并提供简单的迁移脚本（批量将 `data.valueField` 移动到 `props.valueField`）。
- **涉及文件：** `01-node-protocol.md`（DataRef 定义）、`03-component-registry.md`、`schemas/node.schema.json`、`schemas/component-registry.json`、`05-scenarios/grid-dashboard.md`。

### A4. 补充 `datasources` + `data.ref` 的示例
- **对应审计条目：** §4.1
- **变更：** 在 `grid-dashboard.md` 场景示例中增加至少一个使用 `datasources` 预声明 + `data.source: ref` 引用的完整片段。
- **涉及文件：** `05-scenarios/grid-dashboard.md`。

### A5. `meta` 增加 `protocolVersion` 字段
- **对应审计条目：** §4.2
- **变更：** `meta` 增加必填字段 `protocolVersion: string`（如 `"0.2"`），Renderer 据此判断解析规则版本，为后续协议演进做兼容判断的锚点。
- **涉及文件：** `01-node-protocol.md`（顶层文档结构）、`schemas/node.schema.json`、所有场景示例。

## B. 直接新增（向后兼容，不扩展表达式边界）

### B1. Node 增加可选 `id` 字段
- **对应审计条目：** §3.1
- **用途：** 页面内定位（滚动、埋点）、增量更新/局部 patch 的锚点。
- **变更：** Node 结构新增可选字段 `id: string`（页面内唯一，Renderer 侧做重复校验）。
- **涉及文件：** `01-node-protocol.md`、`schemas/node.schema.json`。

### B2. 解决 `tabs` 的 children 归属问题（采用方案 A）
- **对应审计条目：** §2.4
- **决策：** 采用审计报告中的**方案 A**——在 `tabs.props.items` 内直接内嵌各 tab 的内容节点，不依赖顶层 `children` 做归属匹配，避免仅为这一个组件引入通用的 `key` 字段。
  ```yaml
  type: tabs
  props:
    items:
      - key: tab1
        label: 概览
        content: { type: section, children: [...] }   # 内嵌 Node
      - key: tab2
        label: 详情
        content: { type: table, ... }
  ```
- **涉及文件：** `03-component-registry.md`（`tabs` 契约重写）、`schemas/component-registry.json`。

### B3. 补充 `actions` 完整契约
- **对应审计条目：** §2.5
- **变更：** 新增独立小节（可作为 `01-node-protocol.md` 的子节，或拆为新文档 `07-actions-contract.md`，视条目最终大小决定），至少定义：
  - 支持的 `type`：`request` / `navigate` / `modal` / `custom`（`custom` 仅限白名单前端预注册的动作，不接受任意代码）。
  - `request` 类型的完整字段：`method`/`url`/`bodyMapping`/`onSuccess`/`onError`。
  - `onSuccess`/`onError` 支持的语义级行为：`toast`/`navigate`/`reload`/`closeModal`。
- **决策备注：** `custom` 类型仅做"预注册白名单函数名引用"，不允许内联脚本，以保持与"配置而非代码"的一致原则。
- **涉及文件：** 新增 `07-actions-contract.md`（或在 `01` 中扩展）、`schemas/`下新增 `action.schema.json`、`05-scenarios/form-with-reactions.md`（补充 `onSuccess`/`onError` 示例）。

### B4. `span` 提升为通用 props
- **对应审计条目：** §3.7
- **变更：** `span` 不再是 `section` 专属字段，任何作为 `grid` 直接子节点的 Node 均可在自身 `props` 中声明 `span`，避免仅为占栏而强制包一层 `section`。
- **涉及文件：** `03-component-registry.md`（`grid` 章节补充说明"子节点通用支持 `span`"）。

### B5. 表单字段补充 `placeholder` / `description` / `tooltip`
- **对应审计条目：** §4.4
- **决策：** 采纳，且与 A/B 组一起在同批次发布（审计报告放在 P3，我们认为改动成本低、性价比高，合并进本批）。
- **涉及文件：** `03-component-registry.md`（`input`/`inputNumber`/`select` 契约）、`schemas/component-registry.json`。

### B6. i18n `labelKey` 支持
- **对应审计条目：** §3.3
- **变更：** `props` 中 `label`/`title`/`content` 类字段均可用对应的 `xxxKey` 形式替代（如 `labelKey`），Renderer 优先查 i18n 词典，找不到则 fallback 到非 Key 版本。
- **涉及文件：** `01-node-protocol.md`（新增小节说明 i18n 约定）、`03-component-registry.md`（各组件补充 `xxxKey` 可选字段）。

### B7. `select` 支持远程动态选项
- **对应审计条目：** §3.4
- **变更：** 新增 `props.optionsSource`（与 `options` 二选一），结构：`url`/`params`/`labelField`/`valueField`/`searchable`。`params` 中允许引用 `$deps.*`（复用现有联动依赖机制，不新增表达式能力，仅做参数级值替换，不做条件判断）。
- **空值行为规则（基于 v0.2-plan-feedback 的 #3，补充明确规定）：** 当 `params` 中引用的 `$deps.*` 值为 `null` 或 `undefined` 时，该参数从最终请求的 query/body 中**整体省略**（不传空字符串、不传 `null` 字面量），避免后端把"参数不存在"和"参数为空字符串"混为两种不同的业务语义。
- **涉及文件：** `03-component-registry.md`（`select` 契约 + 空值省略规则）、`04-datasource-contract.md`（补充"选项类接口"响应契约 + 参数省略规则，与 `table`/`chart` 等其他 `$deps` 参数场景保持同一约定）。

### B8. 组件级 `states`（空态/加载态/错误态文案定制）
- **对应审计条目：** §3.5
- **变更：** Node 新增可选字段 `states: { loading, empty, error }`，仅允许语义级文案/插画标识（`text`/`illustration`/`fallbackText`），不允许样式类字段。
- **范围约束（基于 v0.2-plan-feedback 的 #2）：** `states` 仅对 `supportsData: true` 的组件有意义（如 `table`/`chart`/`statCard`/`text`），对 `section`/`grid`/`tabs` 等无数据状态的纯布局节点无意义。在 `component-registry.json` 的每个组件定义中增加 `supportsStates: boolean` 标注，Renderer 与 CI 校验均可据此拒绝在不支持的组件上声明 `states`（而不是静默忽略）。
- **涉及文件：** `01-node-protocol.md`、`schemas/node.schema.json`、`03-component-registry.md`（补充 `supportsStates` 说明）、`schemas/component-registry.json`（为每个组件标注 `supportsStates`）。

### B9. `component-registry.json` 增加 `deprecated`/`since` 标注能力
- **对应审计条目：** §5.4
- **变更：** 为组件 props 定义结构增加可选的 `deprecated`/`deprecatedMessage`/`since` 元信息，供前端在开发环境输出迁移警告。
- **涉及文件：** `schemas/component-registry.json`、`03-component-registry.md`（说明标注规则）。

### B10. 表格行级操作显隐的过渡方案（提前落地，不等 C3 的 ADR）
- **对应审计条目：** §3.6 的过渡方案部分（基于 v0.2-plan-feedback 的 #4）
- **决策修正：** 原计划把过渡方案写在 C3（暂缓）条目的备注里，导致它被"暂缓"标签一起拖住；实际上过渡方案本身不依赖任何表达式引擎扩展，不需要等 ADR，应作为独立条目随 v0.2.0 正式发布。
- **方案：** 后端在表格行数据中直接下发语义化的布尔字段（如 `canRefund: true`/`canEdit: false`），`table.props.actions` 中的每个行内操作通过 `visibleField` 声明依据哪个数据字段判断显隐：
  ```yaml
  actions:
    - key: refund
      label: 退款
      visibleField: canRefund   # 取当前行数据中的同名字段作为显隐依据
  ```
  这是"数据驱动显隐"而非"表达式驱动显隐"，不引入 `$row` 变量，也不复用 `reactions`/`when` 语法，因此不受 C3 中"表达式作用域混淆"风险的影响。
- **涉及文件：** `03-component-registry.md`（`table.props.actions` 契约补充 `visibleField`）、`05-scenarios/data-table.md`（补充完整示例）、`schemas/component-registry.json`。

## C. 需先补 ADR，暂不直接实现

以下条目方向认可，但涉及扩展表达式引擎的作用域或影响协议核心边界，**必须先形成 ADR 并评审通过**，才能进入 B 组正式实现，不在 v0.2 首批直接落地。

### C1. 节点级条件渲染 `visibleWhen` + `$context` 命名空间
- **对应审计条目：** §2.3、§5.1
- **风险：** 一旦引入 `$context.user`/`$context.features` 等页面级上下文变量，联动表达式实际承载的信息面会显著扩大，可能在事实上突破 ADR-0001/0002 中"联动仅做简单声明式判断"的既有边界。
- **要求先产出：** `decisions/0003-context-namespace-and-visible-when.md`，需回答：
  - `$context` 的数据来源、注入时机、是否可被前端环境篡改（安全边界）。
  - `visibleWhen` 与现有 `reactions` 的关系——是否复用同一套白名单表达式解析器。
  - 是否要为此单独限定：`$context` 只读、不允许出现在 `fulfill`/`otherwise` 的赋值目标中。
- **暂定归属：** 若 ADR 通过，`visibleWhen` 归入 `01-node-protocol.md` 的 Node 结构（第 6 个可选字段），`$context` 命名空间归入 `02-reaction-expression.md`。

### C2. 权限控制 `permissions`
- **对应审计条目：** §3.2
- **优先级调整：** 审计报告放在 P2，**本计划建议提升为高优先级评审项**（中后台系统普遍存在多角色场景），但由于其判定逻辑本质上依赖 C1 的 `$context.user`，两者应合并评审、统一设计，不单独抢跑实现。
- **要求先产出：** 在 `decisions/0003-context-namespace-and-visible-when.md` 中一并讨论 `permissions` 与 `$context` 的关系，或视讨论结果拆分独立 ADR。

### C3. 表格行级联动 `$row`
- **对应审计条目：** §3.6
- **风险：** 引入 `$row` 意味着表达式作用域从"表单内字段"扩展到"任意数据行上下文"，需要明确它与表单 `$deps` 的隔离边界，避免同一表达式解析器在不同场景下变量语义混淆。
- **要求先产出：** `decisions/0004-row-level-scope.md`，需回答：
  - `$row` 仅在 `table` 的 `columns`/`actions` 内可用，是否需要显式声明（如 `scope: row`）来防止误用。
  - 行级联动是否复用 `reactions` 结构，还是需要独立的 `rowReactions` 概念。
  - `$row` 方案与 B10 的"数据驱动显隐"方案（`visibleField`）是否长期共存，还是 `$row` 方案成熟后逐步替代 B10（如替代需提供迁移路径）。
- **过渡方案已提前发布：** 见 B10（`visibleField`），在 ADR 通过前，行级显隐通过该方案表达，不存在协议空窗期。

### C4. 表达式引擎新增 `compute` 字段（纯计算，非表单联动）
- **对应审计条目：** §5.2
- **说明：** 审计报告本身已给出谨慎的演进路径（不扩展 `when`，新增独立的 `compute` 字段，且明确排除函数调用能力）。本计划采纳该路径本身作为**未来演进方向**记录在案，但不在 v0.2 中实现，等到出现真实、迫切的只读计算需求时再启动，避免过度设计。
- **要求先产出（触发时）：** `decisions/0005-compute-field.md`。

## D. 暂不采纳 / 降级说明

| 条目 | 原报告优先级 | 本计划处理 | 理由 |
|---|---|---|---|
| §3.2 权限控制 | P2 | 提升为 C 组高优先级评审项 | 中后台刚需，但需与 `$context` 一并设计，见 C2 |
| §4.4 placeholder/description/tooltip | P3 | 提升为 B5，随批次直接发布 | 改动成本低、纯新增可选字段，无需等待 |
| §5.3 `component-registry.json` CI 自动生成 | 建议列为 v0.2 高优先级 | **有意延后至 v0.3**，非遗漏 | 需要先确定前端组件 Props 的 TS 类型定义规范并搭建生成工具链，属于工具链建设而非协议内容变更，不应绑定协议本身的发版节奏。v0.2 阶段 `component-registry.json` 继续手动维护，但 B9 的 `deprecated`/`since` 标注和本计划附带的验收清单（要求 Schema 与文字文档逐字段核对）作为手动维护阶段的临时防漂移手段 |

其余条目维持审计报告原判断，未做优先级调整。

## 发布顺序建议

1. **v0.2.0**：A 组全部 + B 组全部（B1–B10），一次性发布，附带迁移说明（尤其是 A3 的破坏性字段迁移）。
