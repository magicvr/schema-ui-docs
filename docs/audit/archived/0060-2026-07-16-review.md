## 审视范围

纳入本轮的判断标准是：某个协议字段、默认值、校验规则或执行算法是否可能让独立的前端 Renderer 与后端消费者对同一页面配置产生不同请求、不同结果或不同接受/拒绝结论。

明确排除 Cookie/CSRF、Authorization、宿主 origin 策略、运行时并发/内存预算、MCP worker 稳定性、npm 网络审计和 CI 机器环境。这些问题不属于协议字段和协议执行语义本身。

## 已复核的协议修复

### V236：DataRef 只读边界

- DataRef 和页面级 API datasource 只允许 `GET`，缺省 method 也按 `GET` 处理。
- `params` 仅编码到 query，DataRef 不隐式定义请求体。
- Schema、L2、JS/Python request reference 和非法 method fixture 使用同一拒绝码 `DATA_REF_METHOD_NOT_READ_ONLY`。
- 需要写操作或 JSON body 的命令式请求必须使用 Action；body-based read 需要另行 ADR。

对应证据：[`ADR-0013`](../../decisions/0013-dataref-read-only.md)、[`node.schema.json`](../../schemas/node.schema.json)、[`page.schema.json`](../../schemas/page.schema.json) 和 request conformance fixtures。

### V237：Action 重试、幂等和未知结果

- `retryPolicy` 仅允许 `never` 或 `idempotent`，缺省为 `never`。
- `idempotent` 的同一次逻辑调用在重试中复用同一个 `Idempotency-Key`；页面 DSL 不携带静态 key。
- timeout/network interruption 的结果是 `unknown`，不能被解释成服务端未执行。
- JS/Python request、upload、action-outcome reference 与 fixtures 对齐；业务服务是否真正去重属于接入方验收，不在本轮协议文档复核中代替判断。

对应证据：[`ADR-0014`](../../decisions/0014-action-replay-semantics.md)、[`07-actions-contract.md`](../../07-actions-contract.md) 和 actions/uploads/request fixtures。

### V238：RowAction 标量和 own-property

- `$row.*` 解析结果必须是 string、finite number、boolean 或 null；path/query/body 不能隐式携带对象或数组。
- row 字段按 own-property 读取，缺失字段统一拒绝。
- `__proto__`、`prototype`、`constructor` 路径被统一拒绝，L2 与两种 reference 的结果一致。

### V239：DataRef body 语义

当前协议明确选择 query-compatible read API：DataRef 不增加隐式 body 字段；需要 JSON body 的读取接口不能复用 `params`，必须新增独立协议设计和迁移证据。

### V240：协议 URL grammar

请求、上传、远程选项加载和导航相关 URL 统一为 baseURL 下的单斜杠相对路径；绝对 URL、协议相对 URL、空白和反斜杠形式在 Schema、L2 和 reference 中拒绝。

## 延续问题

### 0059/V241（P1，已关闭）：v2.0 版本边界已跟随协议语义收紧

**分类：** 协议兼容性 / 版本发布边界

**证据：**

1. 根 `package.json`、MCP `package.json` 和对应 lockfile 已统一为 `2.0.0`；`npm run release:check` 当前报告 `protocolVersion: "2.0"`。
2. 当前补丁将原先可接受的非 GET DataRef、绝对/非协议相对 URL 变为拒绝，并为 Action/Upload 引入 `retryPolicy`、`unknown` 和幂等 key 的执行约束。
3. [`09-v1-release-goals.md`](../../09-v1-release-goals.md) §5 的 MAJOR 规则已被遵守；v2 文档、Schema、fixtures、MCP 和迁移指南作为同一版本候选物同步更新。
4. [`10-v2-release-goals.md`](../../10-v2-release-goals.md)、[`1.0-to-2.0.md`](../../migrations/1.0-to-2.0.md) 和 version-negotiation fixtures 共同定义了 v1.0 到 v2.0 的唯一迁移边界。

**影响：**

- v1.0 页面继续由 v1.0 Renderer 消费，或先经过显式 migration adapter；v2.0 Renderer 对未迁移的 v1.0 页面明确返回 `UNSUPPORTED_PROTOCOL_VERSION`。
- v2.0 的 Action 超时、重试和未知结果语义只对声明 `2.0` 的页面生效，不会静默改变 v1.0 消费者的结果解释。
- package、lockfile、Schema、fixtures、版本协商、迁移文档和 MCP 索引已形成同一发布边界。

**处理结果：** 已选择方案 A，将 V236–V240 定义为下一 MAJOR `2.0`，并完成版本协商、迁移、fixtures 和发布元数据同步。方案 B 未采用，因此没有在同一 `1.0` 入口引入双语义。V241 已关闭；0060 仅等待用户确认归档。
