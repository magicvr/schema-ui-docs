## 发现的问题

### V199（🟡）V197 后组件表与 Schema 未同步「完整单个 `$deps.*` / 禁模板拼接」

**位置：**

- `docs/03-component-registry.md` `OptionsSource.params` 字段说明
- `docs/04-datasource-contract.md` §9 请求说明
- `docs/schemas/node.schema.json` `DataRef.params.description`
- `docs/schemas/component-registry.json` `select.props.optionsSource.description`

**当前值（摘录）：**

```text
# 03 OptionsSource.params
请求参数，可引用 `$deps.*`（复用 `reactions` 依赖机制，仅做值替换，不做条件判断）。

# 04 §9
params 中的值可引用 `$deps.*`（表单内其他字段的当前值）。

# node.schema.json DataRef.params
值可引用 $deps.*（仅做值替换，不做条件判断）

# component-registry.json optionsSource
params 可引用 $deps.*，值为 null/undefined 时该参数整体省略
```

**问题：** 0048/V197 已将权威规则收紧为：params 字符串要么是不含 `$` 的字面量，要么整段精确匹配单个 `$deps.<path>`；L3a 对模板拼接报 `DATA_PARAMS_VARIABLE`。入口组件表与机器可读 Schema 描述仍停留在「可引用 / 仅做值替换」，未禁止拼接，也未指向 §3.1/§10.7 的完整匹配口径。生成器或读者若只读 `03`/Schema，仍可能写出 `prefix-$deps.x`，随后被 L3a 拒绝。

**影响：** 🟡 中等。实现与权威正文（`02`/`04`§3.1/`06`/`08`）已正确，但常用入口与 Schema 描述漂移，属易误用。

**建议修复：**

1. `03` `OptionsSource.params`：改为「仅允许字面量或完整单个 `$deps.*` 整值替换，禁止模板拼接」，并指向 `04` §3.1。
2. `04` §9：请求句同步「完整单个 `$deps.*` / 禁止模板拼接」，并交叉引用 §3.1。
3. `node.schema.json` / `component-registry.json`：description 同步上述口径。
4. 可选：`01` `data.params` 注释补「完整单个 `$deps.*`」以与 §3.1 对齐（若改则一并列入 checklist）。
5. 无需改 L3a 实现或 MCP 测试（V197 已覆盖）；修复后做文本抽查与既有 MCP 测试回归。

**需要同步的文件：**

- `docs/03-component-registry.md`
- `docs/04-datasource-contract.md`
- `docs/schemas/node.schema.json`
- `docs/schemas/component-registry.json`
- 可选：`docs/01-node-protocol.md`
- `docs/CHANGELOG.md` Unreleased

**与已归档审计的关系：** 0048/V197 已修 L3a 与 `02`/`04`§3.1/`06`/`08`，**漏同步组件入口与 Schema 描述**，属新发现的残留，不是旧问题回退。
