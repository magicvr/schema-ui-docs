# 审视报告 0059：协议边界与执行语义复审

## 结论

本轮复审聚焦前后台共同消费的协议本身，跟踪 V236–V240 五项问题及其版本边界。V236–V240 的实现、Schema、L2、reference 和 fixtures 已完成；V241 已通过将这批收紧语义作为 `2.0.0` / `protocolVersion: "2.0"` 候选版本原子发布而关闭。审计材料保持 active，仅等待用户确认后归档，不代表仍有开放协议风险。

本轮候选实现采用的协议边界是：

- DataRef 是只读、query-only 的数据加载契约，只允许 `GET`。
- 变更 Action 通过 `retryPolicy: never | idempotent` 表达重试和幂等语义；超时/网络中断结果为 `unknown`。
- RowAction 的 `$row.*` 映射只接受标量结果，读取只走 own-property，并拒绝 `__proto__`、`prototype`、`constructor` 路径。
- 请求、上传、选项加载和导航相关 URL 统一为 baseURL 下的单斜杠相对路径。

这些规则已同步到协议正文、机器可读 Schema、L2 校验、JS/Python reference 和共享 fixtures。版本协商已增加 v2.0 正例及 v1.0 页面被 v2.0 Renderer 拒绝的反例；完整 MCP 测试的 122 个断言全部通过，但当前环境仍报告一个 Vitest `onTaskUpdate` worker 超时，该异常属于测试工具链，不计入协议风险。

## 范围判定

纳入本轮的依据是某个字段或执行规则是否会改变前后台对同一页面配置的解释：

1. Schema 是否允许危险或含义不完整的协议形态。
2. Renderer 与后端消费者是否会对 HTTP method、请求体、URL、重试和结果状态产生不同解释。
3. RowAction、DataRef 和 Action 的 JS/Python reference 是否按同一契约执行。
4. L0/L1、L2 和 conformance 是否能拒绝协议已经明确禁止的输入。

明确排除 Cookie/CSRF、Authorization 注入、宿主 origin 策略、运行时并发/内存预算、MCP worker 状态和 npm 网络审计等问题；它们不属于本项目作为共享协议的字段和执行语义。

## 问题闭环

### V236（P1）DataRef 自动加载可执行变更请求

**原风险：** 旧契约允许 DataRef 使用 `POST`、`PUT`、`PATCH`、`DELETE`。由于 DataRef 会在节点或页面数据加载阶段执行，配置错误可能把退款、审批或删除接口变成自动加载请求。

**闭环证据：**

- [01-node-protocol.md](../../01-node-protocol.md) 和 [04-datasource-contract.md](../../04-datasource-contract.md) 将 DataRef 明确定义为只读 GET/query-only 契约。
- [node.schema.json](../../schemas/node.schema.json) 与 [page.schema.json](../../schemas/page.schema.json) 将 DataRef/API datasource 的 `method` 枚举收紧为 `GET`。
- [0013-dataref-read-only.md](../../decisions/0013-dataref-read-only.md) 固化了非 GET DataRef 的标准拒绝码 `DATA_REF_METHOD_NOT_READ_ONLY`，并规定 body-based read 必须另行 ADR。
- L2 同时覆盖节点级 `data.source: api` 和页面级 `datasources.*`；JS/Python request reference 与 `data-ref-post-rejected` fixture 返回相同拒绝结果。
- 六个官方场景和合法的表格、图表、远程选项加载均保持通过。

**结论：** v2.0 候选实现已完成。自动加载路径不再具有隐式变更语义；写操作必须进入用户触发的 Action。

### V237（P1）Action 缺少重试、幂等和未知结果语义

**原风险：** 客户端超时不等于后端未执行。旧契约只有可重试错误，没有定义同一逻辑调用的身份、服务端去重或“请求可能已提交”的结果，前后端可能重复创建、扣款或上传。

**闭环证据：**

- [action.schema.json](../../schemas/action.schema.json)、[07-actions-contract.md](../../07-actions-contract.md) 和 [08-renderer-spec.md](../../08-renderer-spec.md) 增加 `retryPolicy: never | idempotent`，默认值为 `never`。
- [0014-action-replay-semantics.md](../../decisions/0014-action-replay-semantics.md) 定义 invocation id、`Idempotency-Key`、服务端去重边界、冲突处理、Abort 以及 timeout/network 的 `unknown` 结果。
- JS/Python request、upload reference 对 `idempotent` 使用同一 invocation id；Action outcome reference 对超时和网络中断输出 `outcome: unknown`。
- fixtures 覆盖幂等头、缺少 invocation id、非法策略、上传逐文件 key 和未知结果，JS/Python runner 均通过。

**结论：** v2.0 候选实现已完成。真实业务服务是否正确实现去重属于消费者接入验收，不在本轮协议审计范围内。

### V238（P2）RowAction 标量约束未闭合到执行结果

**原风险：** 协议只允许扁平标量映射，但旧 reference 可能把 `$row.profile` 等对象直接放入 body；同时使用原型链读取会使不存在的业务字段被解释成运行时属性。

**闭环证据：**

- [07-actions-contract.md](../../07-actions-contract.md) 和 [08-renderer-spec.md](../../08-renderer-spec.md) 明确 path/query/body 的值只能是 string、finite number、boolean、null 或单个 `$row.*` 引用，解析后仍必须是标量。
- [validate-l2-components.js](../../../scripts/validate-l2-components.js) 在静态阶段拒绝对象/数组和原型链保留路径。
- JS reference 使用 `hasOwnProperty`，Python reference 使用字典 own-key 读取；两端均拒绝嵌套对象、数组、缺失路径和保留路径。
- request fixtures 覆盖 `row-action-nested-body-rejected`、`row-action-prototype-path-rejected` 以及合法嵌套 scalar 路径；MCP 回归覆盖同一 L2 边界。

**结论：** 候选实现已完成；协议声明、静态校验和执行结果的类型边界一致。

### V239（P2）DataRef 无法表达 body-based read API

**原风险：** 旧契约把所有 DataRef `params` 编码进 query，却允许非 GET 方法，既不能安全表达自动写请求，也不能明确表达需要 JSON body 的只读接口，容易诱发实现方私自扩展。

**闭环证据：**

- 本轮选择并固化 query-compatible read API 方案，而不是为 DataRef 增加隐式 body。
- [0013-dataref-read-only.md](../../decisions/0013-dataref-read-only.md)、[01-node-protocol.md](../../01-node-protocol.md) 和 [04-datasource-contract.md](../../04-datasource-contract.md) 明确 `params` 只进入 GET query，DataRef 不定义 body 字段。
- 需要 body 的命令式请求统一使用 Action；未来若需要 body-based read，必须新增独立字段、方法集合、序列化和 fixtures 的 ADR。
- Schema、L2、JS/Python reference 和 request fixtures 对该选择保持一致。

**结论：** 候选实现已完成；该能力缺口已转化为明确的协议边界，不再由 Renderer 自行推断。

### V240（P2）URL 目标和解析范围不统一

**原风险：** 旧契约在不同位置同时出现相对路径和完整绝对 URL 表述，消费者可能采用不同的 baseURL、scheme 或跨 origin 解释。

**闭环证据：**

- [08-renderer-spec.md](../../08-renderer-spec.md) 固化 URL grammar：只接受 `/` 开头且不以 `//` 开头、无空白和反斜杠的单斜杠相对路径；Renderer 按 baseURL 拼接。
- Page/Node/Action Schema 同步该 pattern，覆盖 DataRef、页面 datasource、request/upload/navigate action 和 navigate outcome。
- L2 覆盖节点 DataRef、页面 datasource、optionsSource、直接上传地址、Action URL 和 navigate URL。
- JS/Python request/upload reference 拒绝绝对 URL、协议相对 URL、空白和反斜杠；MCP fixture 已覆盖绝对 URL 反例。

**结论：** v2.0 候选实现已完成；协议 URL 的目标范围和拒绝规则已统一。

### V241（P1）已关闭：候选修复已进入 v2.0 版本边界

**证据：**

1. [09-v1-release-goals.md](../../09-v1-release-goals.md) §5 明确规定：收紧既有合法输入、改变默认值或执行结果属于 MAJOR；本轮按该规则选择 v2.0，而不是修改同一 v1.0 入口的行为。
2. 根协议包、MCP 包、lockfile、核心规范、官方场景和 algorithm fixtures 已统一为 `2.0.0` / `protocolVersion: "2.0"`。
3. [10-v2-release-goals.md](../../10-v2-release-goals.md) 与 [1.0-to-2.0.md](../../migrations/1.0-to-2.0.md) 定义了迁移和发布门禁；版本协商 fixture 覆盖 v2.0 接受及 v1.0 页面被 v2.0 Renderer 拒绝。
4. `release:check` 已检查 v2.0 迁移文档、78 个 versioned cases、包版本一致性和 fixture digest。

**处理结果：** v1.0 发布物和 v2.0 发布候选物拥有明确版本边界。旧页面继续交给 v1.0 Renderer，迁移后的页面才交给 v2.0 Renderer；同一个版本入口不会隐式混用两套语义。

**必须决策：**

- 已选择方案 A：将 V236–V240 定义为 `2.0` MAJOR 候选，并同步 package/lockfile、协议版本、fixtures、版本协商、迁移文档和发布门禁。
- 方案 B 不再适用于当前实现；v1.0 兼容性由版本协商和显式迁移 adapter 保持，而不是在同一入口引入双语义。

**当前状态：** 已关闭。报告和 checklist 暂保留在 active 区，等待用户确认审计资料归档；未执行归档移动。
