## 发现的问题

---

### U1（🟡）`form` 条件必填约束挂错层级，L2 可能放过无效配置

**位置：** `schemas/component-registry.json` `components.form`

**对应正文：** `03-component-registry.md` `form` 组件说明

**问题：** `03-component-registry.md` 明确声明：

- 默认模式（`mode` 未指定或 `mode: default`）提交时执行 `submitAction`，因此 `submitAction` 在默认模式下必填。
- 搜索模式（`mode: search`）提交时刷新 `targetTable`，因此 `targetTable` 在搜索模式下必填，`submitAction` 被忽略。

但 `component-registry.json` 中 `mode`、`submitAction`、`targetTable` 都定义在 `form.props` 内，而当前 `allOf.if` 挂在 `form` 组件对象层，并直接检查顶层 `mode`：

```json
"allOf": [
  {
    "if": {
      "properties": {
        "mode": { "const": "search" }
      },
      "required": ["mode"]
    },
    "then": { "required": ["targetTable"] },
    "else": { "required": ["submitAction"] }
  }
]
```

字段位置验证结果：

```text
form_has_top_level_allOf=True
form_props_has_allOf=False
form_props_keys=title,titleKey,submitAction,mode,targetTable,additionalProperties
```

这说明条件判断与实际字段不在同一层级。若 L2 校验器按注册表结构消费该配置，`mode: search` 却缺少 `targetTable`、默认模式缺少 `submitAction` 的配置可能不会被机器约束拦住。

**影响：** 🟡 中等。正文契约清晰，但机器可读注册表无法可靠阻断无效 `form` 配置，可能造成 Renderer 运行时才暴露行为缺口。

**建议：** 将条件必填规则移动到 `form.props` 内，或改写为检查 `props.mode` / `props.targetTable` / `props.submitAction` 的结构。修复后建议用至少两个反例验证：

- `props.mode: search` 且缺少 `props.targetTable` 应失败。
- 默认模式缺少 `props.submitAction` 应失败。

### U2（🟡）`OutcomeBehavior` 的条件必填只写在正文，未进入 `action.schema.json`

**位置：** `schemas/action.schema.json` `definitions.OutcomeBehavior`

**对应正文：** `07-actions-contract.md` §7 `OutcomeBehavior`

**问题：** `07-actions-contract.md` 明确声明：

| behavior | 必填附加字段 |
|---|---|
| `toast` | `message` |
| `navigate` | `url` |
| `reload` | 无 |
| `closeModal` | 无 |

但 `action.schema.json` 当前只要求 `behavior`：

```json
"required": ["behavior"]
```

字段验证结果：

```text
outcome_required=behavior
outcome_has_conditionals=False
```

因此 `{ "behavior": "toast" }` 或 `{ "behavior": "navigate" }` 这类不完整 outcome 可能通过 Schema 校验，但 Renderer 执行时缺少必要提示文案或跳转地址。

**影响：** 🟡 中等。Action 主体类型的条件必填已经在 schema 顶层通过 `allOf.if/then` 表达，但 `OutcomeBehavior` 没有同等约束，会造成同一文件内约束强度不一致。

**建议：** 在 `OutcomeBehavior` 内补充条件约束：

- `if behavior == toast then required ["message"]`
- `if behavior == navigate then required ["url"]`

并保留 `reload` / `closeModal` 无附加字段的现有语义。

### U3（🟢）`table.props.actions` 总描述仍保留旧 `visibleField` 过渡措辞

**位置：** `schemas/component-registry.json` `table.props.actions.description`

**对应正文：** `03-component-registry.md` RowAction、ADR-0004

**问题：** `visibleField` 字段自身的描述已经更新为 `scope: row` 的 `visibleWhen` 语法糖，并说明显式 `visibleWhen` 优先。但 `actions` 数组的总描述仍写：

```text
每项支持 visibleField（since 0.2，审计 §3.6 过渡方案，数据驱动显隐）
```

当前正文和 ADR-0004 已将 `visibleField` 定义为兼容语法糖，而不是独立的过渡显隐机制。该总描述虽然不影响结构校验，但会让基于注册表生成提示、悬停说明或 AI 配置建议的工具继续继承旧语义。

**影响：** 🟢 低。属于描述漂移，不改变协议行为，但会影响机器可读文案的一致性。

**建议：** 将总描述改为更中性的当前语义，例如：

```text
行内操作；可使用 visibleField 兼容语法糖或显式 visibleWhen 表达行级显隐
```

如需更精确，可同时标注新项目推荐显式 `scope: row` 的 `visibleWhen`。
