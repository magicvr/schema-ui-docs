# 审计报告 0032：DataRef/权限静态校验缺口 + MCP 文档/错误分类漂移 + Action 示例不可执行

## 背景

本轮审计基于 0031 已归档后的再次全量文档、Schema、校验脚本与 MCP 实现复审。0031 中发现的表格内嵌表达式 `$ref` 结构约束、Renderer L3 清单、`internalError` 对象结构和 0030 checklist 状态问题在当前工作区已确认修复，本轮不重复报告 V67-V70。

本轮重点交叉核对：`DataRef` 来源形态与 `responseMapping` 语义校验是否一致，`permissions.*` 的“仅 `$context.*`”边界是否真正由 L3a 执行，MCP 工具错误分类、出参示例和实施计划是否跟随实现同步，以及官方文档中的完整 YAML 示例是否仍能通过当前校验链。

## 发现的问题

### V71（🟡）`source: ref` 的本地 `responseMapping` 绕过 L2 列表/分页语义校验

**位置：**
- `docs/01-node-protocol.md` §3.3：`source: ref` 仅携带 `ref`，并可在引用目标为 API 数据源时携带本地 `responseMapping` 覆盖响应解析
- `docs/04-datasource-contract.md` §4.1 / §4.1.1：列表类 `DataRef` 响应映射声明 `list` / `total` 语义
- `docs/06-validation.md` §1：L2 负责 `responseMapping` 条件必填语义
- `scripts/validate-l2-components.js`：`validateResponseMapping()` 在 `data.source !== 'api'` 时直接返回

**当前值：** `node.schema.json` 已允许 `source: ref` 携带 `responseMapping`，正文也明确这是引用 API 数据源时的本地覆盖能力。但 L2 只检查 `source: api`，不会处理 `source: ref` 的本地映射。

**复现（已实测）：** 一个 `table` 使用 `data.source: ref` 引用页面级 API datasource，并声明 `responseMapping.total` 但缺少 `responseMapping.list`。该配置在 `npm run validate -- <tmp.yaml>` 的 L0/L1、L2、L3a、L4 中均通过。

**问题：** `source: ref` 被正文纳入 `DataRef` 响应解析能力后，L2 的语义条件仍停留在“只处理内联 API 数据源”。这会让同一条列表/分页响应映射规则在 `api` 与 `ref` 两种合法写法之间产生差异。

**影响：** 🟡 中等。不会改变基础 Schema，但会让 CI/MCP 接受缺少 `list` 的表格映射，Renderer 运行时才暴露数据解析失败。对服务端分页表格尤其容易误导后端和配置生成器。

**建议修复：**
1. L2 判断从 `data.source === 'api'` 扩展为“声明了 `data.responseMapping` 的 `api` 或 `ref` 数据源”。
2. 对 `source: ref` 无法静态确认引用目标是否 API 的情况下，保守按本地 `responseMapping` 已声明即执行列表/分页语义校验；或者解析 `doc.datasources[data.ref]`，仅引用目标为 API 时执行。
3. 补充反例测试：`table + source: ref + responseMapping.total only` 应失败；合法 `source: ref + list/total` 应通过。
4. MCP `validate_content` 补充同类回归，确保工具返回与 CI 一致。

**需要同步的文件：**
- `scripts/validate-l2-components.js`
- `mcp/tests/test-utils.ts`
- `mcp/tests/validate-content.test.ts`
- 可选：`docs/06-validation.md`（若需要明确 `source: ref` 本地映射也进入 L2 条件必填）

### V72（🟡）`table.props.actions[].permissions` 未进入 L3a 扫描

**位置：**
- `docs/03-component-registry.md` `RowAction.permissions`：操作级权限控制，表达式仅允许 `$context.*`
- `docs/schemas/component-registry.json`：`table.props.actions.items.properties.permissions` `$ref: node.schema.json#/definitions/Permissions`
- `scripts/validate-l3a-expressions.js`：表格列扫描 `columns[].permissions`，但 `actions[]` 循环只处理 `visibleWhen` 和 `reactions`

**当前值：** 表格列 `permissions` 会被 L3a 扫描；行内操作 `permissions` 不会被扫描。补齐必填字段后的临时反例 `table.props.actions[0].permissions.view: "$self == true"` 全链路通过。

**问题：** 文档和 DSL 都声明 RowAction 支持权限表达式，但静态校验遍历漏掉了该路径。行级操作按钮是权限判断的高风险入口，漏扫会让非法变量或非法表达式直到 Renderer 才暴露，甚至被不同 Renderer 以不同方式解释。

**影响：** 🟡 中等。权限表达式属于渲染层显隐/可操作控制；虽不能替代后端鉴权，但工具链误报通过会削弱协议对 UI 权限边界的一致约束。

**建议修复：**
1. 在 `actions.forEach` 中增加 `action.permissions` 遍历，与 `columns[].permissions` 使用同一 `location: 'permission'` 规则。
2. 补充 L3a 反例：RowAction `permissions.view` 使用 `$self` / `$row` / `$deps` 均应失败。
3. MCP `validate_content` 增加 RowAction permissions 回归，避免只覆盖列级权限。

**需要同步的文件：**
- `scripts/validate-l3a-expressions.js`
- `mcp/tests/test-utils.ts`
- `mcp/tests/validate-content.test.ts`
- 可选：`docs/06-validation.md`（若补充 L3a 遍历范围说明）

### V73（🟡）`permissions.*` 的“仅 `$context.*`”规则未完整执行，仍放过 `$self` / `$row.*` / `$parentRow.*`

**位置：**
- `docs/01-node-protocol.md` §3.9：`permissions.*` 表达式为 `$context.*` 表达式
- `docs/02-reaction-expression.md` §10.2 与附录 A：节点 `permissions.*` 只允许 `$context.*`，`$self` / `$row.*` / `$parentRow.*` 均为禁止
- `scripts/validate-l3a-expressions.js`：`location === 'permission'` 仅检查 `hasDepRef`

**当前值：** L3a 对 `permissions.*` 只拒绝 `$deps.*`。补齐必填字段后的临时反例 `body.permissions.view: "$self == true"` 全链路通过。

**问题：** 正文中的“仅允许 `$context.*`”被实现成了“只禁止 `$deps.*`”。这留下了 `$self`、`$row.*`、`$parentRow.*` 等变量的绕过空间，也让权限表达式与变量可见性矩阵不一致。

**影响：** 🟡 中等。接入方可能把业务字段或行数据混入权限判断，造成权限与业务状态边界不清；不同 Renderer 若按文档实现，会与 CI/MCP 接受结果不一致。

**建议修复：**
1. L3a 在 `location: 'permission'` 下拒绝任何非 `$context.*` 变量，而不是只拒绝 `$deps.*`。
2. 保留 `$context.user.*` / `$context.features.*` 的合法路径，并与 V74 的根命名空间白名单校验组合。
3. 补充节点、列级、行级 action 三类 permissions 反例测试。

**需要同步的文件：**
- `scripts/validate-l3a-expressions.js`
- `mcp/tests/test-utils.ts`
- `mcp/tests/validate-content.test.ts`
- 可选：`docs/06-validation.md`（若自检清单要从“是否混入 `$deps.*`”强化为“是否只使用 `$context.*`”）

### V74（🟡）`$context` 根命名空间白名单未被 L3a enforce

**位置：**
- `docs/02-reaction-expression.md` §2：变量命名空间只列出 `$context.user.*` 与 `$context.features.*`
- `docs/02-reaction-expression.md` §12：新增 `$context` 根命名空间必须走 ADR，禁止项目层面自行扩展
- `scripts/validate-l3a-expressions.js`：`ALLOWED_NS_PREFIXES` 只按 `$context.` 前缀放行

**当前值：** L3a 接受任意 `$context.*` 根路径。补齐必填字段后的临时反例 `permissions.view: "$context.tenant.id == 't1'"` 全链路通过。

**问题：** 文档把 `$context.user` / `$context.features` 定义为协议级白名单，但实现只检查 `$context.` 前缀，没有校验第二段根命名空间。接入方可绕过 ADR 流程直接使用 `$context.tenant`、`$context.org` 等项目私有根。

**影响：** 🟡 中等。短期是校验与文档不一致；长期会导致 `$context` 结构碎片化，削弱跨项目配置可移植性。

**建议修复：**
1. L3a 增加 `$context` 根命名空间校验，仅允许 `$context.user.*` 与 `$context.features.*`。
2. 对 `$context` 本身或 `$context.user` 这类边界路径给出明确行为：要么允许访问对象根，要么要求至少一段字段，并同步文档。
3. 补充 `$context.tenant.*` 反例与 `$context.user.roles`、`$context.features.newDashboard` 正例。

**需要同步的文件：**
- `scripts/validate-l3a-expressions.js`
- `mcp/tests/test-utils.ts`
- `mcp/tests/validate-content.test.ts`
- 可选：`docs/02-reaction-expression.md`（若要明确是否允许 `$context.user` 根对象访问）

### V75（🟡）MCP 子脚本异常/非 JSON 输出被归类为协议违规，而不是 `internalError`

**位置：**
- `docs/decisions/0007-mcp-protocol-reader-validator.md` D11：脚本缺失、依赖缺失、临时目录不可写等 MCP 内部异常应填入 `internalError`
- `mcp/src/core/validation-runner.ts` `runLayerScript()`：子脚本输出无法解析为 JSON 时返回一条 layer violation

**当前值：** 0031 V69 已将 `internalError` 改为结构化对象；但 `runLayerScript()` 的子进程异常路径仍会把非 JSON 输出映射成 `layers.L2/L3a/L4` 的违规项：`[Lx] 无法解析校验脚本 JSON 输出`。

**问题：** 子脚本缺失、依赖缺失、脚本崩溃或输出非 JSON 属于 MCP 服务内部故障，不是调用方页面配置违反 L2/L3a/L4 协议。当前分类会让 `summary`、`suggestedDocs` 和客户端处理都偏向“配置错误”。

**影响：** 🟡 中等。正常路径不受影响，但当镜像打包遗漏脚本、依赖损坏或子脚本崩溃时，MCP 会返回误导性协议违规，降低可诊断性。

**建议修复：**
1. 让 `runLayerScript()` 能返回或抛出结构化内部错误；`validateContent()` 汇总后设置 `internalError`。
2. 保留脚本正常输出中的 `violations` / `parseErrors` 映射，不把工具自身非 JSON 输出伪装成用户配置违规。
3. 补充单测：模拟子脚本非 JSON 输出或异常退出，断言 `internalError.message` 存在且对应 layer 违规为空。

**需要同步的文件：**
- `mcp/src/core/validation-runner.ts`
- `mcp/tests/validate-content.test.ts`
- 可选：`docs/mcp/v1-design.md`（若需补充内部错误分类示例）

## 与已归档审计的关系

- V71 与 0028 V49 相关：V49 决定 `source: ref` 可携带本地 `responseMapping`，但未覆盖 L2 对该本地映射的条件必填语义。

- V72/V73 与 0031 V67 相关：V67 修复 table 内嵌 `permissions` 的 `$ref` 结构约束，未覆盖 RowAction permissions 的 L3a 遍历和“仅 `$context.*`”变量规则。

- V75 与 0031 V69 相关但不重复：V69 修复 `internalError` 类型结构，本轮对象是子脚本异常分类仍被放入 layer violation。
- V76/V77 是 MCP 文档与当前实现状态漂移，不属于 0031 的错误对象结构修复。
- V78 与 0027 V46 同类（示例缺 table 必填 props），但本轮位置是 `07-actions-contract.md` §3.1 行级后端请求示例，不是 Renderer 数据依赖示例。

## 建议处理顺序

## 防止复发的建议

1. 对每个新增 DSL `$ref` 或新增挂载点，除结构校验外，必须明确 L3a 是否遍历表达式内容。
2. 权限表达式相关规则用“允许集合”实现，避免只禁止单个变量导致新变量绕过。
3. `DataRef` 每次扩展来源分支时，同步复查依赖组件类型的 L2 语义规则是否也适用于新分支。
4. MCP 工具错误应区分“用户配置违规”和“工具链内部故障”；测试需覆盖至少一条子工具异常路径。
5. 文档中的完整 `meta/body` YAML 示例应定期抽取并跑 `validate-all`，不要只校验场景目录。
