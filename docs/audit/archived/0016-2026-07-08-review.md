# 审计报告 0016：DSL 必填关系 + 行级依赖语义 + Renderer 表达式示例漂移

## 背景

本轮审计基于 0015 已归档后的再次全量文档复审。0015 中发现的 Renderer `dateRangePicker` 示例、根 README 目录树、表达式作用域边界和校验路径问题在当前工作区已修复，本轮不重复报告这些已完成项。

本轮新增发现集中在三类此前未充分覆盖的表面：组件注册 DSL 与正文的必填语义是否一致、`scope: row` 下 `dependencies` 的机器可读描述是否足够清晰、以及 Renderer 表达式示例是否仍忠实表达当前协议的运算符和变量白名单。

## 发现的问题

### V14（🔴）`text` 组件的 `content` / `contentKey` 必填关系在 DSL 内部自相矛盾

**位置：** `schemas/component-registry.json` `text.props.content` 与 `text.anyOf`；`03-component-registry.md` `text` 组件属性表

**当前值：** 正文写 `content / contentKey` 必填，语义上表示二者至少提供一个。但 DSL 中同时存在两套互相冲突的约束：

```json
"content": { "type": "string", "required": true },
"contentKey": { "type": "string", "required": false },
"anyOf": [
  { "required": ["content"] },
  { "required": ["contentKey"] }
]
```

**问题：** `content.required: true` 表示必须提供 `content`；组件级 `anyOf` 又表示只提供 `contentKey` 也合法。工具如果读取 props 级 required，会判定 `contentKey` 单独使用非法；如果读取组件级 `anyOf`，又会判定合法。

**影响：** 🔴 关键。该文件是组件注册表的机器可读 DSL，冲突会直接影响表单编辑器、校验器和文档生成器对 `text` 组件 i18n 写法的判断。

**建议修复：** 将 `text.props.content.required` 改为 `false`，由组件级 `anyOf` 单独表达 `content` / `contentKey` 至少提供一个。正文可保留 `content / contentKey` 必填表述。

### V15（🟡）`scope: row` 下 `dependencies` 的语义在 reaction schema 中仍偏向 `$deps.*`，容易误导实现者

**位置：** `schemas/reaction.schema.json` `dependencies.description`；`03-component-registry.md` `visibleField` 展开说明；`schemas/component-registry.json` `visibleField.description`；`05-scenarios/data-table.md` 行级示例

**当前值：** ADR-0004 已明确 `scope: row` 下 `dependencies` 声明的是当前行内字段，并且 `visibleField` 等价展开为：

```yaml
visibleWhen:
  scope: row
  dependencies: [canRefund]
  when: "$row.canRefund == true"
```

但 `reaction.schema.json` 对 `dependencies` 的描述仍强调 `$deps.*` 变量必须列出，容易让读者理解为 `dependencies` 永远只服务于 `$deps` 注入。

**问题：** 正文和 ADR 的设计是正确的：`scope: form` 下 `dependencies` 对应 `$deps.*`；`scope: row` 下 `dependencies` 对应 `$row.*` 的行内字段依赖。但机器可读 schema 的描述没有写出这个分支，会让实现者误判 `scope: row` 示例中的 `dependencies: [canRefund]` 是错误写法，或反过来在行级表达式中错误注入 `$deps.canRefund`。

**影响：** 🟡 中等。协议主文档有正确规则，但 schema 描述偏窄，容易影响静态校验器和代码生成器实现。

**建议修复：** 更新 `reaction.schema.json` 的 `dependencies.description`：明确 `scope: form` 时声明 `$deps.*` 来源字段，`scope: row` 时声明 `$row.*` 行内字段路径；并说明 `scope: row` 不注入表单级 `$deps`。

### V16（🟡）Renderer expr-eval 示例把 `contains` 表达成函数模型，与协议的运算符语义冲突

**位置：** `08-renderer-spec.md` §5.2 最小集成示例与 §5.4 安全约束

**当前值：** `02-reaction-expression.md` 将 `contains` 定义为二元比较运算符，且表达式语言禁止函数调用。但 Renderer 示例中写：

```js
// 注册 contains 运算符
parser.functions.contains = (arr, val) => {
  return Array.isArray(arr) && arr.includes(val)
}
```

同时安全约束表写“除 `contains` 外不允许任何函数调用”。

**问题：** 协议语法是 `$context.user.roles contains 'admin'`，不是 `contains($context.user.roles, 'admin')`。示例把实现适配层写成 `parser.functions.contains`，再配合“除 contains 外”的措辞，会误导 Renderer 支持一个协议未定义的函数调用例外。

**影响：** 🟡 中等。实现者可能基于示例开放 `contains(...)` 函数语法，造成不同 Renderer 的表达式语言不一致，也削弱“禁止函数调用”的安全边界。

**建议修复：** 将示例改为“把协议中的 `a contains b` 在适配层编译/映射为安全的包含判断”，并把安全约束表改为“不允许任何函数调用”。若继续使用 expr-eval，应明确这是实现适配细节，不改变协议语法。

---

### V17（🟡）Renderer 最小集成示例未覆盖当前协议的完整变量白名单

**位置：** `08-renderer-spec.md` §5.2 最小集成示例；对照 `02-reaction-expression.md` 变量可见性矩阵

**当前值：** 示例主要展示 `$deps.*` 和 `$context.*` 的替换与注入，但当前协议变量白名单还包括 `$self`、`$row.*`、`$parentRow.*`，并且这些变量受表达式位置和 `scope` 严格约束。

**问题：** 作为 Renderer 实现规范中的“最小集成示例”，如果只展示表单和上下文变量，却不说明行级变量由调用方按位置注入，读者可能误以为 Renderer 表达式执行器只需支持 `$deps` / `$context`。这会让 `table.columns` / `table.actions` 的 `scope: row` 示例在实现层缺少明确入口。

**影响：** 🟡 中等。不会改变协议正文，但会影响 Renderer 实现完整性，尤其是行级显隐、禁用和嵌套表格表达式。

**建议修复：** 二选一：

1. 在示例标题或注释中明确“本示例仅覆盖 form/context 场景，`$self` / `$row` / `$parentRow` 由调用方按表达式位置注入并受静态校验约束”；
2. 或补充一个 `scope: row` 的调用示例，展示 `row`、`self`、`parentRow` 如何进入表达式求值上下文。

## 建议处理顺序

1. **V14（🔴）**：优先修正 `component-registry.json` 中 `text.content.required`，避免机器可读 DSL 继续输出冲突约束。
2. **V16（🟡）**：修正 Renderer `contains` 示例和函数调用安全约束，避免实现者扩大协议语法。
3. **V15（🟡）**：补充 `reaction.schema.json` 的 `dependencies` 分支说明，让 `scope: row` 示例和机器描述一致。
4. **V17（🟡）**：补充 Renderer 示例的变量覆盖说明或行级求值示例。

---

## 防止复发的建议

1. **组件 DSL 同时检查 props 级 required 与组件级组合约束：** 使用 `anyOf` 表达二选一时，对应 props 不应再单独标为必填。
2. **Schema 描述应随作用域扩展同步分支语义：** 当同一字段在不同 `scope` 下语义不同，机器可读 description 需要显式写出分支。
3. **Renderer 示例必须区分协议语法和实现适配：** 示例可以展示底层库适配，但要明确不改变协议允许的表达式形态。
4. **变量白名单更新后回扫 Renderer 示例：** 新增 `$row` / `$parentRow` / `$self` 这类变量后，Renderer 规范应同步说明注入边界。
