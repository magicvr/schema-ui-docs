# 审计报告 0044：表达式冲突语义、表格 scope 表述、row dependencies 权威文档、版本叙事与场景覆盖

## 背景

本轮基于 0043 已归档后的再次全量复审。0043 中的 tabs content 完整 Node 执行、嵌套结构错误分类、YAML 非有限数值、固定 select `labelKey`、统一入口退出码 2 与审计复合身份索引问题均已确认落地，本轮不重复报告 V155-V161。

本轮完整读取协议正文、场景、ADR、Schema/组件 DSL、L0-L4 校验脚本、MCP 文档/实现以及全部审计文件；对四个官方场景完整 YAML 执行 L0-L4 统一校验（全部通过）；对 `$row` dependencies 声明格式与表单内 `visibleWhen.dependencies` 条件必填执行最小正反例。新问题集中在五类：

1. Renderer 规范与表达式权威文档 / ADR-0006 对 `value` 冲突规则的表述冲突；
2. 表格列/操作字段表过度要求 `scope: row`，与同页作用域说明及 L3a 不一致；
3. L3a 已执行的 `$row` dependencies 完整点路径规则未写入 `01`/`02` 权威正文；
4. 表格 `scope: form` 下 `required`/`value` 语义未闭合，以及 search/upload 等高风险能力缺可复制场景；
5. 版本叙事、组件页脚、ADR 执行层标注、CHANGELOG/package 元数据等文档卫生问题。

扫描活跃区和归档区全部 `review` / `checklist` / `plan` 后，历史最大审计编号为 `0043`，本轮使用 `0044`。历史问题定义标题 165 个、唯一 V 数字 161 个，全局最大问题编号为 `V161`，因此本轮从 `V162` 连续编号到 `V174`。

## 发现的问题

### V162（🔴）`08` 的 `value` 冲突规则与 `02` / ADR-0006 矛盾

**位置：** `docs/08-renderer-spec.md` §5.3；`docs/02-reaction-expression.md` §14.1；`docs/decisions/0006-expression-evaluation-order.md` D3 / D1。

**当前值：**

- `08` §5.3：*「同一目标字段出现多处 `value` 写入时，按文档中 Node 的深度优先遍历顺序后写优先」*。
- `02` §14.1 / ADR-0006 D3：`fulfill.value` / `otherwise.value` **仅作用于当前字段**，不支持跨字段写入；同一字段上多条 `reactions` 按 **数组顺序** 后写优先。
- ADR-0006 D1 明确采用稳定快照模型，避免「组件树遍历顺序」影响协议行为。

**问题：** Renderer 实现规范发明了一套跨 Node 深度优先合并语义，与表达式权威文档和 ADR 直接冲突。实现者若只读 `08`，会做出与协议决策相反的调度。

**影响：** 🔴 严重。不同 Renderer 或同一实现的不同模块可能对同配置给出不同字段终值，破坏声明式协议的确定性。

**建议修复：** 重写 `08` §5.3 末句，对齐 `02` §14.1 与 ADR-0006 D3：仅同字段 `reactions` 数组顺序后写优先；禁止暗示跨 Node / 深度优先 value 合并。可交叉引用 `02` §14.1。

**与已归档审计的关系：** 0012/A2 与 0037/V106 已收敛跨字段 value 与「不同节点写同一目标」不可表达问题；本项是 `08` 残留的旧表述未同步。

**需要同步的文件：** `docs/08-renderer-spec.md`；必要时在 `docs/02-reaction-expression.md` §14.1 增加「Renderer 规范必须与本节一致」的反向引用。

### V163（🟡）ColumnDef / RowAction 字段表过度要求 `scope: row`

**位置：** `docs/03-component-registry.md` ColumnDef 与 RowAction 字段表；对照同页「作用域说明」、`docs/01-node-protocol.md` §3.5、`docs/02-reaction-expression.md` §9、L3a。

**当前值：**

- 字段表：`visibleWhen` / `reactions` 写 *「需声明 `scope: row`」*、*「行内字段联动规则，需声明 `scope: row`」*。
- 同页作用域说明：`scope: form` 为默认，可用于 `$deps.*`（表格在 form 内时）与 `$context.*`。
- `01` §3.5：*「使用 `$row.*` 或行级 `$self` 时必须显式 `scope: row`；仅使用 `$deps.*` / `$context.*` 时可使用默认 `scope: form`」*。
- L3a：独立表格列/操作在 `scope: form` 下允许仅含 `$context.*` 的表达式。

**问题：** 字段表强制语气与协议真实规则不一致，读者会认为一切列/操作表达式都必须 `scope: row`。

**影响：** 🟡 中等。生成器可能给纯权限/功能开关场景硬加 `scope: row` 并引入无意义的 `$row` dependencies，或把合法 form-scope 配置判为非法。

**建议修复：** 字段表改为条件表述：读取 `$row.*` 或列级 `$self` 时必须 `scope: row`；默认 `scope: form` 用于 `$context.*` / 表单内 `$deps.*`。与 `01` §3.5 对齐。

**与已归档审计的关系：** 0021/V28a 已补充独立表格 `scope: form` 下 `$deps` 拒绝说明；本项是字段表强制措辞未同步该边界。

**需要同步的文件：** `docs/03-component-registry.md`。

### V164（🟡）`$row.*` 的 `dependencies` 声明规则未写入表达式权威文档

**位置：** `docs/02-reaction-expression.md` §2 / §8 / §9；`docs/01-node-protocol.md` §3.5 / §3.8 注意；`docs/06-validation.md` L3a；`scripts/validate-l3a-expressions.js`。

**当前值：**

- L3a 将 `$row.<path>` 的 path 完整后缀与 `dependencies` 做精确包含匹配（如 `$row.canRefund` → `canRefund`，`$row.meta.canRefund` → `meta.canRefund`，`$row.__index` → `__index`）。
- 实测：`dependencies: [canRefund]` + `when: "$row.canRefund == true"` **通过**；`dependencies: ["$row.canRefund"]` **被拒**（`UNDECLARED_ROW_DEP`）。
- `06` L3a 仅有一句 *「`$row.*` 按完整点路径在 `dependencies` 中精确声明」*，未解释「不含 `$row.` 前缀」。
- `01` 关于 dependencies 的注意几乎只提 `$deps.*`；`02` 列出 `$row.__index` / `$row.__key` 但未给出 dependencies 键写法。

**问题：** 机器校验已强制执行的规则不在表达式权威正文，按 `02` 单独编写时易写成带 `$row.` 前缀的非法 dependencies。

**影响：** 🟡 中等。CI/L3a 会拒绝「看起来正确」的配置，增加接入摩擦；`__index` / `__key` 的声明方式也不明确。

**建议修复：** 在 `02` §8 或 §9 增加权威规则与正反例；`01` §3.5 补一句 row 作用域 dependencies 为「行字段点路径（无 `$row.` 前缀）」；`06` L3a 描述改为同样明确。

**与已归档审计的关系：** 行级作用域由 0004/ADR-0004 引入，后续校验增强未回填权威语法说明。

**需要同步的文件：** `docs/02-reaction-expression.md`、`docs/01-node-protocol.md`、`docs/06-validation.md`。

### V167（🟡）表格列/操作在 `scope: form` 下的 `required`/`value` 语义未闭合

**位置：** `docs/02-reaction-expression.md` §9.3；`docs/03-component-registry.md`；`scripts/validate-l2-components.js`（`validateStateMap`）。

**当前值：**

- §9.3 与 L2 仅在 `scope: row` 时禁止 `fulfill`/`otherwise` 的 `required`/`value`。
- 列/操作默认 `scope: form` 时仍可声明 `required`/`value`，L2 不拒绝。
- 列与行操作不是表单字段，协议未定义「当前字段」写入目标，Renderer 无明确语义。

**问题：** 合法通过校验的配置在运行时无定义行为，或被各实现自行解释。

**影响：** 🟡 中等。配置可通过 CI 但渲染结果不确定；与「value 仅作用于声明 reaction 的当前字段」模型冲突。

**建议修复：** 决策二选一并落地：

1. **推荐：** 无论 scope，表格 `columns[]`/`actions[]` 的 reactions 一律仅允许 `visible`/`disabled`；或  
2. 明确 form-scope 下列/操作 reactions 允许的状态键与语义，并补 L2 与正反例。

**与已归档审计的关系：** ADR-0004 / 0031 仅闭合 row 作用域 `required`/`value` 禁令；未覆盖 form-scope 挂载点。

**需要同步的文件：** `docs/02-reaction-expression.md`、`docs/03-component-registry.md`、`docs/06-validation.md`、`scripts/validate-l2-components.js`、`mcp/tests/validate-content.test.ts`。

### V169（🟡）search form / upload 等高风险能力缺少端到端场景

**位置：** `docs/05-scenarios/**`；对照 `docs/03-component-registry.md`、`docs/07-actions-contract.md`、L2 规则。

**当前值：**

- 官方四场景（grid / data-table / form-reactions / row-backend-actions）与 `00` 声明一致，完整 YAML 均通过 L0-L4。
- `form.mode: search` + `targetTable` 必须解析到 API 数据源、`upload` + `actions.upload` / `actionRef` 等规则仅有 `03` 内联片段，无整页可复制示例。
- 相关 L2 约束密集（能力声明、submitAction 忽略、actionRef 约束唯一来源等）。

**问题：** 生成器与接入方容易漏 `requiredCapabilities`、错误使用 `submitAction` 或 upload 双模式混用。

**影响：** 🟡 中等。不破坏既有场景正确性，但提高高风险能力的误用概率。

**建议修复：** 新增（或扩展）至少两个可复制场景：

1. 搜索表单 + 服务端分页表格（`mode: search`、`targetTable`、API data）；  
2. 上传字段（`action` 直接 URL 与 `actionRef` + `actions.upload` 至少覆盖一种完整页，另一种可作对照片段）。

**需要同步的文件：** `docs/05-scenarios/`、`docs/05-scenarios/README.md`、`docs/00-overview.md`（若扩展「覆盖场景」表述）。
