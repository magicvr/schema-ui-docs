# 审计报告 0040：组件 DSL 嵌套封闭性、MCP 异常边界与运行时兼容、归档证据链

## 背景

本轮审计基于 0039 已归档后的再次全量复审。0039 中的表达式校验边界、`$parentRow` 暂缓策略、`$context` 生命周期、Renderer 初始化契约和 MCP 完整工具文本预算均已确认修复，本轮不重复报告 V117-V124。

本轮完整读取协议正文、场景、ADR、机器可读 Schema/DSL、L0-L4 校验脚本、MCP 文档/实现/测试/打包配置及审计索引，并执行最小反例、MCP 构建测试和链接检查。新问题集中在三类边界：

1. 组件 DSL 的嵌套对象字段表没有统一表达和执行封闭性；
2. MCP 对临时目录失败和 Node 运行时下限的承诺没有形成可执行闭环；
3. 早期归档清单状态和相对链接未遵守当前审计证据链约定。

扫描全部 83 份 `review` / `checklist` / `plan` 后，历史最大问题编号为 `V124`，因此本轮问题从 `V125` 连续编号。当前活跃区为空，历史最大审计编号为 `0039`，本轮使用 `0040`。

## 发现的问题

### V125（🟡）组件 DSL 的部分嵌套对象放行协议字段表之外的属性

**位置：** `docs/schemas/component-registry.json` 的 `tabs.props.items[]`、`table.props.pagination`、`select.props.options[]`；`scripts/validate-l2-components.js` 的 `validateNestedObject()`。

**当前值：** 组件正文为这些嵌套对象给出了封闭字段表，但对应 DSL 规格未声明 `additionalProperties: false`。L2 的递归校验只有在该标记显式为 `false` 时才拒绝未知字段。

**问题：** 以下两类最小页面均通过完整 L0-L4 校验：

- `table.props.pagination: { mode: server, unexpected: true }`；
- `select.props.options: [{ label: A, value: a, unexpected: true }]`。

同一盲区还影响 `tabs.props.items[]`。未知字段会被当成合法组件配置进入 Renderer，与“组件 props 由注册 DSL 字段表约束”的 L2 定位不一致。

**影响：** 🟡 中等。配置拼写错误或私有扩展可以静默穿过官方校验链，不同 Renderer 可能选择忽略、消费或报错，削弱组件契约的互操作性。

**建议修复：**

1. 为应当封闭的嵌套规格补充 `additionalProperties: false`；
2. 明确 DSL 的默认开放/封闭策略，避免新增嵌套对象时继续遗漏；
3. 增加 `pagination`、`tabs.items[]`、`select.options[]` 未知字段反例及合法正例；
4. 更新 `03-component-registry.md` / `06-validation.md` 的 DSL 解释边界；
5. 在 MCP `validate_content` 测试中覆盖完整 L0-L4 行为。

**与已归档审计的关系：** 0037/V99 修复的是 `validateNestedObject()` 未执行嵌套字段级 `required: true`，只证明 `pagination: {}` 会因缺少 `mode` 被拒绝；未覆盖嵌套对象的未知字段封闭性。本条是同一解释器的另一条独立约束缺口。

**需要同步的文件：**

- `docs/schemas/component-registry.json`
- `scripts/validate-l2-components.js`（若统一默认策略需要实现支持）
- `docs/03-component-registry.md`
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`

### V126（🟡）MCP 创建临时目录失败时绕过结构化 `internalError`

**位置：** `mcp/src/core/validation-runner.ts` 的 `validateContent()`；ADR-0007 D11。

**当前值：** `fs.mkdtempSync()` 在 `try` 块之外执行。ADR-0007 明确把“临时目录不可写”列为 MCP 内部异常，并要求返回 `passed: false`、结构化 `internalError` 和内部错误摘要。

**问题：** 将 `TEMP` / `TMP` 指向不存在路径后调用 `validateContent()`，实际直接抛出：

```text
ENOENT: no such file or directory, mkdtemp '...schema-ui-mcp-XXXXXX'
```

调用方收不到稳定的 `ValidateContentResult`，MCP 工具层可能把异常交给 SDK 通用错误处理，违反 ADR 的对外错误格式。

**影响：** 🟡 中等。容器只读文件系统、临时目录权限错误或磁盘异常时，工具返回契约失效，AI 客户端无法按 `internalError` 分支处理。

**建议修复：**

1. 将临时目录创建纳入异常捕获；
2. 仅在目录已成功创建后执行 `finally` 清理；
3. 返回不含堆栈和内部路径细节的结构化 `internalError`；
4. 通过可注入的临时目录创建器或受控 mock 增加失败测试；
5. 执行 MCP build/test/tools smoke，并核对 ADR-0007 与 v1 design 无需改口径。

**与已归档审计的关系：** 0032/V75 修复子脚本异常/非 JSON 输出被误归类为协议违规；当前问题发生在子脚本启动前的临时目录创建阶段，不属于 V75 已覆盖路径。

**需要同步的文件：**

- `mcp/src/core/validation-runner.ts`
- `mcp/tests/validate-content.test.ts`
- 如实现策略变化，核对 `docs/mcp/v1-design.md`

### V127（🟡）MCP 包声明 Node `>=18`，但编译产物无法在 Node 18.19 启动

**位置：** `mcp/package.json` 的 `engines.node`；`mcp/src/server.ts` 的 JSON import attributes；根 `README.md`、ADR-0007 与 MCP 设计文档的运行时声明。

**当前值：** 包与文档统一承诺 Node.js `>=18`。`server.ts` 使用：

```ts
import packageJson from '../package.json' with { type: 'json' };
```

TypeScript 输出保留 `with { type: 'json' }`。Docker 基于 Node 24，因此镜像内正常。

**问题：** 使用声明范围内的 Node 18.19 解析编译后的 MCP server，实测失败：

```text
Unexpected token 'with'
```

本地 npm/node 运行方式是文档保留的开发与调试入口，当前 engines 范围会允许安装到实际无法启动的运行时。

**影响：** 🟡 中等。依赖 `engines` 选择运行时的消费者会在启动阶段失败；Docker 测试因使用 Node 24 无法发现该问题。

**建议修复：** 二选一并保持单一口径：

1. 若确需支持 Node 18，避免在运行时使用该语法，例如通过 `createRequire` 或读取 package JSON 获取版本；
2. 若保留 import attributes，则把包、根 README、MCP 文档与 ADR 的最低 Node 版本提高到实际支持该语法的版本；
3. 增加最低受支持 Node 版本的启动/smoke 验证，避免只测 Docker 当前版本；
4. 执行 build/test/tools smoke，并按最终决策验证最低版本启动。

**与已归档审计的关系：** 历史审计未覆盖 Node 运行时下限与编译产物语法的交叉验证；0038/V116 只处理 MCP SDK 依赖版本策略。

**需要同步的文件：**

- `mcp/src/server.ts` 或 `mcp/package.json`
- `README.md`
- `docs/mcp/README.md`
- `docs/mcp/v1-design.md`
- `docs/decisions/0007-mcp-protocol-reader-validator.md`
- MCP 启动/smoke 测试或 CI 配置

## 建议处理顺序

1. **先处理 V127：** 决定继续支持 Node 18 还是提高最低版本，避免实现和文档沿不同方向修改。
2. **再修 V125：** 统一组件 DSL 嵌套对象封闭策略，并用 MCP 反例固定行为。
3. **随后修 V126：** 闭合 MCP 临时目录异常边界并补测试。
4. **最后处理 V128/V129：** 按历史事实回填清单状态，修复归档链接并执行全量检查。

## 防止复发的建议

1. 为组件 DSL 解释器建立表驱动契约测试，覆盖每类嵌套对象的 required、enum、`$ref` 与 unknown-property 行为。
2. MCP 对外错误协议应覆盖资源获取阶段，而不只覆盖资源使用阶段；临时目录创建、Schema 读取和子进程启动都应落入结构化异常边界。
3. `engines.node` 应由最低版本启动测试证明，Docker 当前版本通过不能替代支持下限验证。
4. 归档前执行空复选框扫描和移动后 Markdown 链接检查；未来触发型条目也必须使用带解释的终态，而不是保留裸 `[ ]`。
