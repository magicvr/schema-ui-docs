# Schema-Driven UI 协议第三次审视：MVP 就绪度评估

> 审视日期：2026-07-07 | 审视版本：v0.2（含 0001/0002 审计修复）| 审视人：AI 助手
>
> 本次审视的视角不同：不再逐字段检查协议正确性（0001/0002 已完成），
> 而是评估**协议是否已足够作为前后端团队共同使用的 MVP 实践约定**。
> 核心判断标准：一个后端工程师 + 一个前端工程师拿着这套文档，能否在一个迭代内产出可工作的页面。

---

## 一、总体结论

协议在**概念框架层**已经相当成熟——单树模型、表达式白名单、分层校验体系、ADR 决策记录
都是高质量的架构设计。但从"文档协议"到"团队可以开工的 MVP 约定"，差距不在设计深度，而在**操作性**。

**核心问题：协议描述了"配置长什么样"，但没描述"配置怎么来"、"Renderer 怎么跑"、"缺了哪些必要组件"。**

以下按四类缺口展开评估，每类附 MVP 阻断性判定。

---

## 二、缺口一：缺少前端 Renderer 实现规范

### 现状

协议详细规定了后端如何编写 YAML/JSON 配置（Node 结构、组件契约、数据源），但前端
Renderer 的核心行为没有任何规范文档。目前"前端负责编写解析器（Renderer）、核心组件库、
皮肤样式"这句话是协议中关于 Renderer 的全部描述。

### 缺失内容

#### R1. 组件注册机制

- 当前状态：`00-overview.md` 术语表中定义了"组件注册表"概念，`03-component-registry.md`
  列出 `type` → 契约映射。但没有说明运行时的注册方式。
- MVP 需要回答：注册是运行时注入（`registerComponent('statCard', StatCard)`）还是编译期
  静态绑定？注册表是全局单例还是按页面实例隔离？

#### R2. 数据加载协调策略

- 当前状态：每个 Node 可独立声明 `data.source: api`，但未定义多个数据节点在一页中的加载行为。
- 场景：一个 dashboard 页面有 4 个 `statCard` + 1 个 `chart` + 1 个 `table`，各自有独立 API。
- MVP 需要回答：
  - 所有 API 是否并行发起？
  - 某个 API 失败是否阻断兄弟节点的渲染（stale-while-render vs all-or-nothing）？
  - 是否支持声明数据依赖（如 `table` 的 API 参数依赖 `select` 的当前值）？

#### R3. 错误边界与降级策略

- 当前状态：`states` 字段定义了组件级空态/加载态/错误态的**展示文案**，但未定义
  Renderer 在整棵树中的错误传播行为。
- MVP 需要回答：树中某个节点渲染/数据加载失败——
  - 是整页白屏崩溃（最差，但也是最常见的"没定义"行为）？
  - 只影响该节点，兄弟节点正常渲染？
  - 还是影响该节点所在子树？

#### R4. 表达式引擎沙箱实现指引

- 当前状态：`02-reaction-expression.md` 规定了白名单语法，`06-validation.md` 提到 L3
  校验，但没有给前端 Renderer 实现者提供具体的沙箱化方案指引。
- MVP 建议：至少推荐一个白名单表达式解析库（如 `expr-eval`），并说明集成方式。

### MVP 阻断性判定

| 缺失项 | 阻断 MVP？ | 理由 |
|---|---|---|
| R1 组件注册机制 | 否 | 前后端可通过口头约定 + 代码评审弥合，有经验的前端团队可自行设计 |
| R2 数据加载策略 | **是** | 如果不约定，前后端对"页面加载行为"的预期会不一致（后端以为 6 个 API
并行，前端做成串行，性能体验完全不同） |
| R3 错误边界 | 弱阻断 | `states` 已覆盖部分语义，团队可先用"组件级兜底"临时方案 |
| R4 沙箱实现指引 | 否 | `expr-eval` 等库的接入是前端纯技术决策 |

> **建议：** 新增 `docs/08-renderer-spec.md`，至少约定 R2（数据加载策略）+ R3（错误边界）。
> R1 和 R4 可附带提及，不强求规约级别的详细设计。

## 三、缺口二：缺少后端配置生成指南

### 现状

协议文档回答了"YAML 长什么样"，但后端工程师的实际工作流是：

1. 打开编辑器
2. 写 YAML
3. 校验它是否正确
4. 提交给前端（或存入配置中心）

第 3 步和第 4 步目前没有指引：

- 没有说明如何用 `schemas/*.json` 做本地校验（需要什么工具？`ajv` CLI？VS Code 插件？）
- 没有说明 `protocolVersion` 的版本协商流程（前端怎么知道后端送来了什么版本？不匹配时
  前端的行为是什么——拒绝渲染？降级渲染？报警？）
- 没有环境变量管理约定：`url: /api/orders` 在 dev/staging/prod 环境如何切换？后端是
  写入不同文件、用模板变量替换、还是由 Renderer 注入 baseURL？

### 缺失内容

#### G1. 配置校验工具链指引

- 当前状态：`06-validation.md` 描述了 CI 流程，但面向的是"已接入 CI 的团队"。
- MVP 需要：一份"后端开发者本地校验 5 分钟上手指南"——至少列出推荐工具和命令。

#### G2. 版本协商规范

- 当前状态：`meta.protocolVersion` 必填（since 0.2），`01-node-protocol.md` 有兼容说明。
- 未定义：Renderer 如何向配置的生成方（后端）宣告自己的支持版本范围？不匹配时如何处理？
- MVP 至少需要约定"不匹配时的失败模式"（拒绝渲染并报错，还是尽力按最近版本解析）。

#### G3. 环境变量 / baseURL 管理

- 当前状态：所有场景示例中 `url` 是硬编码的相对路径。
- MVP 需要约定：baseURL 是由 Renderer 统一注入，还是在 YAML 中写完整 URL？如果是注入，
  是 Renderer 初始化参数还是全局配置？

### MVP 阻断性判定

| 缺失项 | 阻断 MVP？ | 理由 |
|---|---|---|
| G1 校验工具链指引 | 弱阻断 | 团队可在 CI 中配置好校验后通知后端，但本地无校验会拉长反馈环 |
| G2 版本协商 | **是** | 没有版本协商，一旦协议升级（如 v0.2→v0.3），不知哪方先改、旧版
配置如何处理，必然造成线上事故 |
| G3 环境变量管理 | 否 | 可通过 Renderer 统一注入 baseURL 临时方案 |

> **建议：** 在 `00-overview.md` 或新增 `docs/08-renderer-spec.md` 中增加版本协商与环境变量
> 章节。校验工具链指引可在 `06-validation.md` 中追加一小节。

## 四、缺口三：高优先级能力卡在 ADR 等待区（C 组）

### 现状

0001 审计中标记的 C1/C2/C3 条目，在 0002 审计时（E1）仍然无进展。这三个条目不是锦上添花，
而是中后台系统的**标配需求**——缺失意味着协议无法覆盖多数真实业务场景。

### 逐项评估

#### C1 + C2：节点条件渲染 + 权限控制（耦合项）

- **场景：** "管理员看到编辑按钮和统计卡片，普通用户只看到只读列表"。
- **当前能否实现？** 不能。`reactions` 仅存在于表单字段，`visibleWhen` 未定义，
  `$context`（用户角色/权限）命名空间不存在。
- **MVP 阻断性：** **是**。除非系统只需一种角色，否则 MVP 第一页就会撞上这个限制。
- **建议：** 优先启动 `decisions/0003-context-namespace-and-visible-when.md`，
  合并讨论 C1+C2。在 ADR 中明确 `$context` 的数据注入时机和安全边界。

#### C3：表格行级联动 `$row`

- **场景：** 表格中"库存量 < 10 的行的库存列标红"。
- **当前能否实现？** B10 的 `visibleField` 覆盖了行级操作显隐场景，但无法覆盖
  列级动态样式/格式化（如条件标色）。这是纯展示需求，不涉及表达式能力扩展。
- **MVP 阻断性：** 弱阻断。大多数 MVP 可以先接受无行级条件样式。
- **建议：** B10 的 `visibleField` 已作为过渡方案在 v0.2 发布，C3 可排入 v0.3，
  在此之前通过 `tagMap` + 后端下发语义字段（如 `rowStatus: warning`）间接满足
  部分需求。

### MVP 阻断性判定

| 条目 | 阻断 MVP？ | 理由 |
|---|---|---|
| C1 visibleWhen + C2 permissions | **是** | 单角色系统极少，权限控制是第一页就需要的功能 |
| C3 $row 行级联动 | 否 | B10 visibleField + tagMap 可过渡 |

## 五、缺口四：缺失关键组件类型

### 现状

当前 11 个 `type` 覆盖了布局、展示、表格、表单四大类，但缺少中后台 MVP 几乎必然
需要的几个高频组件。

### 缺失组件清单

| 组件 | 优先级 | 典型场景 | 替代方案 |
|---|---|---|---|
| **`datePicker`** | 高 | 订单列表时间范围筛选、报表日期选择 | `input` 手动输入日期字符串（体验极差） |
| **`dateRangePicker`** | 高 | 同上 | 两个 `datePicker` 组合（不优雅但可行） |
| **弹窗内容 Node 化** | 高 | `actions` 中 `type: modal` 的弹窗内容如何用 YAML 描述？ | 当前 `modal` 只有 `modalId` 引用，弹窗内容
依赖前端硬编码——这不 Schema-Driven |
| **`searchForm`** | 中 | 表格上方筛选区，行为与 `form` 不同（不提交而是触发表格刷新） | 用 `form` + `custom` action 模拟（可接受） |
| **`upload`** | 中 | 表单附件上传 | 暂无替代（除非 MVP 不做上传功能） |

### MVP 阻断性判定

| 组件 | 阻断 MVP？ | 理由 |
|---|---|---|
| `datePicker` / `dateRangePicker` | **是** | 几乎任何中后台系统都有时间筛选需求，`input` 手动输入不可接受 |
| 弹窗内容 Node 化 | **是** | 没有弹窗的 Schema-Driven UI 不能称为 Schema-Driven |
| `searchForm` | 否 | `form` + `custom` action 可替代 |
| `upload` | 否 | 取决于 MVP 是否包含上传场景 |
