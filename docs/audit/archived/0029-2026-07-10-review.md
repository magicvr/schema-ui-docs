# 审计报告 0029：表达式交叉引用 + reactions 术语边界 + 作用域/校验缺口 + method 缺省语义

## 背景

本轮审计基于 0028 已归档后的再次全量文档复审。0028 中发现的 `DataRef` 互斥、`data.params` 作用域矩阵、ADR-0004 字段残留、`tagMap` DSL、Action Schema 类型隔离与 `04` 元数据漂移问题在当前工作区已确认修复，本轮不重复报告 V49-V54。

本轮重点交叉核对：表达式规范内部章节引用是否与当前 §9/§10 结构同步，总纲/核心协议对 `reactions` 的术语边界是否仍与表格行级能力冲突，ADR-0003 与 v0.2.5 `$context.user` 最小字段集是否一致，表格列 `$deps.*` 在正文/组件文档/L3a 之间是否对齐，以及 `optionsSource.params`、`DataRef.method` / upload `method` 缺省语义是否进入机器契约或权威正文。

## 发现的问题

### V55（🔴）`02-reaction-expression.md` §2 禁止事项交叉引用漂移到错误的 §9.x

**位置：** `docs/02-reaction-expression.md` §2「禁止事项」

**当前值：**

```text
- permissions.* …（见 §9.3）
- 非表单节点的 visibleWhen …（见 §9.2）
- 表格 actions 的 scope:row 中使用 $self …（见 §9.4）
```

当前 §9 实际结构为：

| 章节 | 实际内容 |
|---|---|
| §9.1 | 作用域隔离规则 |
| §9.2 | `$self` 的作用域语义 |
| §9.3 | `fulfill`/`otherwise` 状态键限制 |
| §9.4 | **不存在** |

对应静态校验规则实际在：

- `permissions.*` 禁 `$deps` → **§10.2**
- 非表单 `visibleWhen` 禁 `$deps` → **§10.1**
- 表格 `actions` 禁 `$self` → **§10.3**（语义说明在 §9.2）

**问题：** 读者点到 §9.2/§9.3 会落到错误规则；§9.4 更是断锚。这是协议主文档内的破坏性引用漂移。

**影响：** 🔴 严重。实现方与 AI 生成配置时会按错误章节理解禁止规则，导致静态校验边界被误读。

**建议修复：** 将三处引用分别改为 §10.2 / §10.1 / §10.3，并删除不存在的 §9.4。

### V56（🔴）术语/协议边界矛盾：`reactions`「仅表单」vs 表格列/行内 `reactions`

**位置：**
- `docs/00-overview.md` §3 术语表
- `docs/01-node-protocol.md` §3 / §3.5
- 对照：`docs/03-component-registry.md` `table` ColumnDef/RowAction、`docs/decisions/0004-row-level-scope.md`

**当前值：**
- 总纲：`reactions`「**仅用于表单场景内**，字段之间的显隐/必填等状态联动」
- `01`：`reactions`「**仅字段类 Node 可用**」「**仅表单场景内的字段类 Node 使用**」
- `03` / ADR-0004：表格 `columns[]` / `actions[]` 可声明 `reactions`，且需 `scope: row`

**问题：** 权威术语与核心协议把 `reactions` 收成「表单字段专属」，但组件契约与 ADR 已把它扩展为表格行级联动挂载点。以 `01`「冲突以本文档为准」为准，表格行级 `reactions` 会被读成非法；以 `03`/ADR 为准，总纲/`01` 术语过时。这是实质性协议边界冲突。

**影响：** 🔴 严重。配置生成与 Renderer 对「表格列/行内能否写 reactions」会产生相反理解。

**建议修复：** 在 `00`/`01` 将 `reactions` 定义为「声明式联动规则数组」，并写明挂载点：表单字段（默认 `scope: form`）+ 表格列/行内操作（`scope: row`，且 `fulfill` 受限）。`01` §3.5 将「字段类」改为「支持 reactions 的组件位置（见组件注册表）」。

### V57（🟡）ADR-0003 与正文不同步：`$context.user` 最小字段集

**位置：** `docs/decisions/0003-context-namespace-and-visible-when.md` D1  
对照：`docs/02-reaction-expression.md` §11.1、`docs/CHANGELOG.md` v0.2.5 C3

**当前值：**
- ADR-0003 D1：`$context.user`「字段集合由具体项目扩展，但协议只规定“这是一个只读对象，**不规定其内部字段**”」
- `02` §11.1：协议级必须字段 `$context.user.id` / `name` / `roles`
- CHANGELOG v0.2.5 C3：明确最小字段集

**问题：** ADR 状态为 accepted / 已合并，但 D1 正文仍停留在「不规定内部字段」，与 v0.2.5 后的强制最小字段集冲突。接入方若以 ADR 为准，会误以为 `roles` 等可不提供。

**影响：** 🟡 中等。不会直接破坏 Schema，但会误导基于 ADR 的实现与权限表达式编写。

**建议修复：** 更新 ADR-0003 D1：协议规定最小字段 `id`/`name`/`roles`；项目可扩展但不得覆盖最小集名称；并注明 v0.2.5 后生效。

### V58（🟡）表格列 `scope: form` + `$deps.*`：表达式矩阵 / 组件文档 / L3a 三方不一致

**位置：**
- `docs/02-reaction-expression.md` 附录 A
- `docs/03-component-registry.md` table「作用域说明」
- `scripts/validate-l3a-expressions.js` 表格列扫描

**当前值：**
- 附录 A：`表格列 scope: form` → `$deps.*` **✅**（无“必须在 form 内”前提）
- `03`：仅当表格位于 `form.children` 内时 `$deps.*` 才合法；独立表格即使 `scope: form` 也应静态拒绝
- L3a：表格列固定 `hasFormContext: false`，但对 `tableColumn` **不会**因“非表单”拒绝 `$deps`；只要写在 `dependencies` 里就会过

**问题：** 合法/非法边界在文档与工具间漂移，独立表格上的 `$deps` 可能通过 L3a。

**影响：** 🟡 中等。独立表格误用 `$deps.*` 会静默通过静态校验，运行时却读不到表单字段。

**建议修复：** 选定 `03` 的 form 内限制为权威：附录 A 加「仅表格位于 `form.children`」；L3a 在遍历时继承 `inFormCtx`，对 `tableColumn`/`tableAction` 的 `scope: form` + `$deps` 在 `!inFormCtx` 时拒绝。

### V59（🟡）`select.optionsSource.params` 未纳入 L3a，与文档声明冲突

**位置：**
- `docs/02-reaction-expression.md` §1 / §2 / §10.6
- `docs/03-component-registry.md` `select`
- `docs/04-datasource-contract.md` §3.1 / §9
- `docs/06-validation.md` L3a 描述
- `scripts/validate-l3a-expressions.js`（仅 `scanDataParams(node.data.params)`）

**当前值：** 文档称 `optionsSource.params` 可 `$deps.*` 值替换且规则对齐 `data.params`；`06` L3a 只写了 `data.params`；脚本全仓无 `optionsSource` 扫描。

**问题：** 写在 `optionsSource.params` 上的非法 `$row.*` 或非表单 `$deps.*` 会被 L3a 静默放过。

**影响：** 🟡 中等。与 V50「把 data.params 纳入 L3a」同类缺口，挂载点换成了 `select.props.optionsSource.params`。

**建议修复：** L3a 在节点上扫描 `props.optionsSource.params`，复用 `scanDataParams` 规则；`06` L3a 行补充该检查点；必要时在 §10.6 或附录 A 显式覆盖 `optionsSource.params`。

### V60（🟡）`DataRef.method` 缺省语义未定义

**位置：**
- `docs/schemas/node.schema.json` `DataRef` `oneOf` → `source: api` 仅 `required: ["source","url"]`
- `docs/01-node-protocol.md` §3.3、`docs/04-datasource-contract.md`
- 示例：`docs/05-scenarios/grid-dashboard.md`、`docs/08-renderer-spec.md` §2.1

**当前值：** Schema 中 `method` 可选；正文列出枚举但未写可省略及默认值；示例多处省略 `method`。

**问题：** 机器契约允许无 `method`，人类文档与示例默认「像是 GET」，但无权威默认。不同 Renderer 可能分别当 GET、报错或 POST。

**影响：** 🟡 中等。属契约歧义，会导致跨实现请求方法不一致。

**建议修复：** 在 `01`/`04` 明确：`source: api` 时 `method` 缺省为 `GET`；Schema description 同步标注 default；保留示例可省略写法。

### V61（🟡）`upload` action：`method`「默认 POST」仅在文档，Schema 无 default

**位置：** `docs/07-actions-contract.md` §7 vs `docs/schemas/action.schema.json` `UploadAction`

**当前值：** 文档写 `method: POST | PUT  # 默认 POST`；Schema 中 `method` 可选 enum，无 `default`，`required` 仅 `["type","url"]`。

**问题：** 与 V60 同类：文档承诺默认 POST，机器契约不表达。

**影响：** 🟡 中等（场景较窄于 DataRef）。实现可能对缺省 method 产生分叉。

**建议修复：** Schema 增加 `"default": "POST"`，并在 description 中写明缺省按 POST。

- V55 与 V26/V44 同属表达式章节引用修复，但本轮对象是 `02` 内部 §2→§9/§10 自引用漂移，不是 `01`→`02` 的 §13/§14 外链。

- V57 与 V22 同属 ADR-0003 同步，V22 修的是 `permissions` 禁 `$deps` 措辞，本轮是 `$context.user` 最小字段集。
- V58 与 V28/V36/V50 相关，但本轮是「列表达式 `$deps` + form 上下文」三方仍未对齐，不是默认 scope 或 `data.params`。
- V59 与 V50 同类（参数值替换进 L3a），挂载点为 `optionsSource.params`，此前未覆盖。
- V60 与 0008 T1 相关，T1 只同步 method 枚举，未定义缺省语义。
- V61 与 V53 相关，V53 做了 type 分支隔离，未处理 upload method 默认值。
- V62 是 `08` 内部未加文档前缀的模糊 § 引用，此前交叉引用修复多在 `01`/`02`/`04` 之间。
- V63 与 V42/V54 同属元数据同步，是场景示例 frontmatter 的新触达漂移。

## 建议处理顺序

1. **先修 V55 / V56**：错误章节指针 + `reactions` 术语边界（影响面最大）。
2. **再修 V57 / V58 / V59**：ADR 最小字段、表格列 `$deps` 三方对齐、`optionsSource.params` L3a。
3. **契约补齐 V60 / V61**：`method` 缺省。
4. **收尾 V62 / V63**：`08` § 引用、场景 `last_updated`。

## 防止复发的建议

1. 章节重排后全文搜索「见 §」类交叉引用，优先核对 `02` 内部自引用。
2. 扩展 `reactions` 挂载点时，同步更新 `00` 术语表与 `01` §3.5，而不仅改 `03`。
3. 参数值替换类挂载点（`data.params`、`optionsSource.params`、未来同类字段）统一进 L3a 扫描清单。
4. 可选字段若有协议级默认值，必须同时写入正文与 Schema `default`/`description`。
