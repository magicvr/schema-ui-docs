# 审计报告 0023：校验工具链执行缺陷——reserved 误杀 + responseMapping 误判 + 文档命令不可用

## 背景

前面 22 轮审计全部聚焦于**文档文字层面**的一致性、术语准确性、交叉引用正确性。在全部 22 轮归档后，本轮首次将焦点转向**校验工具链的可执行正确性**——实际运行 `npm run validate`，用文档自带的三个场景示例（`docs/05-scenarios/`）作为输入，暴露纯文档审计无法触及的代码级缺陷。

## 审视范围

| 范围 | 说明 |
|---|---|
| L2 校验器（`validate-l2-components.js`） | `getDeclaredFields()` 的 `reserved` 集合是否误杀业务字段；`validateResponseMapping()` 的服务端分页检查是否覆盖合法默认用法 |
| `06-validation.md §5` | 本地校验指南中的命令是否可实际执行，参数是否存在，与 CI 工具链是否同步 |
| 场景示例 | `05-scenarios/form-with-reactions.md`、`05-scenarios/data-table.md` 是否可通过 L2 校验 |

## 发现的问题

### V32（🔴）`getDeclaredFields()` 的 `reserved` 集合误杀 6 个表单组件的 `required` 业务字段

**位置：** `scripts/validate-l2-components.js`，`getDeclaredFields()` 函数

**当前值：**

```js
const reserved = new Set(['additionalProperties', 'required', 'allOf', 'anyOf', 'oneOf']);
return Object.keys(propsSpec).filter(k => !reserved.has(k));
```

**问题分析：**

`reserved` 集合的设计初衷是过滤 `component-registry.json` 中表示"字段必填"的结构性 `required`（如 `{type:"object", required:["field"]}`），防止它被当作普通字段名进入"已声明字段"列表，从而避免 `additionalProperties: false` 将其判为非法。

但 `component-registry.json` 中 `input` / `inputNumber` / `select` / `datePicker` / `dateRangePicker` / `upload` 六个表单组件的 `props` 里**同时也定义了一个同名的 `required` 业务字段**：

```json
"required": { "type": "boolean", "required": false }
```

这个字段表达的是"该表单输入项是否必填"的业务语义，与 DSL 结构关键字 `required` 仅键名相同、语义完全不同。由于 `getDeclaredFields()` 使用纯字符串匹配过滤，这个业务字段被错误地从"已声明字段"列表中移除。当用户在配置中使用 `required: true` 时，`additionalProperties: false` 检查就会报：

```
body.children[0].props.required：不允许的额外字段 "required"
（组件 "input" 的 props 不接受额外属性）
```

**影响范围：** 6 个表单组件（`input` / `inputNumber` / `select` / `datePicker` / `dateRangePicker` / `upload`），以及所有在 `props` 中使用 `required: true` 的配置实例。如果表单代码生成器默认产出 `required: false`（表示非必填），则所有代码生成产出的表单节点都会触发此误报。

**根因：** `reserved` 集合用键名做纯字符串匹配，没有考虑键名在 DSL 中存在二义性的可能性。`additionalProperties` / `allOf` / `anyOf` / `oneOf` 四个键名在任何组件中都不可能是业务字段，但 `required` 不是——它在表单组件中具有业务含义。

**建议修复：** 从 `reserved` 集合中移除 `required`。原因：
- `validateProps()` 中的必填检查通过 `spec.required === true` 处理，不依赖 `getDeclaredFields` 过滤；
- `validateNestedObject()` 中的 `required` 数组处理通过 `Array.isArray(spec.required)` 判断，同样不依赖此过滤；
- 移除后 `required` 作为业务字段名可正常进入 `declaredFields`，不再被 `additionalProperties: false` 误杀。

### V33（🟡）`validateResponseMapping()` 服务端分页检查未区分"未声明 responseMapping"与"声明了但缺 total"

**位置：** `scripts/validate-l2-components.js`，`validateResponseMapping()` 函数

**当前值：**

```js
if (type === 'table' && props.pagination && props.pagination.mode === 'server') {
    if (!rm || !rm.total) {
      violations.push({
        path: `${nodePath}.data.responseMapping.total`,
        message: 'table 使用 pagination.mode=server 时，必须在 responseMapping 中声明 total',
      });
    }
}
```

**问题分析：**

`04-datasource-contract.md §4.1.1` 明确写道：

> 未声明 `responseMapping` 时，Renderer 按协议默认字段名解析……服务端分页总数读取 `total`

这意味着当 API 返回 `{list: [...], total: 100}` 且使用默认字段名时，开发者可以**完全不写 `responseMapping`**。本仓库 `05-scenarios/data-table.md` 的官方示例就是这么写的：

```yaml
data:
  source: api
  url: /api/orders
  params:
    page: "$props.pagination.page"
    pageSize: "$props.pagination.pageSize"
# 没有 responseMapping——走默认 list/total
```

但当前校验代码对 `rm` 做 `!rm` 真值判断，当 `responseMapping` 完全不存在时也会触发报错——这与协议文档允许的默认用法直接矛盾。

相比之下，同一函数中对 `rm.list` 的检查已正确处理了此场景：

```js
if (type === 'table' && rm !== undefined && !rm.list) { // rm !== undefined 才是正确写法
```

**建议修复：** 将条件中的 `!rm || !rm.total` 改为 `rm !== undefined && !rm.total`，与上方的 `list` 检查保持一致的判断模式。

### V34（🟢）`06-validation.md §5` 本地校验指南命令不可用 / 参数不存在

**位置：** `docs/06-validation.md` §5.1 和 §5.4

**问题 1（§5.1）：** 文档声称：

> `ajv-cli` 会自动加载 `$ref` 引用的子 schema（如 `node.schema.json`、`reaction.schema.json`），无需额外配置。

实测执行 `npx ajv validate -s docs/schemas/page.schema.json -d my-page.yaml` 报错 `can't resolve reference node.schema.json from ...`。`ajv-cli` 并无自动解析 `$ref` 的能力，必须通过 `-r` 参数显式传入被引用的 schema 文件。仓库自带的 `scripts/validate-all.js` 中 `runAjv()` 函数已正确处理了 `-r`：

```js
const output = execFileSync(
  process.execPath,
  [ajvCliIndex, 'validate', '-s', pageSchemaPath,
   '--allow-union-types', '--strict=false',
   '-r', path.resolve(..., 'node.schema.json'),
   '-r', path.resolve(..., 'action.schema.json'),
   '-r', path.resolve(..., 'reaction.schema.json'),
   '-d', ...patterns],
  ...
);
```

文档命令未同步此正确用法。

**问题 2（§5.4）：** CI 示例中包含：

```yaml
npx ajv validate -s docs/schemas/page.schema.json -d "pages/**/*.yaml" --strict-refs=true
```

`--strict-refs` 参数在 `ajv-cli@5.0.0` 中不存在，实测报 `error: parameter --strict-refs is unknown`。该参数可能是 `ajv` 库本身的选项，但 `ajv-cli` 命令行并未暴露此参数。照抄该示例会直接导致 CI 挂掉。

**建议修复：**
1. §5.1：删除"无需额外配置"声明，改为显式 `-r` 参数用法说明，并给出完整命令示例。
2. §5.4：删除 `--strict-refs=true`，替换为正确的 `-r` 参数集。

## 教训

1. **纯文档审计有盲区。** 22 轮审计全部在文档文字层面做交叉核对，无法发现校验脚本中的执行逻辑缺陷。实际执行工具链（哪怕是手动跑一遍 `npm run validate`）能暴露完全不同类型的 bug。
2. **保留关键字集合需要上下文感知。** 使用纯字符串集合做过滤时，必须确认被过滤的键名在所有上下文中确实不可能作为业务字段出现。`required` 在 JSON Schema / DSL 语境中是结构关键字，但在表单组件的 props 中同时具有业务含义，二者不能混为一谈。
3. **脚本与文档应绑定同步。** `validate-all.js` 中 `runAjv()` 使用 `-r` 参数的做法是正确的，但 `06-validation.md` 中教后端同学手动执行的命令没有同步这个正确做法。建议在 CI/文档生成流程中加入"脚本中的命令用法与文档中的命令示例自动对账"的检查。
