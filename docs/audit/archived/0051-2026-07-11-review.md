# 审计报告 0051：L3a 异常分类与 `contains` 操作数约束

## 背景

本轮基于 0050 已归档后的再次全量复审。0050 中的 `page.schema.json` DatasourceDeclaration.params 描述残留（V202）与 Renderer 嵌套 `$deps` / `$context` 路径示例（V203）已确认落地；当前协议正文、Schema、分层校验脚本、MCP 实现与测试仍保持该轮修复结果。

本轮发现两处此前未覆盖的 L3a 边界：

1. 合法页面对象内部的 `table.props.columns` / `actions` 类型错误会触发 L3a JavaScript 异常，并被 MCP 提升为 `parseError`。
2. 协议和 ADR 均要求 `contains` 右操作数为字面量，但 L3a 仍接受变量等动态操作数。

扫描活跃区与归档区后，历史最大审计编号为 `0050`，本轮使用 `0051`。扫描所有 `*-review.md`、`*-checklist.md` 与已有 `*-plan.md` 后，全局最大问题编号为 `V203`，因此本轮从 `V204` 连续编号到 `V205`。

## 发现的问题

### V204（🟡）L3a 将合法 JSON 的表格字段类型错误误归类为 `parseError`

**位置：**

- `scripts/validate-l3a-expressions.js` → `walkNode()` 的表格列/操作表达式扫描
- `mcp/src/core/validation-runner.ts` → L2/L3a/L4 子脚本 `parseErrors` 合并
- 对照 `docs/mcp/v1-design.md` §3 返回结构与错误分类

**当前值：**

```javascript
if (node.type === 'table' && node.props) {
  const { columns = [], actions = [] } = node.props;
  columns.forEach((col, ci) => {
    // ...
  });
  actions.forEach((action, ai) => {
    // ...
  });
}
```

解构默认值只会替换 `undefined`，不会替换显式的对象、字符串、数字或 `null`。当语法合法的 JSON 页面写入 `table.props.columns: {}` 或 `table.props.actions: {}` 时，L0/L1 或 L2 能正确识别字段类型错误，但 L3a 随后执行 `.forEach()` 抛出 `TypeError`。子脚本将该异常放入 `parseErrors`，MCP 再提升为顶层 `parseError`，摘要也变成“解析失败”。

**问题：** 输入本身是合法 JSON/YAML，错误发生在页面对象内部字段类型。`docs/mcp/v1-design.md` 已明确：页面内部类型错误属于结构违规，L2-L4 应跳过不适用检查，不得把脚本类型异常提升为 `parseError`。当前行为破坏了错误分类契约，并向调用方暴露内部 JavaScript 异常文本。

**影响：** 🟡 中等。MCP/AI 客户端会把可定位、可自动修复的结构错误误判为语法解析失败，导致摘要、建议文档与自动修复策略偏离真实问题。

**建议修复：**

1. 在 L3a 表格扫描前分别用 `Array.isArray()` 守卫 `columns` 与 `actions`；类型不合法时跳过该数组的表达式扫描，由 L0/L1/L2 承担结构违规。
2. 不改变合法数组中的列、操作、`visibleWhen`、`reactions` 与 `permissions` 扫描行为。
3. 增加 MCP 回归测试，至少覆盖 `columns` / `actions` 为 object，并建议补 string / `null`；断言保留结构违规且 `parseError === null`、`internalError === null`。
4. 在 `docs/06-validation.md` 或 `docs/mcp/v1-design.md` 的实现边界中补充/确认 L3a 对结构不适用字段跳过扫描的规则。
5. 在 `docs/CHANGELOG.md` Unreleased 记录修复及测试范围。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `mcp/tests/validate-content.test.ts`
- `docs/06-validation.md` 或 `docs/mcp/v1-design.md`
- `docs/CHANGELOG.md`

**与已归档审计的关系：** 0043/V156 修复的是 L2 对 `node.props` 非普通对象缺少守卫；本项发生在 L3a 对 `table.props.columns/actions` 缺少数组守卫。根因类别相邻，但触发层、字段和修复位置不同，是 V156 未覆盖的后续边界，不是重复报告。

### V205（🟡）L3a 未执行 `contains` 右操作数必须为字面量的协议约束

**位置：**

- `docs/02-reaction-expression.md` §3 运算符白名单
- `docs/decisions/0003-context-namespace-and-visible-when.md` D5a
- `scripts/validate-l3a-expressions.js` → `checkTokenSyntax()`

**当前值：**

协议正文和 ADR 均定义：`contains` 左操作数为数组，右操作数为字面量。L3a 当前仅将 `contains` 纳入普通二元比较运算符状态机；操作符后接受所有通用操作数，包括 `VAR`。因此以下表达式能通过完整 L0-L4：

```yaml
when: "$deps.roles contains $deps.targetRole"
dependencies: [roles, targetRole]
```

带括号的动态右操作数同样没有专门禁令：

```yaml
when: "$deps.roles contains ($deps.targetRole)"
dependencies: [roles, targetRole]
```

**问题：** 协议没有定义动态右操作数语义，但静态校验批准了该表达式。不同 Renderer 可能接受、拒绝，或采用不同求值模型，导致表达式语言的互操作性与可审计性下降。

**影响：** 🟡 中等。CI 与 MCP 会放行正文明确禁止的表达式，错误只能在 Renderer 运行时暴露，并可能出现跨实现差异。

**建议修复：**

1. 在 L3a token 语法检查中为 `contains` 增加专门约束：右操作数必须是字符串、数字、布尔或 `null` 字面量。
2. 拒绝变量、分组表达式及其他动态右操作数；错误路径继续指向原表达式字段，并返回稳定、可理解的 L3a 违规代码/消息。
3. 保持合法字面量正例：`'admin'`、`1`、`true`、`null`。
4. 增加脚本或 MCP 正反例测试，覆盖直接变量与括号变量两种绕过方式。
5. 同步 `docs/06-validation.md` 的 L3a 语法检查清单，并在 `docs/CHANGELOG.md` Unreleased 记录修复。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `mcp/tests/validate-content.test.ts`
- `docs/06-validation.md`
- `docs/CHANGELOG.md`

**与已归档审计的关系：** 0016/V16 修复的是 Renderer 示例把 `contains` 错写成函数模型；0039/V121 修复的是多个比较运算符链式连续使用。本项针对单个 `contains` 的右操作数类型约束，此前未报告或修复。

## 建议处理顺序

1. **V204**：先增加 L3a 数组守卫，稳定错误分类并补 MCP 回归。
2. **V205**：在现有 token 状态机中执行字面量约束，补合法/非法表达式测试。
3. 同步 `06-validation.md`、CHANGELOG 与审计清单；执行 L3a 针对性样例、MCP build/test/smoke:tools；用户确认后再归档。

## 防止复发

- 分层脚本遍历 L0/L1 已判定为非法类型的字段前，统一使用类型守卫；语义层不得依赖解构默认值处理显式错误类型。
- 表达式运算符新增语义约束时，测试矩阵除“是否识别运算符”外，还必须覆盖左右操作数类型、分组绕过与链式组合。
- MCP 回归应同时断言违规层和顶层错误分类，避免子脚本异常被错误映射为 `parseError`。
