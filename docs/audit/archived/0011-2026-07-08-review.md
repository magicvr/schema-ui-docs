## 发现的问题

### W1（🟡）`06-validation.md` 将 L3 表达式校验定位为运行时，与当前静态拒绝规则不一致

**位置：** `06-validation.md` §1 校验层级、§2 CI 建议流程

**对应规范：** `02-reaction-expression.md` §10、`08-renderer-spec.md` §5.4

**问题：** `06-validation.md` 的校验层级表将 L3 联动表达式校验的时机写为“前端 Renderer 运行时”，流程图中也将 L3 放在运行时阶段：

```text
运行时（前端）
  → L3 联动表达式沙箱解析（防止表达式注入）
```

但当前表达式规范已经多处声明相关规则应在静态校验阶段拒绝，例如：

- `$deps.*` 出现在非表单 `visibleWhen` 中时静态校验拒绝。
- `permissions.*` 中出现 `$deps.*` 时静态校验拒绝。
- `scope: row` 下 `$self` 出现在 `actions` 中时静态校验拒绝。
- `scope: row` 下 `fulfill` / `otherwise` 出现 `required` / `value` 时静态校验拒绝。

`08-renderer-spec.md` §5.4 也明确写为 Renderer 加载页面配置时执行静态校验，并在无法通过时直接拒绝渲染。

**影响：** 🟡 中等。接入方可能误以为 L3 只是运行时兜底或表达式求值阶段的安全处理，从而没有在配置加载前阻断非法协议，导致错误配置进入运行时后才暴露。

**建议：** 将 `06-validation.md` 中 L3 拆分或改写为：

- L3a：表达式静态校验，时机为 Renderer 加载配置时；CI 可选前置执行。
- L3b：表达式运行时求值，时机为交互/数据变化时，仅在已通过静态校验的表达式上执行。

至少应将当前 L3 的时机从“前端 Renderer 运行时”改为“Renderer 加载配置时 / CI 可选前置”，并调整流程图，避免与 `02` / `08` 冲突。

### W2（🟡）已接受 ADR-0004 仍保留跨作用域变通方案，与当前正文边界不一致

**位置：** `decisions/0004-row-level-scope.md` D3、后果章节

**对应规范：** `02-reaction-expression.md` §9.1

**问题：** 当前正文已经收紧为：v0.2 不提供同时依赖表单级字段和行内字段的跨作用域联合判断协议级语法，推荐后端预计算为行数据字段，或通过 ADR 新增标准机制。

但 ADR-0004 仍保留如下变通描述：

```text
由业务方在表格组件之外自行判断 ... 再通过组件的静态 props 传入表格组件 ...
```

该 ADR 状态为 accepted，并声明已合并入 `01-node-protocol.md` / `02-reaction-expression.md`。读者回溯设计决策时，容易把这段“props 传入”理解为当前协议认可的可落地路径；而当前正式正文并未定义该机制的挂载位置、字段名、读取方式或与行级表达式的交互规则。

**影响：** 🟡 中等。不会破坏现有 Schema，但会误导实现方把组件私有扩展当成 v0.2 标准能力，削弱正文已收紧的作用域边界。

**建议：** 同步收紧 ADR-0004：

- D3 中保留“跨作用域不支持”的决策结论。
- 删除或改写“props 传入”作为实际变通方案的表述。
- 如需保留历史讨论，应明确该方案未标准化、不属于 v0.2 协议能力。
- 与 `02-reaction-expression.md` §9.1 保持一致：当前可落地建议为后端预计算行字段，或另起 ADR。

### W3（🟢）数据表格场景开头仍保留 `visibleField` 旧过渡措辞

**位置：** `05-scenarios/data-table.md` 开头变更提示

**对应规范：** `03-component-registry.md` RowAction、ADR-0004 D6

**问题：** 场景文档开头仍写：

```text
actions[].visibleField ... 演示“数据驱动显隐”——由后端在行数据中下发语义化布尔字段 ... 前端据此判断行内操作是否展示，不引入 $row 表达式。
```

但同一文档后文已经将 `visibleField` 定义为 `scope: row` 的 `visibleWhen` 语法糖，组件注册表和 ADR-0004 也采用该当前语义。开头的“不引入 `$row` 表达式”属于旧过渡方案措辞残留。

**影响：** 🟢 低。示例本身仍可用，但读者可能误以为 `visibleField` 是独立于 `$row` / `visibleWhen` 的旧机制，而不是当前推荐写法的兼容语法糖。

**建议：** 将开头变更提示改为当前语义，例如：

```text
actions[].visibleField 演示兼容语法糖：解析阶段等价展开为 scope: row 的 visibleWhen；新项目推荐直接使用 $row 表达式写法。
```

### W4（🟢）当前规范与 Schema 描述仍残留过程性审计编号

**位置：** `07-actions-contract.md`、`schemas/component-registry.json` 多处描述

**问题：** 当前规范和机器可读描述中仍有过程性审计编号，例如：

```text
对应审计条目 §2.5，计划条目 B3
```

```text
由 data.valueField 迁移而来（审计 §2.1）
```

```text
远程动态选项（审计 §3.4）
```

审计编号适合保留在 `CHANGELOG.md` 或 `audit/archived/` 中作为历史追溯，不适合作为当前规范正文或 Schema hover 描述的一部分。特别是 `component-registry.json` 的描述可能被 IDE、文档生成器或 AI 配置生成工具直接消费，过程性编号会降低机器可读说明质量。

**影响：** 🟢 低。属于描述层质量问题，不影响结构校验，但会影响当前规范的可读性和自动提示质量。

**建议：** 将当前规范和 Schema 描述改为结果导向措辞：

- `07-actions-contract.md` 顶部只说明本文件定义顶层 `actions` 完整结构。
- `statCard.valueField.description` 只说明 `since 0.2` 后由 `props.valueField` 指定展示取值字段。
- `select.optionsSource.description` 只说明远程动态选项与 `$deps.*` 参数空值省略规则。
- 历史审计编号留在 `CHANGELOG.md` / `audit/archived/`。
