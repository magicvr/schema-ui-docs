# 审计报告 0041：Action 请求语义、表达式与数据校验边界、MCP 返回契约、归档证据链

## 背景

本轮审计基于 0040 已归档后的再次全量复审。0040 中的组件 DSL 嵌套封闭性、MCP 临时目录异常边界、Node 18 运行时兼容和归档链接修复均已确认落地，本轮不重复报告 V125-V129。

本轮完整读取协议正文、场景、ADR、机器可读 Schema/DSL、L0-L4 校验脚本、MCP 文档/实现/测试/打包配置及全部审计索引，并执行最小正反例、真实 MCP transport 调用、构建测试、Docker smoke 和归档证据链扫描。新问题集中在四类边界：

1. Action 与数据引用在机器校验中可通过，但缺少唯一可执行语义；
2. L3a/L4 对嵌套路径、数组和开放业务 map 的遍历不完整或不够结构化；
3. MCP 的全局返回预算和解析错误分类没有覆盖 `validate_content` 的完整输入空间；
4. 归档索引与历史章节锚点仍有小范围证据链缺口。

扫描活跃区与归档区全部 `review` / `checklist` / `plan` 后，历史最大问题编号为 `V129`，因此本轮从 `V130` 连续编号。当前活跃区为空，历史最大审计编号为 `0040`，本轮使用 `0041`。

## 发现的问题

### V130（🔴）普通表单 `GET` request 没有可执行的字段传输语义

**位置：** `docs/07-actions-contract.md` §3；`docs/schemas/action.schema.json` 的 `RequestAction`；`docs/08-renderer-spec.md` Action 执行策略；L2 Action 引用校验。

**当前值：** 普通表单可通过 `submitAction` 引用 `type: request` Action。该 Action 允许 `method: GET`，同时正文和 Schema 只定义 `bodyMapping`：缺省时表单字段按原字段名组成 JSON 请求体，存在映射时组成映射后的请求体。

**问题：** Fetch 标准不允许 GET 请求携带 body，而普通表单 Action 没有 `queryMapping` 或等价的 query 传输结构。最小页面使用 `form.props.submitAction` 引用 `method: GET` 且声明 `bodyMapping` 时可通过 L0-L4，但 Renderer 无法同时遵守 HTTP 执行约束和 Action 字段传输契约。

**影响：** 🔴 严重。协议和机器契约共同接受不可按正文实现的配置；不同 Renderer 只能选择丢弃字段、私自转 query、修改 method 或运行时报错，形成破坏互操作性的分叉。

**建议修复：** 在 v0.2 保守选择唯一行为：普通表单 `submitAction` 引用 `type: request` 时拒绝 `method: GET`，继续使用 POST/PUT/PATCH/DELETE 的 JSON 请求体；若未来确有 GET 表单场景，再通过 ADR 定义 query 映射。同步 Action 正文、Renderer 执行策略、L2 语义校验、测试和 Unreleased 说明。

**与已归档审计的关系：** V85 修复 `submitAction` 引用存在性，V103 收紧 `bodyMapping` 值类型，均未检查引用后的 HTTP method 与字段传输方式是否可执行。行级请求已有独立 `requestMapping.query`，但该结构不属于普通表单 Action，不能作为隐式补救。

**需要同步的文件：**

- `docs/07-actions-contract.md`
- `docs/08-renderer-spec.md`
- `docs/06-validation.md`
- `scripts/validate-l2-components.js`
- `mcp/tests/validate-content.test.ts`

### V131（🟡）非表单 `visibleWhen` 可绕过 `$self` 变量禁令

**位置：** `docs/02-reaction-expression.md` §9.1/§10.1；`scripts/validate-l3a-expressions.js` 的 `validateExpression()`。

**当前值：** 规范明确非表单节点 `visibleWhen` 只能访问 `$context.*`，不得访问 `$deps.*` 或没有绑定对象的 `$self`。L3a 的非表单 `visibleWhen` 分支只检测 `hasDepRef`。

**问题：** 普通 `section.visibleWhen.when: "$self == true"` 经 L3a 返回零违规。该表达式没有可注入的当前字段或单元格值，运行时语义未定义。

**影响：** 🟡 中等。无绑定变量穿过官方静态校验后，不同 Renderer 可能注入 `undefined`、抛错或沿用错误上下文。

**建议修复：** 将非表单 `visibleWhen` 检查收敛为变量白名单：只允许 `$context.user.*` / `$context.features.*`；至少显式拒绝 `$self`、`$deps.*`、`$row.*` 和 `$parentRow.*`。增加表单内/外正反例并同步 L3a 文档。

**与已归档审计的关系：** V11/V105 修正文作用域措辞，V117 修受控 `$self.start/end`，均未覆盖普通 Node `visibleWhen` 的精确 `$self` 漏检。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`

### V132（🟡）L3a 将合法的 `$row` 嵌套依赖路径截断为顶层字段

**位置：** `docs/schemas/reaction.schema.json` 的 `dependencies`；`docs/02-reaction-expression.md` 行级依赖规则；`scripts/validate-l3a-expressions.js` 的 `rowDepFields` 提取。

**当前值：** 机器描述规定 `scope: row` 的 `dependencies` 列出 `$row.*` 使用的行内字段路径。L3a 却对 `$row.customer.status` 执行首段截断，得到 `customer` 后再与 dependencies 比较。

**问题：** `dependencies: ["customer.status"]` 配合 `when: "$row.customer.status == 'active'"` 被错误报告为未声明 `$row.customer`。该配置满足点路径语法并能由行对象直接求值。

**影响：** 🟡 中等。合法嵌套行数据无法通过 CI；实现方可能被迫把依赖声明降级为顶层字段，与机器描述的“字段路径”含义冲突。

**建议修复：** 行级依赖按 `$row.` 后的完整点路径比较，并保留精确匹配。补充嵌套路径正例、仅声明父路径反例、多个路径和普通顶层字段回归测试；同步 L3a 文档说明。

**与已归档审计的关系：** V97 修复 `$row` 使用位置与缺少依赖声明，未覆盖多段点路径的提取精度。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `docs/02-reaction-expression.md`
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`

### V133（🟡）params 数组元素绕过 L3a 变量限制

**位置：** `docs/02-reaction-expression.md` §10.6；`docs/06-validation.md` L3a 规则；`scripts/validate-l3a-expressions.js` 的 `scanDataParams()`。

**当前值：** `data.params`、`optionsSource.params` 和顶层 `datasources.*.params` 只允许字面量或合法表单上下文中的 `$deps.*` 值替换。扫描器递归普通对象，但遇到数组时既不递归也不检查字符串元素。

**问题：** 表单内 `data.params.owners: ["$context.user.id"]` 及类似数组嵌套值可通过 L0-L4；数组中的 `$row.*`、`$self` 等同样绕过变量限制。

**影响：** 🟡 中等。协议禁止的变量可藏入常见的多值 query/body 参数结构，导致静态约束和 Renderer 行为漂移。

**建议修复：** `scanDataParams()` 对数组逐项递归，路径使用稳定的 `[index]` 表示；合法字面量数组保持可用。补充顶层数组、对象内数组、嵌套数组、合法 `$deps.*` 和非法变量回归测试。

**与已归档审计的关系：** V50/V59/V65 补齐不同 params 挂载点的扫描，均未覆盖值为数组时的递归边界。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`

### V134（🟡）L4 将开放业务 map 的键误判为 CSS 属性

**位置：** `scripts/lint-l4-banned-props.js` 的 `checkObjectForBannedKeys()`；`docs/03-component-registry.md` 的 `optionsSource.params` 与 RowAction `requestMapping`；`docs/06-validation.md` L4 边界。

**当前值：** L4 从 `node.props` 无差别深度递归，仅对 `tagMap` 的动态映射键做特判。`optionsSource.params`、`requestMapping.path/query/body` 等开放业务 map 仍按键名匹配 CSS 禁用词。

**问题：** 合法 `optionsSource.params: { color: red }` 被报告为禁用 CSS 属性。相同误报可影响名为 `width`、`height`、`style` 等的后端参数或路径占位符。

**影响：** 🟡 中等。合法业务 API 契约会被 CI 拒绝；业务字段命名被不必要地绑定到前端样式禁用词集合。

**建议修复：** 将 L4 改为结构感知遍历：在已知开放业务 map 层跳过键名判定，同时仍递归其对象/数组值中真正属于协议结构的部分。建立集中式开放 map 路径规则，并为 `optionsSource.params`、DataRef params 与 RowAction requestMapping 增加正反例。

**与已归档审计的关系：** V119 只修复 `tagMap` 业务键。当前问题位于请求参数和映射对象，是同类算法在其他已定义开放 map 上的残留。

**需要同步的文件：**

- `scripts/lint-l4-banned-props.js`
- `docs/06-validation.md`
- 相关 L4 或 MCP 回归测试

### V135（🟡）`source: ref` 引用静态 datasource 时仍允许 API 专属 `responseMapping`

**位置：** `docs/01-node-protocol.md` §3.3；`docs/schemas/node.schema.json` 的 DataRef `source: ref` 分支；`scripts/validate-l2-components.js` 的 `getEffectiveResponseMapping()` / `validateResponseMapping()`。

**当前值：** 正文限定本地 `responseMapping` 用于 `source: api`，或 `source: ref` 指向 API datasource 时覆盖其映射。Schema 的 ref 分支无法检查目标类型，L2 解析生效映射时也不检查 `datasources[ref].source`。

**问题：** `data.source: ref` 指向 `{ source: static, value: [] }` 时仍可声明 `responseMapping.list`，完整 L0-L4 全部通过。Renderer 对静态值是否应用响应路径没有唯一答案。

**影响：** 🟡 中等。静态数组可能被错误路径解析为空，也可能被不同实现直接忽略映射，形成运行时差异。

**建议修复：** 在可解析跨引用关系的 L2 中拒绝静态目标上的本地 `responseMapping`，并防御静态 datasource 自身不可能继承映射的情况。Schema 保持结构表达能力，但在描述中明确该条件由 L2 执行；补 MCP 反例。

**与已归档审计的关系：** V71/V88/V91 修复 ref 映射的本地校验、继承和优先级，未执行“引用目标必须是 API”这一前置条件。

**需要同步的文件：**

- `scripts/validate-l2-components.js`
- `docs/01-node-protocol.md`
- `docs/06-validation.md`
- `docs/schemas/node.schema.json`（描述同步）
- `mcp/tests/validate-content.test.ts`

### V136（🟡）通用 i18n 契约承诺 `titleKey`，table DSL 却拒绝

**位置：** `docs/01-node-protocol.md` i18n 约定；`docs/03-component-registry.md` 通用约定与 table props 表；`docs/schemas/component-registry.json` 的 table props。

### V137（🟡）日期组件 `min` / `max` 的 ISO 日期约束只存在于文字说明

**位置：** `docs/03-component-registry.md` 的 `datePicker` / `dateRangePicker` 字段表；`docs/schemas/component-registry.json` 对应 props DSL；`scripts/validate-l2-components.js` 的字符串字段校验。

**当前值：** 文档要求 `min` / `max` 使用 ISO 8601 `YYYY-MM-DD`。机器 DSL 只声明 `type: string`，L2 因此接受任意字符串。

**问题：** `min: definitely-not-a-date`、`max: 31/12/2026` 可通过完整校验链。不同 Renderer/日期库可能忽略、宽松解析或报错。

**影响：** 🟡 中等。公开格式约束不可执行，日期边界在运行时表现不稳定。

**建议修复：** 扩展组件 DSL/L2 对字符串 `pattern` 的受控支持，或增加日期字段专用语义校验；至少拒绝不符合 `YYYY-MM-DD` 的字符串，并决定是否同时检查真实日历日期。补闰日、非法月份/日期和合法边界测试，更新 DSL 支持关键字说明。

**与已归档审计的关系：** V18 规范化了 DSL 支持关键字集合，V99/V125 分别补嵌套 required 和封闭性；此前没有为日期格式建立机器约束。

### V138（🟡）`protocol.validate_content` 可突破统一的 20KB 工具返回预算

**位置：** ADR-0007 D10；`mcp/src/tools/validate-content.ts`；`mcp/src/tools/utils.ts`；`mcp/src/types.ts` 的校验结果类型。

**当前值：** ADR 规定单次工具返回的最终文本不超过 20KB。V124 只在 `protocol.get_doc` 的最终响应边界实现预算；`validate_content` 仍将全部违规对象直接格式化为 JSON。

**问题：** 真实 stdio MCP 调用输入 300 个未知组件节点时，输入约 10KB，L2 返回 300 条违规，最终工具文本约 43KB；增加违规数时结果继续线性增长。

**影响：** 🟡 中等。工具会突破已承诺的上下文预算，大型错误页面反而可能淹没 AI 上下文或触发客户端截断。

**建议修复：** 不截断 JSON 字符串，而是在结构层限制每层返回的违规项数量，并返回 `totalViolations`、`omittedViolations` / `truncated` 等统计字段。预算必须以最终 `content[0].text` 的 UTF-8 字节数验证，保留每层至少一条代表性违规和 suggested docs。

**与已归档审计的关系：** V124 明确并修复 `get_doc` 的完整工具文本预算。本条证明同一 ADR 的全局承诺尚未施加到 `validate_content`，不是 V124 已修复路径的复发。

- `mcp/src/core/validation-runner.ts`
- `mcp/src/tools/validate-content.ts` 或通用预算工具
- `mcp/src/types.ts`
- `mcp/tests/validate-content.test.ts`

### V139（🟡）合法 JSON 根标量被误归类为 `parseError`

**位置：** `mcp/src/core/validation-runner.ts` 的 `parseInput()`、L0/L1 调用和子脚本结果合并；ADR-0007 D11。

**当前值：** 预解析会正确接受 JSON `null`、数组和字符串等语法合法值。随后 runner 仍调用假定根值为页面对象的 L2/L3a/L4 脚本，并把子脚本因访问 `doc.body` 等产生的 parseErrors 合并为最终 `parseError`。

**问题：** 输入 `content: "null"`、`format: json` 时，同时得到 L0/L1 `must be object` 和 `parseError: Cannot read properties of null...`。合法 JSON 被标为解析失败，且违反 ADR “parseError 时 layers 为空；结构失败时 parseError 为 null”的分类约定。

**影响：** 🟡 中等。AI 客户端无法区分语法错误与页面根结构错误；内部 TypeError 还通过错误消息暴露为用户输入问题。

**建议修复：** 预解析返回解析后的值；若根值不是普通对象，只执行 L0/L1 并跳过假定页面对象的 L2-L4，保持 `parseError: null`。同时让根脚本直接处理 null/标量时不抛异常作为纵深防线。补 JSON null/数组/字符串、空 YAML 和非法语法回归测试。

**与已归档审计的关系：** V75 修复子脚本异常与协议违规的分类，V126 修临时目录异常；当前问题是语法合法的根标量被子脚本二次误分类，历史未覆盖。

**需要同步的文件：**

- `mcp/src/core/validation-runner.ts`
- `mcp/tests/validate-content.test.ts`
- 视纵深策略修改 `scripts/*.js`

### V140（🟢）归档索引遗漏整个 0011 审计

**问题：** 0011 的历史结论无法从唯一归档索引发现，归档编号序列和索引不完整。
