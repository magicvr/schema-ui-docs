# 审计报告 0008：残余文档一致性与链接审视

## 背景

前七轮审计已经覆盖协议语义、组件契约、Renderer 行为、Schema 结构和版本元数据等主要风险。本轮审计基于对当前文档体系的再次全量浏览与自动化检查，重点查找仍未收敛的残余问题：本地 Markdown 断链、正文示例与 Schema 不一致、机器可读描述文本滞后，以及审计索引自身的元数据漂移。

本轮未发现新的阻断级协议设计缺口；所有 JSON 文件均可正常解析。`responseMapping` 字段映射仍属于 `audit/README.md` 中已登记的活跃跟踪项，本报告不重复列为新增问题。

## 审视范围

| 范围 | 说明 |
|---|---|
| Markdown 本地链接 | 遍历全部 `.md` 文件，检查相对链接目标文件是否存在 |
| JSON 可解析性 | 遍历全部 `.json` 文件，确认可被 `JSON.parse` 解析 |
| 正文与 Schema 一致性 | 抽查 `01`/`03`/`06`/`07`/`08` 与 `schemas/` 的关键字段约束 |
| 审计元数据 | 检查 `audit/README.md` 编号规则与当前归档状态是否一致 |

## 发现的问题

### T1（🟡）`01-node-protocol.md` 中 `DataRef.method` 示例仍停留在 GET/POST

**位置：** `01-node-protocol.md` §3.3 `data` 示例

**问题：** `data` 示例中仍写为：

```yaml
method: GET | POST
```

但当前 `schemas/node.schema.json` 的 `DataRef.method` 已支持 `GET | POST | PUT | DELETE | PATCH`，并且 `07-actions-contract.md` 的 `request.method` 也使用同一枚举。

**影响：** 🟡 中等。Schema 与 Action 契约已经正确，但核心协议正文示例会误导读者，以为 `data.source: api` 只能使用 GET/POST。该问题不影响机器校验，但影响协议阅读和配置生成。

**建议：** 将 `01-node-protocol.md` 示例更新为 `method: GET | POST | PUT | DELETE | PATCH`，并保持描述与 `node.schema.json` 一致。

### T2（🟡）当前规范文档中仍存在本地断链

**位置：** `03-component-registry.md`、`CHANGELOG.md`

**问题：** 自动化链接检查发现当前规范文档中仍有相对路径指向不存在的文件：

| 文件 | 当前链接 | 问题 |
|---|---|---|
| `03-component-registry.md` | `./audit/0001-2026-07-07-plan.md` | 文件已归档到 `audit/archived/` |
| `03-component-registry.md` | `./audit/0001-2026-07-07-plan.md#a3-...` | 同上，且锚点所在文件路径错误 |
| `CHANGELOG.md` | `./docs/audit/0003-2026-07-07-review.md` | 从 `docs/` 出发会解析为 `docs/docs/audit/...` |

**影响：** 🟡 中等。读者从当前规范跳转历史计划/审计记录时会失败；同时这会降低文档站点或静态检查工具的可信度。

**建议：** 将当前规范中的链接修正为实际归档路径，例如 `./audit/archived/...` 或从所在文件正确计算的相对路径。

### T4（🟢）`component-registry.json` 中 `visibleField` 描述文本滞后

**位置：** `schemas/component-registry.json` `table.props.actions.items.properties.visibleField.description`

**问题：** Schema 描述仍写着 `visibleField` 是“取当前行数据中的同名布尔字段作为显隐依据，不引入 $row 表达式”。但当前 `03-component-registry.md` 和 ADR-0004 已将 `visibleField` 定义为 `scope: row` 的 `visibleWhen` 语法糖，解析阶段等价展开为：

```yaml
visibleWhen:
  scope: row
  dependencies: [<field>]
  when: "$row.<field> == true"
```

**影响：** 🟢 低。Schema 结构本身未出错，但机器可读描述会误导使用 Schema 生成提示、悬停说明或 AI 辅助生成配置的工具。

**建议：** 更新 `visibleField.description`，说明它是 `scope: row` 的 `visibleWhen` 语法糖，并注明与显式 `visibleWhen` 同时声明时以显式 `visibleWhen` 为准。

### T5（🟢）Renderer 运行时校验与 CI 校验的处理策略需要补充边界说明

**位置：** `08-renderer-spec.md` §1.2、`06-validation.md` §1

**问题：** `08-renderer-spec.md` 描述运行时组件契约验证时，倾向于“开发环境告警并兜底渲染”：

- 未知 `props` 字段输出开发警告，但不阻断渲染。
- 缺失必填 `props` 字段输出开发警告，并使用默认值或兜底占位。

而 `06-validation.md` 中 L2 组件契约校验定位为后端 CI 阶段的结构性校验，语义上应阻断非法配置合并。两者并非真正冲突，但当前文档缺少一句明确区分，读者可能误解为非法 props 在所有阶段都只是 warning。

**影响：** 🟢 低。实现方大多可以自行理解 CI 与运行时的差异，但协议文档作为协作契约应明确两者职责边界。

**建议：** 在 `08-renderer-spec.md` §1.2 或 `06-validation.md` §1 增补说明：CI/提交前校验应阻断不合法配置；Renderer 运行时校验是兜底防线，主要用于开发环境告警和生产降级。

### T6（🟢）`audit/README.md` 编号规则示例已过期

**位置：** `audit/README.md` “编号规则”表格

**问题：** 编号规则示例仍写“当前活跃区最大 `0004`、归档区最大 `0003`，则下一份为 `0005`”。当前归档区最大编号已经是 `0007`，本轮审计应为 `0008`。

**影响：** 🟢 低。规则本身仍正确，但示例值过期，可能误导后续创建审计文件的维护者。

**建议：** 将示例改成不依赖当前仓库状态的抽象表达，或更新为当前实际状态：归档区最大 `0007`，下一份为 `0008`。
