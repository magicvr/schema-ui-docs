## 🔴 关键问题

### V32（🔴）`validate-l2-components.js` `getDeclaredFields()` 的 `reserved` 集合误杀表单组件 `required` 业务字段

**问题：** `getDeclaredFields()` 中定义了一个 `reserved` 集合用于过滤 DSL 结构关键字：

```js
const reserved = new Set(['additionalProperties', 'required', 'allOf', 'anyOf', 'oneOf']);
```

该集合的初衷是排除 `component-registry.json` 中表示"字段必填"的**结构性** `required`（如 `{type:"object", required:["field"]}`）。但 `input` / `inputNumber` / `select` / `datePicker` / `dateRangePicker` / `upload` 六个表单组件的 `props` 中**同时也定义了一个叫 `required` 的业务字段**（类型为 `boolean`，表示"是否必填"）。这个业务字段的键名恰好与保留关键字重合，于是被 `filter(k => !reserved.has(k))` 排除出"已声明字段"列表，导致 `additionalProperties: false` 检查判为非法。

**实际影响：** 文档中每个表单组件示例只要写了 `required: true`（这是最常见的表单用法），过 L2 校验就会报 `不允许的额外字段 "required"`。`05-scenarios/form-with-reactions.md` 官方旗舰示例也会因此报错。

**处理方案：** 修改 `getDeclaredFields()` 的逻辑——将 `reserved` 集合缩小或改为上下文感知过滤：

1. `additionalProperties`、`allOf`、`anyOf`、`oneOf` 这四个键在 `component-registry.json` 的 DSL 中是纯结构关键字，在任何组件的 `props` 中都不可能作为业务字段出现，可以保留在过滤集中。
2. 但 `required` 在 DSL props 中有双重含义——既可以是标准 JSON Schema 风格的 `required: ["fieldA"]` 数组，也可以是表单组件的 `required: true/false` 布尔业务字段。因此**仅当 `required` 的值为数组时才应被过滤**（作为结构关键字），**值为布尔类型时不应被过滤**（作为业务字段）。

**具体代码修改方案：**

```js
function getDeclaredFields(propsSpec) {
  const reserved = new Set(['additionalProperties', 'allOf', 'anyOf', 'oneOf']);
  return Object.keys(propsSpec).filter(k => {
    if (reserved.has(k)) return false;
    // required 有双重含义：数组是结构关键字，布尔是表单业务字段
    if (k === 'required') {
      const spec = propsSpec[k];
      if (spec && typeof spec === 'object' && Array.isArray(spec.type ? spec.type : spec)) return false; // 结构关键字
      // 如果是 { type: "boolean", ... } 则是业务字段，保留
      return true;
    }
    return true;
  });
}
```

简化方案：直接检查 `propsSpec[k]` 的类型，如果 `required` 对应的值是 `{type: "boolean", ...}` 则保留，如果是 `["fieldA"]` 则过滤。更简单的方式：只过滤 `additionalProperties`、`allOf`、`anyOf`、`oneOf` 这四个纯结构关键字，`required` 不做过滤——因为 `validateProps()` 中的 `required` 字段检查已经通过 `spec.required === true` 正确处理了 DSL 风格的必填标记，不需要在 `getDeclaredFields` 中额外过滤 `required`。

**推荐采用最简修复：** 将 `required` 从 `reserved` 集合中移除。原因是：
- `validateProps()` 中已通过 `if (spec.required === true && props[field] === undefined)` 正确处理了 DSL 的必填语义；
- `validateNestedObject()` 中已通过 `Array.isArray(spec.required) ? spec.required : []` 处理了数组风格的 required；
- 因此 `getDeclaredFields` 不再需要过滤 `required` 来避免干扰。

**涉及文件：** `scripts/validate-l2-components.js`

### V33（🟡）`validateResponseMapping()` 对"未声明 `responseMapping` 走默认字段名"的合法场景误报

**问题：** `validateResponseMapping()` 中服务端分页的检查逻辑：

```js
if (type === 'table' && props.pagination && props.pagination.mode === 'server') {
    if (!rm || !rm.total) {
      violations.push({...});
    }
}
```

`04-datasource-contract.md §4.1.1` 明确说明："未声明 `responseMapping` 时，Renderer 按协议默认字段名解析……服务端分页总数读取 `total`"。这意味着**完全不写 `responseMapping`、直接返回 `{list, total}` 是合法的默认用法**。但当前代码的无条件 `!rm || !rm.total` 检查没有区分"完全不声明 responseMapping（走默认）"和"声明了 responseMapping 但漏填 total"两种情况。

同样，列表类的 `rm.list` 检查也有类似问题但影响较小：`if (type === 'table' && rm !== undefined && !rm.list)` 已正确处理了"未声明 rm 时不报"的情况。服务端分页的检查也应按此模式修正。

**处理方案：** 将服务端分页的检查改为：**仅当 responseMapping 已声明但缺少 total 时才报错**，不报"未声明 responseMapping"的情况。

```js
// table 服务端分页：已声明 responseMapping 时必须有 total
if (type === 'table' && props.pagination && props.pagination.mode === 'server') {
    if (rm !== undefined && !rm.total) {
      violations.push({...});
    }
}
```

**涉及文件：** `scripts/validate-l2-components.js`

## 🟢 轻微问题

### V34（🟢）`06-validation.md §5` 本地校验指南命令与实际行为不一致

**问题：** 三处文档与实测不符：

1. **§5.1 声称"`ajv-cli` 会自动加载 `$ref` 引用的子 schema……无需额外配置"**——实测 `npx ajv validate -s docs/schemas/page.schema.json -d my-page.yaml` 会报错 `can't resolve reference node.schema.json`，需要手动加 `-r` 参数。但仓库自带的 `scripts/validate-all.js` 中 `runAjv()` 已正确处理了 `-r` 引用（传入了 3 个 `-r` 参数），只是文档没同步。

2. **§5.4 CI 示例中的 `--strict-refs=true` 参数在当前 pin 的 `ajv-cli@5.0.0` 中不存在**——实测会报 `error: parameter --strict-refs is unknown`。该参数在 `ajv-cli` 中从未存在过（可能是 `ajv` 库的选项，但 `ajv-cli` 不支持）。照抄会让 CI 直接挂掉。

**处理方案：**

1. §5.1：删除"`ajv-cli` 会自动加载 `$ref` 引用的子 schema……无需额外配置"这句话，替换为正确用法说明（需要显式传入 `-r` 参数），并给出完整命令示例：

```bash
npx ajv validate -s docs/schemas/page.schema.json \
  -r docs/schemas/node.schema.json \
  -r docs/schemas/action.schema.json \
  -r docs/schemas/reaction.schema.json \
  -d "pages/**/*.yaml"
```

2. §5.4 CI 示例：删除 `--strict-refs=true`，替换为正确的 `-r` 参数用法。

**涉及文件：** `docs/06-validation.md`
