# 审计报告 0048：params `$deps.*` 整值替换边界 + 跨文档 fragment 漂移

## 背景

本轮基于 0047 已归档后的再次全量复审。0047 中的表格 actions `$self` 口径残留、L3a 注释与 CHANGELOG/元数据卫生项已确认落地，本轮不重复报告 V192-V196。

## 发现的问题

### V197（🟡）params 中嵌入式 `$deps.*` 被误当作合法值替换

**位置：** `scripts/validate-l3a-expressions.js` 的 `scanDataParams()`；`docs/04-datasource-contract.md` §3.1/§3.2；`docs/02-reaction-expression.md` §10.7；`mcp/tests/validate-content.test.ts` params 数组测试。

**当前值：** `scanDataParams()` 使用未锚定正则从任意字符串中提取变量引用。只要提取结果以 `$deps.` 开头且节点位于表单上下文，便直接接受。因此下列参数可通过 L0-L4：

```yaml
params:
  owner: prefix-$deps.ownerId
```

**问题：** 协议只定义 params 值为字面量或单个 `$deps.*` 的“值替换”，并规定依赖值为 `null` / `undefined` 时整体省略该参数；未定义字符串模板拼接、转义、类型转换，也未说明空值时是否保留前后缀。允许嵌入式引用会使不同 Renderer 分别采取整值替换、子串替换、保留原字符串或拒绝配置。

**影响：** 🟡 中等。同一合法性判定下可能产生不同 query 参数，且空值省略规则无法确定，属于实现互操作边界缺口。

**建议修复：** params 字符串只允许以下两类：

- 不含协议变量标记的普通字面量；
- 整个字符串完整匹配单个合法 `$deps.<path>`。

字符串一旦包含 `$` 变量语法但不能完整匹配单个 `$deps.*`，以 `DATA_PARAMS_VARIABLE` 拒绝。该规则应统一覆盖 `data.params`、`select.props.optionsSource.params` 与 `datasources.*.params` 的递归对象/数组值；页面级 `datasources.*.params` 中完整 `$deps.*` 仍按现有 `NON_FORM_DATA_PARAMS` 拒绝。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`：收紧完整值匹配。
- `docs/02-reaction-expression.md`、`docs/04-datasource-contract.md`、`docs/06-validation.md`：明确整值替换与禁止模板拼接。
- `mcp/tests/validate-content.test.ts`：覆盖完整引用、嵌入式引用、普通字面量以及 optionsSource/datasources 路径。
- `docs/CHANGELOG.md`：在 Unreleased 记录校验边界修复。

**验证方式：**

- `"$deps.ownerId"` 通过；
- `"prefix-$deps.ownerId"` 与 `"$deps.ownerId-suffix"` 以 `DATA_PARAMS_VARIABLE` 失败；
- `"prefix-owner"` 作为普通字面量通过；
- 执行 `npm run validate:l3a -- <fixture>`、`npm --prefix mcp test` 与 `npm --prefix mcp run build`。

**与已归档审计的关系：** V19 定义 params 空值整体省略；V59 将 `optionsSource.params` 纳入 L3a；V100/V178 对 `requestMapping` 已采用“含 `$` 时必须整段匹配”的规则。本条是相同风险在 params 值替换路径中的未覆盖位置，不是历史问题回退。

### V198（🟢）`04` 指向表达式 §10.1 的 Markdown fragment 已失效

**位置：** `docs/04-datasource-contract.md` §3.2 指向 `docs/02-reaction-expression.md` §10.1 的链接。

**当前值：** 链接 fragment 为：

```text
#101-deps-出现在非表单-visiblewhen-中
```

当前目标标题已变为：

```text
### 10.1 非表单 / 表单 `visibleWhen` 的变量白名单
```

**问题：** 相对文件目标存在，但 fragment 不再对应当前标题。读者点击后只能打开文档，不能定位到所引用的作用域规则。

**影响：** 🟢 轻微。不改变协议行为，但降低跨文档导航准确性。

**建议修复：** 将链接更新为当前 §10.1 标题生成的 fragment，并执行 Markdown 相对链接的文件目标与 fragment 双重检查。

**需要同步的文件：** `docs/04-datasource-contract.md`。

**与已归档审计的关系：** V111 修复过当前正文中的其他 fragment，V141 补充了“文件存在但 fragment 失效”的检查。本条是 §10.1 后续改名产生的新漂移位置。

1. **V197**：先确定整值替换规则，收紧 L3a，并补充 MCP 回归测试与协议说明。

## 防止复发

- 将所有“变量值替换”位置统一为显式的完整匹配策略，避免通过搜索变量子串意外引入模板语言。
- 为 params 的三种挂载路径共享同一组正例/反例，覆盖对象、数组和空值语义。
- Markdown 链接检查同时验证文件目标和 fragment；高频改名章节变更时搜索旧 fragment。
