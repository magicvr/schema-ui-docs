## 发现的问题

### V64（🔴）`actions[].type: modal` 的 `content` Node 不进入 L2 / L3a / L4

**位置：**
- `scripts/validate-l2-components.js`（`validateNode` / `validateRowActionRefs` / `validateRequiredCapabilities` 仅遍历 `body` 与 `tabs.items[].content`）
- `scripts/validate-l3a-expressions.js`（`validatePage` 仅 `doc.body`）
- `scripts/lint-l4-banned-props.js`（`scanPage` 仅 `doc.body`）
- 对照：`docs/07-actions-contract.md` §5、`docs/schemas/action.schema.json` `ModalAction.content`、`docs/06-validation.md` L2/L3a/L4 描述
- 间接影响：MCP `protocol.validate_content` 复用上述脚本，存在同样盲区

**当前值：**
- Schema 层：`modal.content` 通过 `$ref: node.schema.json` 做 L0/L1 结构校验。
- 脚本层：L2/L3a/L4 均不递归进入 `doc.actions[*].content`。
- 协议层：`07` 明确 `content` 是完整 Node（可为 `form`/`text`/`table` 等），应遵守组件契约与表达式规则。

**复现（已实测）：**

```yaml
actions:
  openForm:
    type: modal
    modalId: medium-dialog
    content:
      type: form
      props: { title: broken, submitAction: noop }
      children:
        - type: input
          props: { field: x, label: X }
          reactions:
            - dependencies: []
              when: "$row.a == 1"   # body 下会被 L3a 拒绝
              fulfill: { visible: true }
        - type: unknownWidget       # body 下会被 L2 拒绝
          props: { foo: 1 }
```

- 上述配置：`validate-all` / L2 / L3a / L4 全部 **exit 0**。
- 将同一 `reactions.when: "$row.a == 1"` 放到 `body` 下：L3a 报 `SCOPE_ISOLATION`（exit 1）。

**问题：** 弹窗内容 Node 是协议合法挂载点，却绕过组件契约、表达式作用域与 CSS 禁用词校验。L0/L1 只能拦 Schema 结构（如 `fulfill` 非法键），拦不住未知 `type`、`$row` 误用、能力声明与 `requestMapping` 等语义规则。

**影响：** 🔴 严重。生成器/人工配置可在 `modal.content` 中写入 body 侧会被拒绝的非法配置；CI 与 MCP `validate_content` 会误报“通过”，破坏“与 CI 同版本脚本一致”的校验承诺。

**建议修复：**
1. L2：在页面扫描中遍历 `doc.actions`，对 `type: modal` 且存在 `content` 的 action 调用 `validateNode(content, actions.<id>.content, ...)`；行级 actionRef / capability 扫描同样进入该子树。
2. L3a：`validatePage` 对每个 `actions[*].content` 调用 `scanNode(..., hasFormContext=false, tableDepth=0)`。
3. L4：`scanPage` 对每个 `actions[*].content` 调用 `scanNode`。
4. `06-validation.md` L2/L3a/L4 描述补充“覆盖 `actions[].type: modal` 的 `content` Node”。
5. 增加最小正/反例验证；MCP 测试可补充 `modal.content` 反例，确保 `validate_content` 分层报错。

**需要同步的文件：**
- `scripts/validate-l2-components.js`
- `scripts/validate-l3a-expressions.js`
- `scripts/lint-l4-banned-props.js`
- `docs/06-validation.md`
- 可选：`mcp/tests/validate-content.test.ts`（反例回归）

### V65（🟡）页面级 `datasources.*.params` 中的 `$deps.*` 未纳入 L3a

**位置：**
- `scripts/validate-l3a-expressions.js`（仅扫描节点 `data.params` / `optionsSource.params`，不扫描 `doc.datasources`）
- `scripts/lint-l4-banned-props.js` 注释写“datasources 的 params 也要扫描”，实现未做（L4 本身不负责 `$deps`，但说明实现与注释漂移）
- 对照：`docs/04-datasource-contract.md` §3.2、`docs/02-reaction-expression.md` §10.6 / 附录 A、`docs/06-validation.md` L3a

**当前值：**
- 协议：`$deps.*` 在 `data.params` / `optionsSource.params` 中**仅表单上下文**合法；非表单上下文静态拒绝。
- 页面级 `datasources` 是顶层预声明，**不处于 form 上下文**。
- L3a 只从 `body` 递归，从不读取 `doc.datasources`。

**复现（已实测）：**

```yaml
datasources:
  orders:
    source: api
    url: /api/orders
    params:
      status: $deps.status   # 无表单上下文
body:
  type: table
  data: { source: ref, ref: orders }
  props: { rowKey: id, pagination: { mode: none }, columns: [{ field: id, label: ID }] }
```

- 全链路 `validate-all` **exit 0**。

**问题：** 与 V50/V59「参数值替换必须进 L3a」同类，但挂载点换成页面级 `datasources`。配置可静默通过，运行时 `$deps.*` 无绑定对象。

**影响：** 🟡 中等。不会破坏 Schema 结构，但会放过明确的作用域配置错误；与 04/02 正文承诺不一致。

**建议修复：**
1. L3a `validatePage` 遍历 `doc.datasources`，对每个 `DataRef.params` 调用 `scanDataParams(..., hasFormContext=false)`（页面级预声明永远非 form 上下文）。
2. `06-validation.md` L3a 行补充 `datasources.*.params`。
3. 可选：`02` §10.6 / 附录 A 或 `04` §1 明确“页面级 `datasources.params` 不允许 `$deps.*`”。
4. 修正 L4 注释，避免“已扫描 datasources”的虚假描述（L4 若仍不扫 params 键名，应改注释；若需扫 props 嵌套则另论）。
5. 用最小非法样例验证 exit≠0 且规则为 `NON_FORM_DATA_PARAMS`（或等价）。

**需要同步的文件：**
- `scripts/validate-l3a-expressions.js`
- `docs/06-validation.md`
- 可选：`docs/02-reaction-expression.md`、`docs/04-datasource-contract.md`
- `scripts/lint-l4-banned-props.js`（注释一致性）

### V66（🟢）ADR-0004 附录矩阵仍未同步「表格列 scope:form 仅 form 内」前提

**位置：** `docs/decisions/0004-row-level-scope.md` 附录「变量可见性矩阵」  
对照：`docs/02-reaction-expression.md` 附录 A（V58 已修）、`docs/03-component-registry.md` 作用域说明

**当前值：**

| 使用位置 | `$deps.*` |
|---|---|
| 表格列 `scope: form` 表达式 | ✅ |

无“仅表格位于 `form.children` 内”限定，也无“独立表格列拒绝 `$deps`”行。

**问题：** V58 已把正文权威矩阵收紧，ADR-0004 附录仍是旧口径。以 ADR 为准的读者会认为独立表格列可用 `$deps.*`。

**影响：** 🟢 轻微。不改变运行时 Schema，但 ADR 与正文不一致；同类于 V57 的 ADR 残留同步问题。

**建议修复：** 将 ADR-0004 附录该行改为与 `02` 附录 A 一致：
- 「表格列 `scope: form`（仅表格位于 `form.children` 内）」→ `$deps.*` ✅
- 新增「独立表格列 `scope: form`」→ `$deps.*` ❌（静态校验拒绝）

**需要同步的文件：**
- `docs/decisions/0004-row-level-scope.md`

## 与已归档审计的关系

- V64 与 0004 审计中 `tabs.items[].content` / `modal.content` Schema `$ref` 工作相关，但此前只保证 L0/L1 结构引用，从未要求 L2/L3a/L4 递归进入 `actions.modal.content`。本轮是脚本遍历边界的新缺口。
- V65 与 V50（`data.params` 进矩阵/L3a）、V59（`optionsSource.params` 进 L3a）同类，挂载点为页面级 `datasources`，此前未覆盖。
- V66 与 V58 直接相关：正文附录 A 已修，ADR-0004 附录遗漏同步；亦与 V51（ADR-0004 字段名残留）同属 ADR 滞后。

## 建议处理顺序

1. **先修 V64**：`modal.content` 盲区影响 L2/L3a/L4 与 MCP 一致性，严重度最高。
2. **再修 V65**：补齐 `datasources.params` L3a，并同步文档/注释。
3. **收尾 V66**：ADR-0004 附录矩阵对齐 `02` 附录 A。

## 防止复发的建议

1. 任何“完整 Node”挂载点（`body`、`tabs.items[].content`、`modal.content`、未来同类字段）必须进入同一套 L2/L3a/L4 遍历清单，并在 `06-validation.md` 显式列出。
2. 参数值替换类挂载点（`data.params`、`optionsSource.params`、`datasources.*.params`）统一进 L3a，新增挂载点时同步矩阵与脚本。
3. 正文附录 A 变更时，全文搜索 ADR 中的变量可见性矩阵并同步，避免决策文档残留旧口径。
4. 为 `modal.content` 与 `datasources.params` 各保留一条 MCP/脚本反例回归测试，防止盲区复发。
