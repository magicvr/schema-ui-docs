# 审计报告 0038：版本说明漂移 + schemas 入口索引误导 + MCP SDK 依赖版本策略

## 背景

本轮审计基于 0037 已归档后的再次全量复审。0037 中的 MCP UTF-8 截断、导言搜索、filename 映射、表格 scope 口径、组件 DSL 校验边界等问题在当前工作区已确认修复，本轮不重复报告 V96–V113。

本轮新增问题集中在三类“入口与发布侧一致性”表面：

1. 最新 PATCH 版本的发布说明未继续反映当前仓库状态；
2. 入口文档仍把 `docs/schemas/` 整体等同于标准 JSON Schema；
3. MCP 包内部核心 SDK 依赖仍采用浮动版本策略，与仓库对外强调的稳定接入口径不一致。

当前历史最大问题编号为 `V113`，因此本轮问题从 `V114` 连续编号。

## 发现的问题

### V114（🟡）`v0.2.8` 发布说明已落后于当前仓库状态

**位置：** `docs/CHANGELOG.md` 顶部 `v0.2.8` 条目；对照 `README.md`、`docs/00-overview.md`、`docs/mcp/README.md` 当前版本说明。

**当前值：** `CHANGELOG.md` 中的 `v0.2.8` 仅说明“基于审计 0034–0035”的修复，修复列表停留在 V85/V86/V88；但当前仓库正文、脚本、MCP 文档与测试已包含 0036/0037 落地后的行为与规则。

**问题：** 当前“最新 PATCH 版本说明”没有继续反映最新已合入的协议/脚本/MCP 变更，导致读者从版本记录读取到的内容落后于仓库实际状态。

**影响：** 🟡 中等。不会直接导致校验错误，但会削弱 CHANGELOG 作为发布说明与交叉引用锚点的可信度，也会让“当前最新补丁到底包含哪些修复”变得模糊。

**建议修复：**
1. 先明确当前仓库语义：
   - 若 0036/0037 代表已并入 `v0.2.8` 的补丁内容，则补齐 `v0.2.8` 条目，覆盖当前实际状态；
   - 若这些内容应属于下一个 PATCH，则新增 `v0.2.9` 或 `Unreleased` 段，避免继续把新增修复挤进旧条目。
2. 同步核对 `README.md`、`docs/00-overview.md`、`docs/mcp/README.md` 中对“当前最新 PATCH”的表述，确保入口与 CHANGELOG 同步。
3. 如涉及 MCP 对外版本示例，也同步确认镜像 tag 与文档示例是否需要调整。

**与已归档审计的关系：** 最接近 `0035 / V89`，但 `V89` 修的是“缺少 `v0.2.8` 版本闭环”；本条是“已有 `v0.2.8` 条目，但其内容再次落后于当前仓库状态”，不属于重复问题。

**需要同步的文件：**
- `docs/CHANGELOG.md`
- 视最终版本决策可能还需同步：`README.md`、`docs/00-overview.md`、`docs/mcp/README.md`

### V115（🟡）入口文档仍把 `docs/schemas/` 整体描述为标准 JSON Schema

**位置：** `README.md`（仓库说明与目录树）；`docs/00-overview.md` 文档地图中的 `schemas/` 行。

**当前值：**
- `README.md` 将 `docs/` 描述为“协议、场景示例、JSON Schema、ADR 与校验说明”；
- 目录树把 `docs/schemas/` 标成“机器可读 JSON Schema”；
- `docs/00-overview.md` 文档地图把 `schemas/` 概括为“机器可读的 JSON Schema”。

**问题：** 当前入口描述忽略了 `docs/schemas/component-registry.json` 已被正式定义为**自定义 DSL、非标准 JSON Schema** 的事实。读者或工具若只从入口索引阅读，会被误导为 `docs/schemas/*.json` 全部都能被标准 JSON Schema 工具等价处理。

**影响：** 🟡 中等。不会改变协议正文，但会误导新接入者、文档读者与 AI 工具的认知路径，弱化 0017 对 DSL 边界的修复效果。

**建议修复：**
1. 将入口层表述改为“机器可读 Schema / JSON Schema 与组件 DSL”或等价措辞；
2. 在 `README.md` 与 `docs/00-overview.md` 中明确：
   - `page.schema.json` / `node.schema.json` / `action.schema.json` / `reaction.schema.json` 是标准 JSON Schema；
   - `component-registry.json` 是组件注册 DSL，不应直接按 AJV 标准 Schema 处理。
3. 如需保持简介简短，可在入口层补一句“详见 `03-component-registry.md` 对 DSL 边界的说明”。

**与已归档审计的关系：** 最接近 `0017 / V18`，但 `V18` 修的是 `03-component-registry.md` 与 `06-validation.md` 对 DSL 解释边界的说明；本条是**README / 总纲入口层**仍未同步该边界，不属于重复问题。

**需要同步的文件：**
- `README.md`
- `docs/00-overview.md`

### V116（🟢）MCP 核心 SDK 依赖仍使用浮动版本 `latest`

**位置：** `mcp/package.json` 的 `dependencies["@modelcontextprotocol/sdk"]`；对照 `mcp/package-lock.json`。

**当前值：** `mcp/package.json` 仍声明：

```json
"@modelcontextprotocol/sdk": "latest"
```

同时 `mcp/package-lock.json` 当前已锁到具体发布版本。

**问题：** 仓库对外文档持续强调接入应固定 PATCH 版本、避免无意漂移，但 MCP 包内部最核心运行时依赖仍采用浮动版本策略。虽然 lockfile 使当前构建可复现，但在后续刷新 lockfile、依赖升级或维护时，源码层面并没有清晰表达“本版本实际针对哪个 MCP SDK 验证过”。

**影响：** 🟢 轻微。短期内不会立刻破坏当前构建，但会增加未来依赖升级与问题回溯的模糊性，和项目发布稳定性口径不完全一致。

**建议修复：**
1. 将 `@modelcontextprotocol/sdk` 从 `latest` 改为明确版本范围（优先固定具体 PATCH，或至少固定兼容区间并说明策略）；
2. 同步更新 `mcp/package-lock.json`；
3. 重新执行 MCP 构建、测试与 smoke，确认与当前代码行为一致。

**与已归档审计的关系：** 未见历史审计直接覆盖“**MCP 内部依赖版本策略**”这一层面；与 0035/0036 的协议版本与镜像 tag 漂移属于相邻主题，但问题层次不同。

**需要同步的文件：**
- `mcp/package.json`
- `mcp/package-lock.json`
- 如需记录依赖策略，也可补充到 `docs/mcp/README.md`

## 与已归档审计的关系

- **V114 ↔ 0035 / V89：** 同属版本说明/发布闭环主题，但 `V89` 修的是“缺少 `v0.2.8`”，本轮是“已有 `v0.2.8` 但内容再次滞后”。
- **V115 ↔ 0017 / V18：** 同属组件 DSL 边界主题，但 `V18` 修的是正文内部说明，本轮是入口索引层残留误导。
- **V116：** 历史审计未直接覆盖 MCP 内部依赖版本策略，仅与版本漂移主题相邻，不重复。

## 建议处理顺序

1. **先处理 V114：** 先澄清当前仓库应对应的最新 PATCH / unreleased 语义，避免后续文档同步没有锚点；
2. **再处理 V115：** 修正入口层 schemas 描述，收敛读者与工具的第一印象；
3. **最后处理 V116：** 固化 MCP SDK 依赖策略，并执行最小必要的 MCP 验证。

## 防止复发的建议

1. 只要正文、脚本或 MCP 行为继续在当前 PATCH 上演进，就必须同步维护 CHANGELOG 或显式引入 `Unreleased` 段，避免“版本号不变但说明失真”。
2. 入口索引类文档（`README.md`、`00-overview.md`）应视为“认知入口”，凡正文已确立的重要边界（如 DSL vs JSON Schema）都要同步到入口层，而不是只留在深层章节。
3. 面向发布的核心运行时依赖尽量避免 `latest`；即便有 lockfile，也应在 manifest 中表达明确的依赖策略，降低后续维护与追溯成本。
