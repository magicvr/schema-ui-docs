## 发现的问题

### V18（🟡）`component-registry.json` 自定义 DSL 混用 JSON Schema 组合关键字，但支持边界未被规范化

**位置：** `schemas/component-registry.json`；`03-component-registry.md` §“机器可读注册表”；`06-validation.md` L2 组件契约校验

**当前值：** `03-component-registry.md` 已明确 `component-registry.json` 是自定义 DSL，非标准 JSON Schema，不应直接用 AJV 校验。但该 DSL 内部同时使用两类约束语义：

```json
"submitAction": { "type": "string", "required": false },
"mode": { "type": "string", "enum": ["default", "search"], "required": false },
"targetTable": { "type": "string", "required": false },
"allOf": [
  {
    "if": { "properties": { "mode": { "const": "search" } }, "required": ["mode"] },
    "then": { "required": ["targetTable"] },
    "else": { "required": ["submitAction"] }
  }
]
```

同一文件中还存在 `anyOf`、`oneOf`、`allOf`、`if` / `then` / `else`、数组形式 `required`、`additionalProperties: false` 等标准 JSON Schema 风格关键字。

**问题：** 当前文档只说明它“不是标准 JSON Schema”，但没有说明 L2 校验器必须支持哪些 JSON Schema 风格的组合关键字，也没有解释字段级布尔 `required` 与数组形式 `required` 的不同语义。实现者如果只按字段表解析 DSL，会漏掉 `form.mode=search` 时 `targetTable` 必填、默认模式 `submitAction` 必填、`select.options/optionsSource` 二选一、i18n 文案二选一等关键约束。

**影响：** 🟡 中等。协议正文基本正确，但机器可读 DSL 是 L2 校验、编辑器提示和生成器的重要输入。解释边界不清会导致不同工具对同一组件契约给出不同结论。

**建议修复：** 二选一：

1. 在 `03-component-registry.md` 与 `06-validation.md` 明确 `component-registry.json` 的 DSL 约束模型，列出必须支持的组合关键字子集，并说明字段级 `required: true/false` 与组合约束中的 `required: [...]` 分别如何解释；
2. 或把组合约束改写为 DSL 自有字段，例如 `requiresOneOf`、`requiresWhen`、`exclusiveProps`，避免自定义 DSL 与 JSON Schema 关键字混用。

### V19（🟢）`DataRef.params` 空值规则的交叉引用指向 `04-datasource-contract.md §8`，章节号已漂移

**位置：** `01-node-protocol.md` `DataRef` 结构示例；`04-datasource-contract.md` §3.1 与 §8

**当前值：** `01-node-protocol.md` 的 `DataRef` 示例中写道：

```yaml
params: map        # 【可选】请求参数映射，值可引用 $deps.*，空值规则见 04-datasource-contract.md §8
```

但 `04-datasource-contract.md` 中通用空值省略规则位于 §3.1：`data.params / optionsSource.params 中 $deps.* 的空值省略规则`。§8 是 `select.optionsSource` 远程动态选项契约。

**问题：** 交叉引用把通用 `data.params` 规则指向了 `select.optionsSource` 专门章节。虽然 §8 也涉及 `optionsSource.params`，但它不是 `DataRef.params` 的通用规则入口。

**影响：** 🟢 轻微。不会改变协议语义，但会让读者误以为空值省略规则主要属于 `select.optionsSource`，削弱 `data.params` 与 `optionsSource.params` 共用规则的清晰度。

**建议修复：** 将 `01-node-protocol.md` 中该注释改为指向 `04-datasource-contract.md §3.1`；如需保留 `select` 关联，可补充“`select.optionsSource.params` 同样适用”。

## 防止复发的建议

1. **为自定义 DSL 建立关键字白名单：** 每次向 `component-registry.json` 添加组合约束时，同步确认该关键字是否在 DSL 规范中列为必支持。
2. **区分标准 JSON Schema 与 DSL 借用语法：** 如果借用 JSON Schema 关键字，应明确语义是否完全等同 JSON Schema。
3. **章节引用跟随标题而非手写编号：** 对跨文档规则引用优先使用 Markdown 锚点链接，减少章节重排后的编号漂移。
