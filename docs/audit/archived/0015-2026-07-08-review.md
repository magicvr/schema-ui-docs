## 发现的问题

### V9（🔴）`08-renderer-spec.md` 示例仍引用已废弃/不存在的 `$deps.dateRange`

**位置：** `08-renderer-spec.md` §2.3 数据依赖声明示例

**当前值：** 示例中 `dateRangePicker` 声明 `startField: dateFrom`、`endField: dateTo`，但目标表格的 `data.params` 写成：

```yaml
params:
  status: $deps.orderStatusFilter
  dateFrom: $deps.dateRange
```

**问题：** `03-component-registry.md` 已明确规定：`dateRangePicker` 拥有两个字段值，其他字段依赖它时应在 `dependencies` 中填写 `startField` / `endField` 的值，并通过 `$deps.<startField>` / `$deps.<endField>` 访问。也就是说，本例应引用 `$deps.dateFrom` 和 `$deps.dateTo`，而不是 `$deps.dateRange`。

**影响：** 🔴 关键。该示例位于 Renderer 实现规范，读者很可能直接复制。错误的 `$deps.dateRange` 会让实现者误以为 `dateRangePicker` 仍提供合并字段，导致搜索参数缺失或不同 Renderer 自行兼容出分叉行为。

**建议修复：** 将示例改为显式传递两个独立参数：

```yaml
params:
  status: $deps.orderStatusFilter
  dateFrom: $deps.dateFrom
  dateTo: $deps.dateTo
```

### V11（🟡）`scope: form` 默认场景表述容易误导非表单节点 `visibleWhen`

**位置：** `02-reaction-expression.md` §5 作用域表

**当前表述：** `scope: form` 的默认场景包含“表单 `reactions`、页面 `visibleWhen`”。

**问题：** 该表述容易让读者误以为页面级或其他非表单节点的 `visibleWhen` 可以像表单表达式一样读取 `$deps.*`。但 `01-node-protocol.md` 已规定：非表单节点 `visibleWhen` 不绑定表单字段，只允许使用 `$context.*`。

**影响：** 🟡 中等。协议正文存在正确规则，但作用域表的摘要表达容易形成错误心智模型，尤其会影响 Renderer 静态校验规则：非表单节点 `visibleWhen` 中出现 `$deps.*` 应被拒绝。

**建议修复：** 将默认场景改为更精确的表述，例如“表单字段 `reactions`、表单内 `visibleWhen`”，并补充说明：非表单节点 `visibleWhen` 虽默认属于 form 级表达式，但只能访问 `$context.*`，不得访问 `$deps.*`。

### V12（🟡）表格列 `scope: form` 下 `$self` 可见性语义未定义

**位置：** `02-reaction-expression.md` §6 变量可见性矩阵

**当前值：** 变量矩阵允许“表格列 `scope: form` 表达式”访问 `$self`。

**问题：** `$self` 在正文中有两个清晰语义：

- 表单字段自身值；
- `scope: row` 下的当前列单元格原始值。

但表格列在 `scope: form` 下没有当前行/当前单元格上下文，也不是表单字段，因此 `$self` 没有明确绑定对象。若不收紧，Renderer 可能分别解释为 `undefined`、列配置、整列字段值或当前单元格值，导致实现分叉。

**严重性：** 🟡 中等。该问题不一定影响常规配置，但会影响静态校验器和 Renderer 对边界配置的处理一致性。

**建议修复：** 将“表格列 `scope: form` 表达式”的 `$self` 改为不可用；如果确有使用需求，应另行定义它的绑定对象，而不是复用当前未定义语义。

### V13（🟢）`06-validation.md` 校验命令路径与仓库实际目录不一致

**位置：** `06-validation.md` §3.1 AJV 示例与 VS Code YAML 配置示例

**当前值：** 文档示例使用：

```bash
ajv validate -s schemas/page.schema.json -d examples/page.json
```

以及：

```json
"schema": "schemas/page.schema.json"
```

**问题：** 当前仓库中 schema 实际位于 `docs/schemas/page.schema.json`。如果读者从仓库根目录直接复制命令，会找不到 schema 文件。除非文档明确要求从 `docs/` 目录执行，否则该路径应写成 `docs/schemas/page.schema.json`。

**严重性：** 🟢 轻微到中等。它不影响协议内容，但影响本地验证命令的可复制性。

**建议修复：** 二选一：

1. 将命令和 VS Code 配置路径统一改为 `docs/schemas/page.schema.json`；
2. 或在命令前明确说明“以下命令需在 `docs/` 目录下执行”。

## 防止复发的建议

1. **示例应纳入协议变更回归检查：** 修改组件契约时，使用关键词回扫所有示例中的旧字段名和旧 `$deps.*` 写法。
2. **入口目录树随新增 ADR 更新：** 新增 `docs/decisions/NNNN-*.md` 后，同步检查根 README 的 `decisions/` 列表。
3. **校验命令应注明执行目录：** 所有可复制命令都应明确“从仓库根目录执行”或“从 `docs/` 目录执行”。
4. **变量矩阵只列已定义语义：** 若某变量在某作用域下没有明确绑定对象，矩阵中应标为不可用，而不是留给 Renderer 自行解释。
