# 审计报告 0065：ADR-0023 权限继承提案可接受性评审

## 结论

**方向可接受，当前正文尚不可直接 `accepted`。**

ADR-0023 作为 **post-v2.2 的新 MINOR 候选**，在版本纪律、opt-in 边界、单调收紧与「禁止从实现细节推断意图」上与章程、Admin 轨道原则及 ADR-0003 遗留问题对齐，**未发现会静默破坏 v2.2 合法页面的 P0**。

但对照已接受的 ADR-0020/0021/0022 质量条，正文仍有 **4 项 P1** 使跨 Renderer 的可观测行为无法唯一确定；另有 **4 项 P2** 会在落地门禁时变成分叉或文档漂移。**在 P1 以开放问题裁决并写回 ADR 正文之前，不得将状态改为 `accepted`，也不得进入 `protocol-manifest.json` 的 `authority.semanticSpecs`。**

本文档是过程记录，不是协议权威来源。权威仍以已接受 ADR、`01`/`03`/`07`/`08` 与 v2.2 制品为准；0023 在 `proposed` 期间不改变任何已发布页面合法性。

## 总体评估

### 已对齐、应保留的设计

| 点 | 评价 |
|---|---|
| `proposed` + 明确「非 v2.2 合法字段」 | 符合「先定边界再写字段」与 0064 版本纪律 |
| 显式 `permissionCascade`，禁止「容器上写 `permissions.edit` 即隐式继承」 | 保护旧页可观测行为；与 D1 不追溯修改 v2.2 一致 |
| 有效权限单调 AND | 与 ADR-0003 D3d「只能收紧」同构 |
| `permissionIntent` 禁止从 key/method/URL/文案推断 | 直接堵住多 Renderer 私货分叉的主路径 |
| capability `permissions.inheritance` + 新 MINOR | 符合轨道「MINOR + capability」策略 |
| 非目标（安全边界、自定义键自动继承、跨 modal/跨页） | 范围克制，MVP 锚点清晰 |
| 落地门禁 M1–M6 | 与 PROJECT_CHARTER 原子交付一致 |

### 不可直接接受的原因（摘要）

1. **结构路径未闭合**：`effectivePermission` 的「从根到 t」对 Node `children` 清楚，对 `table.props.toolbar` / `actions[]` / `tabs.items[].content` 等 **DSL 嵌套挂载点** 的祖先链未定义；`table` 自身是否可作为 cascade 边界也只靠示例暗示。
2. **表单 `edit` 目标集合未闭合**：哪些组件 type 在 cascade 下变为只读/禁用、`form.mode=search` 是否参与、`submitAction` 是否自动视为 edit 目标，均未达到可写 L2/fixtures 的粒度。
3. **`permissionIntent` 挂载面未闭合**：RowAction / toolbar / actionButton 已点名，但与 navigate/modal、`form` 提交入口、列级操作的关系仍可被 Host 用私货补全。
4. **质量条缺口**：0020–0022 接受前有 **开放问题裁决表**；0023 仅有「接受前复核点」，未把上述歧义收成可关闭的 OQ。

### V287（P1）cascade 结构路径未闭合 — 开放

**现象：** 有效权限公式写「从根到 t 的所有 `permissionCascade[k]`」，示例把 `permissionCascade` 写在 `type: table` 上并级联到 `props.toolbar[]`。但 `toolbar` / `actions[]` / `columns[]` **不是** Node `children`；`tabs` 内容在 `items[].content`；`modal` 内容另有根。ADR 只点名「`children`、`tabs.items[].content` 等」，未给出：

- 哪些 type 允许声明 `permissionCascade`（仅 `supports children`？是否包含 `table` / `tabs`？）；
- 对操作入口，路径上的「祖先」是否包含承载它的 table/form Node；
- 跨 `modal.content` / 新 page 根是否硬切断（D5 有非目标句，但未进入公式前提）。

**互操作风险：** 甲 Renderer 只沿 `children` 传播，乙沿 props 内嵌操作传播 → 同 YAML 不同禁用集。

**建议裁决方向（供 ADR 写回，非本审计权威）：**

1. 定义 **权限结构树** = Node 树边（`children`、`tabs.items[].content`、规范已定义的其它嵌套 Node 位置）∪ **操作挂载边**（至少 `table.props.actions[]`、`table.props.toolbar[]`；`actionButton` 作为普通 Node 走 Node 边）。
2. `permissionCascade` 仅允许出现在「对上述边至少有一条出边」的位置；L2 白名单 type 表写死。
3. 公式中的祖先 = 权限结构树上从 page body 根到目标的路径（含声明 cascade 的 table 节点本身）。

**关闭条件：** ADR-0023 增加结构路径定义 + 允许 cascade 的 type 表；接受时 M2/M4 能据此写 L2。

### V288（P1）表单 `edit` 目标白名单未闭合 — 开放

**现象：** D4 称 `edit`「对表单字段和 form 的非搜索提交入口按组件契约生效」，但未列出：

- 哪些 `type` 算可编辑字段（input/select/…/upload？只读展示 Node？）；
- `form.mode=search` 下 cascade `edit=false` 是否禁用筛选字段与「查询」提交；
- 默认模式 `submitAction` 是否在无 `permissionIntent` 时仍受 form 级 cascade `edit` 约束；
- `recordSource` 回填只读字段、已有 `disabled` 与 cascade 的合成。

业务锚点 1（编辑页 form 只读）依赖该集合确定，否则无法写 conformance 向量。

**建议裁决方向：**

1. MVP：仅 `form.mode` 缺省/default 的子树；`mode=search` **不**参与 `edit` 继承（搜索不是「编辑记录」锚点）。
2. 可编辑目标 = 组件注册表中「表单字段类且支持用户写入」的 type 白名单 + 该 form 的提交入口（见 V289）。
3. 合成：`effective edit = false` ⇒ 与现有 `permissions.edit=false` 相同呈现（只读/禁用），且提交不得发出。

**关闭条件：** ADR 给出 type 表或明确「以 `03` 字段类 + 支持 reactions/value 写入者为准」的可判定规则；search 模式一句话。

### V289（P1）`permissionIntent` 挂载点矩阵未闭合 — 开放

**现象：** 操作入口需 `permissionIntent: edit | delete` 才参与级联；已举 RowAction、toolbar、actionButton。未裁决：

| 挂载点 | 是否允许 `permissionIntent` | 与 action type 关系 |
|---|---|---|
| `table.props.actions[]` | ？ | request / navigate / 本地 key？ |
| `table.props.toolbar[]` | 示例为 delete | request 批量与非批量？ |
| `actionButton` Node | 点名 | navigate/modal/request？ |
| form 默认提交控件 | 未提 | 是否免 intent、仅吃 cascade edit？ |
| 列上无独立操作 DSL | — | MVP 不涉及？ |

**风险：** 「编辑」行按钮实为 `navigate`，若不允许 `permissionIntent: edit`，容器只读时仍可点进编辑页；若允许却无矩阵，各实现不一致。

**建议裁决方向：**

1. MVP 允许 intent 的挂载点白名单写死（至少：RowAction、`table.toolbar`、`actionButton`）。
2. `permissionIntent` 与 `actionRef` 目标 type **解耦**（intent 描述入口业务意图，不推断 HTTP）；`navigate` 到编辑页可标 `edit`。
3. form 默认提交：**不**使用 `permissionIntent` 字段，而由「处于 cascade edit 的 form 子树内的默认提交入口」直接吃 `effectivePermission(formSubtree, edit)`（或等价：提交控件视为隐式 edit 目标）。
4. 未在白名单的位置出现 `permissionIntent` → L2 拒绝。

**关闭条件：** ADR 增补挂载点 × intent ×（可选）action type 表。

---

### V290（P1）中间节点「有 permissions 无 cascade」语义 — 开放

**现象：** 公式只 AND「路径上声明了 cascade 的祖先源权限」与「目标本地 permissions」。因此：

```yaml
type: section
permissions:
  edit: "false"   # 无 permissionCascade
children:
  - type: input
    props: { field: name }
```

按公式，input **不**继承 section 的 `edit=false`。这与 D1/D2「禁止隐式继承」一致，但：

- section 自身 `edit=false` 的可观测效果是什么（容器通常无编辑态）？
- 文档读者是否会误以为「写在容器上的 permissions.edit 历来级联」？

**建议裁决方向：** 显式写清：

1. 无 `permissionCascade` 时，容器上的 `edit`/`delete` **仅作用于该挂载点自身的既有语义**（若该 type 无自效应则为 **no-op 于后代**，保持 v2.2）；
2. 需要影响后代时必须声明 `permissionCascade.keys`；
3. 示例对比「仅本地」vs「cascade」。

**关闭条件：** ADR D2/D3 各增一小节 + 至少一组正/反例意图（接受后进 fixtures）。

### V291（P2）缺少开放问题裁决表 — 开放

0020–0022 均在 accepted 前关闭 OQ 表。0023 的「接受前复核点」未编号、未裁决。应将 V287–V290 及必要 P2 收成 `OQ-23-*`，接受时状态为已关闭。

### V292（P2）与 `08` 执行时序交叉引用 — 开放

`08` 行内：`visibleWhen` → `permissions` → `disabled` → `confirm` → 构造请求。0023 要求在确认/请求前应用有效权限，但未写明：

- `effectivePermission` 替换/扩展「permissions 判定」步骤；
- toolbar / page Trigger / form submit 是否同一顺序；
- 与 `requiresSelection` disabled 的合成（OR 禁用，仍 fail-closed）。

建议在 ADR 增加「执行时序」段，接受时同步 `08`。

---

### V293（P2）capability 门控粒度 — 开放

页面使用 cascade/intent 须声明 `permissions.inheritance`。边界：

- 仅写 `permissionIntent`、路径上无任何 `permissionCascade`：intent 对继承无贡献——L2 是 **拒绝** 还是 **允许但警告**？
- 仅有 cascade、子树无 intent 且无表单 edit 目标：是否允许（合法但无操作级联）？

建议：字段出现即要求 capability + 版本下限；「intent 无祖先 cascade」MVP **允许**（intent 仅在存在祖先 cascade 时改变有效权限），避免强迫空标。

---

### V294（P2）隐藏 vs 禁用 — 开放

`01` 对 `delete` 写「禁用/隐藏」；0023 把呈现交给组件契约，只强制「不可执行」。可接受，但须写明：

- 互操作只约束 **可执行性**（confirm 前失败、不发请求）；
- 隐藏与禁用差异 **不得** 成为第二个权限通道；
- 推荐示例统一一种呈现（如 delete 入口 disabled 或 hide 二选一写进 `03`）。
