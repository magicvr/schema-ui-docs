# 审计报告 0039：表达式边界、嵌套表格可表达性、Renderer 初始化契约与 MCP 返回预算

## 背景

本轮审计基于 0038 已归档后的再次全量复审。0038 中的版本说明、schemas 入口索引与 MCP SDK 依赖策略已确认修复，本轮不重复报告 V114-V116。

本轮采用正文、机器契约、L0-L4 校验脚本、MCP 实现与真实构建/测试的交叉核对方式，并对疑似历史同类项回查相关 review/checklist 正文。新问题集中在四类边界：

1. 正文公开的表达式语法与 L3a 实际接受范围不一致；
2. 表单上下文和业务映射对象未被校验器按其真实语义区分；
3. `$parentRow`、`$context` 与 Renderer 初始化参数在不同权威文档之间没有形成可实现的闭环；
4. MCP `protocol.get_doc` 的 20KB 预算仅覆盖正文，没有覆盖完整工具响应。

当前历史最大问题编号为 `V116`，因此本轮问题从 `V117` 连续编号。

## 发现的问题

### V117（🔴）`dateRangePicker` 公开的 `$self.start` / `$self.end` 被 L3a 拒绝

**位置：** `docs/03-component-registry.md` 的 `dateRangePicker` reactions 说明；`scripts/validate-l3a-expressions.js` 的 `ALLOWED_NS_PREFIXES`、`tokenize()` 与变量白名单检查。

**当前值：** 组件正文明确规定 `dateRangePicker` 自身 reactions 中的 `$self` 是 `{ start, end }` 对象，可通过 `$self.start` / `$self.end` 访问。L3a 却只把精确 `$self` 作为合法变量；`$self.start` 会匹配 `$self` 前缀后留下 `.start`，被词法检查判为非法后缀。

**问题：** 正文公开的合法表达式无法通过项目唯一的 L3a 静态校验入口。最小表单页面使用 `when: "$self.start < '2026-01-01'"` 时，L0/L1、L2、L4 可通过，但 L3a 报 `SYNTAX`。

**影响：** 🔴 严重。协议定义和执行校验直接矛盾，Renderer 或 CI 采用当前脚本时无法实现正文承诺的双字段组件自身联动。

**建议修复：**

1. 将 `$self` 的词法规则改为同时接受精确 `$self` 与受控属性路径；
2. 只在 `dateRangePicker` 自身的表单字段 reactions 中允许 `$self.start` / `$self.end`，不得把任意 `$self.*` 扩展到普通字段或表格列；
3. 增加合法双属性正例、未知属性反例、普通组件属性路径反例和现有精确 `$self` 回归测试；
4. 同步 `docs/06-validation.md` 的 L3a 边界说明。

**与已归档审计的关系：** 0007/S2 只补齐了 `dateRangePicker` 的正文语义，并认定无需修改表达式规范；此前未验证该语法能否通过 L3a。本条是正文能力未落入实现的剩余缺口，不重复报告已解决的语义定义问题。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `docs/06-validation.md`
- 相关脚本回归样例或 MCP `validate_content` 测试

### V118（🟡）表单上下文 `visibleWhen.dependencies` 条件必填未被执行

**位置：** `docs/01-node-protocol.md` §3.8；`docs/schemas/node.schema.json` 的 `VisibleWhen`；`scripts/validate-l2-components.js` 的 `validateVisibleWhen()`；`scripts/validate-l3a-expressions.js` 的 visibleWhen 扫描。

**当前值：** 正文规定节点位于 `form` 子树时 `visibleWhen.dependencies` 必填，非表单上下文才可省略。机器 Schema 只要求 `when`，L2 只在 `dependencies` 已存在时检查类型，L3a 在缺失时默认为空数组。

**问题：** 表单内节点声明 `visibleWhen: { when: "$context.features.beta == true" }` 时，完整 L0-L4 校验链会返回通过，与正文的条件必填规则不一致。

**影响：** 🟡 中等。缺少显式依赖声明会削弱静态可审计性，并使第三方实现无法区分“合法无依赖”与“表单配置漏写”。

**建议修复：**

1. 保持 `VisibleWhen` 基础 Schema 对非表单上下文可选的表达能力；
2. 在按 Node 树位置遍历的 L2 中执行表单上下文条件必填；
3. 在 L3a 增加同一防线或确保统一入口不会绕过；
4. 增加表单内缺失反例、非表单 `$context.*` 正例和表单内空数组正例；
5. 同步 `docs/06-validation.md` 与 MCP 反例测试。

**与已归档审计的关系：** 0037/V105 修正的是“不能根据 dependencies 是否存在反推上下文”的正文措辞，未增加按 Node 树位置执行条件必填的校验。本条是该规则的实现缺口。

**需要同步的文件：**

- `scripts/validate-l2-components.js`
- `scripts/validate-l3a-expressions.js`（若采用双层防线）
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`

### V119（🟡）L4 将合法 `tagMap` 业务值键误判为 CSS 属性

**位置：** `docs/03-component-registry.md` 的 `ColumnDef.tagMap`；`docs/schemas/component-registry.json` 的 tagMap DSL；`scripts/lint-l4-banned-props.js` 的 `scanNode()` / `checkObjectForBannedKeys()`。

**当前值：** `tagMap` 是“任意数据值到 `{ text, tone }`”的映射。L4 从整个 `node.props` 开始无差别深度递归，并按每一层对象键名匹配 CSS 禁用词。因此合法映射键 `color`、`width` 等会被当成样式属性。

**问题：** `tagMap.color: { text: Color, tone: info }` 通过 L0/L1、L2、L3a，却被 L4 报为 `body.props.columns[0].tagMap.color (color)`。此处 `color` 是后端行数据枚举值，不是 props 字段。

**影响：** 🟡 中等。合法业务数据会被 CI 拒绝，且任何恰好与禁用词同名的枚举值都会触发误报。

**建议修复：**

1. 将 L4 扫描改为结构感知：检查协议 props 字段和状态映射，但跳过明确的数据字典键；
2. 对 `tagMap` 仍扫描每个映射项内部，防止 `{ text, tone, color }` 之类非法字段绕过；
3. 增加 `tagMap.color` 正例与映射项内部 `color` 反例；
4. 同步 `docs/06-validation.md` 对 L4 深度扫描边界的说明。

**与已归档审计的关系：** 0028/V52 修的是 tagMap 映射项必须有 `text` 且禁止额外字段；当前问题是 L4 把映射字典的业务键当成 props，属于不同校验层的误报。

**需要同步的文件：**

- `scripts/lint-l4-banned-props.js`
- `docs/06-validation.md`
- 相关正反例测试或验证夹具

### V120（🔴）协议公开 `$parentRow`，但组件 DSL 无法表达嵌套表格

**位置：** `docs/02-reaction-expression.md` §2 与附录 A；`docs/03-component-registry.md` 的 table 章节；`docs/schemas/component-registry.json` 的 `table.supportsChildren`；`scripts/validate-l2-components.js` 的 children 能力检查。

**当前值：** 表达式规范将 `$parentRow.*` 定义为嵌套表格内 `scope: row` 的正式变量，L3a 还维护了 `tableDepth` 并允许内层表格访问。与此同时，table 的 DSL 明确 `supportsChildren: false`，且列定义、展开行或其他字段中没有承载子 Node 的挂载点。

**问题：** 将内层 table 放入外层 table.children 时，L3a 正例可通过，但 L2 必然报“table 不支持 children”。当前协议不存在另一条能同时通过 L1/L2/L3a 的嵌套表格结构，正式公开的 `$parentRow` 能力不可构造。

**影响：** 🔴 严重。权威表达式规范公开了机器契约无法表达的能力；不同实现可能私自设计展开行结构，造成不可互操作的协议分叉。

**建议修复：** 先通过本轮计划明确二选一决策：

1. 若 v0.2 保留嵌套表格能力，则定义唯一、受控的嵌套挂载结构，同步组件正文、DSL、L2/L3a、Renderer 规范和完整正反例；
2. 若 v0.2 暂不支持嵌套表格，则从当前正式变量白名单、矩阵、L3a 可达分支和 RowAction 映射中移除或明确保留为未来能力，不得继续宣称可用；
3. 无论选择哪条路径，都应回补 0026/V43c 的证据链说明，解释旧 L3a-only 正例为何不足以证明全链路可表达。

**与已归档审计的关系：** 0026/V43c 曾要求“若当前协议结构无法表达则明确保守拒绝”，但完成证据只运行了 L3a 嵌套正例，没有运行 L2。当前全链路证明该历史项没有覆盖可表达性，不是重复报告已修复的非法作用域问题。

**需要同步的文件：**

- `docs/02-reaction-expression.md`
- `docs/03-component-registry.md`
- `docs/07-actions-contract.md`（若影响 `$parentRow` requestMapping）
- `docs/08-renderer-spec.md`
- `docs/schemas/component-registry.json`
- `scripts/validate-l2-components.js`
- `scripts/validate-l3a-expressions.js`
- 相关场景与 MCP 校验测试

### V121（🟡）非链式比较运算符 `contains` 可被 L3a 连续使用

**位置：** `docs/02-reaction-expression.md` §3；`scripts/validate-l3a-expressions.js` 的 `checkTokenSyntax()`。

**当前值：** 正文明确 `contains` 是二元、非链式运算符，不存在 `a contains b contains c`。L3a 状态机只验证操作数与二元运算符交替，因此 `$deps.roles contains 'admin' contains true` 被视为完整 token 序列并通过。

**问题：** 解析器接受正文禁止且没有结合性定义的表达式，把运行时结果留给不同 Renderer 自行解释。

**影响：** 🟡 中等。非法表达式能通过 CI，并可能在不同表达式引擎中产生不同结果或运行时异常。

**建议修复：**

1. 在 L3a 语法阶段拒绝无括号的连续比较运算符，至少覆盖 `contains` 与其他比较运算符组成的比较链；
2. 明确带逻辑运算符分隔的多个比较仍合法；
3. 增加 `contains` 链、混合比较链反例，以及 `a contains b && c == d` 正例；
4. 同步 `docs/06-validation.md` 的表达式语法检查清单。

**与已归档审计的关系：** 0037/V98 收紧了空表达式、畸形变量路径等语法，但未覆盖比较运算符非结合性。本条是新的语法边界。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `docs/06-validation.md`
- MCP 或脚本表达式回归测试

### V122（🟡）Accepted ADR 对 `$context` 生命周期定义冲突

**位置：** `docs/decisions/0003-context-namespace-and-visible-when.md` D1、后果与遗留问题；`docs/decisions/0006-expression-evaluation-order.md` D2；`docs/02-reaction-expression.md` §14。

**当前值：** ADR-0003 规定 `$context` 是实例初始化时一次性注入的非响应式快照，变化只能通过整页重新挂载刷新。后续 ADR-0006 和正式表达式正文却把“上下文刷新”列为触发同一 Renderer 实例下一轮求值的普通事件。

**问题：** 两份 accepted ADR 和正式正文无法同时实现：同实例替换 context 后，按 ADR-0003 应继续使用旧快照或要求重挂载；按 ADR-0006/正文则应进入新求值轮次并读取新 context。

**影响：** 🟡 中等。Renderer 的公共生命周期行为没有唯一答案，权限和功能开关更新可能在不同实现中表现不同。

**建议修复：** 先决定权威口径：

1. 若保留一次性注入，则从 ADR-0006 与正文触发源中删除“上下文刷新”，并明确重挂载会创建全新实例/快照；
2. 若允许实例内刷新，则修订 ADR-0003，定义宿主更新入口、刷新原子性以及 `$context.user` / `$context.features` 的更新粒度；
3. 同步 Renderer 初始化接口和求值调度章节，增加生命周期示例；
4. 更新被决策取代的 ADR 段落状态或追加 superseded-by 说明，避免保留两个并列权威口径。

**与已归档审计的关系：** 历史审计未直接覆盖该冲突；它来自 ADR-0006 合并稳定快照模型时对 ADR-0003 生命周期边界的扩张。

**需要同步的文件：**

- `docs/decisions/0003-context-namespace-and-visible-when.md`
- `docs/decisions/0006-expression-evaluation-order.md`
- `docs/02-reaction-expression.md`
- `docs/08-renderer-spec.md`

### V123（🟡）Renderer 初始化参数总表遗漏正式请求配置字段

**位置：** `docs/08-renderer-spec.md` §2.6 与附录 A。

**当前值：** §2.6 的 `RendererRequestConfig` 正式定义 `baseURL`、`credentials`、`requestTimeout`、`requestInterceptor`、`onAuthFailure`。附录 A 自称“Renderer 初始化参数一览”，但 `RendererOptions` 只包含 `baseURL`，没有组合或展开其余四项。

**问题：** 接入方若以附录的公共 TypeScript 接口为准，会在类型层拒绝正文要求支持的认证、超时与请求拦截配置。

**影响：** 🟡 中等。Renderer 公共初始化契约内部不闭合，容易造成实现只支持 `baseURL` 或各自发明配置嵌套方式。

**建议修复：**

1. 明确 `RendererRequestConfig` 与 `RendererOptions` 的组合方式（顶层展开或命名字段二选一）；
2. 更新附录接口，覆盖全部正式初始化字段及默认值；
3. 交叉核对正文所有 `new Renderer({...})` 示例；
4. 如采用类型复用，避免正文出现两份可独立漂移的字段列表。

**与已归档审计的关系：** 0003/G3 只确认 `baseURL` 属于 Renderer 初始化参数；本条涉及 v0.2.5 后新增的四个请求配置字段，历史未覆盖。

**需要同步的文件：**

- `docs/08-renderer-spec.md`
- 视版本记录策略同步 `docs/CHANGELOG.md` 的 `Unreleased`

### V124（🟡）MCP `protocol.get_doc` 未按完整工具响应执行 20KB 预算

**位置：** `mcp/src/core/protocol-index.ts` 的 `getDoc()`；`mcp/src/tools/utils.ts` 的 `jsonResponse()`；`mcp/tests/search.test.ts`；`docs/mcp/v1-design.md` §3.2；ADR-0007 D10。

**当前值：** `getDoc()` 将 `content` 本身截到 20,480 字节，然后返回 `docId`、`title`、`path`、`matchedSection`、`truncated`、`availableSections`。工具层再对整个对象执行带缩进 JSON 序列化，没有总响应预算。测试只断言 `doc.content <= 20KB`。

**问题：** 实测 `component-registry` 的正文为 20,480 B，但完整 MCP 文本为 21,513 B；`renderer-spec` 的完整 MCP 文本为 22,819 B。设计文档和 ADR 约束的是“单次工具返回”，不是对象中的单个字段。

**影响：** 🟡 中等。大文档仍突破对外声明的上下文预算，章节越多，`availableSections` 带来的超额越明显。

**建议修复：**

1. 在完整 `protocol.get_doc` 工具文本层执行字节预算，或由 `getDoc()` 根据固定序列化格式为元数据预留空间；
2. 保证截断提示、元数据和 `availableSections` 均计入 20KB，并保持 UTF-8 边界安全；
3. 测试实际 `handleGetDoc()` 返回的 `content[0].text` 字节数，而不只测试 `doc.content`；
4. 覆盖章节数量较多、中文多字节和精确 section 不截断场景；
5. 同步细化 MCP 设计文档与 ADR，明确预算对象是完整工具文本。

**与已归档审计的关系：** 0033/V83 增加了 `availableSections`；0037/V108 修复了正文按字符而非 UTF-8 字节截断的问题。V108 checklist 的“总字节不超限”实际只由 `doc.content` 断言覆盖。本条是完整工具响应预算未被测试和实现覆盖的剩余边界。

**需要同步的文件：**

- `mcp/src/core/protocol-index.ts`
- `mcp/src/tools/docs.ts` 或 `mcp/src/tools/utils.ts`（按最终预算层选择）
- `mcp/tests/search.test.ts`
- `docs/mcp/v1-design.md`
- `docs/decisions/0007-mcp-protocol-reader-validator.md`

## 建议处理顺序

1. **先决策 V120 与 V122：** 两项决定 `$parentRow` 可达结构和 `$context` 生命周期，不能先以局部脚本补丁固化未决语义。
2. **再修 V117、V118、V121：** 在已确定的表达式边界上收敛 L2/L3a，并补正反例。
3. **随后修 V119：** 将 L4 改为结构感知扫描，避免业务数据误报且不削弱禁用词防线。
4. **补齐 V123：** 用单一公共类型闭合 Renderer 初始化接口。
5. **最后修 V124：** 在工具边界统一预算完整序列化响应，并运行 MCP 全套验证。

## 防止复发的建议

1. 正文新增表达式变量形态或运算符规则时，必须同时增加通过统一入口的正例和反例，不能只验证单层脚本。
2. 任何“仅在某上下文必填”的规则，都应在能感知 Node 树位置的校验层执行，并在 Schema 说明中明确为何不能只靠 L1。
3. 对 map/dictionary 类型建立明确的“键是数据还是协议字段”元信息，递归 lint 不应只凭键名判断语义。
4. 协议公开嵌套或引用能力前，应提供一个通过 L0-L4 的最小可构造页面作为验收证据。
5. Accepted ADR 被后续决策扩展或替代时，应显式更新旧 ADR 的相关段落或状态，避免并列权威定义冲突。
6. MCP 体积限制测试应落在最终 transport/tool response 边界，而不是中间对象字段。
