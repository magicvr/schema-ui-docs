# 审计报告 0018：responseMapping 语义校验边界 + 文档元数据漂移

## 背景

本轮审计基于 0017 已归档后的再次全量文档复审。0017 中发现的组件注册 DSL 解释边界和 `DataRef.params` 空值规则引用漂移在当前工作区已修复，本轮不重复报告 V18/V19。

本轮重点核对 ADR-0005 `responseMapping`、`node.schema.json` 的结构约束、`04-datasource-contract.md` 的响应映射正文，以及 `06-validation.md` 的校验层级说明。新增发现集中在一个语义校验边界缺口和一个轻量元数据漂移。

## 发现的问题

### V20（🟡）`responseMapping` 的条件必填语义没有落入校验清单

**位置：** `04-datasource-contract.md` §4.1.1；`schemas/node.schema.json` `ResponseMapping`；`06-validation.md` §1 与 §6；`decisions/0005-response-mapping.md` D3/D4

**当前值：** ADR-0005 与 `04-datasource-contract.md` 均明确：

- `responseMapping.list` 对列表类接口必填；未声明映射时默认读取响应体 `list`。
- `responseMapping.total` 在 `table.props.pagination.mode: server` 时必填；未声明映射时默认读取响应体 `total`。

`schemas/node.schema.json` 中 `ResponseMapping` 只能表达结构下限：

```json
{
  "properties": {
    "list": { "type": "string", "pattern": "..." },
    "total": { "type": "string", "pattern": "..." }
  },
  "additionalProperties": false,
  "minProperties": 1
}
```

`06-validation.md` 的人工自检目前只写到：

```md
- [ ] `data.responseMapping.list` / `total` 是否为合法点路径，且映射结果类型符合组件预期？
```

**问题：** `node.schema.json` 缺少组件上下文，无法单独判断“当前 Node 是否为列表类接口”或“当前 table 是否为服务端分页”。这本身合理，但 `06-validation.md` 没有把该差异明确交给 L2/L4 或人工 review。结果是一个只声明 `responseMapping.total`、没有 `list` 的 table 列表接口可以通过 L1 结构校验，也不一定会被现有校验清单拦住。

**影响：** 🟡 中等。协议正文和 ADR 决策是清楚的，但校验文档没有把 Schema 表达力之外的条件必填规则落地，容易让接入方误以为 `minProperties: 1` 就是完整契约。

**建议修复：** 在 `06-validation.md` 中补充语义校验项：列表类接口声明 `responseMapping` 时必须提供 `list`；`table.props.pagination.mode: server` 时必须提供 `total`。同时说明该规则不能只靠 `node.schema.json` L1 完成，需要结合组件类型和 props 在 L2/L4 或人工 review 中检查。

### V21（🟢）0017 修复链触达的文档未同步 `last_updated`

**位置：** `03-component-registry.md` frontmatter；`06-validation.md` frontmatter；0017 V18 修复记录

**当前值：** 0017 checklist 显示 V18 已完成：

- 在 `03-component-registry.md` 中补充 `component-registry.json` 的 DSL 关键字白名单。
- 明确字段级布尔 `required` 与组合约束数组 `required` 的语义差异。
- 在 `06-validation.md` 中说明 L2 组件契约校验必须处理组合约束。

当前两个文档正文均已包含这些修复内容，但 frontmatter 仍为：

```yaml
last_updated: 2026-07-07
```

**问题：** 文档内容已在 2026-07-08 的 0017 修复链中发生协议解释层面的更新，但更新时间元数据未同步。相比正文语义错误，这属于轻量元数据漂移。

**影响：** 🟢 轻微。不会影响协议执行，但会削弱读者判断文档新旧程度的可靠性，也会让后续审计误判哪些文档已吸收 2026-07-08 的修复。

**建议修复：** 将 `03-component-registry.md` 与 `06-validation.md` 的 `last_updated` 同步为 `2026-07-08`。`07-actions-contract.md` 虽仍为 `2026-07-07`，但本轮未找到 0017 修复链直接触达该文件的证据，因此不纳入本项。

## 防止复发的建议

1. **标注 Schema 表达力边界：** 凡是标准 JSON Schema 只能表达字段形状、无法表达组件上下文条件的规则，应在 `06-validation.md` 明确落入 L2/L4 或人工 review。
2. **条件必填规则成对维护：** 正文出现“某场景必填”时，同步检查 Schema、组件 DSL、校验文档是否至少有一处能执行该规则。
3. **修复链同步元数据：** 审计 checklist 中凡涉及正文更新的文档，完成项应包含 frontmatter `last_updated` 核对。
