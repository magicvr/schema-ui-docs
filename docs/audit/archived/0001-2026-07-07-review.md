# Schema-Driven UI 协议审视与建议

> 审视日期：2026-07-07 | 审视版本：v0.1 | 审视人：AI 助手

---

## 一、总体评价

协议整体设计**质量很高**，体现出几个突出的架构品味：

1. **单棵 Node 树的决策非常正确**。避免了 RJSF 式双树对齐的维护噩梦，降低了后端的心智负担。ADR-0001 和 ADR-0002 的理由充分且有说服力。
2. **严格收敛表达式引擎能力**。将 `when` 限定为纯比较+逻辑运算，拒绝算术/函数调用，守住了"配置而非代码"的底线。
3. **分层校验体系（L1-L4）**设计合理，从 CI 静态检查到运行时沙箱，覆盖了不同阶段的错误发现需求。
4. **文档组织清晰**，有总纲、术语表、ADR、场景示例、机器可读 Schema，层次分明。

以下按"设计缺陷 / 未覆盖场景 / 文档问题 / 长远建议"四个维度展开。

## 二、设计缺陷（建议优先修复）

### 2.1 `data.valueField` 的位置不当

**现状：** `valueField` 定义在 `DataRef` 中（`node.schema.json`），但仅在 `statCard` 场景使用。`table` 通过 `columns[].field` 映射、`chart` 通过 `xField`/`yField` 映射——它们都不需要 `valueField`。

**问题：** `valueField` 本质是组件级的"如何从响应中取值"的配置，放在 `data` 中污染了数据源抽象的纯度，且与 `table`/`chart` 的取值方式不一致。

**建议：** 将 `valueField` 从 `DataRef` 移除，改为 `statCard` 的 `props` 字段：

```yaml
# 现状（不推荐）
data:
  source: api
  url: /api/stats/order-count
  valueField: total

# 建议
data:
  source: api
  url: /api/stats/order-count
props:
  valueField: total   # 移到 props，与其他组件的 field 映射方式统一
```

### 2.2 JSON Schema 中 CSS 禁用词的校验逻辑有误

**现状：** `node.schema.json` 使用 `"not": { "required": [...] }` 来禁止 CSS 属性。

**问题：** JSON Schema 的 `required` 语义是"必须全部存在"，因此 `not: { required: [margin, padding] }` 的含义是"不允许 margin 和 padding **同时存在**"——单独出现一个仍然合法。这与设计意图"任何一个 CSS 属性都不允许出现"不符。

**建议：** 改为在 `props` 上使用 `additionalProperties` 配合白名单，或改为 `"not": { "anyOf": [{ "required": ["margin"] }, { "required": ["padding"] }, ...] }`。最可靠的做法是将 L4 校验放在自定义 lint 脚本中，不依赖 JSON Schema 的 `not`/`required` 组合。

### 2.3 缺少节点级条件渲染能力

**现状：** 可见性控制仅存在于表单字段的 `reactions` 机制中。非表单节点（如 `section`、`statCard`、`chart`）没有条件显隐能力。

**问题：** 以下常见需求无法表达：
- 根据用户角色隐藏某个统计卡片
- 根据数据是否为空决定是否展示某个图表区域
- A/B 测试中按条件展示不同的页面区块

**建议：** 为 Node 增加一个可选的 `visible` 字段，支持表达式求值：

```yaml
# 方案A：简单布尔/表达式
visible: "$context.userRole == 'admin'"
visible: "$data.total > 0"

# 方案B：与 reactions 统一的 when 语法
visibleWhen: "$deps.someField == 'yes'"
```

这需要在表达式中引入一个新的命名空间 `$context`（页面级上下文，如用户信息、权限等）和 `$data`（当前节点的数据）。

### 2.4 `tabs` 组件的 children 匹配机制未定义

**现状：** `03-component-registry.md` 描述 `tabs` 的 `items` 为 `array<{key,label}>`，"实际内容通过 children 中同 key 的子节点匹配"。但 Node 结构中没有 `key` 字段。

**问题：** 无法从协议层面知道一个 child Node 对应哪个 tab。以下写法不合法（Node 只有 5 个字段，不能加 `key`）：

```yaml
type: tabs
props:
  items:
    - { key: tab1, label: 概览 }
    - { key: tab2, label: 详情 }
children:
  - type: section    # 这个对应哪个 tab？无法表达
    key: tab1         # ❌ Node 不允许自定义字段
```

**建议：** 两种方案：
- **方案 A（推荐）：** 在 `tabs.props.items` 中直接内嵌 children，不依赖顶层 `children`。
- **方案 B：** 为 Node 增加可选的 `key` 字段（这也会解决其他场景的标识需求，见 §3.1）。

### 2.5 `actions` 定义不够完整

**现状：** `actions` 仅在场景示例中出现一次（`createOrder`），协议正文未给出完整的 ActionDef 结构。

**问题：** 以下问题无明确答案：
- 除了 `type: request`，还支持哪些 action 类型？（导航跳转？打开弹窗？执行自定义函数？）
- 请求成功/失败后的行为如何定义？（跳转页面？Toast 提示？刷新数据源？）
- `request` action 的完整参数结构是什么？（headers？body 映射？）

**建议：** 补充一份 `actions` 的完整契约文档，至少覆盖：
```yaml
actions:
  someAction:
    type: request | navigate | modal | custom
    # request 类型：
    method: POST
    url: /api/xxx
    bodyMapping:         # 请求体字段映射
      userName: "$form.customerName"
    onSuccess:
      toast: 操作成功
      navigate: /orders   # 或 reload: true / closeModal: true
    onError:
      toast: "$response.message"
```

## 三、未覆盖的常见场景

### 3.1 Node 缺少 `id` / `key` 字段

**影响场景：**
- 前端需要定位到某个节点（如"滚动到第三节"、埋点上报）
- `tabs` 的 children 匹配（见 §2.4）
- 增量更新 / 局部 patch 场景

**建议：** 为 Node 增加可选的 `id: string` 字段，作为页面内的唯一标识。

### 3.2 缺少权限/访问控制表达

**影响场景：** 中后台系统的刚需——不同角色看到不同内容、不同操作按钮。

**建议：** 在 Node 级别和 action 级别支持权限声明：
```yaml
permissions:
  requireAny: [admin, manager]    # 满足任一角色
  # 或 requireAll: [order:read, order:write]
```

前端 Renderer 在渲染前根据当前用户权限过滤节点和操作按钮。

### 3.3 缺少国际化（i18n）支持

**现状：** 所有 `label`、`title`、`content` 均为硬编码字符串。

**问题：** 多语言场景下，后端需要为每种语言生成一份完整配置，维护成本极高。

**建议：** 支持 i18n key 引用：
```yaml
props:
  label: "客户名称"            # 静态文案（保持兼容）
  labelKey: "field.customerName"  # i18n key（与 label 二选一）
```

前端 Renderer 优先使用 `labelKey` 查 i18n 词典，fallback 到 `label`。

### 3.4 缺少下拉选项的动态加载（远程搜索/级联）

**现状：** `select` 的 `options` 仅支持静态数组。

**问题：** 无法表达"下拉选项从 API 动态加载"（如省市区级联、用户搜索）。

**建议：** 为 `select` 增加数据源绑定能力：
```yaml
type: select
props:
  field: city
  label: 城市
  optionsSource:           # 新增：远程选项源
    url: /api/cities
    params:
      province: "$deps.province"
    labelField: name
    valueField: id
    searchable: true       # 支持远程搜索
```

### 3.5 缺少空态/加载态/错误态的组件级定制

**现状：** 错误处理统一按 HTTP 状态码映射固定文案（04 文档 §5），加载和空态未提及。

**问题：** 不同组件可能需要不同的空态文案或加载占位样式。

**建议：** 为 Node 增加可选的 `states` 配置：
```yaml
states:
  loading:
    text: 加载中...
  empty:
    text: 暂无数据
    illustration: empty_box   # 语义级插画标识
  error:
    fallbackText: 数据加载失败
```

### 3.6 缺少联动表达式中 `$deps` 的跨层级引用

**现状：** `reactions` 仅支持同表单内的兄弟/祖先字段。

**问题：** 无法表达"表格中某行的操作按钮根据该行数据状态变化"（行级联动）。

**建议：** 在表格场景中增加行级作用域的 `$row` 变量（但这需要谨慎设计，避免与表单的 `$deps` 混淆）。

### 3.7 缺少 `span` 在非 `section` 节点上的支持

**现状：** `span` 仅在 `section` 的 props 中定义。

**问题：** 如果想把一个 `statCard` 直接放在 `grid` 下并占两栏，必须包一层 `section`，产生不必要的嵌套。

**建议：** 将 `span` 提升为 Node 级别的通用 props（或至少支持所有 `grid` 的直接子节点）。

## 四、文档与规范一致性问题

### 4.1 `datasources` + `data.ref` 无示例

`01-node-protocol.md` 和 `04-datasource-contract.md` 都提到了 `data.source: ref`，但三个场景示例中一个都没有使用。建议在 `grid-dashboard.md` 中补充一个 `ref` 引用示例。

### 4.2 协议版本号不在配置中

当前 YAML 配置中不包含协议版本号（`meta` 只有 `pageId`/`title`/`description`）。如果协议升级，Renderer 无法判断配置是哪个版本编写的。建议在 `meta` 中增加 `protocolVersion` 字段。

### 4.3 `text` 组件的 `data` 说明矛盾

`03-component-registry.md` 中 `text` 标注 `supportsData: false`（注释说"如需动态文本用 `data.value`"），但 `supportsData: false` 意味着 Renderer 不会为它处理数据。这个矛盾需要澄清——要么改为 `supportsData: true` 并规范取值方式，要么明确 `text` 只能通过 `reactions` 动态改值。

### 4.4 `input`/`inputNumber` 缺少 `placeholder`、`description`、`tooltip`

这三个是表单字段的高频需求，当前均不支持。建议补充：
- `placeholder: string`
- `description: string`（字段下方的帮助文案）
- `tooltip: string`（标签旁的提示图标）

## 五、长远架构建议

### 5.1 考虑引入 `$context` 命名空间

随着权限（§3.2）、i18n（§3.3）、条件渲染（§2.3）的加入，需要一种统一的"页面级上下文"机制。建议定义：

| 变量 | 来源 | 示例 |
|---|---|---|
| `$context.user` | Renderer 注入 | `$context.user.role` |
| `$context.locale` | Renderer 注入 | `$context.locale` |
| `$context.features` | 运行时特性开关 | `$context.features.newDashboard` |
| `$data` | 当前节点 data 返回值 | `$data.total` |

这些变量可用于 `visibleWhen`、`permissions` 等新字段的表达式求值。

### 5.2 表达式引擎的渐进式演进策略

当前表达式引擎不支持算术和函数调用是**正确的设计决策**。但如果未来确实遇到用不同 Node 结构表达过于冗长的场景（如"金额 = 单价 × 数量"的只读计算结果展示），建议的演进路径是：

1. **不扩展 `when` 语法**——保持其纯布尔表达式的定位。
2. 新增 `compute` 字段（独立于 `reactions`），用沙箱化的受限表达式做纯数据计算。
3. 始终拒绝函数调用能力——这是不可逾越的红线。

### 5.3 `component-registry.json` 的自动化生成

文档已提到"建议由 CI 从前端组件的 TypeScript Props 类型定义自动生成本文件"。建议将此作为 v0.2 的高优先级事项落地——手动维护的注册表必然会与代码实现漂移。

### 5.4 考虑增加 `deprecated` / `since` 版本标注

当组件 props 发生变更时（如字段改名、新增必填字段），需要一个机制让 Renderer 在开发环境给出警告。建议在 `component-registry.json` 的 props 定义中增加：

```json
"title": {
  "type": "string",
  "required": false,
  "deprecated": "v0.3",
  "deprecatedMessage": "请改用 heading 字段",
  "since": "v0.1"
}
```

## 七、总结

v0.1 作为初版协议，核心骨架（单树 Node、语义 props、严格表达式收敛、分层校验）设计得非常扎实。两个 ADR 论证充分，给未来的协议演化打下了良好的决策基础。

主要风险集中在：部分组件（`tabs`、`actions`）的契约不够完整，可能导致前端 Renderer 实现时各团队自行补充约定，产生方言分化。以及 JSON Schema 中 `not`/`required` 的误用可能在 CI 校验中产生漏报。

建议在 v0.2 优先修复 P0/P1 项，补充 P2 的能力缺口，即可覆盖中后台系统中 90% 以上的常见页面需求。
