## 二、新发现的问题

以下问题按严重度降序排列。

---

### M1. [🔴] `action.schema.json` 结构性缺陷——`additionalProperties: false` 导致所有合法 action 校验不通过

**位置：** `docs/schemas/action.schema.json`，顶层 `additionalProperties: false`。

**当前状态：**
```json
{
  "type": "object",
  "properties": {
    "type": { "type": "string", "enum": ["request", "navigate", "modal", "custom"] }
  },
  "required": ["type"],
  "allOf": [ /* if/then 分支中各字段在 then.properties 中声明 */ ],
  "additionalProperties": false
}
```

**问题：** JSON Schema 的 `additionalProperties` 只认**同一层级 `properties` 里声明过的字段**，不识别 `allOf`/`if-then` 分支的 `then.properties` 中声明的字段。只要走 `additionalProperties: false` 这条路径，`method`/`url`/`bodyMapping`/`onSuccess`/`onError`/`modalId`/`content`/`handler` 等字段永远会被判定为"多余字段"而拒绝。

**ajv 实测结果（以 `05-scenarios/form-with-reactions.md` 中的 `createOrder` 为例）：**
```
additionalProperty: 'method'
additionalProperty: 'url'
additionalProperty: 'onSuccess'
additionalProperty: 'onError'
```

**影响：** 🔴 **严重**。如果按 `06-validation.md` 描述的 CI 流程（L0 `page.schema.json` → 引用 `action.schema.json`）接入，协议自己的文档示例都会被 CI 拒绝。这个坑极其隐蔽——前四轮审计全部是"读文档比对语义"，没有任何人真正拿 ajv 跑过 Schema 校验。

**修复建议：** 将 `method`/`url`/`bodyMapping`/`onSuccess`/`onError`/`modalId`/`content`/`handler` 提到顶层 `properties` 中统一声明（各自标为可选），`allOf`/`if-then` 只负责按 `type` 做 `required` 约束，不再依赖 `then.properties` 补字段声明。

### M2. [🔴] `01-node-protocol.md §5` "最小可用页面"示例缺少 `protocolVersion`

**位置：** `docs/01-node-protocol.md §5`，"完整示例（最小可用页面）"。

**当前内容：**
```yaml
meta:
  pageId: hello_page
  title: 示例页面
body:
  type: section
  props:
    title: 你好
  children:
    - type: text
      props:
        content: 这是一个最小示例
```

**问题：** 缺少 `meta.protocolVersion`。而 `page.schema.json` 里 `meta.required` 明确是 `["pageId", "title", "protocolVersion"]`（对应文档 §2 "必填（since 0.2）"的声明）。ajv 实测报错：`must have required property 'protocolVersion'`。

**影响：** 🔴 一份标榜"最小可用"的示例，连自己协议的必填字段都没给全，容易被读者/AI 助手直接抄成一个不合法配置。

**修复建议：** 给该示例补上 `protocolVersion: "0.2"`。

### M4. [🟢] `07-actions-contract.md ` 示例中出现了未定义的字符串插值语法

**位置：** `docs/07-actions-contract.md §5.1`，`confirmRefund` 示例。

**当前内容：**
```yaml
content:
  type: text
  props:
    content: 确认对订单 #{$deps.orderId} 发起退款？
```

**问题：** 这里的 `#{$deps.orderId}` 是**字符串插值语法**，但：
1. `02-reaction-expression.md` 是 `$deps.*` 唯一的权威语法定义来源，里面只定义了 `$deps` 在 `when` 表达式和 `params` 值替换里的用法，**完全没有提到可以在普通文案字符串里做插值**；
2. `03-component-registry.md` 里 `text.props.content` 就是一个普通 `string`，没有说它支持模板插值；
3. 更关键的是，这个 `modal` 的 `content` 是一个独立的 `form` Node，该 Node 树上下文里**没有任何地方声明 `dependencies: [orderId]`**，而 §3.10 明确要求"`$deps.*` 变量必须在对应位置的 `dependencies` 数组中显式声明"。这条示例既没写 `dependencies`，也没有 `when`/`reactions` 这类载体去承载这个声明。

**影响：** 🟢 这是一个"孤例特性"——只在这一个例子里出现，前后都没有配套的语法规则、白名单说明或校验规则来支撑它。AI 助手或后端开发者如果照抄这个例子，很可能会做出协议实际不支持、Renderer 也无法正确解析的配置。

**修复建议（二选一）：**
- **方案 A：** 在 `02-reaction-expression.md` 里正式补一节"文案插值语法"（定义插值边界、允许出现的位置、插值语法格式、以及和 `dependencies` 的关系），并在 `03-component-registry.md` 的 `text.props.content` 说明中标注支持插值。
- **方案 B（保守）：** 把这个示例改回普通静态文案，去掉未定义语法。
