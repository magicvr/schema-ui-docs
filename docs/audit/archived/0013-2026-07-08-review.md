# 审计报告 0013：版本/since 标注漂移复发与残留修复

## 背景

前 12 轮审计已对文档体系进行了系统性修复。本轮审计基于用户在通读核心协议文档、Schema 和 ADR 后的交叉检查，发现 **3 个此前未被记录的实质性问题**，且均为同一类"版本/since 标注漂移"缺陷在新一轮更新后的 **复发或残留**。

## 审视范围

| 范围 | 说明 |
|---|---|
| 版本声明一致性 | `00-overview.md` §5 与 `CHANGELOG.md` 的对照 |
| 文档内部标注自洽性 | `07-actions-contract.md` §5 代码块注释 vs 正文 vs `action.schema.json` |
| Schema 与正文同步 | `schemas/component-registry.json` 中 `since` 标注与 `03-component-registry.md` 正文的对照 |

---

## 发现的问题

### V1（🔴）`00-overview.md` §5 版本声明已过期，与 CHANGELOG 脱节

**位置：** `00-overview.md` §5 版本与稳定性

**当前值：**
> 当前协议版本：`v0.2`（最新补丁版本 **`v0.2.3`**，见 [CHANGELOG.md](../../CHANGELOG.md)）。

**问题：** `CHANGELOG.md` 已记录 **v0.2.4**（审计 0012 产出的 ADR-0005 `responseMapping` 标准化 + ADR-0006 表达式求值时序模型补丁）。`01-node-protocol.md`、`02-reaction-expression.md`、`04-datasource-contract.md`、`08-renderer-spec.md` 正文均已带有 `(since 0.2.4)` 的新增内容，唯独 `00-overview.md` 的版本声明没有同步更新。`last_updated` 也仍停留在 `2026-07-07`。

**严重性：** 🔴 关键。版本声明是文档的"门面"和读者第一印象，与 CHANGELOG 矛盾会直接降低读者对整个文档体系的信任度。

**根因分析：** 这是审计 **0006 的 M3**（"`00-overview.md` §5 仍写 v0.2.1，但 CHANGELOG 有 v0.2.2"）的 **同类缺陷复发**。0012 审计更新了 4 篇正文文档，却漏掉了本该同步的总纲文档。说明当前流程缺乏一个"发布前版本声明核对"的自动化或人工检查节点。

---

### V2（🟡）`modal.content` 的 `since` 标注在文档内部自相矛盾，且与 Schema 不一致

**位置：** `07-actions-contract.md` §5、`schemas/action.schema.json`

**当前三处值：**

| 位置 | 当前值 |
|---|---|
| `07-actions-contract.md` §5 代码块注释（L65） | `content: Node     # 【可选】直接描述弹窗内容（since 0.2），见下方说明` |
| `07-actions-contract.md` §5.1 正文（L71） | `` `content`（since 0.2.1）`` 提供弹窗的**内容** |
| `schemas/action.schema.json` `content.description`（L42） | `"【可选，since 0.2】弹窗内容，一个完整的 Node（如 form/text/table 等），与 modalId 可共存"` |

**问题：** 三处中有两处说 "0.2"，一处说 "0.2.1"。`CHANGELOG.md` v0.2.1 明确记录"弹窗内容 Node 化：`actions[].type: modal` 新增可选 `content: Node` 字段"——**"0.2.1" 才是正确答案**。代码块注释和 Schema 描述需要修正。

**严重性：** 🟡 中等。不影响协议行为，但损害文档的可信度和可追溯性。

---

### V3（🟡）`component-registry.json` 中 table 列/行操作的 `since` 标注系统性落后一个版本（残留未修复）

**位置：** `schemas/component-registry.json` — `columns[].*` 与 `actions[].*` 的 `description` 字段

**当前值：**

| 字段 | Schema 行号 | 当前值 | 应为 |
|---|---|---|---|
| `columns[].visibleWhen` | L164 | `(since 0.2)` | `(since 0.2.1)` |
| `columns[].reactions` | L165 | `(since 0.2)` | `(since 0.2.1)` |
| `columns[].permissions` | L166 | `(since 0.2)` | `(since 0.2.1)` |
| `actions[].visibleField` | L190 | `(since 0.2)` | `(since 0.2.1)` |
| `actions[].visibleWhen` | L192 | `(since 0.2)` | `(since 0.2.1)` |
| `actions[].reactions` | L193 | `(since 0.2)` | `(since 0.2.1)` |
| `actions[].permissions` | L194 | `(since 0.2)` | `(since 0.2.1)` |

**问题：** `03-component-registry.md` 正文在审计 **0006 的 M6** 中已将 `RowAction`/`ColumnDef` 的 `since` 标注从 `(since 0.2)` 更正为 `(since 0.2.1)`（这批能力由 ADR-0003 和 ADR-0004 在 v0.2.1 合并，CHANGELOG 可核实），但机器可读的 `schemas/component-registry.json` 中对应的 7 处 description 全部未同步更新，仍然写着 `(since 0.2)`。

**严重性：** 🟡 中等。不影响协议行为，但导致机器可读 Schema 与正文文档不一致，对依赖 Schema 的自动化工具和 IDE 提示造成误导。

**根因分析：** 这是审计 **0006 的 M6**（"since 粒度模糊"）**未被彻底修复的残留**——当时的修复只改了 `03-component-registry.md` 正文表格中的 5 处，没有同步更新机器可读 Schema 里的对应描述，导致文档和 Schema 从 v0.2.1 起就一直不一致。7 轮后续审计（0007–0012）均未发现此问题。
