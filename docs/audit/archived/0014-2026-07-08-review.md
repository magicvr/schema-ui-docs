# 审计报告 0014：审计证据链自洽性 + 被排查盲区遗漏的版本声明问题 + Schema 格式本质矛盾 + ADR 收尾遗漏

## 背景

前 13 轮审计已对文档体系进行了系统性修复，尤其对"版本/since 标注一致性"（0006、0013）做了多轮深度扫描。本轮审计基于用户在通读全量文档、Schema、CHANGELOG、ADR 以及全部 13 份归档审计记录后的交叉核对，发现 **5 个此前从未被触及的新问题**。

其中 V4 和 V5 与过去 13 轮的核心关注点（版本声明一致性、审计流程自洽性）是**同一类缺陷**，但出现在此前排查方法覆盖不到的地方；V6 触及了 `component-registry.json` 的**格式本质**问题，此前所有审计都在改这个文件的字段内容，但从未质疑其 Schema 语法本身是否自洽。

### V5（🔴）`08-renderer-spec.md` `applies_to` 版本声明滞后两个补丁版本

**位置：** `08-renderer-spec.md` frontmatter

**当前值：**
```yaml
applies_to: schema-ui-protocol v0.2.1
```

**问题：** 这是审计 0006（M5）当年刻意设置的，理由是"该文档在 v0.2.1 才首次引入"。但 v0.2.4 之后，该文档正文已新增：
- **§2.5** `responseMapping` 应用阶段 — 标注 `(since 0.2.4)`
- **§5.3** 表达式批量提交与循环保护 — 标注 `(since 0.2.4)`

CHANGELOG D4 确认："`08-renderer-spec.md` 增加 `responseMapping` 应用阶段、表达式批量提交与循环保护要求。"

**严重性：** 🔴 关键。`applies_to` 落后正文内容两个补丁版本——这是过去 13 轮审计反复揪出来的"版本声明与实际内容脱节"模式（0006-M3、0013-V1），只是因为 `08` 是全项目唯一一个把 `applies_to` 写成 patch 级别（`v0.2.1`）而非其余文档统一使用的 `v0.2` 裸版本号的文档，历次"检查是否都写 v0.2"的排查方式天然会跳过它。

**根因分析：** 这是一个**排查方法论盲区**。0013 的 V1 专门检查了"版本声明一致性"，但只覆盖了 `00-overview.md` §5 的版本字符串，没有覆盖各文档 frontmatter 的 `applies_to` 字段。而且 `08` 的 `applies_to` 是唯一一个写成 `v0.2.x` patch 格式的，其余文档（01/02/04/06/07）都是 `v0.2`——所以 grep 搜 `applies_to.*v0.2$` 不会命中它，搜 `applies_to.*v0.2.1` 又不会与其他文档比较发现异常。

### V6（🟡）`schemas/component-registry.json` Schema 方言声明与自身语法自相矛盾

**位置：** `docs/schemas/component-registry.json`

**当前值：**
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "https://internal/schema-ui/component-registry.json",
  ...
  "components": {
    "grid": {
      ...
      "props": {
        "columns": { "type": "number", "required": true },
        "additionalProperties": false
      }
    }
  }
}
```

**问题分析：** 该文件自称是 JSON Schema draft-07，并且确实在内部使用了合法的 JSON Schema `$ref` 语法（如 `"content": { "$ref": "node.schema.json" }`），但它的**整体结构完全不是合法的 draft-07 文档**：

| 特征 | 标准 JSON Schema 做法 | `component-registry.json` 实际做法 |
|------|----------------------|-----------------------------------|
| 顶层结构 | `properties`/`definitions`/`$defs` 等 schema 词汇 | 自造键 `"components"` |
| 必填性 | `"required": ["fieldA", "fieldB"]` 数组语法 | `"required": true` 内联布尔值 |
| `additionalProperties` | 模式关键字，位于 `"type": "object"` 同级 | 作为 `props` 对象内的普通兄弟字段 |

**对比其他四个 Schema 文件：** `page.schema.json`、`node.schema.json`、`action.schema.json`、`reaction.schema.json` 都严格遵循 JSON Schema 语法（使用 `properties`/`required` 数组/正确的 `additionalProperties` 位置）。`component-registry.json` 是**唯一**一个语法不合规的文件。

**实际影响：** 如果用 `ajv` 或其他标准 JSON Schema 校验器处理这个文件，`components` 和内联 `required` 会被当成无意义的普通属性直接忽略，不会产生该文件"看起来"应有的约束效果。文档中反复称该文件是"机器可读的权威版本"、"CI 从组件的 TypeScript Props 类型定义自动生成"，但它的格式本质是一种自定义 DSL，只是套了个 JSON Schema 的壳子。

**严重性：** 🟡 中等。不影响协议运行时的行为（Renderer 有各自的组件注册表实现），但文档对该文件的性质描述与实际格式能力之间存在本质落差。对于依赖标准 JSON Schema 校验器的自动化流程，这会导致虚假的安全感。

**根因：** 这个文件采用 JSON Schema 的语法外壳（`$schema` + `$id` + `title` + `description`）和局部 `$ref` 引用，但整体结构早在项目早期就被设计成了"组件目录"而非"组件 Schema"。过去 13 轮审计有大量条目在修正这个文件的字段内容（since 标注、描述措辞、字段增删），但从未有人质疑过它的**格式本身**是否自洽。

### V7（🟡）ADR-0005 / ADR-0006 状态未随合入完成更新

**位置：** `decisions/0005-response-mapping.md`、`decisions/0006-expression-evaluation-order.md`

**当前值：**
- ADR-0005：`已接受(Accepted)—— 需合并入 01-node-protocol.md、04-datasource-contract.md、08-renderer-spec.md 与 schemas/node.schema.json。`
- ADR-0006：`已接受(Accepted)—— 需合并入 01-node-protocol.md、02-reaction-expression.md 与 08-renderer-spec.md。`

**问题：** "需合并入"暗示尚未落地。但根据 CHANGELOG v0.2.4（D1-D4）和对正文的核对：
- ADR-0005 的 `responseMapping` 已在 `01-node-protocol.md` §3.3（DataRef 字段表）、`04-datasource-contract.md` §7、`08-renderer-spec.md` §2.5、`node.schema.json` 中落地。
- ADR-0006 的表达式求值时序模型已在 `01-node-protocol.md` §3.10（重新定义）、`02-reaction-expression.md` §9（新增）、`08-renderer-spec.md` §5.3 中落地。

**对比：** ADR-0003 和 ADR-0004 的状态行早已正确更新为`已合并入`。0005/0006 的"需合并入"会让读者误以为这两份 ADR 还停留在"已批准但未落地"阶段。

**严重性：** 🟡 中等。不影响协议行为，但损害 ADR 体系的可信度和信息时效性。读者查找 ADR 时无法快速区分"已合入"和"待合入"的决策。

**根因分析：** ADR 合入后的状态更新缺乏标准化的收尾步骤。从 0003/0004 的例子看，正确的做法是：ADR 内容全部合入正文后，回到 ADR 自身将"需合并入"改为"已合并入"。0005/0006 在 0012 审计的合入流程中遗漏了这个收尾步骤。
