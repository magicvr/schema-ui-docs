# 审计报告 0033：upload.actionRef 能力声明提示缺失 + permissions 文档表述残留 + MCP get_doc 示例/截断偏差 + v1-design 状态漂移

## 背景

本轮审计基于 0032 已归档后的再次全量文档、Schema、校验脚本与 MCP 实现复审。0032 中发现的 `source:ref` 本地 `responseMapping` 校验缺口、RowAction `permissions` 遍历、`$context` 白名单执行、`internalError` 结构化、category 示例同步、v1 实施计划状态和 `07` §3.1 示例不可执行在当前工作区已确认修复，本轮不重复报告 V71-V78。

本轮重点交叉核对：`upload.props.actionRef` 的 `actions.upload` 能力要求是否在入口文档/DSL 中完整提示，`permissions` 的 `scope` 字段是否被错误地描述为可通过 `scope` 声明求值作用域，扩展权限动作键声明机制是否存在文档与实现的缺口，以及 MCP `get_doc` 的示例 docId、截断建议和设计文档状态是否与实现同步。

## 发现的问题

### V79（🟡）`upload.props.actionRef` 的 `actions.upload` 能力声明提示未同步到组件文档/DSL/action 入口

**位置：**
- `docs/03-component-registry.md` upload 表格 `actionRef` 行
- `docs/schemas/component-registry.json` upload `actionRef` description
- `docs/07-actions-contract.md` §7 upload action 说明
- 对照：`docs/01-node-protocol.md` §2 `meta.requiredCapabilities` 已完整说明；`scripts/validate-l2-components.js` `validateRequiredCapabilities()` 已对 upload action 和 `upload.props.actionRef` 正确执行校验

**当前值：**
- `03-component-registry.md` upload 组件的 `actionRef` 仅写"引用顶层 actions 中 type=upload 的动作 id"，无 `requiredCapabilities` 提示。
- `component-registry.json` upload `actionRef` description 同样只写引用关系。
- `07-actions-contract.md` §7 upload 类型说明未提醒使用 `upload.props.actionRef` 时时需声明 `actions.upload`。

**对照：** 同一文中 `RowAction.actionRef` 的文档已正确写明"使用时页面必须声明 `meta.requiredCapabilities: [actions.row.request]`"。

**问题：** L2 校验器实际已对 `upload` action 和 `upload.props.actionRef` 执行 `actions.upload` 能力声明检查。但入口级文档（组件注册表 DSL、upload 组件表格、upload action 说明）没有同步提示，生成器/AI 助手以这些文档为准时可能产出被校验拒绝的页面。

**影响：** 🟡 中等。不会影响校验脚本行为，但组件注册表是 AI/工具读取 `upload` 能力的入口，缺少能力声明提示会诱导生成缺少 `requiredCapabilities` 的页面。RowAction `actionRef` 已经写了对等提示，形成前后不对称。

**建议修复：**
1. 在 `03-component-registry.md` upload `actionRef` 行追加"使用时页面必须声明 `meta.requiredCapabilities: [actions.upload]`"。
2. 在 `component-registry.json` upload `actionRef` 的 description 中同步追加该提示。
3. 在 `07-actions-contract.md` §7 开头的说明中追加"使用 `upload.props.actionRef` 时页面必须声明 `actions.upload`"。

**需要同步的文件：**
- `docs/03-component-registry.md`
- `docs/schemas/component-registry.json`
- `docs/07-actions-contract.md`

### V80（🟡）`03-component-registry.md` 作用域说明仍将 `permissions` 描述为可通过 `scope` 声明作用域，与机器 Schema 和 L3a 约束矛盾

**位置：**
- `docs/03-component-registry.md`「作用域说明（since 0.2.1）」段落
- `docs/schemas/node.schema.json` `Permissions` 定义（无 `scope` 字段）
- `docs/decisions/0004-row-level-scope.md` D2：`reactions`/`visibleWhen`/`permissions` 三处均可通过 `scope` 声明求值作用域
- 对照：`scripts/validate-l3a-expressions.js` `PERM_CONTEXT_ONLY` 规则强制 `permissions.*` 仅允许 `$context.*`，且 `node.permissions` 扫描时固定 `scope: 'form'`

**当前值：** `03-component-registry.md` 作用域说明写：
> 列/操作内的 `visibleWhen`/`reactions`/`permissions` 可通过 `scope` 属性声明求值作用域

但 `node.schema.json` 中 `Permissions` 没有 `scope` 属性——它是 map（`view`/`edit`/`delete` + 扩展键），不是 `VisibleWhen`/`Reaction` 那种带 `scope` 字段的对象。

L3a 对 `permissions.*` 的校验固定 `location: 'permission'`，不参与 `scope: row/form` 判定——所有 permissions 表达式都按 `$context.*` 白名单强制执行。

**问题：** 文档声明说 `permissions` 可通过 `scope` 控制求值作用域，但机器 Schema 和校验脚本均不支持此能力。这是 ADR-0004 D2 的遗留措辞——当时将 `permissions` 与 `reactions`/`visibleWhen` 并列描述为"可通过 `scope` 声明"，但后来的 `$context.*` 白名单和 V72-V74 修复已把 `permissions` 固定在 `$context.*` 范围内，不再需要 `scope`。

**影响：** 🟡 中等。不会破坏校验，但读者可能误以为 `permissions` 也能设置 `scope: row` 并读取 `$row.*`——实际 L3a 会拒绝违反 `PERM_CONTEXT_ONLY` 的权限表达式。

**建议修复：**
1. 将 `03-component-registry.md` 的作用域说明从"`visibleWhen`/`reactions`/`permissions`"改为"`visibleWhen`/`reactions`"。
2. 在 ADR-0004 D2 中补充决策更新：`permissions` 的作用域固定为 `$context.*`，不再参与 `scope` 作用域语义。

**需要同步的文件：**
- `docs/03-component-registry.md`
- `docs/decisions/0004-row-level-scope.md`

### V81（🟡）权限扩展动作键声明机制仅存在于文档要求，组件注册表无对应表达能力和校验

**位置：**
- `docs/01-node-protocol.md` §3.9：接入方可在 `component-registry.json` 的组件契约中按需扩展自定义动作键（如 `approve`/`export`）
- `docs/schemas/node.schema.json` `Permissions.additionalProperties` description："扩展动作键（如 approve/export），需在 component-registry.json 的组件契约中声明"
- `docs/decisions/0003-context-namespace-and-visible-when.md` D5：同上说明
- `docs/schemas/component-registry.json`：`table.props.columns.items.properties.permissions` 与 `table.props.actions.items.properties.permissions` 均仅 `$ref` 到通用 `Permissions`，无 `allowedPermissionKeys` 或等价声明字段
- `scripts/validate-l2-components.js` `validatePermissions()`：仅检查 `object` 类型、非空和值类型为 `string`，不检查动作键是否在白名单内

**当前值：**
- 正文要求自定义权限键在组件契约中声明。
- DSL 既无声明入口，L2 也不执行白名单检查。
- 实测 `table.props.actions[].permissions.approve: "$context.user.roles contains 'admin'"` 全链路 `passed: true`。

**问题：** "在组件契约中声明"这个规则在文档/Schema 中存在，但没有工具可以实际执行它。接入方使用 `approve`/`export` 等扩展键时，无法判断是否已声明、从哪个文件中读取声明、如何 CI 校验。

**影响：** 🟡 中等。标准 `view`/`edit`/`delete` 不受影响；扩展键的实际使用频率在当前版本中较低。但如果项目使用自定义权限键并遵循文档要求尝试"在组件契约中声明"，会发现没有入口。

**建议修复：**
1. 决定扩展键策略。选项 A：删除"需在组件契约中声明"的正文要求，放宽为"协议层不限制权限键名，由接入方自行 CI/Review 约束"。选项 B：在 `component-registry.json` 的组件级声明中增加 `allowedPermissionKeys` 字段。
2. 若选 A，同步修改 `01-node-protocol.md`、`node.schema.json` 和 ADR-0003 中的对应措辞。
3. 若选 B，补充 DSL 字段、L2 校验和回归测试。

**需要同步的文件：**
- `docs/01-node-protocol.md`
- `docs/schemas/node.schema.json`
- `docs/decisions/0003-context-namespace-and-visible-when.md`
- 可选：`docs/schemas/component-registry.json`
- 可选：`scripts/validate-l2-components.js`

### V82（🟡）`protocol.get_doc` 示例 `docId` 与实际白名单不一致，调用即失败

**位置：**
- `docs/mcp/v1-design.md` §3.2 输入示例：`"docId": "02-reaction-expression"`
- `mcp/src/core/protocol-index.ts`：白名单实际为 `reaction-expression`

**当前值：** 设计文档示例写 `"docId": "02-reaction-expression"`，但白名单 docId 只有 `reaction-expression`。同一设计文档 §5.1 的白名单表也正确写 `reaction-expression`。

**问题：** 按示例参数调用 `protocol.get_doc` 会返回"未知 docId"。白名单表与示例自相矛盾。

**影响：** 🟡 中等。文档是 AI 客户端理解接口的主要来源；示例不可用会让接入方多一次试错，也可能削弱 AI 工具对 MCP 接口的信任。

**建议修复：**
1. 将示例中的 `docId` 改为 `"reaction-expression"`，与 §5.1 白名单表一致。
2. 如希望 docId 按文件名前缀模式也支持，可以在实现中增加 `02-reaction-expression` 别名——但不推荐，因为白名单会变不透明。

**需要同步的文件：**
- `docs/mcp/v1-design.md`

### V83（🟡）`protocol.get_doc` 截断时未返回"章节建议"，与设计文档承诺不一致

**位置：**
- `docs/mcp/v1-design.md` §3.2：`超出时返回截断提示和章节建议`
- `mcp/src/core/protocol-index.ts` `getDoc()`：截断后返回 `truncated: true`，正文末尾追加 `[内容已截断，请传入更精确的 section]`

**当前值：** 实现只标记了截断状态和提示文字，没有提供"章节建议"——没有返回可用的章节列表、section 名称数组或类似结构。超过 20KB 的文档（如 `03-component-registry.md` ≈26KB、`08-renderer-spec.md` ≈21KB）实际触发截断后，AI 客户端只知道被截断了，但不知道有哪些章节可以请求。

**问题：** 设计文档承诺"章节建议"作为截断时的辅助输出，但实现未提供。这会在实际使用中降低 MCP 工具获取完整文档信息的能力。

**影响：** 🟡 中等。不影响校验功能，但影响协议知识查询体验。AI 客户端读取大文档时无法从工具结果中获得可重试的章节列表。

**建议修复：**
1. 在 `getDoc()` 返回结果中补充一个可选字段如 `availableSections: string[]`，当触发截断时填充当前文档的可用章节标题。
2. 同步更新 `docs/mcp/v1-design.md` 的输出结构说明。
3. 补充 MCP `get_doc` 截断场景测试。

**需要同步的文件：**
- `mcp/src/core/protocol-index.ts`
- `docs/mcp/v1-design.md`
- `mcp/tests/search.test.ts`（或新增 `mcp/tests/docs.test.ts`）

## 与已归档审计的关系

- V79 与 V35（`requiredCapabilities` 能力协商机制）直接相关：V35 建立了机制和 L2 校验，但 `upload.props.actionRef` 的字段级/action 级文档/DSL 未同步。
- V80 与 V72/V73/V74（permissions 变量作用域修复）相关：V72-V74 修了 L3a 扫描和 `$context` 白名单，但 `03` 正文仍保留 `permissions` 可通过 `scope` 作用的旧说法。
- V81 与 V22/V31/V73 同属 `permissions` 边界问题，但既有审计主要覆盖变量作用域和示例键名，未涉及扩展键声明机制。
- V82/V83/V84 与 V77（实施计划状态漂移）同类，是 MCP 文档生态的整体同步问题。

## 建议处理顺序

1. **先修 V79**：纯文档补充，不涉及实现变更，风险最低；对生成器/AI 影响直接。
2. **再修 V80**：纯文档修改 + ADR 同步，但需确认同步口径。
3. **再修 V84**：一行 frontmatter 修改，风险极低。
4. **再修 V82**：文档示例同步。
5. **再修 V83**：需要实现变更，涉及 `protocol-index.ts` 增加 `availableSections` 字段。
6. **最后处理 V81**：需要策略决策（A 或 B），依赖团队共识。

## 防止复发的建议

1. 能力声明类字段（`actionRef` 等）的入口文档/DSL/action 说明新增时，强制同步检查是否有 `requiredCapabilities` 提示——与 L2 校验规则成对出现。
2. 作用域说明中涉及 `permissions`、`visibleWhen`、`reactions` 三者的列表描述变更后，全文搜索"visibleWhen/reactions/permissions"并列出现的段落，确保不会遗漏同步。
3. 新增协议机制时，先在 ADR 中集中定义，正文引用的核心协议文档应在机制稳定后独立写清边界，避免长期依赖 ADR 作为唯一权威来源——ADR 的定位是决策记录，不是持续同步的规范。
4. MCP 文档（设计 + 实施计划 + README）的 status 必须保持一致；一个文件状态变更时，全文搜索 mcp 文档 frontmatter。
