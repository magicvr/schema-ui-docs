# 审计报告 0043：Node 嵌套挂载、校验错误分类、数值与 i18n 边界、CLI 退出码、审计身份

## 背景

本轮基于 0042 已归档后的再次全量复审。0042 中的请求与 Action 语义、组件校验边界、MCP 输出容量和审计编号身份问题均已确认落地，本轮不重复报告 V142-V154。

本轮完整读取协议正文、场景、ADR、Schema/组件 DSL、L0-L4 校验脚本、MCP 文档/实现/测试/打包配置以及全部审计文件，并对候选问题执行最小正反例。新问题集中在四类边界：

1. `tabs.items[].content` 虽声明为完整 Node，但自定义 DSL 的 `$ref` 没有进入 L2 执行；
2. L2 对嵌套结构错误和 YAML 非有限数值的处理破坏错误分类或放过非法值；
3. 通用 i18n 约定与 `select.options[]` 封闭 DSL 不一致，统一入口也丢失调用错误退出码；
4. V153 建立的审计复合身份规则和 0042 plan 范围尚未完整写回归档索引。

扫描活跃区和归档区全部 `review` / `checklist` / `plan` 后，历史最大审计编号为 `0042`，本轮使用 `0043`。历史问题定义标题 158 个、唯一 V 数字 154 个，全局最大问题编号为 `V154`，因此本轮从 `V155` 连续编号到 `V161`。

## 发现的问题

### V155（🔴）`tabs.items[].content` 的完整 Node 契约未被 L2 执行

**位置：** `docs/schemas/component-registry.json` 的 `tabs.props.items[].content`；`scripts/validate-l2-components.js` 的 `validateKnownRef()`、`validateProps()` 与 `validateNode()`。

**当前值：** 组件 DSL 已将 `content` 声明为 `$ref: node.schema.json`，正文称其为完整 Node。但 `validateKnownRef()` 只处理 `VisibleWhen`、`Reaction`、`Permissions` 三类引用，不处理整个 Node；后续递归仅在 `item.content` 为真值时调用 `validateNode()`，而 `validateNode()` 对非对象直接返回。

**问题：** `content: "not-a-node"`、`null`、`false`、`0` 等非法值不会产生 L2 违规。实测字符串值通过统一 L0-L4 校验，说明 0004/N1 只补 DSL `$ref` 后，执行链仍未闭合。

**影响：** 🔴 严重。协议接受一个无法渲染为 Node 的核心内容挂载点，错误被推迟到 Renderer 运行时，不同实现可能崩溃、忽略或产生空标签页。

**建议修复：** 为完整 `node.schema.json` 引用增加受控 L2 校验：值必须是普通对象并进入同一 `validateNode()` 递归；用属性存在性而非真值判断遍历 `content`。补字符串、null、布尔、数字反例和合法 Node 正例。

**与已归档审计的关系：** 0004/N1 只把 `content` 从普通 object 改为 `$ref`；V125 只封闭 tabs item 外壳；本项是 `$ref` 的执行缺口。

**需要同步的文件：** `scripts/validate-l2-components.js`、`docs/06-validation.md`、`mcp/tests/validate-content.test.ts`。

### V156（🟡）合法 JSON 的嵌套结构错误被误报为 `parseError`

**位置：** `scripts/validate-l2-components.js` 的 `validateNode()` / `validateProps()`；`mcp/src/core/validation-runner.ts` 的分层 parseError 合并。

**当前值：** `body.props: null` 会先由 L0/L1 正确报告“必须为 object”，但 L2 解构默认值不会替换显式 null，随后访问 `props.columns` 或执行 `Object.entries(props)` 抛异常。子脚本把异常写入 `parseErrors`，MCP 再将其提升为顶层 `parseError`。

**问题：** 输入语法完全合法，却同时得到结构违规和解析失败，违反 MCP 契约中“parseError 仅表示 YAML/JSON 语法解析失败”的分类边界。

**影响：** 🟡 中等。客户端会把可定位的结构错误误当作无法解析，摘要、suggested docs 和自动修复路径均会偏离。

**建议修复：** L2 对 Node 和 props 增加普通对象守卫；结构不合法时由 L0/L1 承担类型违规，L2 跳过不适用的组件语义检查且不得抛异常。补 null、字符串、数组 props 回归测试，断言 `parseError: null`。

**与已归档审计的关系：** V139 修复合法根标量误分类，本项发生在合法页面对象内部的嵌套值。

**需要同步的文件：** `scripts/validate-l2-components.js`、`mcp/tests/validate-content.test.ts`，必要时补充 `docs/mcp/v1-design.md` 的测试说明。

### V157（🟡）YAML 非有限数值绕过组件 DSL 的 `number` 和 `minimum`

**位置：** `scripts/validate-l2-components.js` 的 `checkType()` / `checkNumberBounds()`；`docs/schemas/component-registry.json` 的数值字段。

**当前值：** L2 仅以 `typeof value === "number"` 判断数值。`js-yaml` 会把 `.nan`、`.inf`、`-.inf` 解析为 JavaScript number，而 NaN 与任何 minimum 比较都为 false。

**问题：** `grid.props.columns: .nan` 和 `upload.props.maxSize: .nan` 可绕过数值与下界约束；实测 `.nan` 通过完整 L0-L4。

**影响：** 🟡 中等。布局列数、文件大小上限等运行时参数可成为不可序列化或无意义的值，Renderer 行为不确定。

**建议修复：** DSL 的 `number` 类型统一要求 `Number.isFinite(value)`；仅对有限数执行 minimum。补 `.nan`、`.inf`、`-.inf` 反例及 0/正数正例。

**与已归档审计的关系：** V151 增加 minimum 支持，但未处理 YAML 可产生的非有限 number。

**需要同步的文件：** `scripts/validate-l2-components.js`、`docs/03-component-registry.md`、`docs/06-validation.md`、`mcp/tests/validate-content.test.ts`。

### V158（🟡）通用 i18n 约定与 `select.options[]` 封闭 DSL 矛盾

**位置：** `docs/01-node-protocol.md` §6；`docs/03-component-registry.md` 通用约定与 select；`docs/schemas/component-registry.json` 的 `select.options[]`。

**当前值：** 协议规定 props 中语义为展示文案的 `label` / `title` / `content` 可由对应 `xxxKey` 替代，并以组件契约为具体范围。`select.options[]` 的 `label` 同样是固定选项展示文案，但 DSL 只允许并强制要求 `label`，同时以 `additionalProperties: false` 拒绝 `labelKey`。

**问题：** 按通用约定编写 `{ labelKey, value }` 会同时触发 label 缺失和 labelKey 额外字段违规，固定选项无法使用统一 i18n 机制。

**影响：** 🟡 中等。正文允许而官方 L2 拒绝同一配置，生成器和 Renderer 无法稳定实现固定选项国际化。

**建议修复：** 为 option item 增加 `labelKey`，把 `label` / `labelKey` 改为至少一个，保留 `value` 必填和外壳封闭性；同步 select 字段表与测试。

**与已归档审计的关系：** 0001/B6 建立通用 i18n；V125 封闭 option 外壳；两者组合后暴露本项遗漏。

**需要同步的文件：** `docs/03-component-registry.md`、`docs/schemas/component-registry.json`、`mcp/tests/validate-content.test.ts`。

### V159（🟡）统一校验入口吞掉分层脚本的调用错误退出码 2

**位置：** `scripts/validate-all.js` 的 `runScript()` 和最终 `process.exitCode`。

**当前值：** L2/L3a/L4 对无匹配文件均返回退出码 2，聚合器也保留 `err.status`，但最终只按 overallPass 返回 0 或 1。

**问题：** glob 拼错或没有输入文件会被统一入口表示为“存在协议违规”，CI 无法区分调用错误和内容错误。

**影响：** 🟡 中等。自动化可能把未执行任何有效校验误当作普通失败并继续错误处理，削弱 CI 诊断可靠性。

**建议修复：** 聚合器只要任一分层结果为 code 2，就最终返回 2；真实内容违规仍返回 1。AJV 调用错误也应显式携带 code，避免分类再次丢失。增加无参数、无匹配和内容违规三类退出码测试或验证记录。

**与已归档审计的关系：** V154 统一文件匹配形式，但未处理无匹配时的退出码传播。

**需要同步的文件：** `scripts/validate-all.js`、`docs/06-validation.md`。

## 建议处理顺序

1. 修复 V155-V156 的 L2 遍历和对象守卫，先恢复结构边界与错误分类。
2. 修复 V157-V158 的 DSL 数值和固定选项 i18n 契约，并补 MCP 回归测试。
3. 修复 V159 的统一入口退出码传播并执行命令级验证。
4. 修复 V160-V161 的归档索引证据链。
5. 执行官方场景、聚焦反例、MCP build/test/tools smoke 和文档/JSON 完整性检查。

## 防止复发的建议

1. 自定义 DSL 的每一种 `$ref` 都必须有显式执行器或明确声明只作说明，不能静默忽略。
2. 所有分层脚本应把语法解析错误、结构违规和内部异常保持为互斥分类。
3. YAML number 在进入业务约束前统一归一为有限数，避免 JavaScript 特殊数值渗透。
4. 新增通用字段约定时，应扫描所有嵌套展示对象，而不只扫描组件顶层 props。
5. 聚合 CLI 必须传播子命令的调用错误身份，并用命令级测试锁定 0/1/2。
6. 审计身份规则修改后，应扫描主要入口索引而不只修正文中的单个引用。
