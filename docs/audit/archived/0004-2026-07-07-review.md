## 二、新发现的问题

以下问题按"Schema 与文档同步遗漏 / 文档完整性 / 文档质量"三个维度分类。

---

### 维度一：Schema 与文档同步遗漏（L2 校验缺口）

#### N1. [🔴] `component-registry.json` 中 `tabs.props.items.content` 未引用 Node Schema

**位置：** `docs/schemas/component-registry.json`，`tabs` 组件定义，`props.items` 数组的 items 定义。

**当前状态：**
```json
"content": { "type": "object", "description": "完整 Node" }
```

**问题：** 虽然文字标注为"完整 Node"，但类型仅定义为 `"type": "object"`，没有通过 `$ref` 指向 `node.schema.json`。这意味着 L2（组件契约）校验无法对 `tabs` 的 `items[].content` 做 Node 结构合法性检查。相比之下，`action.schema.json` 中 `modal.content` 已正确使用 `"$ref": "node.schema.json"`。

**影响：** 如果用户在 `tabs` 的 `content` 中写了不合法的 Node 结构（如漏写 `type`、混入 CSS 属性），校验无法发现，只能到 Renderer 运行时才暴露。

**修复建议：** 将 `"type": "object"` 替换为 `"$ref": "node.schema.json"`，与 `action.schema.json` 保持一致。

---

#### N2. [🔴] `component-registry.json` 中 `table.props.columns.items` 缺少 `visibleWhen`/`reactions`/`permissions`/`labelKey` 字段

**位置：** `docs/schemas/component-registry.json`，`table` 组件定义，`props.columns.items`。

**当前状态：** 仅定义了 `field`/`label`/`format`/`tagMap` 四个字段。

**问题：** `03-component-registry.md` 中 ColumnDef 表明确列出了以下可选字段（since 0.2）：

| 字段 | 文档状态 | Schema 状态 |
|---|---|---|
| `field` | ✅ 已定义 | ✅ 已定义 |
| `label` | ✅ 已定义 | ✅ 已定义 |
| `labelKey` | ✅ 已定义 | ❌ **缺失** |
| `format` | ✅ 已定义 | ✅ 已定义 |
| `tagMap` | ✅ 已定义 | ✅ 已定义 |
| `visibleWhen` | ✅ 已定义（since 0.2） | ❌ **缺失** |
| `reactions` | ✅ 已定义（since 0.2） | ❌ **缺失** |
| `permissions` | ✅ 已定义（since 0.2） | ❌ **缺失** |

**影响：** L2 校验无法检测到 `columns` 中 `visibleWhen`/`reactions`/`permissions` 的拼写错误或格式错误。

**修复建议：** 在 `columns.items.properties` 中补充 `labelKey`、`visibleWhen`（`$ref` 到 `node.schema.json#/definitions/VisibleWhen`）、`reactions`（`$ref` 到 `reaction.schema.json` 的数组）、`permissions`（`$ref` 到 `node.schema.json#/definitions/Permissions`）。

---

#### N3. [🔴] `component-registry.json` 中 `table.props.actions.items` 缺少多个字段和约束

**位置：** `docs/schemas/component-registry.json`，`table` 组件定义，`props.actions.items`。

**问题：** `03-component-registry.md` 中 RowAction 表定义了（since 0.2）：

| 字段 | 文档状态 | Schema 状态 |
|---|---|---|
| `key` | ✅ 必填 | ✅ 有定义，但无 `required` 数组约束 |
| `label` / `labelKey` | ✅ 至少一个必填 | ⚠️ 仅 `label` 有定义，`labelKey` 有但缺少约束 |
| `confirm` | ✅ 可选 | ✅ 已定义 |
| `visibleField` | ✅ 可选（since 0.2） | ✅ 已定义 |
| `visibleWhen` | ✅ 可选（since 0.2） | ❌ **缺失** |
| `reactions` | ✅ 可选（since 0.2） | ❌ **缺失** |
| `permissions` | ✅ 可选（since 0.2） | ❌ **缺失** |

此外，`actions.items` 缺少：
- `required` 数组（`key` 应必填）
- `additionalProperties: false`（`columns.items` 有，但 `actions.items` 没有）

**影响：** L2 校验无法检测行内操作定义的字段级别错误。

**修复建议：**
1. 补充 `visibleWhen`/`reactions`/`permissions`/`labelKey` 字段定义。
2. 补充 `required: ["key"]`。
3. 补充 `additionalProperties: false`。

### 维度二：文档完整性不足

#### N4. [🟡] `01-node-protocol.md §3.5` reactions 结构示例未包含 `scope` 属性

**位置：** `docs/01-node-protocol.md §3.5`，reactions 结构 YAML 示例。

**当前内容：**
```yaml
reactions:
  - dependencies: [string]     # 依赖的字段名（同表单内的兄弟/祖先字段）
    when: string                 # 条件表达式
    fulfill: StateMap             # 条件为真时应用的状态
    otherwise: StateMap             # 【可选】条件为假时应用的状态
```

**问题：** 自 v0.2（ADR-0004）起，`reactions` 支持 `scope: form|row` 属性，用于表格列/行内操作的表达式作用域声明。但 §3.5 的结构示例中未包含 `scope` 字段。读者如果仅阅读 `01-node-protocol.md`（被引用频率最高的文档），无法知晓 `scope` 属性的存在。

`scope` 的实际定义在 `02-reaction-expression.md §9` 中，但核心文档的结构示例没有体现它——对于"只读核心文档"的读者来说，这是一个信息缺口。

**修复建议：** 在 `reactions` 结构示例中增加 `scope` 字段（可选，默认 `form`），并加注释说明。

---

#### N5. [🟡] `08-renderer-spec.md` 状态 `draft` 与 CHANGELOG v0.2.1 发布状态矛盾

**位置：** `docs/08-renderer-spec.md` 文件头；`docs/CHANGELOG.md`。

**当前状态：** `08-renderer-spec.md` 的 frontmatter 标记为 `status: draft`。

**问题：** CHANGELOG 中 v0.2.1 已将 `08-renderer-spec.md` 列为正式发布文档（"涉及的文档与 Schema"一节中列出）。该文档的内容（R1 注册机制、R2 数据加载策略、R3 错误边界、R4 沙箱实现指引、G2 版本协商、G3 环境变量管理）属于 v0.2.1 的正式产出。`draft` 状态与"已作为协议正式部分发布"的事实矛盾，可能让读者疑惑该文档是否可信。

**修复建议：** 将状态改为 `stable`，或至少在 CHANGELOG 的 v0.2.1 条目中注明 `08-renderer-spec.md` 仍处于草案阶段、内容可能调整。

---

#### N6. [🟡] `01-node-protocol.md §3.10` 最终可见性公式未提及 `dependencies` 前置条件

**位置：** `docs/01-node-protocol.md §3.10`。

**当前公式：**
```
最终 visible =
  permissions.view（若声明，优先级最高）
  AND visibleWhen.when（若声明，次优先级）
  AND reactions 计算出的 visible（若声明，最低优先级）
```

**问题：** 公式仅提及"信号"层面的 AND 逻辑，但未说明 `visibleWhen.when` 和 `reactions[].when` 中若使用了 `$deps.*` 变量，必须在对应的 `dependencies` 数组中声明。这个前置条件是表达式正确求值的必要条件，缺失可能导致读者写出"表达式语法正确但依赖未声明"的配置。

虽然 `02-reaction-expression.md §8` 和 `06-validation.md §6` 提到了校验要求，但 `01-node-protocol.md §3.10` 作为"最终可见性"的权威定义位置，缺少这个提醒可能让读者产生"声明了 when 表达式就够了"的错觉。

**修复建议：** 在公式下方增加一条注释说明："`visibleWhen.when` 和 `reactions[].when` 中使用的 `$deps.*` 变量必须在对应位置的 `dependencies` 数组中声明，否则表达式无法正确求值（详见 `02-reaction-expression.md §8`）。"

### 维度三：文档质量

#### N7. [🟢] `component-registry.json` 中 `tabs.props.items` 缺少 `required` 数组

**位置：** `docs/schemas/component-registry.json`，`tabs` 组件定义，`props.items`。

**问题：** 文档中 `key` 和 `label`/`labelKey` 均为必填，但 JSON 定义中没有遵循 JSON Schema 标准的 `required` 数组。`items` 的各属性使用项目自定义的 `required: true/false` 标注，但缺少顶层的标准约束。

与 N3 的 `actions.items` 属同类问题——`component-registry.json` 中各处的 `items` 定义的约束力度不统一。

**修复建议：** 在 `tabs.props.items` 的 items 对象中补充 `required: ["key", "label"]`。

---

#### N8. [🟢] CHANGELOG 中 v0.2.0 与 v0.2.1 同一天发布缺少说明

**位置：** `docs/CHANGELOG.md`。

**问题：** v0.2.0 和 v0.2.1 均标注为 `2026-07-07`。v0.2.1 包含大量实质性新增内容（Renderer 规范、日期选择器、搜索表单、上传组件、弹窗 Node 化、版本协商、环境变量管理等），并非纯粹的 bug 修复补丁。从版本号语义（MAJOR.MINOR.PATCH）来看，v0.2.0→v0.2.1 是 PATCH 级别变更，但实际内容量远超典型 PATCH。

建议在 CHANGELOG 中增加说明性注释，解释两个版本在同一天发布的原因，以及 v0.2.1 的性质（"基于 0003 审计的 MVP 就绪度紧急补丁，与 v0.2.0 在同一日完成"）。

**修复建议：** 在 v0.2.1 的日期后添加脚注或括号说明："（与 v0.2.0 同一天——基于 0003 审计发现的 MVP 阻断项紧急补发，非重复发布）"。

## 四、总体评价

经过四轮审计，协议文档的正确性和完整性已经达到较高水平。本轮发现的 8 个问题中：
- **3 个 🔴（Schema 缺口）**：属于 JSON Schema 与文档描述的同步遗漏，修复后即可消除 L2 校验盲区。

- **2 个 🟢（文档质量）**：属于细粒度完善建议。

协议的整体架构设计、表达式引擎边界、组件注册表体系等在四轮审计中均未发现根本性设计问题，表明协议的核心设计质量是扎实的。
