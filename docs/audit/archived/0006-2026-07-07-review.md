# 审计报告 0006：文档元数据一致性审计

## 背景

前五轮审计（0001–0005）主要聚焦于 Schema 与文档的**结构与语义一致性**（字段遗漏、约束冲突、表达式规则等）。本轮审计切换视角，检查**元数据层面的一致性**——CHANGELOG 记录是否完整、版本标注是否准确、frontmatter 是否统一等。这些问题不影响协议使用，但会影响版本追溯、可维护性和读者信任度。

## 审视范围

| 范围 | 说明 |
|---|---|
| CHANGELOG.md | 版本条目完整性、引用准确性 |
| 各文档 frontmatter | `applies_to`、`last_updated` / `date` 等元数据字段 |
| 各文档 `since` 标注 | 功能引入版本标注与 CHANGELOG 的对应关系 |
| 未使用的元信息能力 | 已定义但从未被使用的协议机制 |

---

## 发现的问题

### M1（🔴）CHANGELOG v0.2.0 B8 描述字段名错误

**位置：** `CHANGELOG.md` v0.2.0 条目 B8

**原文：**
> 组件级 `states`（空态/加载态/错误态文案定制），仅对 **`supportsData: true`** 的组件生效。

**问题：** `supportsData: true` 应为 **`supportsStates: true`**。`supportsData` 表示组件是否支持 `data` 字段（数据加载），`supportsStates` 才表示组件是否支持 `states` 字段（状态定制）。这是一个复制粘贴笔误——巧合的是当前所有 `supportsStates: true` 的组件（`statCard`、`chart`、`text`、`table`）恰好也都支持 `data`，因此功能行为未受影响，但描述本身是错误的。

**严重性：** 🔴 关键（错误的字段名会误导读者理解协议能力）。

---

### M2（🔴）CHANGELOG v0.2.1 遗漏 ADR-0004 合并记录

**位置：** `CHANGELOG.md` v0.2.1 "已合并的 ADR" 段落

**原文：**
> 已合并的 ADR：
> - `decisions/0003-context-namespace-and-visible-when.md`

**问题：** 当前文档中大量内容来自 **ADR-0004**（行级作用域 `$row`/`scope: row`/`$parentRow`/`$self` 语义），包括：
- `01-node-protocol.md §3.5` 中 `reactions` 新增 `scope` 属性
- `01-node-protocol.md §3.8` 中 `visibleWhen` 新增 `scope` 属性
- `02-reaction-expression.md §9` 完整的作用域规则、§10 静态校验规则、附录 A 变量可见性矩阵
- `03-component-registry.md` 表格 `columns`/`actions` 中的 `scope: row` 表达式支持
- `component-registry.json` 中 `VisibleWhen` 定义新增 `scope` 枚举

ADR-0004 文档自述"已接受——已合并入 `01-node-protocol.md` / `02-reaction-expression.md`"，但 CHANGELOG 中**完全未提及**。读者按 CHANGELOG 追溯功能引入版本时，会找不到 `$row` 体系的来源。

**严重性：** 🔴 关键（CHANGELOG 版本追溯能力存在重大缺口）。

---

### M3（🔴）`00-overview.md §5` 版本声明滞后于 CHANGELOG

**位置：** `00-overview.md §5` 版本与稳定性

**当前值：**
> 当前协议版本：`v0.2`（最新补丁版本 **`v0.2.1**）

**问题：** `CHANGELOG.md` 已记录 **v0.2.2** 条目（"文档一致性补丁"）。这意味着两者之一存在问题：
- 若 v0.2.2 **已正式发布**，则 `00-overview.md` 应更新为 `v0.2.2`；
- 若 v0.2.2 **尚未正式发布**（仅是计划性条目），则不应出现在 `CHANGELOG.md` 中（CHANGELOG 记录已发生的事实，不是计划表）。

无论哪种情况，两个文件当前状态矛盾。

> **注意：** `CHANGELOG.md` v0.2.2 条目中 F6 就是"`00-overview.md` 版本声明增加 v0.2.1 补丁版本说明"——这说明 F6 修复是在 v0.2.2 发布**之前**的某个时间点完成的，而 v0.2.2 发布后概述未再次更新。

**严重性：** 🔴 关键（版本声明是文档的门面，矛盾会降低读者对整个文档体系的信任度）。

---

### M4（🟡）`07-actions-contract.md §5` 中 `modal.content` 缺少 `since` 标注

**位置：** `07-actions-contract.md §5.1`

**问题：** `modal.content` 字段在 CHANGELOG 中明确记载为 v0.2.1 新增（见 v0.2.1："弹窗内容 Node 化：`actions[].type: modal` 新增可选 `content: Node` 字段"），但在 `07-actions-contract.md §5.1` 中没有任何 `(since 0.2.1)` 标注。该文档中的 `content` 描述仅有一句"【可选，since 0.2】弹窗内容"——标注成了错误的版本号。

**影响：** 读者会误以为 `content` 从 v0.2.0 就已存在。

**严重性：** 🟡 中等。

---

### M5（🟡）`08-renderer-spec.md` frontmatter 与实际版本不符

**位置：** `08-renderer-spec.md` frontmatter

**当前值：**
```yaml
applies_to: schema-ui-protocol v0.2
date: 2026-07-07
```

**问题：**
1. `applies_to` 写的是 `v0.2`，但该文档在 v0.2.1 才首次引入（CHANGELOG 确认，`00-overview.md` 文档地图也标注为 `since 0.2.1`）。
2. 使用 `date:` 而非其他文档统一使用的 `last_updated:`——`07-actions-contract.md`、`03-component-registry.md` 等均使用 `last_updated`。该文档是唯一使用 `date:` 的文档。

**严重性：** 🟡 中等。

---

### M6（🟡）`03-component-registry.md` 表格 RowAction/ColumnDef 字段 `since` 粒度模糊

**位置：** `03-component-registry.md` 表格组件 RowAction/ColumnDef 表格

**问题：** `visibleWhen`、`reactions`、`permissions` 在 `columns` 和 `actions` 中标注为 `(since 0.2)`，但它们实际是在 v0.2.1 通过 ADR-0003（`$context`/`visibleWhen`/`permissions`）和 ADR-0004（`scope: row`/`$row`）引入的。

虽然协议约定 `protocolVersion` 只用 MAJOR.MINOR（不区分 PATCH），`since 0.2` 在严格意义上可以涵盖 v0.2.0/v0.2.1/v0.2.2，但这种标注方式会让读者误以为这些能力在 **v0.2.0 就已存在**。

**严重性：** 🟡 中等。

---

### M7（🟢）`component-registry.json` 中 `deprecated`/`deprecatedMessage` 从未被实际使用

**位置：** `03-component-registry.md` "组件生命周期标注" 章节 + `component-registry.json`

**背景：** CHANGELOG v0.2.0 B9 为 `component-registry.json` 增加了三个元信息字段：`deprecated`、`deprecatedMessage`、`since`。其中 `since` 已被广泛使用（几乎所有 v0.2 新增属性都有标注），但 `deprecated` 和 `deprecatedMessage` 在整个注册表中**没有任何一个组件或字段使用**。

**影响：**
- 该能力已定义但从未被验证，存在"定义但不工作"的风险（如 Renderer 是否真的会读取这些字段？）
- 新维护者可能不知道有这个废弃标注机制可用

**严重性：** 🟢 轻微（能力存在但未使用，不造成当前功能问题）。

---

### M8（🟢）`04-datasource-contract.md §4.1` responseMapping 临时方案格式问题

**位置：** `04-datasource-contract.md §4.1` 字段名映射段落

**问题：** 该节使用多层嵌套 blockquote（`>` → `> >` → `> > >`）来描述 responseMapping 的 "planned" 状态和临时变通方案，视觉上层级复杂，读者容易在不同层级的说明间迷失。

**建议：** 精简为两层结构，或将临时变通方案独立为带标题的独立段落，而非多层引用。

```markdown
<!-- 当前效果 -->
> **字段名映射（planned — 待 ADR 标准化后定版）：**
> > ...
> > **跟踪状态：** ...
> >
> > **临时变通方案：** ...
```

**严重性：** 🟢 轻微。
