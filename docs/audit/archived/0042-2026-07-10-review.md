# 审计报告 0042：请求与 Action 语义、组件校验边界、MCP 输出容量、审计编号身份

## 背景

本轮审计基于 0041 已归档后的再次全量复审。0041 中的普通表单 GET request、表达式与 params 校验、组件 DSL、`validate_content` 最终 20KB 返回预算及归档证据链问题均已确认落地，本轮不重复报告 V130-V141。

本轮完整读取协议正文、场景、ADR、Schema/组件 DSL、L0-L4 校验脚本、MCP 文档/实现/测试/打包配置及全部审计文件，并执行最小正反例、MCP 大输出复现、官方场景校验、MCP 构建和测试。新问题集中在四类边界：

1. DataRef、搜索表单、Action 和 upload 的运行时合并/传输语义没有唯一答案；
2. L2/L4 对“被忽略字段”、URL 模板、业务值对象和数值边界的执行与正文不一致；
3. MCP 最终返回虽已受 20KB 预算控制，但校验子进程 stdout 会先触发默认缓冲区上限；
4. 历史审计 V 编号曾重置，已经破坏问题编号的全局唯一身份。

扫描活跃区和归档区全部 `review` / `checklist` / `plan` 后，历史最大审计编号为 `0041`，本轮使用 `0042`。历史最大问题编号为 `V141`，因此本轮从 `V142` 连续编号到 `V154`。虽然历史 `V1-V4` 存在复用，但不得通过再次重置或改用旧编号修复，本轮仍从全局最大值继续递增。

## 发现的问题

### V142（🔴）非 GET `DataRef.params` 的传输位置没有唯一语义

**位置：** `docs/01-node-protocol.md` §3.3；`docs/04-datasource-contract.md` §3.1；`docs/08-renderer-spec.md` §2.3。

**当前值：** DataRef 允许 `GET | POST | PUT | DELETE | PATCH` 并统一提供 `params`。数据源正文将 `data.params` 定义为最终请求 query，Renderer 规范却写成从最终请求的 query/body 中省略空值；Node 正文只称其为“请求参数映射”。

**问题：** 对 `method: POST` 且带 `params` 的合法 DataRef，协议无法确定参数进入 URL query、JSON body，还是按 method 自动分流。不同 Renderer 可构造不同 HTTP 请求，后端契约也无法据此稳定实现。

**影响：** 🔴 严重。协议接受同一配置却允许互不兼容的线上请求形态，直接破坏 Renderer 互操作性。

**建议修复：** 在 v0.2 选择单一保守规则：DataRef `params` 始终编码为 query，不因 method 改变；DataRef 不定义隐式 body。若未来需要请求体数据源，通过 ADR 增加独立 `body` / `bodyMapping` 字段。同步 Node、数据源、Renderer、Schema 描述、测试和 CHANGELOG。

**与已归档审计的关系：** V60 只补 `method` 缺省 GET；V130 处理普通表单 Action 的 JSON body，不涉及通用 DataRef `params`。

**需要同步的文件：** `docs/01-node-protocol.md`、`docs/04-datasource-contract.md`、`docs/08-renderer-spec.md`、`docs/schemas/node.schema.json`、`docs/CHANGELOG.md`。

### V143（🔴）搜索表单可引用无法重新请求的目标表格

**位置：** `docs/03-component-registry.md` form 搜索模式；`docs/06-validation.md` v0.2.8 引用完整性；`scripts/validate-l2-components.js` 的 `validateDataRefsAndTargetTable()`。

**当前值：** 搜索表单承诺把字段附加到目标表格请求并触发重新加载。L2 只要求 `targetTable` 对应存在的 `type: table` Node。

**问题：** 目标 table 没有 `data`、使用 `source: static`，或 `source: ref` 指向静态 datasource 时均可通过 L2，但不存在可重新发起的 API 请求。对 API ref，真实 URL 也位于 datasource 而非正文所写的 `targetTable.data.url`。

**影响：** 🔴 严重。合法页面可声明协议承诺但运行时无法执行的核心搜索行为；Renderer 只能静默不刷新或运行时报错。

**建议修复：** L2 解析目标 table 的有效数据源，要求其最终为 `source: api`；允许内联 API 和 ref API，拒绝无 data、static 及 ref static。正文改用“目标表格的有效 API 数据源”，并补正反例。

**与已归档审计的关系：** V86 只新增 targetTable 存在性和类型检查，没有验证目标数据源是否可请求。

**需要同步的文件：** `docs/03-component-registry.md`、`docs/06-validation.md`、`scripts/validate-l2-components.js`、`mcp/tests/validate-content.test.ts`、`docs/CHANGELOG.md`。

### V144（🟡）`dateRangePicker` 的 reaction `value` 写入目标未定义

**位置：** `docs/02-reaction-expression.md` §14.1；`docs/03-component-registry.md` dateRangePicker reactions 说明；组件 DSL。

**当前值：** `fulfill.value` / `otherwise.value` 仅写入声明 reaction 的当前字段。`dateRangePicker` 没有单一 `field`，而是 `startField` / `endField` 两个独立提交字段；它同时被声明为支持 reactions，`$self` 为 `{ start, end }`。

**问题：** 在 dateRangePicker 自身 reaction 中声明 `value` 时，Renderer 无法判断应写哪个字段、是否要求对象、是否同时覆盖两端或应拒绝。

**影响：** 🟡 中等。双字段组件的状态写入会在 Renderer 间分叉，并可能覆盖错误的表单值。

**建议修复：** v0.2 保守禁止 dateRangePicker reactions 使用 `value`，继续允许 `visible` / `required` / `disabled`；未来若需要范围回写，应通过 ADR 定义明确的 `{start,end}` 结构和部分更新规则。同步 L2/L3a 或 DSL 语义检查、正文与测试。

**与已归档审计的关系：** V117 仅闭合 `$self.start` / `$self.end` 的读取语义，没有定义双字段写入。

**需要同步的文件：** `docs/02-reaction-expression.md`、`docs/03-component-registry.md`、`docs/06-validation.md`、`scripts/validate-l2-components.js` 或 `scripts/validate-l3a-expressions.js`、`mcp/tests/validate-content.test.ts`。

### V145（🟡）upload 组件与 upload Action 的重复约束没有合并规则

**位置：** `docs/03-component-registry.md` upload；`docs/07-actions-contract.md` §7；`docs/schemas/component-registry.json` 和 `action.schema.json`。

**当前值：** upload 组件和其 `actionRef` 指向的 upload Action 均可声明 `accept`、`maxSize`、`multiple`。组件文档说按 Action 配置执行，Action 文档又说客户端校验由组件完成。

**问题：** 两层值不一致时，协议没有说明文件选择 UI、客户端拦截和请求执行采用组件值、Action 值、较严格值，还是应拒绝冲突。

**影响：** 🟡 中等。同一页面在不同 Renderer 中可能接受不同的文件集合，客户端与服务端提示也可能不一致。

**建议修复：** 明确 `actionRef` 模式下 Action 是上传传输与约束的唯一来源，组件侧重复字段不得声明；直接 `action` 模式继续由组件字段控制。由 L2 跨引用拒绝重复配置，并同步正文、DSL 描述和测试。

**与已归档审计的关系：** V79 处理能力声明，V85 处理 actionRef 存在性和类型，均未处理重复字段合并语义。

**需要同步的文件：** `docs/03-component-registry.md`、`docs/07-actions-contract.md`、`docs/06-validation.md`、`scripts/validate-l2-components.js`、`docs/schemas/component-registry.json`、`mcp/tests/validate-content.test.ts`。

### V146（🟡）标准 HTTP 错误处理与 Action `onError` 的组合顺序未定义

**位置：** `docs/04-datasource-contract.md` §5-§6；`docs/07-actions-contract.md` §3/§8；`docs/08-renderer-spec.md` §7。

**当前值：** 数据源契约规定 `400 + errors` 回填字段并展示 message，401/403 触发宿主钩子，其他状态也有默认处理。Action 同时允许任意 `onError` OutcomeBehavior，Renderer 只写“根据 onError 执行结果行为”。

**问题：** 当标准状态处理与配置行为同时存在时，协议没有说明覆盖、追加或执行顺序。例如 `400 + errors` 与 `onError.navigate` 同时出现，可能在回填后立刻离开页面，也可能完全忽略字段错误。

**影响：** 🟡 中等。后端验证错误、认证失败和业务导航在不同实现中表现不一致。

**建议修复：** 定义统一顺序：先执行不可覆盖的协议级状态处理（401/403 钩子、400 字段回填和安全错误文案），再执行 `onError`；但 400 存在字段错误时禁止自动 navigate/closeModal，以保留修正入口。对 toast message 明确配置值与响应 message 的展示优先级。

**与已归档审计的关系：** 历史 Action 审计覆盖引用、能力、GET 可执行性和错误结构，未定义 HTTP 状态矩阵与 OutcomeBehavior 的组合。

**需要同步的文件：** `docs/04-datasource-contract.md`、`docs/07-actions-contract.md`、`docs/08-renderer-spec.md`、必要时 `docs/decisions/` 新 ADR、`docs/CHANGELOG.md`。

### V147（🟡）部分 `bodyMapping` 是否保留未映射字段未定义

**位置：** `docs/07-actions-contract.md` §3；`docs/08-renderer-spec.md` §7；`docs/schemas/action.schema.json` 描述。

**当前值：** `bodyMapping` 被定义为“表单字段名到请求体字段名的映射”；仅说明完全缺省时所有字段按原名组成 JSON。

**问题：** 当 mapping 只包含部分字段时，未列字段是按原名保留，还是从请求体排除，没有定义。两种解释分别对应局部重命名和请求体白名单，数据泄露边界完全不同。

**影响：** 🟡 中等。请求体内容不确定，可能漏传后端必填字段，也可能意外发送本应排除的敏感字段。

**建议修复：** 采用白名单语义：存在 `bodyMapping` 时只发送显式列出的源字段；缺省时才发送全部字段。同步 Renderer 规范和 Schema 描述，并补完整/部分/空 mapping 示例。

**与已归档审计的关系：** V103 只限制 mapping value 为非空字符串；V130 只处理 method 与 body 可执行性。

**需要同步的文件：** `docs/07-actions-contract.md`、`docs/08-renderer-spec.md`、`docs/schemas/action.schema.json`、`docs/CHANGELOG.md`。

### V148（🟡）搜索模式中被声明为忽略的 `submitAction` 仍被 L2 强制解析

**位置：** `docs/03-component-registry.md` form；`docs/06-validation.md` v0.2.8 规则；`scripts/validate-l2-components.js` 的 `validatePageActionRefs()`。

**当前值：** 正文三次明确 `mode: search` 不执行并忽略 `submitAction`，DSL 也允许该字段存在。L2 只判断字段是否存在，不判断 mode。

**问题：** 搜索表单携带不存在的 `submitAction` 或 GET request 时，L2 仍报引用/method 违规；这与“忽略”语义直接冲突。

**影响：** 🟡 中等。正文允许的兼容配置被官方工具拒绝，且校验结果依赖不会进入执行路径的字段。

**建议修复：** `validatePageActionRefs()` 对 `mode: search` 跳过 submitAction 引用和 method 校验；默认模式保持现有规则。增加 search/default 对照测试。

**与已归档审计的关系：** V85 增加引用完整性、V130 增加 GET 限制时都未按 form mode 分支。

**需要同步的文件：** `scripts/validate-l2-components.js`、`docs/06-validation.md`、`mcp/tests/validate-content.test.ts`。

### V149（🟡）L4 将 `select.options[].value` 的业务对象键误判为 CSS

**位置：** `docs/schemas/component-registry.json` 的 select options value；`scripts/lint-l4-banned-props.js` 的深度扫描；`docs/06-validation.md` L4 边界。

**当前值：** option `value` 没有限制类型，可合法承载对象。L4 从 props 递归扫描所有对象键，只豁免 params/requestMapping 和 tagMap 动态键。

**问题：** `options: [{ label: Red, value: { color: red } }]` 通过 L0-L3a，却在 L4 报 `value.color`。业务值载荷被错误当作样式协议对象。

**影响：** 🟡 中等。合法枚举对象值被 CI 拒绝，业务数据形态被 CSS 词表无意限制。

**建议修复：** 将 `select.options[].value` 识别为不透明业务值，跳过其内部键名扫描；option 外壳仍保持封闭并扫描真正的协议字段。同步 L4 文档和测试。

**与已归档审计的关系：** V134 处理 params/requestMapping 开放 map，V125 处理 option 外壳封闭性，均未处理 `value` 内部业务对象。

**需要同步的文件：** `scripts/lint-l4-banned-props.js`、`docs/06-validation.md`、`mcp/tests/validate-content.test.ts`。

### V150（🟡）非法 URL 花括号标记可绕过 RowAction 完整性校验

**位置：** `docs/07-actions-contract.md` §3.1；`scripts/validate-l2-components.js` 的 `extractUrlPathParams()` / `validateRowRequestAction()`。

**当前值：** 规范只定义 `{name}` 占位符并要求逐一绑定。实现只提取匹配标识符正则的占位符，不检查 URL 中残留的其他 `{...}`。

**问题：** `/orders/{order-id}`、`/{123}` 或未闭合花括号不会进入 placeholders 集合，也不会被拒绝。配置可通过 L0-L4，但 Renderer 无法按协议替换该片段。

**影响：** 🟡 中等。运行时可能向后端发送带原始花括号的错误 URL，且配置错误未在静态阶段暴露。

**建议修复：** 在提取合法占位符前后检查 URL 中所有 `{` / `}` 片段；只允许完整 `{[A-Za-z_][A-Za-z0-9_]*}`，拒绝非法、嵌套、孤立或未闭合标记。补正反例和文档格式说明。

**与已归档审计的关系：** V100 处理 requestMapping 值和 path key 对应关系，没有检查 URL 模板词法完整性。

**需要同步的文件：** `scripts/validate-l2-components.js`、`docs/07-actions-contract.md`、`docs/06-validation.md`、`mcp/tests/validate-content.test.ts`。

### V151（🟡）upload 组件允许负 `maxSize`，与 upload Action 约束冲突

**位置：** `docs/schemas/component-registry.json` upload props；`docs/schemas/action.schema.json` UploadAction；`scripts/validate-l2-components.js` DSL 数值校验。

**当前值：** upload 组件 DSL 只要求 `maxSize` 为 number；同语义的 upload Action Schema 要求 `minimum: 0`。正文均将其定义为字节上限。

**问题：** 组件 `maxSize: -1` 可通过 L0-L4，而 Action 同值会被 AJV 拒绝。负上限没有可执行意义，可能导致所有文件都被拒绝。

**影响：** 🟡 中等。同一协议概念在两条配置路径约束不一致，并把错误推迟到运行时。

**建议修复：** 为组件 DSL/L2 增加受控 `minimum` 数值约束支持，upload `maxSize` 设为 `minimum: 0`；同步 DSL 关键字说明和边界测试。

**与已归档审计的关系：** 历史 upload 审计覆盖 placeholder、actionRef、能力和字段隔离，未覆盖 maxSize 下界。

**需要同步的文件：** `docs/schemas/component-registry.json`、`scripts/validate-l2-components.js`、`docs/03-component-registry.md`、`docs/06-validation.md`、`mcp/tests/validate-content.test.ts`。

### V152（🟡）MCP 校验子进程 stdout 可先于最终裁剪触发默认缓冲区上限

**位置：** `mcp/src/core/validation-runner.ts` 的 `defaultLayerScriptExecutor()`；MCP validate_content 测试和 v1 设计。

**当前值：** 输入上限为 1 MiB，最终工具文本受 20KB 结构化裁剪控制；但 `execFileSync()` 未设置 `maxBuffer`，仍使用 Node 默认 stdout 缓冲区。

**问题：** 约 5,000 个未知组件、输入仅约 174KB 时，L2 stdout 已可触发 `ENOBUFS`。runner 随后得到不完整输出并返回“无法解析校验脚本 JSON 输出”，丢失全部真实违规，`layerStats` 也错误归零。

**影响：** 🟡 中等。远低于公开输入上限的请求无法获得承诺的分层结果；虽然失败关闭，但错误分类和裁剪统计均失真。

**建议修复：** 为子进程设置与最坏情况匹配且有明确上限的 `maxBuffer`，并将 `ENOBUFS` 单独归类为结构化内部错误；更稳妥的长期方案是让子脚本支持结构化违规数量上限，避免先生成无界 stdout。补真实子进程高基数边界测试。

**与已归档审计的关系：** V75 修子脚本异常分类，V138 修最终工具文本 20KB 裁剪，均未覆盖裁剪前的子进程传输容量。

**需要同步的文件：** `mcp/src/core/validation-runner.ts`、`mcp/tests/validate-content.test.ts`、`docs/mcp/v1-design.md`、`docs/decisions/0007-mcp-protocol-reader-validator.md`（如新增内部容量约定）。

### V153（🟡）历史 V1-V4 被重复分配，问题编号身份不唯一

**位置：** `docs/audit/archived/0010-*`、`0013-*`、`0014-*`；后续历史引用，如 0035 review 的“历史上 V1”。

**当前值：** 0010 将 V1-V4 分配给 Renderer/表达式/tabs 问题；0013 再将 V1-V3 分配给版本和 since 问题；0014 又将 V4 分配给审计状态问题。当前 V1-V141 数值集合连续，但有 145 个问题定义。

**问题：** 单独使用 `V1`-`V4` 已无法唯一定位问题，且后续“历史 V1”引用存在实际歧义。直接重写历史编号又会破坏既有引用和归档真实性。
