# 审计报告 0012：ADR 决策缺口汇总审视

## 背景

本轮审计基于对当前文档体系的全量审视，目标是收集所有仍需要通过 ADR 落定的协议决策点，并区分以下两类事项：

- 当前正文或审计索引已明确标注为待 ADR 标准化 / 未定义的问题。
- 已接受 ADR 中刻意留下、未来遇到真实需求后再触发的新决策点。

本轮不重复列入已经由 ADR-0001 至 ADR-0004 落定的事项，例如单棵 Node 树、schema + uiSchema 双树取舍、`$context` / `visibleWhen` / `permissions` 基础机制、`$row` 行级作用域与 `visibleField` 语法糖。

## 当前应优先进入 ADR 的决策点

### A1（🔴）`responseMapping` 响应体字段名映射尚未标准化

**位置：** `04-datasource-contract.md` §4.1、`audit/README.md` 跟踪项

**现状：** 文档已说明当后端无法使用协议默认的 `list` / `total` 字段名时，可在 Renderer 扩展层自行约定映射；但协议层尚未承诺标准语法，且 `audit/README.md` 已登记为待产出 ADR 后标准化。

**需要 ADR 回答：**

- 映射声明放在 `data.params.responseMapping`、`data.responseMapping`，还是 Renderer 全局配置中。
- 是否只覆盖列表接口的 `list` / `total`，还是同时覆盖 `select.optionsSource`、`chart`、`statCard` 等数据形态。
- 映射是节点级、数据源级、还是页面级默认配置。
- 未配置映射时的默认行为、错误提示与兼容策略。

**影响：** 🔴 高。该项已经是活跃跟踪项，且遗留系统 API 字段名不一致是较常见接入问题。

---

### A2（🔴）表达式读写求值时序未定义

**位置：** `01-node-protocol.md` §3.10、`decisions/0003-context-namespace-and-visible-when.md` 遗留问题

**现状：** 当字段同时被 `visibleWhen` / `permissions` 读取，又被同节点或其他节点的 `reactions.fulfill.value` 修改时，协议未定义同一渲染周期内读写谁先发生。当前文档只给出规避建议，没有正式求值模型。

**需要 ADR 回答：**

- `visibleWhen` / `permissions` 读取的是变更前快照、变更后结果，还是分阶段批处理结果。
- `reactions` 中的赋值副作用与可见性判断之间的执行顺序。
- 是否通过静态校验禁止同周期读写冲突，还是允许但定义固定求值模型。
- 多个 `reactions.fulfill.value` 写同一字段时的冲突处理。

**影响：** 🔴 高。该问题影响 Renderer 行为一致性，同一配置在不同实现中可能出现不同 UI 结果。

---

### A3（🟡）跨作用域联合判断缺少标准语法

**位置：** `02-reaction-expression.md` §9.1、`decisions/0004-row-level-scope.md` 遗留问题

**现状：** v0.2 明确不支持在同一表达式中同时依赖表单级字段和行级 `$row` 字段。当前推荐做法是后端预计算行数据字段，或未来通过 ADR 新增机制。

**需要 ADR 回答：**

- 是否引入“表格级传入属性”、显式绑定参数、作用域链，或其他跨作用域机制。
- 表单级字段变化后，行级表达式如何重新求值。
- 跨作用域依赖如何声明、静态校验和调试。
- 与 `$deps` / `$row` 互斥规则如何兼容。

**影响：** 🟡 中等。当前已有后端预计算方案，但复杂业务中可能重复出现。

---

### A4（🟡）`scope: row` 下行内字段联动赋值 / 回写模型未设计

**位置：** `02-reaction-expression.md` §9.3、`decisions/0004-row-level-scope.md` 遗留问题

**现状：** `scope: row` 下 `fulfill` / `otherwise` 仅允许 `visible` 与 `disabled`，静态禁止 `required` 与 `value`。若未来需要行内字段联动赋值，需另开 ADR。

**需要 ADR 回答：**

- 行级表达式是否允许写回行数据，写回目标是单元格、整行、临时 UI 状态还是提交 payload。
- 分页、排序、筛选、远程数据重新加载后，回写状态如何保留或丢弃。
- 与表单级 `reactions.fulfill.value` 的求值模型是否共用。
- 是否采用现有 `reactions` 的受限扩展，还是新增独立结构。

**影响：** 🟡 中等。当前禁止是清晰的，但未来一旦支持，设计面较大。

---

### A5（🟡）`compute` 纯计算字段仍停留在未来方向

**位置：** `audit/archived/0001-2026-07-07-plan.md` C4

**现状：** 历史审计已记录“表达式引擎新增 `compute` 字段”作为未来演进方向，并要求触发时产出 `decisions/0005-compute-field.md`。当前正文未定义该能力。

**需要 ADR 回答：**

- 是否新增独立于 `reactions` 的 `compute` 字段，用于只读派生值。
- 表达式能力是否仍保持受限，是否继续禁止函数调用与任意脚本。
- 计算结果挂载在 Node `props`、`data`、临时上下文，还是只作为展示值。
- 计算字段与校验、提交 payload、`visibleWhen` 依赖之间的关系。

**影响：** 🟡 中等。它不是当前阻断项，但涉及表达式引擎能力边界，不能零散补字段。

---

### A6（🟡）跨组件数据依赖 / 参数传递机制尚未设计

**位置：** `04-datasource-contract.md` §3.2

**现状：** 文档明确 `data.params` 中的 `$deps.*` 只在表单上下文有效。若未来需要一个 `chart` 依赖另一个 `chart` 的筛选结果，应通过独立 ADR 设计参数传递机制，而不是复用 `$deps`。

**需要 ADR 回答：**

- 组件间依赖关系如何声明。
- 加载顺序如何推导，是否需要拓扑排序。
- 上游组件失败、空值、刷新时，下游组件如何处理。
- 是否允许跨组件传递任意数据，还是仅允许显式导出的参数。

**影响：** 🟡 中等。当前协议边界清晰，但该类仪表盘联动场景未来可能出现。

## 未来触发型 ADR 决策点

### B1（🟢）`$context` 响应式刷新与 `user` / `features` 刷新粒度

**位置：** `decisions/0003-context-namespace-and-visible-when.md` 遗留问题

当前 `$context` 是初始化快照。若未来需要页面运行中感知用户角色或功能开关变化，需要 ADR 决定是否支持响应式 `$context`，以及 `user` 与 `features` 是否拆分刷新时机。

### B2（🟢）`permissions.edit` / `delete` 在容器节点上的级联语义

**位置：** `decisions/0003-context-namespace-and-visible-when.md` 遗留问题

当前只明确 `permissions.view=false` 与 `visibleWhen=false` 的容器隐藏级联。若未来需要容器级批量禁用或只读，需要 ADR 决定 `edit` / `delete` / 自定义动作键是否级联，以及与子节点权限冲突时如何仲裁。

### B3（🟢）`$context.features` 粒度与新 `$context` 根命名空间扩展

**位置：** `02-reaction-expression.md` §11、`decisions/0003-context-namespace-and-visible-when.md` 遗留问题

当前只允许 `$context.user` 与 `$context.features`。新增命名空间已要求走 ADR。若未来需要细化 feature flag 结构或新增根级上下文，需要 ADR 统一白名单与迁移策略。

### B4（🟢）三层及以上嵌套表格的跨层引用

**位置：** `decisions/0004-row-level-scope.md` 遗留问题

当前只支持 `$parentRow` 访问直接父级行，不支持跳级。若出现三层及以上嵌套表格的真实需求，需要 ADR 评估是否引入 `$parentRow(2)` 等层级引用方式。

### B5（🟢）`visibleField` 下线策略

**位置：** `decisions/0004-row-level-scope.md` 遗留问题

当前 `visibleField` 被定义为 `scope: row` 的 `visibleWhen` 语法糖并长期保留。若未来与 `$row` 语义冲突或维护成本上升，需要 ADR 决定是否废弃、迁移窗口和兼容策略。

### B6（🟢）同一份数据多套展示的 `variant` 机制

**位置：** `decisions/0002-why-not-two-schema-uischema.md` 权衡章节

当前协议选择单棵 Node 树。若未来出现强烈的“同一数据结构多套展示”诉求，可评估是否通过 Node `variant` 扩展实现局部复用，而不是回到双树模型。

### B7（🟢）错误响应 `code` 的前端程序化消费

**位置：** `04-datasource-contract.md` §5

当前 `code` 仅用于调试日志和错误追踪，Renderer 不据此做登录跳转或特殊 UI。若未来要消费业务错误码，需要场景 ADR 约定可识别 code、注册方式和默认行为。

## 建议处理顺序

1. 优先启动 A1 `responseMapping` ADR：已有活跃跟踪项，接入价值明确。
2. 随后启动 A2 表达式求值时序 ADR：这是 Renderer 一致性的底层模型问题。
3. 将 A3 / A4 / A5 作为表达式引擎边界扩展包统一规划，避免零散新增语法。
4. A6 与 B 组保持需求观察，等出现具体场景后再立项。
