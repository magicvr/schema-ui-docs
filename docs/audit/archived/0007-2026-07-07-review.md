# 审计报告 0007：文档第七轮全面审视

## 背景

前六轮审计（0001–0006）覆盖了协议语义正确性、Schema 与文档一致性、MVP 就绪度缺口、元数据一致性四大维度。本轮审计基于对当前文档体系的**全量再审视**，重点查找前六轮未覆盖到的隐性不一致和遗漏。

## 审视范围

| 范围 | 说明 |
|---|---|
| 组件注册表完整性 | 检查各组件 `props` 字段是否存在遗漏/不一致 |
| 表达式作用域边界 | 检查 `$deps.*` 在非预期位置的使用规则是否完整定义 |

## 发现的问题

### S1（🔴）`select` 组件缺少 `required` 和 `defaultVisible` 字段

**位置：** `03-component-registry.md` select 组件表格 + `schemas/component-registry.json` `select.props`

**问题：** 所有表单字段类组件均支持 `required`（是否必填）和 `defaultVisible`（初始显隐）两个 props，唯独 `select` 组件同时缺失这两个字段。

| 组件 | `required` | `defaultVisible` |
|---|---|---|
| `input` | ✅ | ✅ |
| `inputNumber` | ✅ | ✅ |
| `datePicker` | ✅ | ✅ |
| `dateRangePicker` | ✅ | ✅ |
| `upload` | ✅ | ✅ |
| **`select`** | **❌** | **❌** |

`select` 作为最常用的表单字段之一（尤其在联动场景中作为"触发字段"），不可能没有"必填"和"初始显隐"的需求。这属于组件注册表定义时的遗漏。

**影响：** 🔴 严重。后端开发者无法为 `select` 字段声明 `required: true`，也无法通过 `defaultVisible: false` + `reactions` 控制其显隐（虽然可通过 `options.length === 0` 来模拟隐藏，但不是规范做法）。

### S2（🔴）`dateRangePicker` 的 `reactions` 语义未定义

**位置：** `03-component-registry.md` dateRangePicker 章节 + `schemas/component-registry.json` `dateRangePicker.props`

**问题：** `dateRangePicker` 的 `supportsReactions` 为 `true`，但它拥有 **两个字段值**（`startField` / `endField`），而非单一 `field`。当前文档和 Schema 对以下关键问题均未给出定义：

| 问题 | 具体表现 |
|---|---|
| 其他字段如何通过 `$deps` 引用它的值？ | `dependencies: [??]` — 应填写 `startField` 的值、`endField` 的值、还是两者？ |
| `$deps.xxx` 取到什么类型的数据？ | 是 ISO 字符串、Date 对象、还是数组 `[start, end]`？ |
| `$self` 指什么？ | `dateRangePicker` 没有单一的 `field`，所以 `$self` 的概念在此不适用 |
| 它自己的 `reactions` 如何工作？ | 当它的两个字段之一被外部修改时，如何触发联动？ |

**示例——歧义场景：**

```yaml
# 如下配置中，$deps.dateRange 的值到底是什么？
type: input
props: { field: remark, label: 备注 }
reactions:
  - dependencies: [dateRange]     # ← 这里该写 startField 还是 endField？
    when: "$deps.dateRange == '2026-01-01'"   # ← 比较的是起始日期还是结束日期？
    fulfill:
      visible: false
```

**影响：** 🔴 严重。开发者使用 `dateRangePicker` 的 `reactions` 功能时缺少权威指引，只能自行猜测行为，不同 Renderer 实现可能产生歧义。

### S3（🔴）`02-reaction-expression.md` §9.3 措辞错误

**位置：** `02-reaction-expression.md` §9.3 表格上方正文

**当前文本：**
> `scope: row` 的 `reactions`/`otherwise` 中禁止声明 `required` 或 `value` 状态键（仅允许 `visible` 和 `disabled`）。

**问题：** `reactions` 是数组名（`reactions: [{ fulfill, otherwise }]`），不是 `StateMap` 字段名。正确的字段名是 `fulfill`。当前措辞会将读者导向错误的字段。

**正确应为：**
> `scope: row` 的 `fulfill`/`otherwise` 中禁止声明 `required` 或 `value` 状态键（仅允许 `visible` 和 `disabled`）。

**与 ADR 对照：** ADR-0004 §D2a 表格中的正确措辞为：
> `scope: row` 下的 `reactions`/`otherwise` 仅允许声明 `visible` 和 `disabled` 两个状态键

但 ADR-0004 的 §D2a **正文**本身使用的是 `reactions`/`otherwise`——这意味着错误源头在 ADR 本身，在合并入 `02-reaction-expression.md` 时未被发现。

**影响：** 🔴 严重。术语错误会误导读者认为 `reactions` 元素本身可以直接声明状态键，导致写出不合法的配置。

### S4（🟡）`data.params` 中 `$deps.*` 的作用域边界未定义

**位置：** `04-datasource-contract.md` §3.1 + `08-renderer-spec.md` §2.3

**问题：** 两处文档都描述了 `data.params` 中可以引用 `$deps.*` 做值替换（如 `params: { status: $deps.statusFilter }`），但**完全没有定义该机制的作用域边界**。

对比协议中其他使用 `$deps.*` 的位置：

| 位置 | 作用域规则 | 定义状态 |
|---|---|---|
| `reactions[].when` | 必须有显式 `dependencies` | ✅ 已定义 |
| `visibleWhen.when`（表单内） | 必须有显式 `dependencies` | ✅ 已定义 |
| `visibleWhen.when`（非表单） | 禁止 `$deps.*`，静态校验拒绝 | ✅ 已定义 |
| `permissions.*` | 禁止 `$deps.*`，静态校验拒绝 | ✅ 已定义 |
| `data.params` 值替换 | **未声明任何规则** | ❌ **缺失** |

**未定义的具体问题：**
1. 非表单上下文（独立 `table`/`chart`）的 `data.params` 中是否允许 `$deps.*`？
2. 如果在不允许的位置使用了 `$deps.*`，是静默取到 `undefined`、还是静态校验拒绝？
3. `$deps.*` 在 `data.params` 中的"依赖声明"如何与 `reactions`/`visibleWhen` 的 `dependencies` 声明机制对应？是否需要独立的参数替换作用域声明？

**影响：** 🟡 中等。后端开发者可能写出在非表单上下文中使用 `$deps.*` 的 `data.params`，运行时静默取到 `undefined`，难以排查。

### S5（🟡）`form` 搜索模式示例结构不合法

**位置：** `03-component-registry.md` form 组件章节，搜索模式示例

**当前内容：**
```yaml
body:
  type: form
  props:
    mode: search
    targetTable: orderTable
  children:
    - type: dateRangePicker
    ...

# 搜索结果表格    ← 这里有问题
- type: table
  id: orderTable
```

**问题：** `body` 是单个 Node（类型为 `object`），不是数组。示例中在 `form` 之后直接跟了一个 `- type: table`，这在 YAML 中相当于把 `body` 视为数组——但 `01-node-protocol.md` §2 明确定义 `body` 是 `Node`（单一节点），不是 `[Node]`（Node 数组）。

**正确的做法**是将 `form` 和 `table` 包裹在一个容器节点（如 `section` 或 `grid`）中作为 `body`，二者作为该容器的 `children`：

```yaml
body:
  type: section
  children:
    - type: form
      props:
        mode: search
        targetTable: orderTable
      children:
        - type: dateRangePicker
          ...
    - type: table
      id: orderTable
      ...
```

**影响：** 🟡 中等。示例是搜索模式最核心的文档参考，结构错误会导致读者/AI 助手直接抄写无效配置。

### S6（🟢）`CHANGELOG.md` v0.2.1 中 "searchForm（方案 A）" 暗示存在方案 B 但从未说明

**位置：** `CHANGELOG.md` v0.2.1 条目

**当前文本：**
> - **searchForm（方案 A）：** `form` 新增 `mode: search` + `targetTable`

**问题：** 标记为"方案 A"意味着存在"方案 B"，但文档/ADR 中任何地方都没有讨论或提及方案 B。读者会自然追问"方案 B 是什么""为什么没被采纳"。

影响的文档包括：
- `CHANGELOG.md`（直接来源）
- 归档审计 `0003-*-plan.md`（若计划中有多方案讨论但未记录）

"方案 A"这个措辞在 CHANGELOG（最终记录）中出现不合适——CHANGELOG 记录的是"做了什么"，而不是"讨论过什么方案"。

**影响：** 🟢 轻微。不造成功能问题，但造成读者困惑。

### S7（🟢）`datePicker` / `dateRangePicker` / `upload` 在 Schema 中 `placeholder` 缺少 `since` 标注

**位置：** `schemas/component-registry.json` 中 `datePicker.props`、`dateRangePicker.props`、`upload.props`

**问题：** 三个组件均在 v0.2.1 引入，其属性 `description` 和 `tooltip` 标注了 `since: "0.2"`，但 `placeholder` 字段**没有 `since` 标注**。

```json
// datePicker.props
"placeholder": { "type": "string", "required": false },
"description":  { "type": "string", "required": false, "since": "0.2" },
"tooltip":      { "type": "string", "required": false, "since": "0.2" }
```

`dateRangePicker` 和 `upload` 也有相同的问题。虽然因为组件本身是 v0.2.1 新增（所有字段默认即该版本），标注不一致不会造成行为错误，但对按 `since` 字段做自动化处理的工具不友好。

**影响：** 🟢 轻微。
