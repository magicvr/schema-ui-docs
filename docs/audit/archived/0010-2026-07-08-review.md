### V1（🟡）Renderer 并行加载示例将 `body` 写成数组，违反顶层 `body: Node` 结构

**位置：** `08-renderer-spec.md` §2.1 默认并行加载

**对应规范：** `01-node-protocol.md` 顶层结构、`schemas/page.schema.json`

**问题：** §2.1 示例写法为：

```yaml
body:
  - type: statCard
    props: { label: 今日订单数, valueField: total }
    data: { source: api, url: /api/stats/order-count }
  - type: statCard
    props: { label: 今日收入, valueField: amount }
    data: { source: api, url: /api/stats/revenue }
  - type: chart
    props: { chartType: bar, xField: date, yField: sales }
    data: { source: api, url: /api/stats/sales-trend }
```

但当前协议明确要求顶层 `body` 是单个 `Node`，不是 `Node[]`。因此该示例复制后会生成不符合 `page.schema.json` 的页面配置。

**影响：** 🟡 中等。该示例位于 Renderer 实现规范中，容易被前端实现方或配置生成方直接复制，导致与顶层协议结构不兼容。

**建议：** 将示例包裹在容器节点内，例如 `grid` 或 `section`：

```yaml
body:
  type: grid
  props:
    columns: 3
  children:
    - type: statCard
      props: { label: 今日订单数, valueField: total }
      data: { source: api, url: /api/stats/order-count }
    - type: statCard
      props: { label: 今日收入, valueField: amount }
      data: { source: api, url: /api/stats/revenue }
    - type: chart
      props: { chartType: bar, xField: date, yField: sales }
      data: { source: api, url: /api/stats/sales-trend }
```

### V2（🟡）Renderer `$deps.*` 数据依赖示例与同节作用域边界冲突

**位置：** `08-renderer-spec.md` §2.3 数据依赖声明

**对应规范：** `04-datasource-contract.md` §3.2、`02-reaction-expression.md` 作用域隔离规则

**问题：** §2.3 示例展示了独立 `table` 在 `data.params` 中直接引用 `$deps.*`：

```yaml
type: table
data:
  source: api
  url: /api/orders
  method: GET
  params:
    status: $deps.orderStatusFilter
    dateFrom: $deps.dateRange
```

但同一小节随后明确声明：`$deps.*` 在 `data.params` 中的引用仅在表单上下文有效，非表单上下文（独立 `table`/`chart`）出现 `$deps.*` 时应被静态校验拒绝。

**影响：** 🟡 中等。示例和约束在同一段落内互相冲突，会让接入方无法判断应实现“独立表格依赖筛选字段”还是“独立表格静态拒绝 `$deps`”。

**建议：** 改写示例以显式体现表单上下文，例如使用 `form mode: search` + `targetTable`，或将该 `table` 放在明确的表单/搜索容器语境中。若保留独立 `table` 示例，则应移除 `$deps.*`，改用静态参数或搜索模式数据流说明。

### V3（🟡）跨作用域联合判断的“变通做法”描述了协议尚未定义的机制

**位置：** `02-reaction-expression.md` §9.1 作用域隔离规则

**对应规范：** `reaction.schema.json`、ADR-0004 行级作用域规则

**问题：** 文档声明 v0.2 不支持同时依赖表单级字段和行内字段的跨作用域联合判断，但随后给出变通建议：

```text
由业务方在表格组件之外用 scope: form 的表达式算出静态布尔值，通过组件的静态 props 传入。
```

当前协议没有定义“在表格组件之外计算静态布尔值”的挂载位置，也没有定义该布尔值如何进入表格 `props` 后再被行级表达式读取。该描述会让读者误以为协议已有可落地语法。

**影响：** 🟡 中等。不会破坏现有 Schema，但会误导实现方设计出私有扩展，并将其误认为 v0.2 标准能力。

**建议：** 收紧为负向边界说明：v0.2 不提供跨作用域联合判断的协议级语法。如确有需求，应由后端预计算为行数据字段，或通过 ADR 新增标准机制。

### V4（🟢）`tabs` 当前规范和注册表仍残留“方案 A”历史措辞

**位置：** `03-component-registry.md` `tabs` 组件、`schemas/component-registry.json` `tabs.props.items.description`

**相关说明：** `CHANGELOG.md` 中历史版本条目保留方案编号可以接受；当前规范正文与机器可读描述不宜要求读者理解旧审计方案。

**问题：** 当前规范正文仍写：

```text
v0.2 变更（B2，方案 A）
```

注册表描述仍写：

```text
见方案 A，审计 §2.4
```

这类措辞属于历史决策过程，不是当前协议结果。此前 `CHANGELOG.md` 已有过清理“方案 A”歧义的修复方向，但当前规范和注册表仍有残留。

**影响：** 🟢 低。属于描述层漂移，不影响机器结构校验，但会影响当前规范的可读性和基于注册表生成说明的质量。

**建议：** 将当前规范正文和注册表描述改为结果导向措辞，例如：

```text
v0.2 变更：tabs 内容改为 items[].content 内嵌，不再依赖顶层 children 按 key 匹配。
```

`CHANGELOG.md` 中作为历史发布说明的 `B2（方案A）` 可保留，也可按文档风格统一清理。
