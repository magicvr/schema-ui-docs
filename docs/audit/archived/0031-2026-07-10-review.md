## 发现的问题

### V67（🔴）表格 `columns[]` / `actions[]` 内嵌 `visibleWhen` / `reactions` 的结构约束未被实际校验

**位置：**
- `docs/schemas/component-registry.json`：`table.props.columns.items.properties.visibleWhen` `$ref: node.schema.json#/definitions/VisibleWhen`，`reactions.items` `$ref: reaction.schema.json`；`table.props.actions.items` 同类字段
- `scripts/validate-l2-components.js`：`validateNestedObject()` 只处理 `properties` / `required` / `additionalProperties` / enum / type，不解析 DSL 中的 `$ref`
- `scripts/validate-l3a-expressions.js`：只在 `when` 存在时校验表达式变量与作用域，不校验 `visibleWhen` / `reactions` 的必填字段和 `fulfill` / `otherwise` 状态键结构
- 对照：`docs/02-reaction-expression.md` §9.3 / §10.4、`docs/03-component-registry.md` `table` ColumnDef / RowAction、`docs/06-validation.md` L3a 与自检清单
- 间接影响：MCP `protocol.validate_content` 复用上述脚本，因此同样误报通过

**当前值：**
- 正文：表格列/行内操作的 `visibleWhen` / `reactions` 复用表达式结构；`scope: row` 下 `fulfill` / `otherwise` 仅允许 `visible` / `disabled`，`required` / `value` 应静态拒绝。
- DSL：`component-registry.json` 使用 `$ref` 指向标准 `VisibleWhen` / `reaction.schema.json`，看起来表达了完整结构约束。
- 实现：L0/L1 只把 `props` 当自由对象；L2 不解析 `$ref`；L3a 只校验存在的 `when` 字符串，不检查 `when` 缺失、`fulfill.required` 或 `fulfill.value`。

**复现（已实测）：**

1. 表格列 `visibleWhen` 缺少必填 `when`：

```yaml
body:
  type: table
  props:
    rowKey: id
    pagination: { mode: server }
    columns:
      - field: id
        label: ID
        visibleWhen:
          scope: row
          dependencies: [id]
  data: { source: api, url: /api/items }
```

`node scripts/validate-all.js <tmp.yaml> --json` 返回 `passed: true`，L0/L1、L2、L3a、L4 均无违规。

2. 表格列 `scope: row` 的 `reactions.fulfill.required`：

```yaml
columns:
  - field: id
    label: ID
    reactions:
      - scope: row
        dependencies: [id]
        when: "$row.id == '1'"
        fulfill:
          required: true
```

全链路返回 `passed: true`。

3. 表格行内 action `scope: row` 的 `reactions.fulfill.value`：

```yaml
actions:
  - key: approve
    label: Approve
    reactions:
      - scope: row
        dependencies: [id]
        when: "$row.id == '1'"
        fulfill:
          value: true
```

全链路返回 `passed: true`。

4. 对照：普通 Node `reactions` 缺少 `when` 会被 L0/L1 正确拒绝（`must have required property 'when'`），说明缺口集中在 table props 内嵌表达式对象，而不是 `reaction.schema.json` 本身失效。

**问题：** 表格内嵌表达式对象被写在组件 DSL 的 `props` 结构内，实际校验链没有执行 `$ref` 指向的标准 Schema，也没有在 L2/L3a 补齐等价校验。机器可读契约给出“已引用标准结构”的外观，但 CI / MCP 会接受明显违反正文的配置。

**影响：** 🔴 严重。表格行级表达式是 v0.2.1 后的核心能力；缺少 `when` 或声明 `required` / `value` 这类非法状态键会让 Renderer 面临未定义行为。更严重的是，官方校验和 MCP 会误报通过，削弱“与 CI 同版本脚本一致”的可信度。

**建议修复：**
1. L2：在 `validate-l2-components.js` 中为 DSL `$ref` 增加受控解析，至少覆盖 `node.schema.json#/definitions/VisibleWhen` 与 `reaction.schema.json`；或显式专门校验 `table.props.columns[].visibleWhen` / `reactions[]` 和 `table.props.actions[].visibleWhen` / `reactions[]` 的结构。
2. L3a：补充 `scope: row` 下 `fulfill` / `otherwise` 状态键校验，覆盖 Node.reactions 与表格列/行内操作 reactions；也可把该项放入 L2，但 `06` 中需写清层级归属。
3. 增加反例测试：`columns[].visibleWhen` 缺 `when`、`columns[].reactions[].fulfill.required`、`actions[].reactions[].fulfill.value` 均应失败；普通合法行级显隐示例应通过。
4. MCP 测试补充至少一条 `validate_content` 回归，确保同类配置返回 `passed: false`。

**需要同步的文件：**
- `scripts/validate-l2-components.js`
- `scripts/validate-l3a-expressions.js`（若状态键检查放在 L3a）
- `docs/06-validation.md`（明确表格内嵌表达式结构与状态键检查的层级）
- `mcp/tests/validate-content.test.ts` / `mcp/tests/test-utils.ts`
- 可选：`docs/schemas/component-registry.json` description，注明 DSL `$ref` 必须由 L2 解析而非 AJV 自动处理

### V68（🟡）`08-renderer-spec.md` §5.5 的 L3 静态校验清单落后于当前 L3a 规则

**位置：** `docs/08-renderer-spec.md` §5.5「静态校验（L3）」

**当前值：** §5.5 只列出三条：

```text
- 所有 when 表达式中的 $deps.* 变量必须在 dependencies 声明中。
- permissions.* 表达式中不能出现 $deps.*。
- 非表单上下文的 visibleWhen 中不能出现 $deps.*。
```

**对照：**
- `docs/02-reaction-expression.md` §9 / §10 / 附录 A 已定义完整变量矩阵与静态校验规则。
- `docs/06-validation.md` L3a 已声明覆盖 `data.params`、`select.props.optionsSource.params`、`datasources.*.params`、表格列/操作 `scope: form` 的 form 上下文、`modal.content` 等。
- `scripts/validate-l3a-expressions.js` 当前已实现 `$row` / `$deps` scope 隔离、`$parentRow` 边界、表格 action 禁 `$self`、非 form reactions 禁 `$deps`、data/optionsSource/datasources params 变量限制等。

**问题：** Renderer 规范是前端实现者最直接查阅的文档，但它的 L3 清单仍停留在早期三条规则。虽然 §5.5 开头引用了 `06-validation.md §1`，但随后用“应执行以下静态校验”列出有限清单，容易被理解为完整要求，导致独立 Renderer 按该清单实现时低于当前协议边界。

**影响：** 🟡 中等。不会改变 Schema，但会造成前端实现、MCP/CI 脚本与协议正文三者在 L3a 覆盖面上漂移；这与 V43/V50/V58/V59/V65 等持续补齐 L3a 规则的历史修复方向相反。

**建议修复：**
1. 将 §5.5 改为“最低必须覆盖 `02` §10 与附录 A / `06` L3a 中列出的全部规则”，避免把三条旧规则当完整清单。
2. 补充当前高风险规则列表：scope 隔离、`$parentRow`、表格 action 禁 `$self`、`scope: row` 状态键限制、表格 `scope: form` 的 form 上下文、params 值替换变量限制、`datasources.*.params` 非 form 规则。
3. 若 V67 修复后把表格内嵌表达式结构检查归入 L2/L3a，也同步在 §5.5 或 §5.1 指明。

**需要同步的文件：**
- `docs/08-renderer-spec.md`
- 可选：`docs/06-validation.md`（若 V67 调整校验层级描述）

### V69（🟡）MCP `internalError` 返回结构与 ADR-0007 不一致

**位置：**
- `docs/decisions/0007-mcp-protocol-reader-validator.md` D11
- `mcp/src/types.ts` `ValidateContentResult`
- `mcp/src/core/validation-runner.ts` `validateContent()` / `summarize()`

**当前值：** ADR-0007 D11 明确：

```json
{
  "message": "bad indentation of a mapping entry",
  "line": 12,
  "column": 5
}
```

并写明 `parseError` 与 `internalError` 的最小结构相同，MCP 内部异常时 `internalError` 填入错误对象。

但当前 TypeScript 类型为：

```ts
internalError: string | null;
```

实现中也直接写入字符串：

```ts
internalError: 'format 必须是 yaml 或 json'
internalError: 'content 超过 1MB 限制'
internalError: error instanceof Error ? error.message : String(error)
```

**问题：** 已接受 ADR 承诺的工具返回结构与 MCP 当前实现不一致。`parseError` 是结构化对象，`internalError` 却是字符串，客户端需要分支处理，违反“返回字段兼容策略：已发布字段不得删除或改变含义；未来扩展只能新增字段”的稳定性原则。

**影响：** 🟡 中等。正常校验路径不受影响，但 MCP 调用方在内部异常、输入过大或绕过工具 schema 的非法 format 场景下会收到与 ADR 不一致的结构，削弱工具集成的可预测性。

**建议修复：**
1. 将 `internalError` 类型改为结构化对象（可复用 `ParseError` 或新增 `ToolError`），最小字段为 `message`。
2. `validation-runner.ts` 中所有 `internalError` 赋值改为 `{ message: ... }`。
3. `summarize()` 改为读取 `internalError.message`。
4. 补充 MCP 单测：超大内容或内部错误注入时 `internalError` 为对象且不包含堆栈。
5. 若决定把 format/1MB 归类为输入错误而非 internalError，需要同步 ADR 或 v1 设计；否则按 ADR 结构化即可。

**需要同步的文件：**
- `mcp/src/types.ts`
- `mcp/src/core/validation-runner.ts`
- `mcp/tests/validate-content.test.ts`
- 可选：`docs/mcp/v1-design.md` / `docs/mcp/v1-implementation-plan.md`（若调整输入错误分类）

## 与已归档审计的关系

- V67 与 0004 中“表格列/行内操作补充 `visibleWhen` / `reactions` `$ref`”相关，但此前只把 `$ref` 写入 DSL，没有审计 L2 是否实际解析；也与 0009 中“`reaction.schema.json` 已禁止 row scope required/value”相关，本轮发现该禁止对 table props 内嵌 reactions 没有生效。
- V67 与 0024 V36（表格 `$row.*` 必须显式 `scope: row`）同属表格内嵌表达式校验，但 V36 修的是表达式变量 scope 默认值，本轮修的是结构约束与状态键约束未执行。
- V68 与 V43/V50/V58/V59/V65 等 L3a 覆盖边界修复相关；本轮不是脚本缺口，而是 Renderer 规范的清单未随规则扩展同步。
- V69 与 ADR-0007 / MCP v1 实现直接相关；此前 0030 只确认 MCP 复用校验脚本，没有审计工具错误对象结构。
- V70 与 0014 V4 同类，但当前仅 0030 中一个可选项未勾选；不是 0013 大面积未勾选问题的重复。

## 建议处理顺序

1. **先修 V67**：会导致 CI/MCP 对协议非法表格表达式误报通过，影响最大。
2. **再修 V69**：MCP 对外返回契约应尽快对齐 accepted ADR，避免客户端集成依赖错误结构。
3. **同步 V68**：Renderer 规范补齐当前 L3a 清单，减少独立实现漂移。
4. **收尾 V70**：恢复审计证据链自洽。

## 防止复发的建议

1. `component-registry.json` 中凡出现 `$ref`，L2 校验器必须有明确解析策略或等价专门校验；不得只把 `$ref` 当文档说明。
2. 新增 L3a 规则时，同步更新 `02`、`06`、`08` 三处：表达式权威规则、校验层级说明、Renderer 实现清单。
3. MCP ADR 的返回结构变更应由 TypeScript 类型测试锁定；至少覆盖 `parseError`、`internalError`、各 layer violation 的 shape。
4. 归档前对 checklist 执行 `\[ \]` 检查；如果存在可选未执行项，应显式标记为 `[x]` 并写明“不执行”的决策原因，或不要在索引中写“全部已完成”。
