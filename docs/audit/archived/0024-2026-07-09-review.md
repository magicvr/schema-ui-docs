# 审计报告 0024：MVP 协议稳定性复查——能力协商 + L3a scope + 组件语义边界

## 背景

本轮审计基于对全量项目文档的 MVP 就绪度评估。前置结论是：现有文档已经足以支撑网格看板、服务端分页表格、表单提交与表单联动的 MVP 实现，但仍有若干会影响“前后端共用协议稳定性”的边界问题。

## 发现的问题

### V35（🔴）PATCH 级执行能力缺少显式协商机制

`meta.protocolVersion` 只声明 MAJOR.MINOR（如 `"0.2"`），而 v0.2.5 已新增 `upload` action / `upload.props.actionRef` 这类需要 Renderer 执行支持的能力。若旧 Renderer 也声明支持 `"0.2"`，页面生成方无法表达“本页面需要 v0.2.5 的 upload 执行能力”。

**影响：** 同为 `"0.2"` 的页面和 Renderer 可能在运行时才暴露能力缺失，削弱协议冻结后的兼容性预期。

**处理：** 新增 `meta.requiredCapabilities` 与 Renderer `supportedCapabilities`，当前预定义能力键 `actions.upload`。L2 校验器要求使用 `upload` action 或 `upload.props.actionRef` 的页面显式声明该能力。

### V36（🔴）L3a 表格表达式默认 `scope: row`，放过省略显式 scope 的 `$row.*`

正文与 ADR-0004 均要求 `$row.*` 的使用必须显式声明 `scope: row`。但 `scripts/validate-l3a-expressions.js` 对表格列/行内操作内的 `visibleWhen` / `reactions` 默认使用 `row`，导致省略 `scope` 仍能通过校验。

**影响：** 后端生成器可能产出 CI 通过、但严格 Renderer 拒绝的配置。

**处理：** L3a 中表格列/行内操作表达式默认回到 `form`，从而使 `$row.*` 在省略 `scope: row` 时触发 `SCOPE_ISOLATION`。

### V37（🟡）`text` 组件数据驱动语义与 `content/contentKey` 必填关系需要澄清

`text` 组件支持 `data.source: api` 与 `props.valueField`，但组件契约仍要求 `content` / `contentKey` 二选一。若不解释二者关系，后端可能误以为可以生成纯 API 驱动的 `text` 节点。

**影响：** 不阻断当前 MVP 三个核心示例，但会造成配置生成口径分歧。

**处理：** 明确 `content` / `contentKey` 仍为必填，声明 `data` 时作为加载前/无数据时兜底文案；API 展示值仍由 `valueField` 或默认第一个字段决定。

### V38（🟡）`RowAction.key` 的执行边界需要更明确

此前已统一 `RowAction.key` 不引用顶层 `actions`，但还需要明确点击后的执行归属，避免实现方误以为协议已定义行级后端请求执行模型。

**影响：** 若 MVP 包含“退款/删除/审批”这类行级后端操作，当前协议只能由前端预注册处理器自行实现，不具备标准化后端请求映射。

**处理：** 在 `03-component-registry.md` 与 DSL 描述中明确 `key` 仅用于前端预注册处理器本地分发，后端请求型行操作需后续 ADR 标准化 `rowAction.actionRef` 一类字段。

## 结论

完成本轮修复后，协议更适合作为 MVP 前后端共用约定：结构版本仍由 `protocolVersion` 锚定，PATCH 级执行能力由 `requiredCapabilities` 显式声明，校验脚本能阻断关键误用。行级后端动作仍是明确的后续扩展点，不应纳入当前 MVP 的稳定承诺范围。
