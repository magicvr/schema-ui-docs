## 发现的问题

### V49（🔴）`DataRef` 的“三选一”来源语义未被 Schema 强制，且额外字段未被禁止

**位置：** `docs/01-node-protocol.md` §3.3；`docs/schemas/node.schema.json` `definitions.DataRef`

**当前值：** `01-node-protocol.md` 将 `data` 描述为三选一：

```yaml
data:
  source: static | ref | api
  value: any        # source=static 时的字面量
  ref: string       # source=ref 时，指向 datasources 中的 key
  url: string       # source=api 时，独立请求的地址
```

但 `node.schema.json` 的 `DataRef` 只根据 `source` 补充对应必填字段，没有设置 `additionalProperties: false`，也没有禁止不同来源形态的字段混用。例如以下结构会通过 Schema 的结构层检查：

```yaml
data:
  source: static
  value: []
  url: /api/orders
  method: GET
  params:
    status: paid
```

同理，`source: api` 也可以同时携带 `value` / `ref`，`source: ref` 也可以携带与引用语义不清的额外字段。

**问题：** 正文把数据来源定义为互斥形态，但机器可读 Schema 只表达“某来源至少需要哪些字段”，没有表达“该来源只能携带哪些字段”。这会让 L0/L1 接受语义冲突的 `DataRef`，后续 Renderer 必须自行猜测优先级。

**影响：** 🔴 严重。`data.source` 是核心协议字段，若 Schema 接受混合来源，配置生成方、CI 与 Renderer 对“实际读取 static 值还是发起 api 请求”可能产生不同理解，属于协议层破坏性歧义。

**建议修复：** 在 `DataRef` 中补充 `additionalProperties: false`，并按 `source` 明确互斥字段集合：`static` 仅允许 `source` / `value`；`api` 允许 `source` / `url` / `method` / `params` / `responseMapping`；`ref` 的允许字段需先决策是否支持本地 `responseMapping` 覆盖引用数据源，随后同步写入 `01-node-protocol.md` 与 Schema 条件约束。

### V50（🟡）`data.params` 中 `$deps.*` 的作用域规则未进入表达式规范的使用位置与变量矩阵

**位置：** `docs/02-reaction-expression.md` §1 / §2 / 附录 A；`docs/04-datasource-contract.md` §3.2；`docs/08-renderer-spec.md` §2.3；`docs/06-validation.md` §1

**当前值：** `04-datasource-contract.md` 和 `08-renderer-spec.md` 已明确：`data.params` 中的 `$deps.*` 仅在表单上下文有效，非表单上下文的独立 `table` / `chart` 中出现 `$deps.*` 应静态拒绝。

但 `02-reaction-expression.md` 的“使用位置”表只列出 `reactions.when`、`visibleWhen.when`、`permissions.*`、表格列/操作和 `select.optionsSource.params`，没有列出 `data.params`。变量可见性矩阵也没有 `data.params` 行。`06-validation.md` 的 L3a 描述主要围绕表达式语法与 `when` 字段，没有明确 `data.params` 中 `$deps.*` 这种“值替换但需要作用域静态检查”的归属层级。

**问题：** `data.params` 的 `$deps.*` 不是条件表达式，但它复用了同一套变量命名空间和作用域边界。当前规则分散在 04/08 中，未进入 02 的统一变量表和 06 的校验边界，容易导致实现方只校验 `when` 表达式，漏掉 `data.params` 中的非法 `$deps.*`。

**影响：** 🟡 中等。不会改变字段结构，但会造成静态校验责任不清：独立 `chart.data.params.status: $deps.status` 这类配置在文档语义上应被拒绝，工具链或 Renderer 若只按 02/06 实现，可能不会覆盖。

**建议修复：** 在 `02-reaction-expression.md` 的使用位置表和变量可见性矩阵补充 `data.params` 行，明确其只做值替换、不执行条件判断；在 `06-validation.md` 中说明该检查属于 L2 或 L3a 的静态作用域校验责任；必要时同步校验脚本以覆盖 `data.params`。

### V51（🟡）ADR-0004 仍使用旧的 `columns[].key` / `$row[column.key]`，与当前 `ColumnDef.field` 不一致

**位置：** `docs/decisions/0004-row-level-scope.md` D1 / D2b；`docs/03-component-registry.md` `ColumnDef`；`docs/schemas/component-registry.json` `table.props.columns.items`

**当前值：** ADR-0004 的示例仍写：

```yaml
columns:
  - key: status
    visibleWhen:
      scope: row
      dependencies: [level]
      when: "$row.level == 'high'"
```

同一 ADR 对 `$self` 的定义写作：

```text
$self ... 即 $row[column.key]
```

但当前 `03-component-registry.md` 和 `component-registry.json` 的表格列字段均为 `field`，没有 `key`。

**问题：** 这是 Accepted ADR 中的旧字段名残留。由于 ADR-0004 是 `$row` / `$self` 作用域语义的来源，残留 `key` 会让实现方误以为列定义支持 `key`，或把 `$self` 绑定到不存在的 `column.key`。

**影响：** 🟡 中等。正文与机器 DSL 已经使用 `field`，但 ADR 作为设计依据仍可能被 AI 或实现者引用，导致行级表达式实现和配置生成出现字段名漂移。

**建议修复：** 将 ADR-0004 中列示例的 `key: status` 改为 `field: status`，并将 `$row[column.key]` 改为 `$row[column.field]`。如需保留历史语境，应加一句说明早期草案使用过 `key`，最终协议统一为 `field`。

### V52（🟡）`tagMap` 正文要求 `{text, tone}`，但组件 DSL 未要求 `text` 且未禁止额外字段

**位置：** `docs/03-component-registry.md` `table` / `ColumnDef`；`docs/schemas/component-registry.json` `table.props.columns.items.properties.tagMap`

**当前值：** `03-component-registry.md` 将 `tagMap` 定义为：

```text
format: tag 时，值 → {text, tone} 的映射。tone 可选值：warning|success|neutral|info|danger
```

但 `component-registry.json` 中 `tagMap.additionalProperties` 的项只声明了 `text` 与 `tone` 的类型，没有 `required: ["text"]`，也没有 `additionalProperties: false`。

**问题：** DSL 会接受没有展示文案的映射项，例如 `{ tone: "success" }`，也可能接受额外字段。正文虽然描述了 `{text, tone}` 的形态，但机器契约没有表达最小可展示结构。

**影响：** 🟡 中等。`format: tag` 是表格常用展示能力，若标签映射缺少 `text`，Renderer 无法按协议稳定展示状态文案；若额外字段被接受，也会削弱组件 DSL 的 `additionalProperties` 收敛策略。

**建议修复：** 在 `component-registry.json` 的 `tagMap` 映射项中补充 `required: ["text"]` 与 `additionalProperties: false`。如果协议决定 `text` 可选，则需反向修改 `03-component-registry.md`，明确缺省展示值的规则。

### V53（🟡）`action.schema.json` 未按 `type` 禁止不相关字段，机器契约接受混合 Action

**位置：** `docs/07-actions-contract.md` §3-§8；`docs/schemas/action.schema.json`

**当前值：** `07-actions-contract.md` 分别定义了 `request`、`navigate`、`modal`、`custom`、`upload` 的完整字段。例如 `navigate` 只有 `type` / `url`，`upload` 才有 `fieldName` / `accept` / `maxSize` / `multiple`。

但 `action.schema.json` 把所有 action 字段放在同一个顶层 `properties` 中，只用 `if/then` 补充条件必填，没有为各 `type` 禁止不相关字段。因此以下结构在机器契约上没有被排除：

```yaml
goOrders:
  type: navigate
  url: /orders
  bodyMapping:
    customerName: name
  fieldName: file
```

**问题：** Schema 表达了“某类型至少需要哪些字段”，但没有表达“某类型只允许哪些字段”。这与正文中“各 action 类型完整字段”的边界不一致。

**影响：** 🟡 中等。混合 Action 会让 Renderer 对无关字段是忽略、警告还是报错产生分歧，也会让 AI 生成配置时把不同类型的字段拼接到一起而仍通过 L1。

**建议修复：** 将 `action.schema.json` 改为类型分支式结构（如 `oneOf` + 各类型独立 `required` / `additionalProperties: false`），或在现有 `if/then` 中补充 `not` 约束，禁止当前 `type` 不相关字段。同步确认 `OutcomeBehavior` 条件必填不受影响。

## 与已归档审计的关系

- V49 与 0002 D7、0017 V19、0027 V47/V48 同属数据源契约复查，但本轮对象是 `DataRef` 来源形态的互斥结构，不是 `method` 枚举、`params` 空值规则或响应体分类。
- V50 与 0007 S4、0026 V43 相关，但本轮不是 `$parentRow.*` 可见性或非表单 `visibleWhen` 问题，而是 `data.params` 的 `$deps.*` 规则没有进入统一表达式矩阵和校验层级说明。
- V51 与 0022/0024 中的 RowAction 语义收敛不同，本轮是 ADR-0004 自身残留旧列字段名。
- V52 与 0004/0009/0017/0027 中的组件 DSL 修复同类，但此前未覆盖 `tagMap` 映射项内部约束。
- V53 与 0005/0009 对 `action.schema.json` 的结构修复相关，但本轮聚焦 action 类型分支之间的字段隔离。
- V54 与 0018 V21、0025 V42 同属元数据同步问题，是 0027 修复后的新触达文档漂移。

## 建议处理顺序

1. 优先修复 V49。它影响核心 `DataRef` 互斥语义，且会让 L1 接受协议矛盾配置。
2. 修复 V53。Action 类型混合字段会影响执行契约边界，适合与 Schema 结构一起收敛。
3. 修复 V50。先明确 `data.params` 的静态校验归属，再决定是否补脚本。
4. 修复 V52。补齐 `tagMap` 最小可展示约束。
5. 修复 V51 与 V54。二者为文档同步型问题，改动小、风险低。

## 防止复发建议

- 对“至少需要哪些字段”和“只允许哪些字段”分开审查。Schema 的 `required` 只能表达下限，互斥来源和类型分支必须单独建反例验证。
- 所有 Accepted ADR 中的代码片段应纳入字段名漂移抽查，尤其是早期草案常用 `key` / `field` 这类易变字段。
- 对 `data.params`、`optionsSource.params`、`requestMapping` 这类“不是表达式但使用变量命名空间”的字段建立统一矩阵，避免只校验 `when` 表达式。
- 每轮修复触达带 frontmatter 的协议文档时，把 `last_updated` 作为 checklist 子项处理。
