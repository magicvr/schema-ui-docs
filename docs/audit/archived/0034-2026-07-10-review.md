# 审计报告 0034：L2 引用完整性校验不对称 + 文档元数据漂移

## 背景

本轮审计基于 0033 已归档后的再次全量文档、Schema、校验脚本与 MCP 实现复审。0033 中发现的 `upload.props.actionRef` 能力声明提示、`permissions` 作用域表述、扩展键口径、`protocol.get_doc` 示例/截断、`v1-design.md` 状态漂移在当前工作区已确认修复，本轮不重复报告 V79–V84。

本轮重点交叉核对：页面内各类“字符串引用”字段是否在 L2 中获得对称的存在性/类型校验。已确认 `RowAction.actionRef` 具备完整 L2 规则，但 `form.props.submitAction`、`upload.props.actionRef`、`data.source: ref`、`form.props.targetTable` 仍可携带悬空或不匹配目标且通过全链路校验。

## 发现的问题

### V85（🔴）L2 对 `submitAction` / `upload.props.actionRef` 的引用完整性校验缺失，与 `RowAction.actionRef` 不对称

**位置：**
- `scripts/validate-l2-components.js`：`validateRowActionRefs()` / `validateRequiredCapabilities()` 仅覆盖 `RowAction.actionRef` 与能力声明
- 对照文档：
  - `docs/03-component-registry.md`：`form.props.submitAction` 引用顶层 `actions` id；`upload.props.actionRef` 引用 `type: upload` action
  - `docs/07-actions-contract.md` §1：顶层 `actions` 供 `submitAction` / `upload.props.actionRef` / `table.props.actions[].actionRef` 引用
  - `docs/06-validation.md` §1 L2 / v0.2.7 变更：明确写出 `RowAction.actionRef` 的存在性与类型校验，但未覆盖 `submitAction` / `upload.actionRef`
- 对照实现：`validateRowActionRefs()` 已检查 `RowAction.actionRef` 是否存在、是否 `type: request`、是否有非空 `requestMapping`

**当前值（实测）：**

1. `form.props.submitAction: missing`（顶层无该 action）→ L2 `violations: []`
2. `upload.props.actionRef: notUpload`，且 `actions.notUpload.type: request`（非 `upload`）→ L2 `violations: []`
3. 同时声明 `meta.requiredCapabilities: [actions.upload]` 时，能力检查通过，但目标类型错误仍不被拒绝

**问题：** 协议将三类字段都定义为“按 id 引用顶层 actions”，但 L2 只对 `RowAction.actionRef` 做引用完整性与类型约束。生成器/AI 或人工配置一旦写错 `submitAction` 或 `upload.actionRef`，CI 与 MCP `validate_content` 均无法拦截，运行时才失败。

**影响：** 🔴 严重。同类协议语义下校验能力不对称，会导致错误页面配置进入合并/发布链路；尤其 `upload.actionRef` 指向非 upload action 时，能力协商已“看似满足”，但执行语义仍错误。

**建议修复：**
1. 在 L2 增加页面级 action 引用扫描：
   - `form.props.submitAction`：目标必须存在于 `doc.actions`（类型不限，因默认模式可引用 request/navigate/modal/custom 等）。
   - `upload.props.actionRef`：目标必须存在，且 `type` 必须为 `upload`。
2. 遍历范围与现有 L2 一致：`body`、`tabs.items[].content`、`actions[].type: modal` 的 `content` Node。
3. 同步更新 `docs/06-validation.md` L2 说明与自检清单。
4. 补充 MCP `validate_content` 反例测试（缺失 `submitAction` 目标、`upload.actionRef` 类型错误）。

**需要同步的文件：**
- `scripts/validate-l2-components.js`
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`
- `mcp/tests/test-utils.ts`（如需新增样例 YAML）

### V86（🟡）L2 未校验 `data.ref` / `form.props.targetTable` 的引用存在性

**位置：**
- `scripts/validate-l2-components.js`：无 `datasources` / `targetTable` 引用解析
- 对照文档：
  - `docs/01-node-protocol.md` §3.3：`data.source: ref` 指向 `datasources` 中的 key
  - `docs/03-component-registry.md`：`form.props.targetTable` 声明关联表格 Node `id`
  - `docs/04-datasource-contract.md` §1：`datasources` + `data.ref` 预声明引用
- 对照实现：V71 仅在 `source: ref` 本地声明 `responseMapping` 时做条件必填，**不解析** `doc.datasources[data.ref]` 是否存在

**当前值（实测）：**

1. `data: { source: ref, ref: missing }`（`datasources` 不存在）→ L2 `violations: []`
2. `form.mode: search` + `targetTable: missingTable`（页面无对应 table id）→ L2 `violations: []`

**问题：** 这两类引用与 action 引用同属“页面内标识绑定”。缺少静态存在性校验时，配置错误会推迟到 Renderer 运行时才暴露。`targetTable` 错误尤其隐蔽：搜索提交会静默找不到目标表格。

**影响：** 🟡 中等。不会破坏 Schema 结构合法性，但会让本应在 CI 拦截的配置错误漏过；与 V85 同属引用完整性缺口，但错误频率与破坏面略低于 action 引用。

**建议修复：**
1. L2 扫描 `data.source: ref`：`data.ref` 必须存在于 `doc.datasources`。
2. L2 扫描 `form.props.mode: search` 的 `targetTable`：页面 Node 树中必须存在 `id === targetTable` 且 `type === table` 的节点（遍历范围同 L2）。
3. 同步 `docs/06-validation.md` L2 描述与自检清单。
4. 补充 MCP `validate_content` 反例测试。

**需要同步的文件：**
- `scripts/validate-l2-components.js`
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`
- `mcp/tests/test-utils.ts`（如需新增样例 YAML）

## 与已归档审计的关系

- V85 与 V30/V38/V79 相关：既有审计统一了 `RowAction.key` / `actionRef` 边界，并为 `upload.actionRef` 补了能力声明提示；但 L2 引用完整性仍只覆盖行级 `actionRef`，未覆盖表单提交与上传 action 引用。
- V86 与 V71 相关：V71 对 `source: ref` 的本地 `responseMapping` 做了条件必填，但明确“不解析引用目标”；本轮确认目标不存在时同样可漏过。
- V87 与 V21/V42/V63 同类：正文修订后 frontmatter `last_updated` 未同步。

## 建议处理顺序

1. **先修 V85**：严重校验缺口；优先补齐 `submitAction` / `upload.actionRef` 引用规则与回归测试。
2. **再修 V86**：同一次 L2 引用扫描改造中一并落地，避免二次改脚本。
3. **最后修 V87**：更新 `last_updated`；若 V85/V86 改到 `03`/`07` 相关说明，一并同步日期。

## 防止复发的建议

1. 凡协议中定义为“按 id 引用”的字段（`submitAction`、`actionRef`、`data.ref`、`targetTable` 等），新增或修订时必须同步：入口文档说明 + L2 存在性/类型规则 + MCP 反例测试。
2. 同类引用字段的 L2 规则应成组维护，避免只修最新引入的 `RowAction.actionRef` 而遗漏更早的 `submitAction` / `upload.actionRef`。
3. 审计修复若触达协议正文，检查清单应显式包含 frontmatter `last_updated` 同步项。
