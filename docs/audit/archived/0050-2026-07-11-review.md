# 审计报告 0050：page.schema datasources.params 残留 + Renderer 嵌套 `$deps` 示例

## 背景

本轮基于 0049 已归档后的再次全量复审。0049 中的 V197 入口/Schema 残留（V199）、CHANGELOG 清单（V200）与 `04` `last_updated`（V201）已确认落地；L3a params 完整值匹配、`node.schema.json` DataRef.params、`component-registry.json` optionsSource、`02`/`03`/`04`/`06`/`08` 正文权威口径与 MCP 测试均保持一致。

本轮确认两类此前未完全同步的残留：

1. **V199 后 Schema 漏项**：`page.schema.json` 的 `DatasourceDeclaration.params` 仍为笼统「请求参数映射」，未同步「完整单个 `$deps.*` / 禁止模板拼接」口径。
2. **Renderer 最小集成示例**：`08` §5.2 的 `$deps` 替换正则只捕获首段字段名，无法正确处理协议已允许的嵌套路径（如 `$deps.customer.id`）。

扫描活跃区与归档区后，历史最大审计编号为 `0049`，本轮使用 `0050`。扫描所有 `*-review.md`、`*-checklist.md` 与已有 `*-plan.md` 后，全局最大问题编号为 `V201`，因此本轮从 `V202` 连续编号到 `V203`。

## 发现的问题

### V202（🟡）`page.schema.json` DatasourceDeclaration.params 未同步整值替换口径

**位置：** `docs/schemas/page.schema.json` → `definitions.DatasourceDeclaration.properties.params.description`

**当前值：**

```text
【可选】请求参数映射
```

**对照（已对齐 V197/V199）：**

```text
# node.schema.json DataRef.params
【可选】URL query 参数映射，对所有 HTTP method 均不隐式进入请求体；值仅允许不含 $ 的字面量，或完整单个 $deps.<path> 整值替换（禁止模板拼接，不做条件判断），见 docs/04-datasource-contract.md §3.1 与 docs/08-renderer-spec.md §2.3

# component-registry.json optionsSource
params 值仅允许字面量或完整单个 $deps.<path> 整值替换（禁止模板拼接）…
```

**问题：** 0049/V199 已同步节点级 `DataRef.params` 与 `optionsSource`，但页面级 `datasources.*.params` 的机器可读描述仍停留在旧笼统措辞。L3a 对 `datasources.*.params` 同样执行完整单个 `$deps.*` 匹配与非表单 `NON_FORM_DATA_PARAMS` 拒绝（V65/V197）；读者或工具若只读 `page.schema.json`，会误以为页面级 params 无「整值替换 / 禁模板拼接」约束。

**影响：** 🟡 中等。实现与权威正文（`06` L3a、`04` §3.1、`node.schema.json`）已正确，但页面级 Schema 入口漂移，属易误用。

**建议修复：**

1. 将 `DatasourceDeclaration.params.description` 与 `node.schema.json` DataRef.params 对齐（至少包含：URL query、字面量或完整单个 `$deps.<path>`、禁止模板拼接、指向 `04` §3.1）。
2. 保留现有 `not.required: ["responseMapping"]`（ADR-0005 D1）不变。
3. 可选：在 description 中注明页面级 `datasources` 无 form 上下文，完整 `$deps.*` 会由 L3a 以 `NON_FORM_DATA_PARAMS` 拒绝（避免读者以为页面级可写 `$deps`）。
4. 同步 `docs/CHANGELOG.md` Unreleased；更新相关文档 `last_updated` 若改动触及正文。
5. 实现行为未变；修复后文本抽查 + MCP 测试回归即可。

**需要同步的文件：**

- `docs/schemas/page.schema.json`
- `docs/CHANGELOG.md` Unreleased

**与已归档审计的关系：** 0049/V199 已修 `node.schema.json` 与 `component-registry.json`，**漏同步 `page.schema.json` DatasourceDeclaration.params**，属新发现的残留，不是旧问题回退。

### V203（🟡）Renderer expr-eval 示例无法处理嵌套 `$deps.*` 路径

**位置：** `docs/08-renderer-spec.md` §5.2 最小集成示例（expr-eval）中的变量替换逻辑。

**当前值（摘录）：**

```javascript
const prepared = normalized
  .replace(/\$deps\.(\w+)/g, (_, key) => {
    if (!dependencies || !dependencies.includes(key)) {
      throw new Error(`未声明的依赖字段: ${key}`);
    }
    return JSON.stringify(context.deps[key]);
  })
```

**问题：**

1. 正则 `/\$deps\.(\w+)/g` 只捕获首段标识符。对 `$deps.customer.id` 会匹配 `$deps.customer`，替换为整个 `customer` 对象的 JSON，残留 `.id`，得到非法片段 `{"id":"x"}.id == 'x'`。
2. `dependencies.includes(key)` 也只按首段字段名比对；协议在 form 作用域下 `dependencies` 声明表单字段名（首段），而表达式可写 `$deps.<field>.…` 子路径——示例未展示「声明首段字段、求值完整点路径」的正确模式。
3. `04` §3.1 与 L3a 已允许 params 完整路径如 `$deps.customer.id`；表达式引擎若按此示例实现，嵌套路径会运行时失败。

**复现（示意）：**

| 输入表达式 | 示例替换结果 | 期望 |
|---|---|---|
| `$deps.orderType == 'x'` | `"retail" == 'x'` | 正确 |
| `$deps.customer.id == 'x'` | `{"id":"x"}.id == 'x'` | 错误；应读取嵌套值并得到 `"x" == 'x'` |

**影响：** 🟡 中等。协议正文与 L3a 不依赖该示例；但 `08` 是 Renderer 实现指引，可复制示例会误导接入方实现嵌套 `$deps` 求值。

**建议修复：**

1. 将 `$deps` 替换改为捕获完整点路径（如 `/\$deps\.([A-Za-z_][A-Za-z0-9_]*(?:\.[A-Za-z_][A-Za-z0-9_]*)*)/g`）。
2. 求值时：`dependencies` 校验使用路径首段（与 form 声明字段名一致）；值读取沿点路径从 `context.deps` 下钻（缺失短路为 `undefined`）。
3. 示例注释明确：form 的 `dependencies` 写字段名（无 `$deps.` 前缀）；表达式可访问该字段下的子路径。
4. 同步检查 `$context` 替换是否需同样改为完整点路径（当前 `(\w+)\.?(\w+)?` 仅两段，对 `$context.user.roles` 够用但扩展字段可能不足——若一并修，写清边界）。
5. 更新 `08` `last_updated` 与 CHANGELOG Unreleased。

**需要同步的文件：**

- `docs/08-renderer-spec.md`
- `docs/CHANGELOG.md` Unreleased

**与已归档审计的关系：** 0016/V16 曾修 expr-eval 的 `contains` 函数模型；本条是 `$deps` 路径捕获深度不足的新问题，非 V16 回退。
