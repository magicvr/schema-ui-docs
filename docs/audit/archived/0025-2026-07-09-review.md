# 审计报告 0025：入口索引 + 组件 DSL + 链接与元数据漂移

## 背景

本轮审计基于 0024 已归档后的再次全量文档复审。0024 中发现的 PATCH 级能力协商、L3a 表格表达式默认 scope、`text` 数据兜底语义与 `RowAction.key` 执行边界在当前工作区已确认修复，本轮不重复报告 V35-V38。

本轮重点交叉核对：协议正文与 `docs/schemas/*.json` / `component-registry.json` 的字段一致性、v0.2.7 行级后端动作新增文件在入口索引中的可发现性、文档链接相对路径、以及 frontmatter 元数据是否与正文更新同步。

## 发现的问题

### V39（🟡）根 `README.md` 目录树漏列 v0.2.7 新增行级后端动作示例与 ADR-0008

**位置：** `README.md` 目录结构代码块；`docs/05-scenarios/row-backend-actions.md`；`docs/decisions/0008-row-action-backend-request.md`

**当前值：** 根 README 的 `docs/05-scenarios/` 目录树只列出：

```text
README.md
grid-dashboard.md
data-table.md
form-with-reactions.md
```

`docs/decisions/` 目录树只列到 `0007-mcp-protocol-reader-validator.md`。

**问题：** v0.2.7 已新增行级后端动作端到端示例 `row-backend-actions.md` 和 ADR-0008，但根入口目录树未同步。`docs/05-scenarios/README.md` 和 `CHANGELOG.md` 已引用这些内容，说明根 README 入口索引发生漂移。

**影响：** 🟡 中等。根 README 是项目第一入口，漏列新增场景和 ADR 会降低 `actions.row.request` 能力的可发现性，尤其影响首次接入者和 AI 助手从目录树建立协议地图。

**建议修复：** 在根 README 的目录树中补充 `05-scenarios/row-backend-actions.md` 与 `decisions/0008-row-action-backend-request.md`。遵守 `docs/audit/README.md` 约定，不在根 README 中列举审计文件清单。

### V40（🔴）`upload.props.placeholder` 正文允许但组件 DSL 未声明，导致文档与 L2 校验直接冲突

**位置：** `docs/03-component-registry.md` `upload` 组件 props 表格；`docs/schemas/component-registry.json` `components.upload.props`

**当前值：** `03-component-registry.md` 已声明：

```text
| `placeholder` | string | 否（since 0.2） | 占位提示文案 |
```

但 `component-registry.json` 的 `upload.props` 中只有 `field`、`label`、`labelKey`、`accept`、`maxSize`、`multiple`、`action`、`actionRef`、`description`、`tooltip`、`required`、`defaultVisible` 与 `additionalProperties: false`，没有 `placeholder`。

**问题：** 审计 0020 的 V24 已补正文 `upload.placeholder`，但机器可读 DSL 未同步。由于 `upload.props.additionalProperties: false`，按正文生成的合法配置会在 L2 组件契约校验阶段被拒绝。

**影响：** 🔴 严重。这是协议正文与机器可读契约之间的直接矛盾，会造成“文档允许、CI 拒绝”的破坏性歧义。

**建议修复：** 在 `component-registry.json` 的 `upload.props` 中补充：

```json
"placeholder": { "type": "string", "required": false, "since": "0.2", "description": "占位提示文案" }
```

并用包含 `upload.props.placeholder` 的最小页面样例执行 L2 或全链路校验，确认不再误拒绝。

### V41（🟢）`06-validation.md` 中脚本链接相对路径指向不存在的 `docs/scripts/`

**位置：** `docs/06-validation.md` §1 校验层级表与 §2 CI 建议流程

**当前值：** 文档中多处链接写作：

```markdown
[`scripts/validate-l2-components.js`](../../../scripts/validate-l2-components.js)
[`scripts/validate-l3a-expressions.js`](../../../scripts/validate-l3a-expressions.js)
[`scripts/lint-l4-banned-props.js`](../../../scripts/lint-l4-banned-props.js)
```

**问题：** `06-validation.md` 位于 `docs/` 目录下，`./scripts/...` 会解析为 `docs/scripts/...`，该目录不存在；实际脚本位于仓库根目录 `scripts/`，同文档 §5.1 已正确使用 `../scripts/validate-all.js`。

**影响：** 🟢 轻微。不会改变协议语义，但会造成文档链接断裂，影响读者从校验说明跳转到可执行脚本。

**建议修复：** 将上述链接目标统一改为 `../scripts/...`，展示文本可继续保持 `scripts/...`。

### V42（🟢）部分文档已包含 v0.2.7 内容但 `last_updated` 元数据未同步

**位置：** `docs/06-validation.md` frontmatter；`docs/07-actions-contract.md` frontmatter；`docs/05-scenarios/data-table.md` frontmatter

**当前值：**

- `docs/06-validation.md` frontmatter 为 `last_updated: 2026-07-08`，正文已包含 v0.2.7 行级后端请求 L2 校验说明。

**建议修复：** 将上述文档的 `last_updated` 同步为 `2026-07-09`。

## 与已归档审计的关系

- V39 是 0015 V10（根 README 目录树漏列当前关键文档）的同类复发，但本轮对象是 v0.2.7 新增的 `row-backend-actions.md` 与 ADR-0008，不重复旧问题。
- V40 与 0020 V24 有直接关联：当时补齐了 `03-component-registry.md` 的 `upload.placeholder`，但本轮发现 `component-registry.json` 未同步，是同一修复链的机器契约残留缺口。
- V41 是链接相对路径漂移，历史审计曾处理过若干断链，但本轮定位的是 `06-validation.md` 内脚本链接，不重复旧链接问题。
- V42 是元数据漂移复发，类似 0018 V21，但本轮涉及 v0.2.7 触达的文档。

## 建议处理顺序

1. 优先修复 V40。它会导致合法配置被 L2 校验拒绝，属于正文与机器契约的直接冲突。
2. 修复 V39。入口索引同步成本低，能恢复 v0.2.7 新能力的可发现性。
3. 修复 V41 与 V42。两者为文档质量问题，可在同一轮小改中完成。

## 防止复发建议

- 在新增场景示例或 ADR 时，把根 README 目录树、`docs/05-scenarios/README.md`、`CHANGELOG.md` 的入口索引作为同一 checklist 项检查。
- 组件正文 props 表和 `component-registry.json` 的字段增删应成对校验，尤其是带 `additionalProperties: false` 的组件。
- 对 `docs/*.md` 的相对链接增加轻量链接检查，至少覆盖指向仓库内文件的 Markdown 链接。
- 每次触达稳定文档正文时同步检查 frontmatter `last_updated`。
