# 审计报告 0061：前后端共享协议互操作性与语义闭合复审

## 结论

本轮以项目的最终目标为审计标准：本仓库只交付协议；任意符合协议的前端 Renderer 与后端页面生产方，在少量项目配置下即可互操作；后端通过 YAML/JSON 描述页面语义，前端自行决定视觉实现。

本轮**未发现 P0**。当前协议架构、权威层级、版本协商和基础 Admin 页面模型方向正确。已有协议能够覆盖看板、表格、搜索、基础表单、上传、行级 request Action、简单联动和响应映射等场景。

本轮确认 25 项新增协议边界问题，现已全部完成协议裁决与独立复核：P1 × 20、P2 × 5。复核期间额外发现并修正了 MCP 合法 `pageSize` 测试数据、JS/Python 缺失/null 深相等分叉、L3a 嵌套 form 符号表丢失、interceptor query 改写、request generation 首次编号、reaction baseline/指数/loop 边界、scalar responseMapping、static ref、runtime defaults 与 component format 的 conformance 缺口。可落地语义现已同步到规范、ADR、Schema/L2-L3a、JS/Python reference 和 versioned fixtures；完整 Admin 生命周期按 ADR-0019 明确为 v2.0 后续范围，不再作为当前协议缺陷。

> 本轮已于 2026-07-18 完成复核并归档。问题编号从全局 `V242` 继续，未复用历史编号。

### V242（P1）`bodyMapping` 源字段与目标字段冲突/缺失

**证据：** `docs/07-actions-contract.md:47-65`、`docs/schemas/action.schema.json:24-31`、`scripts/validate-l2-components.js:953-975`、`conformance/reference-js/request-construction.js:139-140`、`conformance/reference-python/request_construction.py:161-165`。

**当前行为：** Schema 只约束 mapping value 是非空字符串；L2 不验证 source key 是否存在于调用 form，也不拒绝多个 source 写入同一 target。

**失败场景：** `missingField: target` 在 JS 中可能序列化为省略字段，在 Python 中为 `target: null`；`legalName: name` 与 `nickname: name` 的最终值依赖遍历和 map 行为。

**建议：** L2 按每个 `form.props.submitAction` 建立字段表，拒绝不存在的 source key；拒绝重复 target key；定义已声明字段运行时为 `undefined` 时的统一行为；增加 JS/Python 负例 fixture。

### V243（P1）普通表单 Action URL 模板无绑定语义

**证据：** `docs/schemas/action.schema.json:18-23`、`docs/07-actions-contract.md:112-141`、`scripts/validate-l2-components.js:823-869,953-974`。

**当前行为：** 普通 request Action URL 允许 `/orders/{id}`，但 `{name}` 替换仅定义在 RowAction `requestMapping.path`。

**影响：** 同一合法页面可能被原样请求、编码花括号或加载时拒绝。

**建议：** v2 当前版本禁止被普通 form 引用的 Action URL 包含模板；未来如需编辑路径，单独增加 form path mapping 契约。

### V244（P1）Form 字段命名空间和 `$deps` 符号表未闭合

**证据：** `docs/schemas/component-registry.json:276-297,352-376,410-475`、`scripts/validate-l2-components.js:1065-1101`、`scripts/validate-l3a-expressions.js:701-733`。

**当前行为：** 同一 form 可声明重复或空的 `field`、`startField`、`endField`；`$deps.*` 只做语法校验，不解析到实际字段。

**失败场景：** 两个控件均使用 `field: status`，或 `params.status: $deps.stauts`。状态、提交、错误回填和 query 结果均可能分叉。

**建议：** 建立 per-form symbol table；字段非空、唯一；日期范围端点不同且不与其他字段冲突；所有 `$deps` 根路径必须解析到该表；补 L2/L3a 负例。

### V245（P1）服务端分页 `pageSize` 缺省值未定义

**证据：** `docs/schemas/component-registry.json:135-152`、`docs/08-renderer-spec.md:174-178`、`docs/decisions/0011-reserved-query-params.md:41-45`。

**失败场景：** 同一 `mode: server` 页面在不同 Renderer 中首次发送 `pageSize=10`、`20` 或不发送。

**建议：** 将 server 模式 `pageSize` 设为必填，或固定协议默认值并要求 Renderer 物化；同步 Schema、ADR 和 fixture。

### V246（P1）静态数据和 `source: ref` 的组件消费语义不完整

**证据：** `docs/schemas/page.schema.json:55-113`、`docs/schemas/node.schema.json:69-145`、`docs/08-renderer-spec.md:73-114`、`docs/03-component-registry.md:92-141`。

**当前行为：** Schema 允许 static datasource 被 Node ref，也允许 table/statCard/chart/text 消费 static data，但没有逐组件定义 scalar/object/list 形状和取值路径。

**建议：** 按组件定义 static 输入形状；静态 ref 同步提供 `value`，不产生网络 loading/error；对 table/statCard/chart/text 分别增加 fixture。若不准备定义，应收窄 static 允许范围。

### V247（P1）标量组件 `responseMapping` 无明确语义

**证据：** `scripts/validate-l2-components.js:711-740` 允许标量组件 mapping；`docs/08-renderer-spec.md:186-193,227-230` 与 `conformance/reference-js/response-mapping.js:24-48` 主要定义 list/total。

**失败场景：** `statCard.valueField: total` 配合包装响应和 `responseMapping` 时，一个实现读取 `list`，另一个忽略 mapping，第三个读取自定义 value。

**建议：** 当前版本禁止标量组件 responseMapping；或新增明确 `value` mapping 并完整同步契约。推荐先禁止。

### V248（P1）空 local `responseMapping` 与 Schema/fixture 冲突

**证据：** `docs/schemas/node.schema.json:156-172` 要求 `minProperties: 1`；`conformance/fixtures/response-mapping/cases.json:110-144` 和两种 reference 却接受 `localMapping: {}` 并回退默认路径。

**建议：** 推荐删除该成功 fixture，使 reference 对空 mapping fail-closed。若要保留，则必须正式定义 `{}` 为清除继承并恢复默认映射，并移除 Schema 的冲突限制。

### V249（P1）RowAction 缺失值未统一 fail-closed

**证据：** `docs/08-renderer-spec.md:519`、`docs/07-actions-contract.md:120-141`、`conformance/reference-js/request-construction.js:71-85`。

**当前行为：** path 缺失明确拒绝，但 body/query 缺失只要求开发日志，可能被省略、转 null 或部分发送。

**建议：** 缺失/undefined 统一返回 `UNRESOLVED_ROW_VALUE` 且不发请求；显式 null 与 query tombstone 分开；mapping 必须原子解析。

### V250（P1）RowAction 未明确继承 retry/idempotency

**证据：** `docs/08-renderer-spec.md:500-521`、`docs/decisions/0014-action-replay-semantics.md:19-24`。

**失败场景：** 行级 POST 超时后重试，可能因未复用 Idempotency-Key 重复执行审批、退款或删除。

**建议：** RowAction 继承目标 request Action 的 `retryPolicy`；一次点击一个 invocation；idempotent retry 复用 key；timeout/network 为 unknown。

### V251（P1）401/403 与通用错误 fallback 优先级冲突

**证据：** `docs/08-renderer-spec.md:118-123,256,340-345`、`docs/04-datasource-contract.md:206-213`。

**建议：** 明确 `401/403 特殊规则 > 协议级错误类别 > fallbackText > 默认错误提示`；401 抑制文案，403 使用固定无权限占位。

### V252（P1）`requestInterceptor` 可破坏 DataRef GET-only 边界

**证据：** `docs/08-renderer-spec.md:238-256` 允许修改 body；`docs/04-datasource-contract.md:40-49` 要求 DataRef GET 无 body。

**建议：** DataRef/optionsSource GET 不得增加 body；拦截器只可补充 header、credentials、abort signal 等传输元数据，不得改变协议已确定的请求语义。

### V253（P1）依赖变化、可见性和请求响应缺少 latest-wins 规则

**证据：** `docs/08-renderer-spec.md:53-54,124-178`、`docs/01-node-protocol.md:212-232`。

**失败场景：** 旧 query 请求晚于新 query 请求返回并覆盖新数据；节点隐藏后旧响应改变其错误/加载状态。

**建议：** 为每个数据节点定义 request epoch；旧 epoch 响应不得提交；明确 hidden/unmounted 请求是否取消、响应是否丢弃；增加乱序 fixture。

### V254（P1）Form hidden/disabled 字段的提交投影未定义

**证据：** `docs/01-node-protocol.md:212-232`、`docs/07-actions-contract.md:47-65`、`docs/08-renderer-spec.md:498-502`。

**失败场景：** required 字段被 reaction 隐藏，或字段被禁用但存在 bodyMapping；不同实现可能继续校验、跳过校验、发送旧值或发送 null。

**建议：** 定义 hidden、disabled、unmounted、上传失败字段是否参与表单校验、bodyMapping 和提交；增加组合 fixtures。

### V255（P1）L2 未复用严格 URL query/Unicode 校验

**证据：** `scripts/validate-l2-components.js:79-88,138-160,767-804`、`docs/decisions/0010-query-serialization.md:35-45,62-71`、`conformance/fixtures/query-serialization/cases.json:104-112`。

**失败场景：** L2/MCP 接受 `%zz`、`%FF` 或孤立 surrogate，但 canonical serializer 拒绝。

**建议：** 所有 request URL 和 query map 复用严格 query parser、UTF-8 和 Unicode scalar 校验；统一错误码并增加 MCP/L2 负例。

### V256（P1）多处 JS/Python 路径解析使用原型链

**证据：** `conformance/reference-js/response-mapping.js:7-15`、`reaction-scheduler.js:8-15`、`conformance/reference-python/response_mapping.py:8-14`、`reaction_scheduler.py:12-18`、`docs/decisions/0005-response-mapping.md:57-68`。

**失败场景：** 缺失 `toString`、`constructor` 或 `__proto__` 时，JS 读取继承属性，Python 返回缺失值，导致 response mapping 或 reaction 结果分叉。

**建议：** 所有 JSON object traversal 使用 own-property；统一保留路径处理和错误结果；补 response/reaction 跨语言 fixtures。

### V257（P1）Request reference 对协议禁止的 GET body 仍构造请求

**证据：** 普通 form GET 被 `docs/07-actions-contract.md:65` 禁止；RowAction GET/DELETE body 被 `docs/07-actions-contract.md:133-142` 禁止；但 JS/Python request reference 在 `conformance/reference-js/request-construction.js:101-153`、`conformance/reference-python/request_construction.py:114-182` 仍构造 body。

**建议：** reference 早期返回统一拒绝；增加 form GET、row GET body、row DELETE body 负例。L2 已有规则不可由 reference 绕过。

### V258（P1）JS/Python 官方场景 reference 对 metadata/path 校验不一致

**证据：** JS `scenario-execution.js:19-28` 读取并校验 `scenarioPath` 及官方 metadata；Python `scenario_execution.py:22-26` 直接信任调用方 metadata。

**失败场景：** 不存在的 scenario path 或 metadata 不匹配在 JS 拒绝，在 Python 通过。

**建议：** Python 执行前解析并校验相同官方场景；增加不存在路径和 metadata mismatch fixtures。

### V259（P1）JS/Python JSON number `1`/`1.0` reaction 语义分叉

**证据：** JS 使用统一 Number；Python 保留 int/float distinction，见两种 `reaction-scheduler` reference 的比较和 commit 逻辑。

**失败场景：** `$deps.count == 1.0` 在 JS 成立、Python 可能不成立；`1` 写为 `1.0` 在两端对“是否变化”的判断也可能不同。

**建议：** 定义语言无关 JSON number 比较和 commit 相等规则；增加跨语言数值字面量 fixture。

### V260（P1）Reaction 初始求值和 true→false 复位语义未定义

**证据：** `docs/02-reaction-expression.md:109-110,272-281`、`docs/03-component-registry.md:295-298`、`docs/decisions/0006-expression-evaluation-order.md:61-64`。

**失败场景：** 页面初始状态是否强制求值不明确；条件从 true 变 false 且无 otherwise 时，状态可保持、恢复 props 基线或恢复组件默认值。

**建议：** 定义 mount 的初始 snapshot/evaluate 时点；定义每个状态键的基线和无 otherwise 时的复位规则；增加状态转换 fixture。

### V261（P1）表达式比较的类型和 coercion 规则未定义

**证据：** `docs/02-reaction-expression.md:52-76`、`docs/decisions/0004-row-level-scope.md:130-135`。

**失败场景：** `1 == "1"`、`true == 1`、`null == undefined`、字符串与数字大小比较在不同宿主语言中结果不同。

**建议：** 定义操作符比较矩阵；推荐严格类型相等；规定缺失值、null、数字、布尔和异类型比较；禁止直接依赖宿主语言默认 coercion。

### V262（P2）Reaction 下一轮和 loop limit 规则未完全统一

**证据：** `docs/08-renderer-spec.md:428-436`、`docs/decisions/0006-expression-evaluation-order.md:57-64`、`conformance/reference-js/reaction-scheduler.js:54-95`。

**问题：** Renderer 使用“建议 10 轮”和较宽泛的变化描述，ADR/reference 按依赖图优化并固定 fixture 轮数。

**建议：** 定义最多 N 轮及第 N 轮后仍需继续时的固定结果；统一哪些变化会触发下一轮。

### V263（P2）自定义组件运行时注册与封闭 Registry 边界不清

**证据：** `docs/08-renderer-spec.md:23-31`、`docs/06-validation.md:14-18`、`scripts/validate-l2-components.js:439-442`、`docs/schemas/component-registry.json:5-6`。

**建议：** 核心 Registry 继续封闭；`Renderer.register` 和业务组件归入 Host Extension。若需要跨项目扩展，增加 manifest、namespace、version、Schema 和 capability；未声明扩展时 fail-closed。

### V264（P2）`dependencies`/`visibleField` 声明结构缺少完整约束

**证据：** `docs/schemas/reaction.schema.json:8-12`、`docs/schemas/node.schema.json:193-205`、`scripts/validate-l2-components.js:243-265`、`scripts/validate-l3a-expressions.js:417-439`、`docs/schemas/component-registry.json:212-216`。

**问题：** dependency 可重复、可带 `$deps.`/`$row.` 前缀且在未使用时通过；`visibleField` 可写成表达式而非路径。

**建议：** 增加 `uniqueItems`、无前缀 token/path grammar、保留路径拒绝；`visibleField` 在 L3a 前规范化并校验。

### V265（P2）默认值、`baseURL` 和必填 props 的 fallback 不确定

**证据：** `docs/08-renderer-spec.md:33-39,234-254,469-484`。

**问题：** `baseURL` 可缺省但拼接规则依赖它；缺少必填 props 时允许“合理默认值”；DataRef GET、upload 等 schema default 也不是运行时物化规则。

**建议：** 请求型 Renderer 要求明确 baseURL 或规定唯一 same-origin fallback；缺少必填 props 时统一 invalid-component placeholder，不允许各实现猜测；明确 schema default 是否由 normalizer 物化。

### V266（P2）Upload、keyword、数字格式等组件边界语义不完整

**证据：** `docs/04-datasource-contract.md:294-318`、`docs/03-component-registry.md:389-410`、`docs/07-actions-contract.md:198-259`、`docs/decisions/0012-upload-execution.md:26-44`、`docs/schemas/component-registry.json:7-18,125-153`。

**问题：** searchable select 的 `keyword` 冲突、upload 空文件/ASCII whitespace/多文件 idempotency key、`currency`/`datetime` 默认语义、`columns/span/pageSize` 数值域仍有未定义边界。

**建议：** 分主题增加 ADR、Schema 和 fixtures；明确 keyword 合并顺序、上传空输入和 key 派生、日期时区/货币代码以及正整数约束。

## 与完整 Admin 生命周期的范围差异

以下不是当前已有规则的实现回归，而是完整通用 Admin 协议需要另行设计的能力：

- 标准页面级 Action 入口（Button/Toolbar/ActionTrigger）；
- 标准行级详情/编辑导航；
- 编辑记录加载、初始值和表单回填；
- 标准详情/record view 组件；
- 完整权限绑定和权限继承；
- 批量操作。

若当前版本不宣称支持上述能力，它们属于明确的后续范围，不计入本轮 P1/P2 缺陷数量。

## 已检查但未发现问题

- 严格 MAJOR.MINOR 版本协商、判定顺序、错误码和 capability 缺失处理：`docs/08-renderer-spec.md:260-326`、`docs/decisions/0009-strict-version-negotiation.md:20-65`、JS/Python reference 和 fixtures 对齐。
- responseMapping 的 local-over-inherited 整体覆盖优先级已一致；问题仅在空 mapping 和标量消费边界。
- DataRef GET-only、query-only、非 GET 拒绝在规范、Schema、L2 和 request reference 中方向一致；reference 仍需对绕过校验的非法 form/row body fail-closed。
- query serialization、保留 query 参数、搜索/分页/排序状态转换已由 ADR-0010/0011 和 fixtures 覆盖。
- Action HTTP outcome 顺序、401/403、400 字段错误和 unknown 网络结果的基本方向一致；本轮只发现 auth fallback 优先级和 RowAction 继承语义需要补齐。
- 仓库不提供生产 Renderer、业务后端或 Host，不构成本项目审计问题。

## 建议处理顺序

本轮协议条目已闭合。下一阶段如需完整 Admin 生命周期，应按 ADR-0019 另开协议设计并新增字段、Schema、capability、ADR 和 fixtures，不通过 Host handler 猜测核心语义。

## 防复发建议

每项协议语义变更必须原子同步：

```text
核心规范/ADR
→ JSON Schema 或组件 DSL
→ L2/L3a/L4
→ JS/Python reference
→ conformance fixture
→ 官方场景（如适用）
→ CHANGELOG/迁移说明
```

新增负例必须至少覆盖：

- 结构校验通过但运行时不应执行的输入；
- 缺失字段、null、undefined 和空集合；
- 原型名称、Unicode scalar、URL percent encoding；
- JS/Python 数字、路径和序列化边界；
- capability 缺失和 Host Extension 未安装。

本轮已将已裁决的协议语义同步到规范、Schema/L2-L3a、JS/Python reference、versioned fixtures、协议发布门禁和 CHANGELOG；生产 Renderer、业务后端和 MCP 不属于本项目协议交付范围。
