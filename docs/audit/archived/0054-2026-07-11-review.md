# 审计报告 0054：L3a params 标签 + 三面 params 文档 + VisibleWhen dependencies + parentRow/MCP 卫生

## 背景

本轮基于 0053 已归档后的再次全量复审。V197–V212（params 整值替换、`$context` 可用位置、`contains` §10.8、reaction dependencies、parentRow 组件表措辞等）已确认落地；六个官方场景模式与 MCP 五工具契约无明显阻断性漂移；版本叙事仍为已发布 `0.2.8` + `Unreleased`。

本轮新问题集中在：

1. L3a 对页面级 `datasources.*.params` 违规时错误标签为 `data.params`；
2. 实现与 `06`/`08` 已按三面 params 执行，权威表达式章 / 数据源契约仍写两面，且 `02` §2 交叉引用落空；
3. V209 只更新了 `reaction.schema.json`，`VisibleWhen.dependencies` 与 `01` §3.8 仍缺「无前缀完整路径」口径；
4. V211 只改了组件表/DSL，`07` 与 L2/L3a 对外消息仍写「暂不支持」；
5. MCP 临时目录清理失败静默，与 ADR-0007「本地警告」不一致。

扫描活跃区与归档区后，历史最大审计编号为 `0053`，本轮使用 `0054`。扫描所有 `*-review.md`、`*-checklist.md` 与已有 `*-plan.md` 后，全局最大问题编号为 `V212`，因此本轮从 `V213` 连续编号到 `V217`。

## 发现的问题

### V213（🟡）L3a 将 `datasources.*.params` 违规文案误标为 `data.params`

**位置：**

- `scripts/validate-l3a-expressions.js` → `scanDataParams()`
- 对照：`validatePage()` 传入路径 `datasources.${sourceId}.params`；`mcp/tests/validate-content.test.ts` datasources 反例

**当前值：**

```js
const isOptionsSource = paramsPath.includes('optionsSource.params');
const paramsLabel = isOptionsSource ? 'optionsSource.params' : 'data.params';
```

页面级扫描调用为 `scanDataParams(..., \`datasources.${sourceId}.params\`, ..., false)`，因此 `path` 正确（如 `datasources.owners.params.keyword`），但 `message` 中的 `${paramsLabel}` 恒为 **`data.params`**。

**问题：** 错误文案与真实挂载点不符。AI/接入方仅读 message 时会去改 Node `data.params`，而不是页面级 `datasources`。MCP 测试只断言 `path` / `rule`，未锁定 message 标签。

**影响：** 🟡 中等。规则执行正确，排错路径误导；MCP `validate_content` 会把错误文案直接返回给客户端。

**建议修复：**

1. 按 `paramsPath` 派生标签：`optionsSource.params` / `datasources.params`（或 `datasources.*.params`）/ 默认 `data.params`。
2. 文件头注释列出三类 params 扫描面。
3. MCP 反例断言 message 含 `datasources`（或具体标签），避免回归。

**需要同步的文件：**

- `scripts/validate-l3a-expressions.js`
- `mcp/tests/validate-content.test.ts`
- `docs/CHANGELOG.md`

**与已归档审计的关系：** V65 把 `datasources.*.params` 纳入 L3a；V197 收紧整值替换；均未单独校正 `paramsLabel` 对 datasources 路径的分支。

### V214（🟡）`02` §10.7 / 附录 A 与 `04` §3.1–§3.2 仍写两面 params，§2 交叉引用落空

**位置：**

- `docs/02-reaction-expression.md` §2（`$context` 可用位置）、§10.7、附录 A
- `docs/04-datasource-contract.md` §3.1–§3.2
- 对照：`docs/06-validation.md` L3a 行、`docs/08-renderer-spec.md` §5.5、L3a 实现、`page.schema.json` DatasourceDeclaration.params

**当前值：**

- `06` / `08` / L3a：三面——`data.params` / `optionsSource.params` / **`datasources.*.params`**
- `02` §10.7 标题与正文、附录 A：仅两面（节点级 `data.params` + `optionsSource.params`）
- `04` §3.1–§3.2：同上，仅节点级两面
- `02` §2：`$context.*`「**不含** … `datasources.*.params` 值替换（见附录 A / §10.7）」——被引用章节并不出现 datasources

**问题：** 权威静态规则章与数据源契约滞后于 CI/Renderer 清单。读者只读 `02`/`04` 会漏掉页面级 params 的整值替换 / 模板拼接 / 非 form 拒绝。§2 主动点名 datasources 并指向 §10.7，造成**交叉引用落空**。

**影响：** 🟡 中等。执行层已拒绝非法 datasources params；文档与入口引用不一致会放大误用与漏读。

**建议修复：**

1. `02` §10.7 标题/正文纳入页面级 `datasources.*.params`（永远非 form；字面量合法，完整 `$deps.*` → `NON_FORM_DATA_PARAMS`，模板 → `DATA_PARAMS_VARIABLE`）。
2. 附录 A 增补 `datasources.*.params` 行。
3. `04` §3.1–§3.2（或 §1 数据源声明）明确页面级 params 与节点级同一整值替换语法，且因无 form 上下文不得出现 `$deps.*`。
4. 确认 §2「见附录 A / §10.7」在补齐后真实可达。

**需要同步的文件：**

- `docs/02-reaction-expression.md`
- `docs/04-datasource-contract.md`
- `docs/CHANGELOG.md`（按需 `last_updated`）

**与已归档审计的关系：** 0030/V65c 曾决策「不额外改 `02`/`04`，以 `06` 为 datasources params 边界说明」。之后 V197/V207 已在 `02` §2 点名 `datasources.*.params` 并链到 §10.7，旧「可不写 `02`」决策被正文突破，本轮闭合该残留。

### V215（🟡）`VisibleWhen.dependencies` Schema 与 `01` §3.8 未写明无前缀路径规则

**位置：**

- `docs/schemas/node.schema.json` → `definitions.VisibleWhen.properties.dependencies`
- `docs/01-node-protocol.md` §3.8 YAML 注释
- 对照：`docs/schemas/reaction.schema.json` dependencies（V209）；`02` §8.1；`01` §3.5 reactions 注释

**当前值：**

- VisibleWhen Schema：`显式声明的依赖字段名。表单上下文中必填，非表单上下文可省略`
- `01` §3.8：`dependencies: [string]   # 显式声明的依赖字段名（表单上下文中必填，非表单上下文可省略）`
- reaction Schema（已修）：form 无 `$deps.` 前缀字段名；row 为 `$row.` 之后完整点路径；可空数组；禁止带前缀字符串

**问题：** 表格列/操作 `visibleWhen` + `scope: row` 复用 VisibleWhen 结构与 L3a 精确路径匹配，需要 `canRefund` / `user.id` / `__index` 等无前缀完整路径。机器可读描述与 Node 协议草图仍像「仅简单字段名」，构成 V209 的 VisibleWhen 侧残留。

**影响：** 🟡 中等。不改变 L3a 行为，但 Schema/入口注释与 reaction 分叉，易误生成 `dependencies: ["$row.canRefund"]` 或省略嵌套路径。

**建议修复：**

1. 对齐 `VisibleWhen.dependencies.description` 与 reaction/V209/V164：form 字段名；row 为 `$row.` 后缀完整点路径；可空数组；禁止 `$deps.`/`$row.` 前缀字符串。
2. `01` §3.8 YAML 注释与 `01` §3.5 reactions 注释同口径（可注明表格 `scope: row` 场景）。

**需要同步的文件：**

- `docs/schemas/node.schema.json`
- `docs/01-node-protocol.md`
- `docs/CHANGELOG.md`

**与已归档审计的关系：** 0044/V164 写清 `$row` dependencies；0053/V209 只回写 `reaction.schema.json`，未覆盖 `VisibleWhen`。

### V216（🟢）V211 残留：`07` 与 L2/L3a 仍写「暂不支持」`$parentRow` / 嵌套表格

**位置：**

- `docs/07-actions-contract.md` §3.1
- `scripts/validate-l2-components.js` requestMapping 违规 message
- `scripts/validate-l3a-expressions.js` 文件头与 `PARENT_ROW_UNSUPPORTED` message
- 对照：`03` / `component-registry.json`（V211 已改为「静态拒绝」）

**当前值：**

- `07`：`v0.2 暂不支持嵌套表格…`
- L3a：`v0.2 暂不支持 $parentRow.*；待嵌套表格挂载结构通过后续 ADR 定义后再开放`
- L2：`v0.2 暂不支持嵌套表格及 $parentRow.*…`

**问题：** 「暂不支持」像软缺能力，易被理解为可写但运行时忽略。权威口径为 v0.2 **静态拒绝** / **全面拒绝**。MCP 客户端经 `validate_content` 直接看到旧措辞。

**影响：** 🟢 轻微。规则已拒绝，仅措辞与用户可见消息不一致。

**建议修复：**

1. `07`、L2/L3a 消息与头注释统一为「v0.2 静态拒绝 `$parentRow.*`（无嵌套表格挂载结构）」。
2. 可选：MCP 测试断言 message 含「静态拒绝」或不再含「暂不支持」。

**需要同步的文件：**

- `docs/07-actions-contract.md`
- `scripts/validate-l2-components.js`
- `scripts/validate-l3a-expressions.js`
- `mcp/tests/validate-content.test.ts`（可选）
- `docs/CHANGELOG.md`

**与已归档审计的关系：** 0053/V211 仅覆盖组件表与 DSL；本项为同一口径在 Action 契约与校验器消息上的残留。

### V217（🟢）MCP 临时目录清理失败静默，未发本地警告

**位置：**

- `mcp/src/core/validation-runner.ts` finally 块
- 对照：ADR-0007 D5、`docs/mcp/v1-design.md`（清理失败应本地警告，不得改变结果结构）

**当前值：**

```ts
} catch {
  // Best-effort cleanup only.
}
```

**问题：** 清理失败被完全吞掉。V126 已要求 best-effort 且不覆盖校验结果，但未落地「本地警告」。与设计/ADR 可观测性约定不符。

**影响：** 🟢 轻微。结果结构正确；仅 tmp 泄漏时缺少 stderr/console 信号。

**建议修复：**

1. `catch` 中 `console.warn`（或等价本地警告），消息不泄露完整内部路径亦可接受稳定短文案。
2. 不改变 `passed` / `layers` / `internalError` 结构。
3. 若可测则补单测；否则在 checklist 记录手工/说明性验证。

**需要同步的文件：**

- `mcp/src/core/validation-runner.ts`
- `mcp/tests/*`（若可注入 rm 失败）
- `docs/CHANGELOG.md`

**与已归档审计的关系：** 0040/V126 闭合临时目录创建失败的 `internalError` 边界；清理失败的「本地警告」仍为设计要求未执行项。

- **V65 / V65c：** 引入 datasources params L3a；曾决定可不改 `02`/`04`。本轮 V214 因 §2 已点名 datasources 并指向 §10.7 而必须闭合。

## 建议处理顺序

1. **V213** — 修正 L3a 标签 + 测试，避免 MCP 继续误导。
2. **V214** — 闭合权威文档与交叉引用。
3. **V215** — 对齐 VisibleWhen 机器/正文描述。
4. **V216 / V217** — 措辞与可观测性卫生。

## 防止复发的建议

1. 新增 params 类挂载点时，同步：`scanDataParams` 标签、`02` §10 / 附录 A、`04`、`06`、`08`。
2. 修改 `dependencies` 路径规则时，**同时**更新 reaction 与 VisibleWhen Schema 及 `01` 两处草图。
3. 统一「静态拒绝」用语时，用全局搜索覆盖脚本 message 与 Action 契约，不仅组件表。
4. ADR 要求的本地警告/日志应在实现 checklist 中单独勾选，避免只测 happy path。
