# 审计报告 0036：responseMapping 继承语义未写回正文 + params 禁令无校验 + Renderer source:ref 规格缺口 + 版本/措辞残留

## 背景

本轮审计基于 0035 已归档后的再次全量文档、Schema、校验脚本与 MCP 实现复审。0035 中发现的 `source: ref` 继承 `datasources.*.responseMapping` 的 L2 条件必填（V88）、v0.2.8 版本声明闭环（V89）、以及 Renderer permissions 白名单描述（V90）在当前工作区已确认修复，本轮不重复报告 V85–V90。

本轮重点交叉核对：V88 修的“生效映射”是否已写回数据源契约与 Renderer 权威正文、ADR-0005 / `01` 对 `params.responseMapping` 的禁止是否有任何机器校验、Renderer §2 是否覆盖 `source: ref` 主路径、以及 v0.2.8 升版后是否仍有默认镜像 tag / 维护者注释残留。

## 发现的问题

### V91（🟡）`responseMapping` 继承语义只写在校验侧，未进入数据源契约与 Renderer 正文

**位置：**
- `docs/04-datasource-contract.md` §4.1.1：仅描述节点上的 `data.responseMapping`
- `docs/08-renderer-spec.md` §2.5：仅写“应用 `data.responseMapping`”
- 对照已落地的权威口径：
  - `docs/01-node-protocol.md` §3.3：`source: ref` 可携带本地 `responseMapping` **覆盖**响应解析
  - `docs/06-validation.md` v0.2.4 / v0.2.8：生效映射 = 本地优先，否则继承 `datasources[data.ref].responseMapping`
  - `scripts/validate-l2-components.js` `getEffectiveResponseMapping()`：已按上述顺序解析

**当前值：**

| 位置 | 当前描述 |
|---|---|
| L2 / `06` | 本地 `data.responseMapping` 优先，否则继承 `datasources[data.ref].responseMapping` |
| `01` §3.3 | `source: ref` 可本地 `responseMapping` 覆盖 |
| `04` §4.1.1 | 只写“可在 `data.responseMapping` 中声明”，**无继承** |
| `08` §2.5 | 只写“应先应用 `data.responseMapping`”，**无继承** |

**问题：** v0.2.8（V88）已把继承纳入正式校验与 MCP 反例，但数据源契约与 Renderer 实现规范仍按“仅看节点本地映射”描述。只读 04/08 的实现方会漏掉继承，与 CI/L2/MCP 行为不一致。

**影响：** 🟡 中等。不直接导致错误配置通过校验，但会让 Renderer 实现与后端配置方对“生效映射从哪来”产生分叉：校验器已按继承拒绝不完整预声明映射，运行时若只读节点本地字段则会忽略 `datasources` 上的合法映射。

**建议修复：**
1. 在 `04-datasource-contract.md` §4.1.1 明确生效映射解析顺序：
   - 节点本地 `data.responseMapping` 优先；
   - 否则当 `data.source: ref` 时继承 `datasources[data.ref].responseMapping`；
   - 无任何映射时沿用默认 `list` / `total`（及 chart 裸数组）语义。
2. 在 `08-renderer-spec.md` §2.5 同步同一解析顺序，并写明继承映射同样只参与响应解析、不得进入请求参数。
3. 如有必要，在 §4.1.1 / §2.5 交叉引用 `06-validation.md` 的 v0.2.8 变更与 ADR-0005。

**需要同步的文件：**
- `docs/04-datasource-contract.md`
- `docs/08-renderer-spec.md`

### V92（🟡）禁止 `params.responseMapping` 仅有文档，无任何校验实现

**位置：**
- `docs/decisions/0005-response-mapping.md` D1：
  > 若 `data.params.responseMapping` 出现，静态校验应拒绝或至少在 L3/L4 阶段报错。
- `docs/01-node-protocol.md` §3.3：协议禁止将 `responseMapping` 放入 `params`
- `docs/06-validation.md` 人工 checklist 已有“未误放入 `data.params`”勾选项
- 对照实现：
  - `docs/schemas/node.schema.json`：`params` 为无约束 `object`（无 `properties` / `not` / `additionalProperties` 限制）
  - `scripts/*`：无 `params.responseMapping` 检查
  - MCP 测试：无对应反例

**当前值：**
- Schema：`params: { type: object, description: ... }`，允许任意键，包括 `responseMapping`
- L1–L4 / MCP：均不会拒绝

```yaml
data:
  source: api
  url: /api/orders
  params:
    responseMapping:   # 协议禁止，但当前全链路可通过
      list: result.records
```

**问题：** 正式禁止项无法被 CI/MCP 拦住。`responseMapping` 可被误塞进请求参数并原样发给后端——正是 ADR-0005 要避免的语义污染。

**影响：** 🟡 中等。属于“文档已定、机器契约未 enforce”的执行缺口；配置错误会在运行时污染请求参数或让映射静默失效。

**建议修复：**
1. 优先在 L2 增加规则：拒绝 `data.params.responseMapping` 与 `datasources.*.params.responseMapping`（页面级 `datasources` 同为 `DataRef`）。
2. 可选加固：在 `node.schema.json` 的 `params` 上增加 `not: { required: ["responseMapping"] }` 或等价约束，使 L1 也能拦截。
3. 同步 `docs/06-validation.md` L2 规则表 / 变更说明。
4. 补充 MCP `validate_content` 反例：节点 `data.params.responseMapping` 与 `datasources.*.params.responseMapping` 均应失败。

**需要同步的文件：**
- `scripts/validate-l2-components.js`（或 L1 Schema，若选择双轨）
- `docs/schemas/node.schema.json`（若落 L1）
- `docs/06-validation.md`
- `mcp/tests/validate-content.test.ts`
- `mcp/tests/test-utils.ts`

### V93（🟡）Renderer 规范缺少 `data.source: ref` / `datasources` 解析与请求协调

**位置：**
- `docs/08-renderer-spec.md` §2.1–§2.4
- 对照：
  - `docs/01-node-protocol.md` §2–§3.3：`datasources` + `source: ref` 为正式模式
  - `docs/04-datasource-contract.md` §1：页面级预声明
  - `docs/05-scenarios/grid-dashboard.md`：官方场景已使用 `datasources` + `ref`

**当前值：** `08` 数据加载章节只覆盖：
- 多个 `data.source: api` 的默认并行请求
- 失败隔离
- `$deps` 参数（表单上下文）
- 本地 `responseMapping`（见 V91）

全篇几乎不定义：
- `source: ref` 如何解析 `datasources[ref]`
- 多节点同 `ref` 是否共享请求 / 缓存
- 失败归属哪个 Node
- 加载态是否与 `source: api` 等价

**问题：** 官方场景与协议主路径已使用 `ref`，但 Renderer 权威实现规范仍以 `api` 为唯一主路径。实现方只能猜测“是否等同内联 api / 是否去重 / 错误态如何绑定节点”。

**影响：** 🟡 中等。不直接破坏现有校验，但会造成多实现分叉；与 V91 叠加时，`ref` 路径的请求与响应解析都会缺少权威规格。

**建议修复：**
1. 在 `08` §2 增补 `source: ref` 行为专节或扩写 §2.1：
   - 解析顺序：读取 `datasources[data.ref]` 的 `url` / `method` / `params` 等请求配置；
   - 加载态 / 错误态与 `source: api` 等价，错误归属消费该 `data` 的 Node；
   - 明确同 `ref` 的请求协调策略（至少给出默认约定：可共享同一进行中的请求结果，或各自独立请求——需选定一种并写死）；
   - 本地 `responseMapping` 覆盖预声明映射的应用点（可与 V91 一并写在 §2.5）。
2. 示例可补充最小 `datasources` + 多节点同 `ref` 场景，或交叉引用 `grid-dashboard`。

**需要同步的文件：**
- `docs/08-renderer-spec.md`
- 可选：`docs/04-datasource-contract.md` §1（若需一句话指向 Renderer 请求协调）

## 与已归档审计的关系

- V91 与 V88/V71 相关：V71/V88 在校验侧完成“本地 + 继承”生效映射，但 04/08 权威正文未写回，属于同一语义的文档落地缺口。
- V92 与 ADR-0005（0012/相关落地）相关：D1 已要求静态校验拒绝 `params.responseMapping`，历史审计补了 `responseMapping` 一等字段与条件必填，但未补“禁入 params”的 enforce。
- V93 与 v0.2.1 R2 / 官方 `grid-dashboard` 相关：Renderer 数据协调最初以 `api` 写就，`datasources`+`ref` 成为主路径后 §2 未扩写。
- V94 与 V89 相关：0.2.8 版本闭环漏掉 smoke 默认 tag。
- V95 与 V88 相关：实现与变更段已更新，总览表/注释未跟到“生效映射”用语。

## 建议处理顺序

1. **先修 V92**：补齐 `params.responseMapping` 的 L2（及可选 L1）拒绝规则与 MCP 反例——这是唯一“文档禁令无 enforce”的执行缺口。
2. **再修 V91 + V93**：同属 `08`/`04` 的 `ref` 与 `responseMapping` 正文补齐，可一并改写 §2 / §2.5 / §4.1.1。
3. **最后修 V94 + V95**：默认 tag 与措辞/注释残留。

## 防止复发的建议

1. 凡 L2/L3a 新增“生效语义”（继承、覆盖、默认），checklist 必须显式包含：权威正文（01/04/08）、`06` 总览表行、脚本头注释、MCP 正反例四类同步项。
2. ADR 中写“静态校验应拒绝”的禁令，落地时必须对应至少一层可执行规则（L1 Schema 或 L2 脚本）+ 一条 MCP 反例；仅写人工 checklist 不算闭环。
3. MCP 版本 bump 时，除 `package.json` / 文档 Docker 示例外，扫描 `mcp/tests/*` 中的默认镜像 tag 与版本字符串。
4. `source: ref` 作为协议主路径时，`08` §2 应与 `source: api` 同等覆盖请求协调、错误归属与响应解析；场景文档引用 `ref` 时可作为回归检查点。
