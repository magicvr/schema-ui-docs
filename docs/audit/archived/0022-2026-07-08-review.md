## 发现的问题

### V30（🔴）`07-actions-contract.md` 将 `table.props.actions[].key` 误写为顶层 `actions` 引用，和 `RowAction`/场景示例直接冲突

**位置：** `docs/07-actions-contract.md` §1“使用位置”；`docs/03-component-registry.md` `table` 组件 `RowAction` 表；`docs/05-scenarios/data-table.md` 主示例

**当前值：**

`07-actions-contract.md` 写道：

```md
顶层文档结构中的 `actions` 字段……供 `form.props.submitAction`、`table.props.actions[].key` 等按 id 引用。
```

而 `03-component-registry.md` 对 `RowAction.key` 的定义仅为：

```md
| `key` | string | 是 | 操作标识 |
```

`05-scenarios/data-table.md` 的主示例则直接声明：

```yaml
actions:
  - key: view
    label: 查看
  - key: refund
    label: 退款
    confirm: 确认发起退款吗？
    visibleField: canRefund
```

但该示例并未提供任何顶层 `actions.view` / `actions.refund` 定义。

**问题：** 三处文档对同一字段给出了互相冲突的语义：

- 若以 `07-actions-contract.md` 为准，则 `table.props.actions[].key` 是对顶层 `actions` 的引用，数据表格场景示例不完整，`RowAction.key` 的“操作标识”表述也不充分；
- 若以 `03-component-registry.md` 和 `05-scenarios/data-table.md` 为准，则 `RowAction.key` 只是行内操作的本地标识，`07-actions-contract.md` 的“按 id 引用”表述错误。

当前协议没有任何其他位置补充“表格行内操作如何绑定顶层 ActionDef”的规则，也没有为 `RowAction` 提供类似 `submitAction` / `actionRef` 的显式引用字段。因此该冲突不是单纯措辞问题，而是**执行语义层面的协议矛盾**。

**影响：** 🔴 严重。实现方若按 `07-actions-contract.md` 理解，会把表格行内操作实现为“引用顶层 ActionDef”；若按 `03-component-registry.md` / 场景示例理解，则会实现为“前端按 `key` 自行分发本地行为”。两种实现彼此不兼容，直接影响协议消费端和配置生成端的一致性。

**建议修复：** 先统一协议口径，再同步修正文档与示例。最小修复路径有两种，需二选一并全局保持一致：

1. **若 `RowAction.key` 只是本地标识**：将 `07-actions-contract.md` §1 中 `table.props.actions[].key` 从“按 id 引用”列表移除，仅保留 `form.props.submitAction` 与 `upload.props.actionRef` 等真实引用场景；必要时在 `03-component-registry.md` 中补一句“`key` 不引用顶层 `actions`，仅供前端识别行内操作类型”。
2. **若 `RowAction.key` 应引用顶层 ActionDef**：则需同步补充 `03-component-registry.md` 的 `RowAction.key` 说明、`05-scenarios/data-table.md` 顶层 `actions` 示例，以及“行内操作如何携带当前行上下文执行 Action”的明确规则。

在当前文档体系下，第 1 条更符合既有正文与示例现状，变更面也更小。

### V31（🟡）`02-reaction-expression.md` 权限示例误用 `permissions.visible`，与标准动作键 `view`/`edit`/`delete` 不一致

**位置：** `docs/02-reaction-expression.md` §11.1 `$context.user` 最小字段集示例；对照 `docs/01-node-protocol.md` §3.9

**当前值：**

```yaml
# 权限表达式示例：仅 admin 可见
permissions:
  visible: "$context.user.roles contains 'admin'"
```

而 `01-node-protocol.md` 已明确协议层预定义的标准动作键为：

```yaml
permissions:
  view: string
  edit: string
  delete: string
```

**问题：** 示例使用了并不存在于当前协议标准键集合中的 `visible`，与正文定义的 `view` / `edit` / `delete` 冲突。读者直接照抄该示例会生成与权威定义不一致的配置。

**影响：** 🟡 中等。属于可复制示例错误。不会改变协议结构本身，但会误导配置编写者在权限控制场景中使用错误键名，导致静态校验失败或实现行为不符合预期。

**建议修复：** 将示例中的 `permissions.visible` 改为 `permissions.view`，与 `01-node-protocol.md` §3.9 的标准动作键保持一致：

```yaml
permissions:
  view: "$context.user.roles contains 'admin'"
```
