# 审计 0042 修复计划

## 目标

在不引入 v0.3 级结构扩张的前提下，为 V142-V154 建立唯一、可校验、可测试的 v0.2 行为。优先采用保守拒绝或单一数据来源，避免 Renderer 私有推断。

## 建议决策

| 问题 | 建议决策 | 理由 |
|---|---|---|
| V142 | DataRef `params` 对所有 method 均编码为 query | 现有字段名和数据源文档已偏向 query；新增 body 字段应另开 ADR |
| V143 | targetTable 的有效数据源必须最终解析为 API | 搜索模式的协议承诺就是重新请求，static 无可执行路径 |
| V144 | v0.2 禁止 dateRangePicker reaction `value` | 双字段部分更新和 null 语义尚未设计，保守拒绝最稳定 |
| V145 | actionRef 模式下上传约束只来自 UploadAction | 可复用 Action 才能作为单一传输契约；组件只保留展示/表单字段配置 |
| V146 | 先执行协议级状态处理，再执行不冲突的 onError | 401/403 安全钩子和 400 字段错误不能被配置静默覆盖 |
| V147 | bodyMapping 存在时采用源字段白名单 | 最小权限、避免敏感字段意外发送；缺省仍保持全字段兼容 |
| V148 | search mode 完全跳过 submitAction 解析 | 与正文“忽略”保持一致，默认表单规则不变 |
| V149 | option value 是不透明业务值 | 与 value 任意类型契约一致，L4 不应解释业务载荷 |
| V150 | URL 中任何花括号必须构成合法 `{identifier}` | 词法先行，再做 mapping 双向完整性检查 |
| V151 | upload maxSize 最小值为 0 | 与 UploadAction 和“字节上限”语义一致 |
| V152 | 设置有界 maxBuffer，并为 ENOBUFS 返回专门 internalError | 先恢复既有输入范围；后续可再设计子脚本流式/限量输出 |
| V153 | 保留历史编号，强制冲突项使用 `NNNN/Vn` | 不改写历史事实，也恢复唯一定位能力 |
| V154 | 抽取共享 pattern 规范化 helper | 避免 L2/L3a/L4 再次分叉 |

## 实施阶段

### 阶段 A：协议决策与正文

1. 落实 V142-V147 的决策并同步权威正文、Renderer 和 Schema/DSL 描述。
2. 只在确实新增公共结构时创建 ADR；纯收敛现有字段解释可记录在本 plan、CHANGELOG 和相关正文。
3. 在进入脚本修改前，为每项写出一个合法正例和一个非法/歧义反例。

### 阶段 B：L2/L4 与组件 DSL

1. V143/V145/V148/V150 使用 L2 的跨引用能力实现。
2. V144 选择 L2 或 L3a 的唯一责任层，不重复报错。
3. V149 将 L4 遍历对象分类扩展为协议对象、开放 map、不透明业务值。
4. V151 在自定义 DSL 中受控实现 `minimum`，并增加关键字解释测试。

### 阶段 C：MCP 容量

1. 为 `execFileSync` 设置显式 `maxBuffer`，错误分类区分执行失败、ENOBUFS 和非法 JSON。
2. 使用真实 L2 子进程构造接近边界的高基数输入，不只注入 mock executor。
3. 继续断言最终 `content[0].text` UTF-8 字节数不超过 20KB，且 `layerStats` 保留真实总数。

### 阶段 D：审计和 CLI

1. V153 只增加冲突说明和复合引用，不批量重写历史标题编号。
2. V154 统一四个脚本的路径匹配，Windows 上执行绝对反斜杠 glob 实测。
3. 扫描审计定义数量、唯一 V 数量、最大 V 和复合引用可定位性。

## 验证策略

| 变更范围 | 最小验证 |
|---|---|
| 文档/Schema | JSON 解析、聚焦示例统一入口、交叉链接检查 |
| L2/L3a/L4 | 每项正反例运行对应分层命令，再跑四个官方场景 |
| MCP | build、完整 test、tools smoke；真实高基数 validate_content |
| Docker | 仅当 MCP 运行/打包行为受影响时构建并 smoke，记录镜像和清理状态 |
| 审计 | V 定义/唯一编号扫描、相对链接与 fragment 检查、`git diff --check` |

## 风险控制

- V142/V147 会改变此前未定义配置的实际请求形态，必须在 CHANGELOG Unreleased 明示。
- V143/V145/V144/V150/V151 会把此前可通过的歧义配置改为静态拒绝，应提供明确迁移提示。
- V146 涉及用户可见错误流程，正文必须给出确定时序示例，不能只写原则。
- V152 不应简单设置无限大缓冲区；必须保留显式上限和可诊断错误。
- V153 不得为追求数字整齐而改写历史审计编号。
