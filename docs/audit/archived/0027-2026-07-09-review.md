# 审计报告 0027：grid span 契约漂移 + Renderer 示例 + 数据响应契约分类

## 背景

本轮审计基于 0026 已归档后的再次全量文档复审。0026 中发现的 `$parentRow.*` 静态校验边界与表达式章节引用漂移在当前工作区已确认修复，本轮不重复报告 V43-V44。

## 发现的问题

### V45（🔴）`grid` 直接子节点通用 `span` 约定与组件 DSL 冲突

**位置：** `docs/03-component-registry.md` 通用约定；`docs/schemas/component-registry.json` 各组件 `props`；`scripts/validate-l2-components.js` 组件契约校验

**当前值：** `03-component-registry.md` 的通用约定写明：

```text
任何作为 `grid` 直接子节点的 Node，均可在自身 `props` 中声明 `span`（语义级占栏数）
```

但 `component-registry.json` 只在部分组件中声明 `span`，例如 `section` / `statCard` / `chart` / `text` / `table`；`form` 与多个字段类组件并未声明 `span`，且其 `props.additionalProperties` 为 `false`。

**实测证据：** 以下配置符合正文“任意 `grid` 直接子节点可声明 `span`”的描述：

```yaml
body:
  type: grid
  props:
    columns: 2
  children:
    - type: form
      props:
        span: 1
        submitAction: submitForm
      children:
        - type: input
          props:
            field: name
            label: 姓名
```

实际执行 `node scripts/validate-l2-components.js <临时样例> --json` 返回：

```json
{
  "path": "body.children[0].props.span",
  "message": "不允许的额外字段 \"span\"（组件 \"form\" 的 props 不接受额外属性）"
}
```

**问题：** 正文把 `span` 定义为 `grid` 直接子节点的通用布局字段，但机器可读组件 DSL 与 L2 校验器仍按“各组件自行声明字段”的方式拒绝未列入 props 表的 `span`。这形成“文档允许、CI 拒绝”的直接矛盾。

**影响：** 🔴 严重。配置生成方按总约定编写合法的网格布局时，可能在 L2 被拒绝；Renderer / AI / 后端团队会对 `span` 是“通用布局字段”还是“少数组件私有 props”产生不同实现口径。

**建议修复：** 优先保留 B4 的既定口径，即 `span` 是 `grid` 直接子节点的通用语义字段。同步补齐 `component-registry.json` 中缺失组件的 `span` 声明，并在 `03-component-registry.md` 中说明 `span` 的通用性及必要时在组件表中补充对应字段。若决定收窄能力，则必须同步改写通用约定，明确只有组件契约显式声明 `span` 的组件可用。

### V46（🟡）`08-renderer-spec.md` 的 `$deps` 数据依赖示例不满足当前 `form` 组件契约

**位置：** `docs/08-renderer-spec.md` §2.3 数据依赖声明；`docs/schemas/component-registry.json` `form.props.allOf`

**当前值：** `08-renderer-spec.md` §2.3 为了说明 `data.params` 中 `$deps.*` 仅在表单上下文有效，给出如下结构：

```yaml
body:
  type: form
  children:
    - type: select
      props:
        field: orderStatusFilter
        label: 订单状态
    - type: table
      data:
        params:
          status: $deps.orderStatusFilter
```

但当前 `form` 组件契约要求：默认模式下必须提供 `submitAction`；若 `mode: search`，则必须提供 `targetTable`。

**实测证据：** 将该片段补成最小页面后执行 `node scripts/validate-l2-components.js <临时样例> --json`，L2 返回：

```json
{
  "path": "body.props.submitAction",
  "message": "条件必填字段 \"submitAction\" 缺失（allOf[0] else 分支）"
}
```

**问题：** 示例想表达“表单上下文中的表格可在 `data.params` 读取 `$deps.*`”，但示例中的 `form` 本身不满足当前组件契约。读者复制该片段补全成页面后会被 L2 拒绝。

**影响：** 🟡 中等。不会改变协议字段定义，但会削弱 Renderer 文档中最关键的数据依赖示例的可执行性，尤其容易误导 AI 生成缺少 `form.props` 条件必填字段的配置。

**建议修复：** 将示例改成当前协议认可的完整结构：要么补齐默认表单的 `submitAction` 与对应顶层 action，要么明确使用 `mode: search` / `targetTable` / 目标表格 `id` 的搜索表单结构。修复后应抽取该示例验证 L2 通过。

### V47（🔴）`select.optionsSource` 响应体格式在 `04-datasource-contract.md` 内部自相矛盾

**位置：** `docs/04-datasource-contract.md` §4.1 与 §9；`docs/decisions/0005-response-mapping.md` D4 / 后果

**当前值：** `04-datasource-contract.md` §4.1 标题为：

```text
列表类接口（配合 `table`/`select` 远程数据源）
```

并给出默认响应体：

```json
{
  "list": [ ... ],
  "total": 128
}
```

同一文件 §9 又明确 `select.optionsSource` 的响应体为裸数组：

```text
返回裸数组（无需 `list`/`total` 包装）
```

ADR-0005 也明确：

```text
`select.optionsSource` 继续默认期望响应体为裸数组；若未来需要对 `optionsSource` 的包裹响应做映射，应在 `optionsSource` 契约中另行标准化，不复用 `data.responseMapping`。
```

**问题：** `select.optionsSource` 被同时归入“`list` / `total` 包装的列表类接口”和“裸数组远程选项接口”。这两个响应体契约互斥，会让后端和生成器对远程选项接口到底返回 `{ list, total }` 还是 `[]` 产生相反理解。

**影响：** 🔴 严重。远程选项是常见基础能力，一旦后端按 §4.1 返回包裹结构，而 Renderer 按 §9 / ADR-0005 读取裸数组，就会导致选项加载失败；反之，后端按 §9 返回裸数组时，读者又可能误以为违反了 §4.1 的列表类接口约定。

**建议修复：** 将 §4.1 的适用范围从“`table` / `select` 远程数据源”改为 `table` / 通过 `DataRef` 消费列表数据的组件；并在 §4.1 或 §9 明确 `select.optionsSource` 是组件私有远程选项契约，默认响应体为裸数组，不复用 `data.responseMapping`。

### V48（🟡）`chart` 与 `responseMapping.list` 的适用口径在 ADR、校验文档和 L2 实现之间不一致

**位置：** `docs/decisions/0005-response-mapping.md` D3 / D4；`docs/06-validation.md` §1 与 §6；`scripts/validate-l2-components.js` `validateResponseMapping()`

**当前值：** ADR-0005 D3 将 `responseMapping.list` 的适用场景写为：

```text
`table`、需要数组数据的 `chart`、未来复用 `DataRef` 的列表组件
```

`06-validation.md` 又泛称：

```text
列表类接口声明 `responseMapping` 时必须提供 `list`
```

但当前 L2 实现明确写为：

```javascript
// 只有 table 是列表类接口；使用 API 数据时若声明了 responseMapping 则必须有 list。
// 其他 supportsData 组件（statCard / text / chart）是单值/聚合数据，不强制要求 list。
```

**实测证据：** 以下 `chart` 声明了 `responseMapping` 但没有 `list`：

```yaml
body:
  type: chart
  props:
    chartType: line
    xField: date
    yField: value
  data:
    source: api
    url: /api/chart
    responseMapping:
      total: result.total
```

实际执行 `node scripts/validate-l2-components.js <临时样例> --json` 返回：

```json
{
  "violations": [],
  "parseErrors": []
}
```

**问题：** ADR 与校验文档暗示 `chart` 可作为需要数组数据的 `responseMapping.list` 消费者，但 L2 当前把 `chart` 排除在列表类接口之外。协议没有明确 `chart` 是否支持通过 `responseMapping.list` 从包裹响应中取数组。

**影响：** 🟡 中等。若后端返回 `{ result: { records: [...] } }` 这类包裹数组，读者按 ADR 配置 `chart.data.responseMapping.list` 可能以为标准支持；但校验文档与 L2 的口径不一致会让实现方无法判断 `chart` 的正确响应格式与校验责任。

**建议修复：** 选择并统一一个口径：若 `chart` 应支持包裹数组，则在 `04` / `08` 明确 `chart` 使用 `responseMapping.list` 的语义，并更新 L2 对 `chart` 声明 `responseMapping` 时的 `list` 检查；若 `chart` 只支持裸数组，则更新 ADR-0005 与 `06-validation.md`，移除“需要数组数据的 `chart`”与泛化“列表类接口”说法。

## 与已归档审计的关系

- V45 与 0001 B4（`span` 提升为通用 props）相关，但本轮不是再次讨论是否需要 `span`，而是发现通用约定没有完整落入组件 DSL 与 L2 校验链。
- V46 与 0010 / 0015 中的 Renderer 示例可执行性问题同类，但本轮定位的是 §2.3 数据依赖示例与当前 `form` 条件必填契约之间的新漂移。
- V47 与 ADR-0005、0018 V20（`responseMapping` 条件必填语义校验）相关，但本轮对象是 `select.optionsSource` 的响应体分类，未在 V20 中解决。
- V48 与 0018 V20、0023 V33（`responseMapping` 默认字段名与 L2 误报）相关，但本轮不是默认 `{ list, total }` 的误报问题，而是 `chart` 是否属于 `responseMapping.list` 消费者的口径分裂。

## 建议处理顺序

1. 优先修复 V45。它会导致正文允许的布局配置被 L2 直接拒绝，属于机器契约冲突。
2. 修复 V47。它会导致后端对 `select.optionsSource` 响应体格式产生互斥理解。
3. 修复 V48。先决定 `chart` 的 `responseMapping.list` 支持边界，再同步 ADR / 校验说明 / L2。
4. 修复 V46。该项是示例可执行性问题，适合在数据依赖口径收敛后一并验证。

## 防止复发建议

- 对“通用 props”建立一张机器可读或脚本可检查的覆盖清单，避免新增通用字段后只同步部分组件。
- 文档示例凡包含 `body:` 的 YAML 块，应至少能被补齐为最小页面并通过 L2；Renderer 文档尤其需要避免不可复制片段。
- `responseMapping` 这类跨组件数据契约应明确“组件类型 -> 默认响应形态 -> 是否支持映射”的矩阵，减少“列表类接口”这类泛称在不同文档中的漂移。
