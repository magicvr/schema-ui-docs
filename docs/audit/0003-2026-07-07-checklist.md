### `datePicker` / `dateRangePicker` 组件

- [ ] **补充 `datePicker` 到组件注册表**
  - 字段拟定：`field` / `label` / `labelKey` / `placeholder` / `format`（日期格式）/

### 弹窗内容 Node 化

- [ ] **`actions[].type: modal` 增加 `content` 字段**
  - 当前：`modal` 仅有 `modalId`，弹窗内容是前端硬编码的——渲染内容不在协议控制范围内，
    这违背了 Schema-Driven 的核心原则
  - 建议：`modal` 类型增加可选 `content: Node`，直接描述弹窗内容（如一个 `form`）：

    ```yaml
    actions:
      confirmRefund:
        type: modal
        modalId: refund-confirm-dialog   # 可选，引用预注册弹窗模板
        content:                          # 新增：直接描述弹窗内容
          type: form
          props:
            title: 确认退款
            submitAction: doRefund
          children:
            - type: text
              props:
                content: 确认对订单 #{$deps.orderId} 发起退款？
    ```
  - 需明确：`content` 和 `modalId` 的关系——`modalId` 提供弹窗模板（尺寸、位置、关闭行为），
    `content` 提供弹窗内容，二者可共存
  - 涉及文件：`docs/07-actions-contract.md`、`schemas/action.schema.json`

## 🟡 强烈建议：不阻断 MVP 但应尽快跟进

### R3. 错误边界策略 → `08-renderer-spec.md`

- [ ] **约定节点级错误边界行为**
  - 单个 Node 渲染/数据加载失败时的行为：
    - 该 Node 展示 `states.error.fallbackText`（若已配置）或默认错误提示
    - 兄弟节点不受影响，继续正常渲染
    - 错误信息通过 `console.error` 输出，不阻断页面
  - 涉及文件：`docs/08-renderer-spec.md`

  - 涉及文件：`docs/08-renderer-spec.md`

  - 涉及文件：`docs/08-renderer-spec.md`

### `searchForm` 组件

- [ ] **评估是否需要独立的 `searchForm` 类型**
  - `searchForm` 与 `form` 的核心差异：提交行为不是发 `submitAction` 请求，而是
    触发表格重新请求（带筛选参数）
  - 方案 A：给 `form` 增加 `mode: search` 属性，配合 `targetTable: <nodeId>` 声明关联表格
  - 方案 B：新增独立 `searchForm` 组件
  - 建议先评估 MVP 是否需要，如果需要则走方案 A（改动量最小）
  - 涉及文件：`docs/03-component-registry.md`（如需新增）

### `upload` 组件

- [ ] **评估是否需要 `upload` 组件**
  - 字段拟定：`field` / `label` / `labelKey` / `accept`（文件类型限制）/
    `maxSize` / `multiple` / `action`（上传地址）
  - 取决于 MVP 是否包含上传场景，可作为 v0.3 内容
  - 涉及文件：`docs/03-component-registry.md`、`schemas/component-registry.json`
