## 发现的问题

### V225（🔴 P0）发布证据中的 fixture digest 与仓库可复算值不一致

**证据**

1. 当前与 tag `v1.0.0`（`d2f0fc0877dc6550c9fe7e3635b25c7ec72b4ddd`）对 `conformance/fixtures/**` 按 `scripts/release-check.js` 同算法计算，digest 均为：

   `sha256:fa480baf15bab7c3f05b9c505298e574754e5d86a199519c9866d2982631175c`

2. 发布叙述与归档审计仍写另一值：

   `sha256:7bf6db4ced24ed59f68a32f3072540b4be179c6aadf5c23042350e6668c66f6b`

   出现位置包括：`docs/audit/archived/0055-2026-07-11-review.md`、`0055-...-checklist.md`、`docs/audit/archived/README.md`（索引）。

3. `scripts/release-check.js` 只**计算并打印** `fixtureDigest`，**从不**与期望 digest 做 `assert`；因此错误证据可以随 CI 绿灯长期保留。

**影响**

- 外部消费者若按 0055/发布记录 pin 错误 digest，会误判“制品不一致”或 pin 到不存在的字节状态。
- v1 可复现发布门禁（V223）在“证据层”失效：digest 变成日志字段而非硬门禁。

**严重度** 🔴 P0（发布完整性）

### V226（🟡 P1）根 CLI 与 MCP 的 L0/L1 错误集合不一致

**证据**

同一最小非法页面 `/tmp/page-l0.yaml`：

```yaml
meta:
  protocolVersion: "1.0"
layout:
  type: page
```

| 路径 | 实现 | 报告 |
|---|---|---|
| `npm run validate -- /tmp/page-l0.yaml` | `scripts/validate-all.js` → `ajv-cli` **未**传 `--all-errors` | 仅 1 条：`must have required property 'body'` |
| MCP `validateContent` | `mcp/src/core/validation-runner.ts` `new Ajv({ allErrors: true, ... })` | 4 条：`body` + `additionalProperties: layout` + `meta.pageId` + `meta.title` |

对照代码：

- `scripts/validate-all.js`：`ajv-cli validate ...` 无 `--all-errors`
- `mcp/src/core/validation-runner.ts:194`：`allErrors: true`

官方场景在双路径上均可通过（pass/fail 一致）；差异在**违规诊断完整性**，会影响 AI/CI 修复循环与 MCP vs 根 CLI 用户体验。

**影响**

- 同一内容在根工具链与 MCP 工具上看到不同违规集合，削弱“单一校验真相源”承诺（ADR-0007 / 06-validation）。
- 未一次暴露全部 L0/L1 问题时，调用方可能多轮试错。

**严重度** 🟡 P1

### V227（🟡 P1）65 个版本化 fixtures 中绝大多数仍标注 `protocolVersion: "0.3"`

**证据**

对 8 个 versioned suite（`fixtureVersion: "1.0"` 包装）统计 `cases[].protocolVersion`：

| 值 | 数量 |
|---|---|
| `"0.3"` | **58** |
| `"1.0"` | **7**（6 个 official scenarios + 1 个 `supported-v1-version`） |

样例：

- `conformance/fixtures/actions/cases.json` → `success-toast` 等为 `0.3`
- `conformance/fixtures/search-table/cases.json` → 全部 `0.3`
- `conformance/fixtures/uploads/cases.json` → 全部 `0.3`

`scripts/release-check.js` 只断言 suite 存在、`fixtureVersion === "1.0"`、case 总数 65，**不**校验 case 级 `protocolVersion` 是否与当前稳定 `"1.0"` 对齐。

CHANGELOG / 总纲宣称稳定协议为 `1.0` / `1.0.0`，且“八类 65 fixtures”为 v1 制品；若生产消费者按 `protocolVersion === "1.0"` 过滤，将只跑到 7 条。

**影响**

- v1 互操作覆盖面被错误缩小（过滤）或语义混淆（0.3 case 在 1.0 包版本下仍表示“当前算法”）。
- 与 `docs/09-v1-release-goals.md` “示例和 Renderer 支持版本统一为 1.0” 的完成表述不完全一致。

**严重度** 🟡 P1

### V228（🟡 P1）根依赖 high severity 且 CI 无 `npm audit` 门禁

**证据**

```text
npm audit  # repo root
fast-json-patch <3.1.1  (high)  GHSA-8gh8-hqwg-xf34
  via ajv-cli >=0.7.0
fix available via npm audit fix --force → ajv-cli@0.6.0 (breaking)
```

- 引入链：`package.json` devDependency `ajv-cli@^5.0.0` → `fast-json-patch@2.2.1`（见 `package-lock.json`）。
- `.github/workflows/mcp-ci.yml` / `mcp-cd.yml` **均无** `npm audit`（或 `audit-level=high`）步骤。
- 根 L0/L1 生产路径依赖 `ajv-cli`（`scripts/validate-all.js`）；MCP 运行时 L0/L1 使用自有 `ajv`，Docker 根层 `npm ci --omit=dev` 不含 `ajv-cli`，故镜像运行时不受此 CVE 直接影响，但**本地/CI 协议校验路径**会安装该依赖。

**影响**

- 开发者与 CI runner 环境存在已知原型污染依赖；无自动门禁阻止进一步恶化。
- `npm audit fix --force` 不可用（降级 ajv-cli 破坏当前 validate 入口）。

**严重度** 🟡 P1

### V229（🟢 P2）fixture suite Schema 枚举过期；两类数组 fixtures 游离于版本化校验

**证据**

`conformance/schemas/fixture-suite.schema.json` 的 `category` enum：

```text
version-negotiation, request-construction, response-mapping, search-table,
reactions, actions, errors, uploads, scenarios
```

- 含已不存在的 **`errors`**（Action 错误并入 `actions` suite，见 `conformance/README.md`）。
- **缺少** 实际存在的 `query-serialization`、`table-query-state`（二者仍为裸数组 JSON）。

`conformance/runner/validate-fixture-suites.js` 显式跳过数组文件：

```js
.filter(filePath => !Array.isArray(JSON.parse(fs.readFileSync(filePath, 'utf8'))));
```

因此 `npm run validate:conformance` 只报 `8 versioned suites`，不验证 16+5 条数组向量的 schema/id 约束。README 称 G1–G3 早期数组格式“暂保留”可解释历史，但 enum 中的 `errors` 与缺失目录名属于**死引用/缺口**。

**影响**

- 新增/改坏数组 fixtures 时，suite 校验不会失败（依赖各自 runner 的 deepEqual）。
- Schema 作为“版本化契约”文档对外部实现有误导。

**严重度** 🟢 P2

**证据**

**影响**

- README 表格同时列出两行“搜索/分页/排序状态”，读者难以判断权威源。

**严重度** 🟢 P2

### V232（🟢 P2）MCP 开发依赖存在 critical/high 审计项，CI 同样不跑 audit

**证据**

```text
npm audit --prefix mcp
vitest <=3.2.5  critical  (via @vitest/mocker / vite / esbuild 链)
vite high (path traversal in optimized deps .map)
esbuild moderate
```

- `mcp/package.json`：`vitest@^2.1.9`（lock 解析为 2.1.9）。
- 上述主要为 **devDependency**；`Dockerfile` 在 build 后 `npm prune --prefix mcp --omit=dev`，生产镜像通常不包含 vitest/vite。
- CI/CD 未对 `mcp` 执行 audit，无法阻止 dev 链进一步引入可被 CI 利用的漏洞。

**影响**

- CI 运行 Vitest 时的供应链风险；对 stdio 生产镜像影响有限。
- 与 V228 同属“无安全门禁”类问题。

**严重度** 🟢 P2
