### V309（🟡）同一 `2.4.0` 多 contentDigest — 已关闭（方案 A）

**现象：**

1. 正式协议 tag / GitHub Release 资产 digest 为 `343d6795…` / contentDigest `7583cefe…`。
2. 当前工作树 `npm run build:protocol` 与 GHCR `schema-ui-mcp:2.4.0` 内置 manifest 的 contentDigest 为 `643ed942…` / 本地 tar `5631d459…`。
3. 差异源：tag 之后 commit `fc07514`（MCP MAJOR.MINOR 共线）修改了进入协议制品的 `docs/CHANGELOG.md`、`docs/RELEASE.md`，**未** bump 协议 `artifactVersion`。
4. MCP 镜像 label 仍为 `io.schema-ui.protocol.artifact-version=2.4.0`，与正式协议 Release **非字节同一快照**。

**冲突双方：**

- 章程/发布叙事：协议制品应可固定 pin（版本 + SHA-256）；消费者按 tag 或 tar digest 固定。
- 当前状态：版本字符串 `2.4.0` 映射到至少两套 contentDigest。

**影响：**

- 从 `main` 重建与 pin Release tar 得到不同字节，破坏「版本号唯一对应 digest」假设。
- MCP 与协议 Release 声明同协议制品版本，却不是同一内置快照（差异目前限于发布元数据文档，**非** schemas/fixtures）。
- 后续若继续在同版本上改制品路径，漂移会扩大且无 fail-closed 门禁。

**已采纳修复（方案 A，2026-07-24）：**

1. 协议制品升至 **`2.4.1`**（`protocolVersion` 仍为 `"2.4"`）；根包 / lockfile / `protocol-manifest.json` / validator 兼容声明同步。
2. MCP 升至 **`2.4.1`**，`schemaUiProtocol.artifactVersion: "2.4.1"`；`release:check:mcp` 与 MCP 128 tests 通过。
3. CHANGELOG `v2.4.1` 记录正式 `v2.4.0` 与 tag 后漂移指纹对照；**未**改写 `v2.4.0` tag / Release 资产。
4. 本工作树 `2.4.1` digests：contentDigest `sha256:d6852ee6ff12a19b00b4acf0b51e221457fe14691a53ff5ed828876877766efe`；artifact `sha256:c027fa6c5b4bcb379a2fc90f6447f0e8df0729df5657fcb5d6a382d9ee3fbb18`。正式远端 pin 以未来 `v2.4.1` Release 为准。

**验证：** `npm run release:check`、`verify:protocol-artifact`、`release:check:mcp`、`check:links`、`npm --prefix mcp test` 均通过；fixtureDigest 与基线一致（机器契约未变）。
