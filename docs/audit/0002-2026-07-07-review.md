## 二、新发现的问题

### D1. [P0] `04-datasource-contract.md` 第 7 节"静态数据"重复出现

**位置：** [04-datasource-contract.md](../04-datasource-contract.md) 第 101 行和第 138 行。

**描述：** `## 7. 静态数据（\`data.source: static\`）` 在第 101 行和第 138 行完全重复出现两次，内容完全相同（包括代码示例）。这导致编号错乱——第 138 行之后本是第 8 节（optionsSource 契约），但已被错误的重复第 7 节覆盖。

**推测原因：** 前次审计实施 A4（补充 datasources 示例）和 B7（optionsSource）的编辑过程中，插入操作不慎将原有第 7 节内容残留。后续的 `04-datasource-contract.md §8` 实际引用了"第 8 节"，但文档里第二个第 7 节之后的编号实际上已无第 8 节。

**影响：**
- 读者阅读到第二个第 7 节时会产生困惑。
- 如果按"8. select.optionsSource"引用该节（如 `03-component-registry.md` 中 `select` 的说明），实际文档中没有编号 8 的标题，引用链断裂。
- 但内容本身并未丢失（只是重复），所以不是内容缺失而是排版混乱。

**修复建议：** 删除第 138 行开始的重复第 7 节（第 8 节 optionsSource 契约实际在更上方已存在，确保保留唯一一份即可）。

### D2. [P1] 顶层文档结构缺少 JSON Schema

**位置：** `schemas/` 目录。

**描述：** `01-node-protocol.md §2` 定义了顶层文档结构：

```yaml
meta:            # 页面元信息
  pageId: string
  title: string
  description: string
  protocolVersion: string   # 必填（since 0.2）

datasources:     # 【可选】页面级预声明数据源
  <sourceId>: DataSourceDef

body: Node        # 页面主体，根 Node

actions:          # 【可选】页面级可复用动作定义
  <actionId>: ActionDef
```

但当前 `schemas/` 目录下只有 `node.schema.json`（校验 Node 节点）、`reaction.schema.json`、`component-registry.json`、`action.schema.json`（校验 Action 定义），**没有一份用于校验**顶层文档结构（`meta` + `datasources` + `body` + `actions` 的整体）的 Schema 文件。

**影响：**
- 实践中，开发者/CI 只能各自把顶层文档拆开分别校验，没有一份"完整的页面文档"Schema。
- `meta.protocolVersion`（since 0.2 必填）无法通过 Schema 自动校验必填性。
- `datasources` 中的 `DataSourceDef` 没有对应的 Schema 定义（当前只在 `04-datasource-contract.md` 和场景示例中展示，但无法自动校验）。
- `body` 的根类型未显式约束为 Node 类型（虽然实践中会由 renderer 校验）。

**修复建议：** 新增 `page.schema.json`，作为顶层文档结构的 Schema：

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "https://internal/schema-ui/page.schema.json",
  "title": "Schema-UI Page Document",
  "type": "object",
  "properties": {
    "meta": {
      "type": "object",
      "properties": {
        "pageId": { "type": "string" },
        "title": { "type": "string" },
        "description": { "type": "string" },
        "protocolVersion": { "type": "string" }
      },
      "required": ["pageId", "title", "protocolVersion"]
    },
    "datasources": {
      "type": "object",
      "description": "页面级预声明数据源，key-value 映射",
      "additionalProperties": { "$ref": "node.schema.json#/definitions/DataRef" }
    },
    "body": { "$ref": "node.schema.json" },
    "actions": {
      "type": "object",
      "description": "页面级动作定义",
      "additionalProperties": { "$ref": "action.schema.json" }
    }
  },
  "required": ["meta", "body"]
}
```

### D3. [P2] `tabs` 在 `01-node-protocol.md §3.4` 中仍被列为使用 `children` 的容器

**位置：** [01-node-protocol.md §3.4](../01-node-protocol.md#34-children可选)，第 3 句。

**原文：**
> 子节点数组，仅布局类容器（`grid`/`section`/`tabs`/`form` 等）使用。

**问题：** v0.2 变更（B2）已将 `tabs` 的 `supportsChildren` 改为 `false`，内容改由 `items[].content` 承载。但此处 `tabs` 仍被列举为"使用 children"的布局类容器，与实际情况冲突。`03-component-registry.md` 和 `component-registry.json` 已正确标注 `supportsChildren: false`。

**影响：** 读者如果只看 `01-node-protocol.md`（核心协议文档，引用频率最高），会误以为 `tabs` 仍然使用 `children`，进而写出错误的配置。

**修复建议：** 将 `tabs` 从该句的列举中移除，或加上脚注说明"since 0.2：`tabs` 内容改由 `items[].content` 承载，不再使用 `children`"。

### D4. [P2] `column` 定义中缺少 `ColumnDef` 的完整 JSON Schema 描述

**位置：** `schemas/component-registry.json` 中 `table.props.columns`。

**当前定义：**
```json
"columns": { "type": "array", "required": true }
```

**问题：** `table.props.columns` 在 `component-registry.json` 中没有定义 `items` 的详细结构（`field`/`label`/`format`/`tagMap` 等），而 `03-component-registry.md` 的文字描述中给出了完整的 `ColumnDef` 表格。这意味着 L2（组件契约）校验无法对 `columns` 做字段级检查，只能检查其类型是否为 `array`。

相比之下，`tabs.props.items` 在 JSON 中给出了完整的 `items` 结构（`key`/`label`/`labelKey`/`content`），`action.schema.json` 的 `OutcomeBehavior` 也有完整定义，`columns` 的定义深度与前两者不一致。

**影响：** 如果后端的 YAML 中 `columns` 拼写错误（如 `filed` 而非 `field`），L2 校验无法发现，只能到渲染阶段才能暴露。

**修复建议：** 为 `table.props.columns` 补充完整的 `items` 定义，包含 `field`/`label`/`format`/`tagMap` 等字段，与 `03-component-registry.md` 的表格保持一致。

### D5. [P2] `04-datasource-contract.md` 中错误响应体的 `code` 字段语义未定义

**位置：** [04-datasource-contract.md §5](../04-datasource-contract.md#5-错误约定)。

**描述：** 第 5 节给出了错误响应体约定：

```json
{ "code": "ORDER_NOT_FOUND", "message": "订单不存在" }
```

但 **`code` 字段在前端的行为未定义**——前端是否用它做程序化判断（如"如果 `code` 为 `TOKEN_EXPIRED` 则自动跳转登录页"）？还是仅展示 `message` 给用户？当前文档只描述了 HTTP 状态码 (`4xx`/`5xx`) 对应的前端行为，未说明 `code` 的业务语义是否被消费。

**影响：** 后端开发者不清楚 `code` 字段是否被前端用于程序化处理，可能导致前端希望用它做判断时后端没有包含，或反之。

**修复建议：** 明确 `code` 字段的用途：
- 若仅用于日志/调试，注明"仅用于调试日志，前端不会据此做程序化判断"。
- 若需要支持程序化判断，补充前端行为映射（如 `"TOKEN_EXPIRED"` → 跳转登录页等），但这可能超出当前协议的范围（建议放在"长远考虑"中）。

### D6. [P3] `reaction.schema.json` 中 `dependencies` 的 `minItems: 1` 与场景矛盾

**位置：** `schemas/reaction.schema.json`，`dependencies` 定义。

**当前定义：**
```json
"dependencies": {
  "type": "array",
  "items": { "type": "string" },
  "minItems": 1,
  ...
}
```

**问题：** `minItems: 1` 要求每个 `reaction` 的 `dependencies` 至少包含一个字段名。但 `02-reaction-expression.md` 允许在表达式中仅使用 `$self`（引用当前字段自身的值），不依赖其他字段，此时 `dependencies` 应为空数组：

```yaml
when: "$self == 'some_value'"
dependencies: []   # 没有依赖其他字段，只引用 $self
```

当前 Schema 会拒绝这种合法配置。

**修复建议：** 将 `minItems: 1` 改为 `minItems: 0`（或直接移除 `minItems` 约束），因为 `dependencies` 为空数组时仅引用 `$self` 是合法的。

### D7. [P3] `node.schema.json` 的 `DataRef` 中 `method` 枚举不全

**位置：** `schemas/node.schema.json`，`DataRef` 定义。

**当前定义：**
```json
"method": {
  "type": "string",
  "enum": ["GET", "POST"]
}
```

**问题：** 虽然常见场景中 GET/POST 覆盖了大多数需求，但后续可能的扩展场景（如 PUT 更新、DELETE 删除）中，`data.source: api` 模式下可能需要其他 HTTP 方法。更重要的是，当前枚举与 `action.schema.json` 中 `request` 类型的 `method` 枚举不一致（后者支持 `GET | POST | PUT | DELETE | PATCH`）。

**影响：** 如果未来某个组件（如一个支持 PUT 的"编辑表单"组件）通过 `data.source: api` 调用接口，会被 Schema 拒绝。两处 `method` 枚举不一致也会造成混淆。

**修复建议：** 将 `DataRef.method` 的枚举与 `action.schema.json` 保持一致，扩展到 `["GET", "POST", "PUT", "DELETE", "PATCH"]`。如果当前确实不希望 `data.source: api` 支持写入方法，应明确注释说明理由并在枚举中只留 GET/POST，同时与 action schema 的差异处加注释说明。
