# 审计报告 0066：协议演进完成度与发布闭环复审

## 结论

**协议能力演进（Admin 轨道 P0 + P1）在工作树语义与机器门禁上可判定为完成；正式发布闭环未完成。**

- 工作树 `package.json` / `protocol-manifest.json` 为 **`2.3.0` / `2.3`**，含 ADR-0023 `accepted` 文稿、`permissions-inheritance` suite、`14-v2.3-release-goals` 与 `migrations/2.2-to-2.3`。
- 本机 **`release:check`、`verify:protocol-artifact`、全量 JS/Python conformance（29 入口）、链接与场景校验均通过**。
- **未发现**会静默破坏合法 `2.2` 页面语义的 P0。
- **发现 2 项 P1 + 3 项 P2**（`V295`–`V299`），均属**发布工程与文档一致性**，不否定 2.3 语义本身的完成定义（见 `14` G1–G4 工程侧）。

本文档是过程记录，不是协议权威来源。权威仍以已接受 ADR、核心规范、Schema、versioned fixtures 与（打 tag 后的）协议制品 digests 为准。

### V295（P1）2.3.0 未提交 — 开放

**现象：** 约 56 个路径已 staged（含 `protocol-manifest.json`、`docs/decisions/0023-*.md` 改为 `accepted`、`permissions-inheritance` fixtures/reference/runner、`14-v2.3-release-goals.md`、`migrations/2.2-to-2.3.md`、L2、CHANGELOG 等），但：

- `HEAD` 仍为 `cc1505a`（制品 2.2.0，ADR-0023 正文 **proposed**）；
- 分支与 `origin/dev` 对齐，**无未推送 commit**。

**风险：** 协作方按 remote 消费时看不到 2.3；本地 digests 与 remote 内容摘要不一致；误把工作树绿当作「已合并」。

**关闭条件：**

1. 将 staged 2.3 原子提交（或等价 PR）合入约定分支；
2. 提交后 `HEAD` 的 `artifactVersion`/`protocolVersion` 为 `2.3.0`/`2.3`，ADR-0023 `status: accepted` 且已列入 `protocol-manifest.json` `authority.semanticSpecs`；
3. 提交后复跑 `release:check` 与 `verify:protocol-artifact` 仍通过。

---

### V296（P1）2.2 / 2.3 无协议 tag — 开放

**现象：** [RELEASE.md](../RELEASE.md) 规定正式协议资产仅在人工 `git tag v*` 后由工作流发布。本地与可见 tag 列表仅有 `v1.0.0`、`v2.1.0`。CHANGELOG 与 `13`/`14` 已写「正式协议制品 `schema-ui-protocol-2.2.0.tar.gz` / `2.3.0.tar.gz`」。

**风险：** 消费者无法 pin 不可变 tag；「已发布」与「工作树可构建」混淆；2.2 提交 `712f991` 与 2.3 提交（待 V295）缺少发布指纹对齐。

**关闭条件：**

1. 在目标 commit 上分别打（或确认存在）`v2.2.0`、`v2.3.0`（顺序：先 2.2 再 2.3，除非发布策略明确跳过中间 tag 且文档同步）；
2. `git push origin` 对应 tag，Protocol Release 工作流成功或等价本地 `release:check:tag` + 制品上传证据；
3. 记录各 tag 的 commit SHA 与 artifact SHA-256（2.3 工作树当前 artifact digest 见基线表，**仅在同内容提交上有效**）。

### V297（P2）`main` 与对外叙事偏离 — 开放

**现象：** `main` / `origin/main` 仍指向 `v2.1.0` 合并点；`dev` 含 2.2 提交与（待提交的）2.3。对外入口若默认读 `main`，会得到落后一个以上 MINOR 的协议。

**关闭条件：**

1. 发布策略二选一并写清：`main` 始终等于最新稳定协议，或文档明确「稳定 tag 优先、`main` 可能滞后」；
2. 若选前者：将已打 tag 的 2.2/2.3 变更合入 `main`；
3. README / 00-overview 不暗示 `main` HEAD 即最新已发布制品，除非事实如此。
