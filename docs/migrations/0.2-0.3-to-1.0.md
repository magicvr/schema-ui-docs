# 从 0.2 / 0.3 迁移到 1.0

本文面向现有页面配置、Renderer 和后端接口维护者。迁移必须先在目标分支完成 L0-L4 与跨实现 fixtures，再切换 `protocolVersion`；不要只替换版本字符串。

## 1. 版本协商

- 页面显式声明 `meta.protocolVersion: "1.0"`。
- Renderer 的 `supportedVersions` 必须精确包含 `"1.0"`；不使用最低兼容版本、范围或最近版本 fallback。
- 缺少 `protocolVersion` 的旧页面只能在标准入口前显式调用 legacy adapter。adapter 输出 `1.0` 页面后，必须重新运行完整校验和 capability 协商。
- 使用上传或行级后端请求的页面分别声明 `actions.upload`、`actions.row.request` capability。

## 2. Query 参数

`data.params`、`datasources.*.params`、`optionsSource.params` 和 `requestMapping.query` 只允许非空 key 与 string、finite number、boolean、`null` 标量。对象和数组不再接受；迁移为多个标量参数，或在 BFF/后端适配层编码复杂结构。

URL 按 ADR-0010 规范化：UTF-8 + RFC 3986 编码，空格为 `%20`，`null` 删除同名参数，最终 key 排序且不重复。不要依赖 `URLSearchParams` 的 `+` 空格或后端框架的数组默认格式。

## 3. 分页与搜索保留名

`page`、`pageSize`、`sort` 是 Renderer 保留参数：

- 从 `data.params` 和 `datasources.*.params` 删除这些 key；
- 搜索表单的 `field` / `startField` / `endField` 不得使用这些名称；
- 若后端使用其他分页参数名，在 BFF 或接口适配层转换。

搜索提交和清空筛选会将 `page` 重置为 `1`；排序变化也重置页码，翻页保留筛选和排序。

## 4. 数据与响应映射

- `responseMapping` 与 `params` 同级，禁止放在 params 内。
- `data.source: ref` 的本地 `responseMapping` 整体覆盖 datasource mapping，不做字段级合并。
- table/chart 声明生效映射时必须映射 `list`；服务端分页 table 还必须映射 `total`。
- `statCard` / `text` 的 `valueField` 位于组件 `props`，不放在 `data`。

## 5. 表单与行级 Action

- 普通表单 `submitAction` 不得引用 GET request；表单字段形成 JSON body。
- 行级后端动作使用 `actionRef` + `requestMapping`；path/query/body 都是扁平 map，只允许字面量或单个 `$row.*` 点路径。
- `$parentRow.*`、模板拼接、表达式和 `$context.*` 注入均静态拒绝。
- `400` 字段错误、`401/403` hook 和安全错误文案先于 `onError`，配置行为不能绕过协议级处理。

## 6. Reaction

Renderer 使用稳定快照和批量提交：本轮读取旧值，本轮末尾提交，下一轮才读取新值。同一字段多条 `value` 写入按 reactions 数组后写优先；深相等 no-op 不触发下一轮；实现必须提供循环上限。

## 7. 上传

UploadAction 默认使用 POST 和 multipart 字段 `file`。`multiple: true` 时每文件一个请求，按选择顺序串行；整批先执行 accept / maxSize 校验，任一失败不提交部分字段值。成功响应优先取非空 `url`，否则取非空 `id`。

`actionRef` 模式的 accept / maxSize / multiple 只从 UploadAction 读取，upload 组件不得重复声明。

## 8. 验证顺序

1. 运行 `npm run validate -- "<page-glob>"` 修复 L0-L4 违规。
2. 运行 `npm run validate:conformance` 和全部 `test:conformance:*` runners。
3. 让生产 Renderer 与生产后端直接消费固定 commit SHA 下的同一 fixtures。
4. 运行 `npm run release:check`；正式 tag 流程使用 `npm run release:check:tag`。
5. 构建候选镜像并执行本地与远端固定版本/SHA smoke。
