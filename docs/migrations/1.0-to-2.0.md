# v1.0 到 v2.0 迁移指南

v2.0 是不兼容的 MAJOR 协议版本。v1.0 页面不能直接交给只支持 `2.0` 的标准 Renderer；调用方应继续使用 v1.0 Renderer，或先使用显式 legacy adapter 完成迁移，再把结果交给 v2.0 Renderer。标准 Renderer 不会自动猜测、回退或串联 adapter。

## 1. 版本声明与协商

将页面元数据从：

```yaml
meta:
  protocolVersion: "1.0"
```

更新为：

```yaml
meta:
  protocolVersion: "2.0"
```

v2 Renderer 应声明 `supportedVersions: ["2.0"]`。旧页面与 v2 Renderer 的版本协商结果必须是 `UNSUPPORTED_PROTOCOL_VERSION`；迁移 adapter 的输出必须重新执行 L0–L4 校验和 v2 协商。

## 2. DataRef 和 query

- DataRef 及页面级 datasource 只允许 `GET`。原先使用 `POST`、`PUT`、`PATCH` 或 `DELETE` 的自动加载配置，应迁移为用户触发的 Action。
- DataRef `params`、远程选项 `optionsSource.params` 和表格请求的 query 仍使用原有标量 query 序列化算法；对象和数组不能通过旧 `params` 字段绕过边界。
- `page`、`pageSize` 和 `sort` 仍是保留 query 名，搜索状态、静态参数和分页状态继续遵守 ADR-0011 的覆盖顺序。
- v2.0 不通过 DataRef 增加隐式 JSON body。需要 body 的读取接口必须另行设计字段、方法和 ADR。

## 3. Action、上传和请求映射

- `type: request` 和 `type: upload` 可声明 `retryPolicy: never | idempotent`，缺省为 `never`。
- `idempotent` 的同一次逻辑调用在重试中复用同一个 `Idempotency-Key`；超时或网络中断结果为 `unknown`，不能当作服务端未执行。
- 行级 Action 的 `requestMapping` 只允许标量值。`$row.*` 使用 own-property 读取，并拒绝 `__proto__`、`prototype` 和 `constructor` 路径。
- 使用 `actions.upload` capability 的页面仍需声明 `meta.requiredCapabilities: [actions.upload]`，上传大小、字段和响应映射按现有 upload Action 契约迁移。

## 4. URL

DataRef、页面 datasource、Action、上传、远程选项和 navigate outcome 的 URL 必须是 baseURL 下以单个 `/` 开头的相对路径。绝对 URL、协议相对 URL、空白和反斜杠形式不能继续作为页面协议字段使用；跨 origin 能力由宿主预注册机制承载。

## 5. 迁移验收

迁移完成后，至少执行：

```bash
npm run validate:scenarios
npm run validate:conformance
npm run test:conformance:version
npm run test:conformance:version:python
npm run test:conformance:request
npm run test:conformance:request:python
npm run test:conformance:uploads
npm run test:conformance:uploads:python
npm run release:check
```

生产 Renderer 和后端消费者必须针对同一 v2.0 fixtures 逐字段比较结果；不能只通过 Schema 校验来证明迁移完成。
