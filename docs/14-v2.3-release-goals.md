# v2.3 发布目标与门禁

本文档记录 **v2.2 → v2.3** 的 MINOR 发布：容器 `edit` / `delete` 权限继承与显式操作 intent（ADR-0023）以双重版本/capability 门控进入核心协议。没有新字段的合法 v2.2 页面保持原有本地权限语义；使用新字段的页面必须声明 `meta.protocolVersion: "2.3"` 和 `permissions.inheritance`。

权威 ADR：[0023](./decisions/0023-container-permission-inheritance.md)。迁移见 [migrations/2.2-to-2.3.md](./migrations/2.2-to-2.3.md)。轨道见 [11-next-admin-lifecycle-goals.md](./11-next-admin-lifecycle-goals.md)。

## 1. v2.2 到 v2.3 的边界

- v2.3 页面声明 `meta.protocolVersion: "2.3"`。
- Renderer 可同时支持 `"2.2"` 与 `"2.3"`，但不得为旧页面推断或自动开启继承。
- 新增预定义 capability：`permissions.inheritance`。
- 新增 Node 字段：`permissionCascade: { keys: [edit | delete] }`，仅 `section` / `grid` / `form` / `tabs` / `table` 可声明，且每个 key 均有同 Node 的 `permissions.<key>` source。
- 新增入口字段：`permissionIntent: edit | delete`，仅 RowAction、toolbar Trigger 和 `actionButton.props` 可声明。
- 权限结构边仅包括 Node `children[]`、tabs 内容、table `actions[]` / `toolbar[]` 和 default form submit；modal content、导航页是新根，`columns[]` 始终只适用本地 `permissions`。
- L2 字段集下限：`permissionCascade` / `permissionIntent` 必须 `protocolVersion >= "2.3"`，且缺 capability 时 fail-closed。

## 2. v2.3 阻断门禁

### G1. 版本与 capability

- [x] ADR-0023 `accepted`，并列入 `protocol-manifest.json` 的 `authority.semanticSpecs`。
- [x] `permissions.inheritance` 作为预定义 capability 写入 Node、页面 Schema、Renderer 和迁移规范。
- [x] 任何 `permissionCascade` / `permissionIntent` 都由 L2 同时要求 `protocolVersion >= "2.3"` 与 capability；无祖先 cascade 的 intent 仍合法但只适用本地权限。
- [x] 没有新字段/能力声明的 v2.2 页面不改变既有 `permissions.edit` / `permissions.delete` 的本地语义。
- [x] 根包、lockfile、manifest、validator 与 MCP 兼容声明对齐 `2.3.0` / `2.3`。

### G2. 结构与静态校验

- [x] Node Schema 定义封闭的 `permissionCascade.keys`；组件 DSL 仅在批准的三类操作入口定义 `permissionIntent`。
- [x] L2 拒绝非容器 cascade、空/重复/非标准 keys、缺失同名 source permission、错误 intent 值和错误挂载。
- [x] L2 保留 `table.props.columns[]` 本地权限边界，拒绝其 `permissionIntent`；default form submit 是隐式 edit，search form 不参与。
- [x] L3a 继续以 `$context.user.*` / `$context.features.*` 白名单扫描所有 `permissions.*` source，不因继承开放新表达式能力。

### G3. 可观测算法与场景

- [x] `permissions-inheritance` JS/Python suite 覆盖 children、tabs、modal/navigate 新根、columns 排除、无 cascade 的历史本地行为、祖先/本地 AND、default/search form 与三类 intent 入口。
- [x] suite 覆盖 capability/version、错误结构挂载及 `visibleWhen → permission → disabled/requiresSelection → confirm → action` 的 fail-closed 顺序。
- [x] 扩展示例 [`permission-inheritance.md`](./05-scenarios/permission-inheritance.md) 展示编辑表单、行操作、toolbar 和 actionButton 的显式 intent 边界。
- [x] 所有非历史 conformance case 与官方场景同步 `protocolVersion: "2.3"`。

### G4. 发布制品

- [x] 迁移文档 `2.2-to-2.3` 明确 opt-in、columns 排除和旧页不变规则。
- [x] CHANGELOG `v2.3.0` 正式节、协议制品版本和 release target 同步。
- [x] `release:check` 固定新增 fixture 类别、case 数与 fixture digest；协议制品构建与验证可复现。

## 3. 发布工程门禁

- [x] `package.json` / lockfile / `protocol-manifest.json` 统一为 `2.3.0` / `2.3`。
- [x] 算法类 fixtures 使用 `protocolVersion: "2.3"`；`version-negotiation` 保留历史页面版本输入。
- [x] 链接检查、场景校验、L0-L4、全量 JS/Python conformance、release check 与制品验证通过。

## 4. 版本纪律

v2.3 之后继续遵守 SemVer：文档勘误为 PATCH；兼容字段 / capability 为 MINOR；破坏合法输入或默认可观测结果为 MAJOR。`permissionCascade` 和 `permissionIntent` 仅在本 MINOR 的显式双重门控下生效；任何扩大 target type、权限键或跨根传播的需求必须先经独立 ADR。

## 5. 完成定义

G1–G4 与发布工程门禁全部关闭，且 `npm run release:check` 与 `npm run verify:protocol-artifact` 通过，即可将 `2.3.0` 视为协议制品完成。v2.2 历史门禁保留在 [13-v2.2-release-goals.md](./13-v2.2-release-goals.md)。
