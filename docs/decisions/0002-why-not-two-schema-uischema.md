# ADR-0002：为什么不采用 RJSF 式的 schema + uiSchema 两棵平行树

## 背景

RJSF（react-jsonschema-form）的核心设计是数据结构（`schema`）与展示方式（`uiSchema`）
分离成两棵平行的树，通过字段路径对齐。这是一种被广泛验证过的"数据/UI 分离"思路。

## 决策

不采用两棵平行树的方式，而是采用单棵 Node 树，在同一个节点上用不同命名空间的字段
（`props` 表展示、`data` 表数据来源）表达"数据"与"展示"的区分。

## 理由

1. **对后端心智负担更友好**：两棵平行树要求后端在编写配置时始终保持两棵树的路径严格对齐，
   一旦其中一棵树增删字段而忘记同步另一棵，容易出现"配置能解析但渲染错位"的隐蔽问题。
   单棵树把"这个字段展示成什么"和"这个字段是什么"放在同一个节点里，天然保证不会错位。
2. **更贴近后端的思维习惯**：后端工程师更习惯"一个资源对象包含它的所有描述信息"，
   而不是维护两份需要手动关联的清单。
3. **场景本身已经通过 `type` 隐含了展示语义**：本协议中 `type` 已经承担了 RJSF 里
   `uiSchema` 的核心职责（决定用哪个组件渲染），因此没有必要再单独拆出一棵树。

## 权衡（Trade-off）

- RJSF 两棵树的优势在于：同一份数据 `schema` 可以配合不同的 `uiSchema` 复用
  （例如同一个数据结构，在详情页和编辑页用不同展示方式）。
- 我们目前的场景（中后台配置页面，一个页面对应一份配置）复用两套展示的需求较弱，
  因此认为放弃这个复用能力、换取"防止路径错位"和"降低心智负担"是合理的取舍。
- 如果未来出现强烈的"同一份数据多套展示"复用诉求，可以在不破坏现有协议的前提下，
  为 Node 增加一个可选的 `variant` 字段来实现局部复用，无需引入第二棵树。
