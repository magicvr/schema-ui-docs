# ADR-0001：为什么用单棵 Node 树，而不是多套结构

## 背景

设计初期讨论过是否要为布局、表格、表单分别设计三套独立的顶层结构
（因为三者字段确实有差异：布局强调 `children` 嵌套，表格强调 `columns`，表单强调 `reactions`）。

## 决策

统一采用同一套 Node 结构（`type`/`props`/`data`/`children`/`reactions`），
差异通过 `type` 对应的组件契约（见 `03-component-registry.md`）来体现，
而不是在协议顶层引入多套并行的结构。

## 理由

1. 前端只需要写一个递归 Renderer，按 `type` 查表分发，不需要为每类场景写单独的解析入口。
2. 后端/AI 助手学习成本更低：只需要记住 5 个字段的通用规则，
   具体某个 `type` 支持哪些 `props` 是"字典查询"而非"语法学习"。
3. 新增场景类型（例如未来的"看板列表 kanban"）时，只需要在组件注册表里新增一项，
   不需要改动协议本身，保持了协议的稳定性。

## 权衡（Trade-off）

- 代价：`props` 字段的具体契约无法在协议层通过统一的强类型约束表达，
  必须依赖 `component-registry.json` 做二级校验（L2 校验，见 `06-validation.md`）。
- 我们认为这个代价可接受，因为它把"协议稳定性"和"组件可扩展性"解耦了。

## 相关联动表达式能力边界的类似取舍

联动表达式引擎（`02-reaction-expression.md`）同样遵循"宁可让后端用不同的 Node 结构表达复杂场景，
也不扩展表达式引擎能力"的原则——例如需要算术运算的联动，应该改为后端下发不同的 Node，
而不是给 `when` 加入 `+`/`-` 等运算符。这样能保证联动规则始终是"声明式数据"而非"图灵完备的代码"，
避免协议退化成一种自定义脚本语言。
