# Schema-UI 项目章程

## 使命

Schema-UI 是前端 Renderer 与后端页面生产方共同遵守的、框架和语言无关的协议。
本仓库的首要交付物是可独立发布、固定版本和校验摘要的协议制品。MCP、验证器、参考实现、
Docker 镜像和其他集成只负责帮助消费协议，不得成为新的协议来源。

## 权威层级

协议问题按下表确定权威来源。不同层发生冲突时必须阻止发布，不能以“当前工具能通过”作为裁决。

| 问题 | 权威来源 | 说明 |
|---|---|---|
| 字段语义、默认值、能力边界 | 核心规范与已接受的协议 ADR | 定义协议含义与设计约束 |
| JSON/YAML 结构、组件字段 | JSON Schema 与组件注册 DSL | 核心规范的机器可读结构投影 |
| 可观测算法结果 | versioned conformance fixtures | 前后端实现必须逐字段产生相同结果 |
| 示例、迁移和发布说明 | 场景、迁移文档、CHANGELOG | 提供说明，不得新增未在前三类中定义的语义 |
| 接受/拒绝实现 | 验证器与 reference | 非规范性辅助实现，只能执行已有协议规则 |
| 查询与集成 | MCP 及其他工具 | 非规范性消费入口，不得复制或扩展协议规则 |

协议发布清单由 [`protocol-manifest.json`](./protocol-manifest.json) 定义。`scripts/`、`validator/`、`mcp/`、
`docs/mcp/` 和 `docs/audit/` 不进入协议制品，也不影响协议内容摘要。

## 依赖方向

```text
核心规范 / ADR
  -> Schema / 组件 DSL / conformance fixtures
  -> 验证器 / reference
  -> MCP / Docker / 编辑器和 CI 集成
```

依赖只能沿箭头方向。协议层不得导入 MCP 或某个具体 Renderer、后端框架的实现。

## 版本与发布

- 协议制品使用独立 SemVer，当前制品版本见 `protocol-manifest.json`。
- 页面声明 `meta.protocolVersion`，表示协议 MAJOR.MINOR，不等于任一工具版本。
- MCP、验证器和其他工具使用各自版本，并显式声明其捆绑或支持的协议制品版本。
- 工具修复不得改变协议制品内容摘要；协议语义变化必须按 SemVer 更新协议版本。
- 每个协议 tag 必须生成可复现的语言无关 `schema-ui-protocol-<version>.tar.gz` 和 SHA-256。

## 变更门禁

协议变更必须回答：

1. 这是语义、结构、行为向量、说明还是工具变更？
2. 是否改变既有合法输入、默认值或可观测结果？
3. 前端和后端消费者分别受到什么影响？
4. 是否需要 ADR、迁移说明、Schema 和 conformance fixture 原子更新？

工具变更必须满足：

- 能引用现有协议需求，且不先于协议定义新语义；
- 只修改工具时，协议制品版本和内容摘要保持不变；
- 删除 `mcp/` 后，协议仍可独立构建、发布和被前后端消费。

## 非目标

- 本仓库不提供具体 UI 框架的生产 Renderer。
- 本仓库不提供业务后端、鉴权、数据存储或页面生成平台。
- MCP 不生成或修改业务页面，不读取宿主项目文件系统，不推断业务接口。
- 审计记录不是协议交付物；已完成问题应沉淀为规范、ADR、迁移或 CHANGELOG。
